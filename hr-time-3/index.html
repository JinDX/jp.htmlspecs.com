1. はじめに

このセクションは非規範的です。

ECMAScript言語仕様書 [ECMA-262] では、 Date オブジェクトを、 1970年1月1日UTCからのミリ秒単位の時刻値を表すものとして定義しています。 ほとんどの場合、この時刻の定義は十分であり、これらの値によって 1970年1月1日UTCから約285,616年以内の任意の瞬間の時刻をミリ秒精度で表現できます。

実際には、こうした時刻の定義はクロックスキューや システムクロックの調整の影響を受けます。時刻値は 常に単調増加するとは限らず、後続値が減少したり 同じ値となることもあります。

例えば、以下のスクリプトは計算した duration に 正の数・負の数・ゼロを記録する場合があります。 

var mark_start = Date.now();
doTask(); // Some task
var duration = Date.now() - mark_start;

特定のタスクにおいては、この時刻定義では不十分なことがあり、それは次の理由によります：

  • 安定した単調クロックを持たず、結果として システムクロックのズレの影響を受ける。
  • ミリ秒未満の時刻分解能が提供されない。

この仕様では Date.now() [ECMA-262] の挙動を変更することは提案していません。 Date.now() はカレンダー時刻の現在値の取得に便利であり、 歴史的にも広く利用されています。 DOMHighResTimeStamp 型、 Performance.now() メソッド、 そして Performance.timeOrigin 属性が上記課題を解決し、ミリ秒未満分解能で単調増加する時刻値を提供します。

サブミリ秒分解能の提供はこの仕様の必須要素ではありません。 実装はプライバシーやセキュリティの理由で 露出するタイマー分解能を制限し、 サブミリ秒タイマーを公開しないこともできます。 サブミリ秒分解能に依存するユースケースは、 その場合満たされないことがあります。

1.1. ユースケース

このセクションは非規範的です。

本仕様は以下のような能力を定義します：安定した単調クロックに基づき、コンテキスト間で比較可能で、 サブミリ秒分解能も選択的に持つタイムスタンプを提供します。

パフォーマンス測定に安定した単調クロックが必要な理由は、 無関係なクロックのズレによって測定値が歪み、意味を失うためです。 たとえばドキュメントへのナビゲーション、リソースのフェッチ、スクリプト実行の経過時間を正確に計測するには、 サブミリ秒分解能で単調増加するクロックが求められます。

コンテキスト間でのタイムスタンプ比較は、例えば Worker とメインスレッド間で作業を同期する際や、 タイムライン全体の統一ビューを作るべくイベントを計測する際に必要不可欠です。

最後に、サブミリ秒タイマーが必要になるユースケースは次の通りです：

  • サブミリ秒単位で作業をスケジューリングできること。 これはメインスレッド上で特に重要であり、 処理が短く規則的な間隔でフレーム描画と干渉しない必要があり、 ユーザーに目に見えるジャンクを避けたい場合に重要です。
  • スクリプトベースのアニメーションのフレームレートを計算する際、 アニメーションが60FPSで描画されているかを判定するにはサブミリ秒分解能が必要です。 これがない場合、開発者は58.8FPS（1000ms / 16）か62.5FPS（1000ms / 17）かしか判定できません。
  • 野生環境下でJSコード（例：User-Timing）の実測値を収集する際、 開発者は関数のサブミリ秒計測値を取得し、早期にパフォーマンス退化を検知したい事があります。
  • アニメーションの特定のタイミングでオーディオを鳴らしたり、 オーディオとアニメーションが完全に同期するよう調整する際、 経過時間を正確に測る必要があります。

1.2. 例

このセクションは非規範的です。

開発者は、Worker や SharedWorker 等異なる time origins のイベントを含む アプリ全体のタイムラインを構築したい場合があります。同じタイムラインに描画するには、 DOMHighResTimeStamp を Performance.timeOrigin 属性で変換できます。 

// ---- worker.js -----------------------------
// Shared worker script
onconnect = function(e) {
  var port = e.ports[0];
  port.onmessage = function(e) {
    // Time execution in worker
    var task_start = performance.now();
    result = runSomeWorkerTask();
    var task_end = performance.now();
  }

  // Send results and epoch-relative timestamps to another context
  port.postMessage({
    'task': 'Some worker task',
    'start_time': task_start + performance.timeOrigin,
    'end_time': task_end + performance.timeOrigin,
    'result': result
  });
}

// ---- application.js ------------------------
// Timing tasks in the document
var task_start = performance.now();
runSomeApplicationTask();
var task_end = performance.now();

// developer provided method to upload runtime performance data
reportEventToAnalytics({
  'task': 'Some document task',
  'start_time': task_start,
  'duration': task_end - task_start
});

// Translating worker timestamps into document’s time origin
var worker = new SharedWorker('worker.js');
worker.port.onmessage = function (event) {
  var msg = event.data;

  // translate epoch-relative timestamps into document’s time origin
  msg.start_time = msg.start_time - performance.timeOrigin;
  msg.end_time = msg.end_time - performance.timeOrigin;

  reportEventToAnalytics(msg);
}

2. 時間の概念

2.1. クロック

クロック（clock） は時の経過を追跡し、 アルゴリズムの手順実行中の 非安全な現在時刻（unsafe current time） を報告できます。 様々な種類のクロックがあります。ウェブプラットフォーム上での全てのクロックは 実時間1ミリ秒につき1ミリ秒を数えようとしますが、正確に一致しない場合の扱いは異なります。

壁時計（wall clock） の 非安全な現在時刻（unsafe current time）

は常にユーザーの時間感覚にできる限り近いものです。コンピュータは時に遅れたり進んだり時刻を失うことがあるため、 wall clock は時折調整する必要があり、そのため unsafe current time が減少することもあり、 パフォーマンス計測やイベント順序記録には信頼できません。ウェブプラットフォームは wall clock を [ECMA-262] の time と共有しています。

単調クロック（monotonic clock） の 非安全な現在時刻（unsafe current time）

は決して減少せず、システムクロックの調整によって変化しません。単調クロックは 1つのユーザーエージェントの実行内にのみ存在し、 異なる実行間のイベント比較には使えません。

単調クロックはユーザーの時間と一致するように調整できないため、 ユーザー表示用の時刻でなく測定用に使うべきです。ユーザー表示用の時刻には 壁時計を使用してください。

ユーザーエージェントは、ブラウザの再起動時や隔離ブラウズセッション（例：シークレットモード）開始時、 既存の設定オブジェクトと通信不可能な environment settings objectを作成した時等に、 estimated monotonic time of the Unix epoch を新しく選ぶことがあります。したがって、開発者は共有タイムスタンプを 全ての過去・現在・未来のコンテキストで単調性が保証される絶対時刻として使うべきではありません。 単調性は、postMessageやBroadcastChannel等の伝達手段で互いに通信できる範囲のコンテキスト間にのみ成り立ちます。

ある状況（例：タブがバックグラウンド化されたときなど）では、 ユーザーエージェントが当該コンテキストでタイマーや繰り返しコールバックをスロットリング あるいは完全凍結することがあります。 このようなスロットリングは単調クロックが返す時刻の分解能や精度には影響しないべきです。

2.2. モーメントと持続時間

各 クロック の 非安全な現在時刻は 非安全なモーメント（unsafe moment） を返します。Coarsen time は、これらの 非安全なモーメント を コース化モーメント（coarsened moment）（以下モーメント）に変換します。 異なるクロックの 非安全なモーメントや モーメントは比較できません。

モーメントおよび 非安全なモーメントは時点そのものを表すため、 それ自体を数値として直接格納することはできません。 実装では通常、モーメント を 他の固定時点からの 持続時間（duration） で表現しますが、仕様記述上は モーメント 自体で扱うべきです。

期間（duration）とは、 同じクロック上の ある時点から 別の時点までの距離を指します。どちらの端点も安全でない時点（unsafe moment）であってはならず、 これにより期間や 期間の差分が[§ 9.1 クロックの分解能]の懸念を緩和します。 期間はミリ秒や秒などで測定します。 すべてのクロックが同じ速度でカウントしようとするため、 期間自体には 特定のクロックが紐付いていません。 また、一つのクロックで得た二つの時点から計算した 期間は、 第二のクロック上の ある時点に加算することで、 その第二クロック上の別の時点を導けます。

a から b までの持続時間（duration from） は次のアルゴリズムによる：

  1. 断言：a と b が同じ クロックで生成されたこと。
  2. 断言：a と b の双方が コース化モーメントであること。
  3. a から b までの時間を 持続時間として返す。もし b が a より前なら負の値になる。

持続時間は DOMHighResTimeStamp として暗黙的に利用できます。 持続時間をタイムスタンプに暗黙変換するには、 持続時間 d について、そのミリ秒数を返します。

3. 仕様書著者のためのツール

単一ページ内（1つのenvironment settings objectのコンテキスト内）での時間計測には、 settingsObject の現在の相対タイムスタンプ（current relative timestamp）を使用します。これは duration from settingsObject の time origin から settingsObject の 現在の単調時刻（current monotonic time） までの時間と定義されます。この値は JavaScript から duration の 暗黙のタイムスタンプ変換を使い DOMHighResTimeStamp として直接公開できます。

単一 UA 実行内で、environment settings object の time origin が比較基準に適さない場合は、moment を environment settings object の 現在の単調時刻 を使って作成してください。environment settings object settingsObject の現在の単調時刻（current monotonic time） を求める手順は次の通り：

  1. unsafeMonotonicTime を 単調クロック（monotonic clock） の 非安全な現在時刻とする。
  2. unsafeMonotonicTime と settingsObject の cross-origin isolated capability で coarsen time を呼び、その結果を返す。

単調クロック からの moment は JavaScript や HTTP で直接表現できません。かわりに、そのような moment 間の duration を公開してください。

複数UA実行にまたがって時間を測定する場合、momentを 現在のコース化壁時計時刻（current coarsened wall time）、または（cross-origin-isolated context で高分解能が必要なときは）environment settings object の 現在の壁時計時刻（current wall time）を使って作成します。 現在のコース化壁時計時刻は、coarsen time を 壁時計（wall clock） の 非安全な現在時刻で呼び出す結果です。

environment settings object settingsObject の 現在の壁時計時刻（current wall time）は次の手順で求めます：

  1. unsafeWallTime を 壁時計（wall clock） の 非安全な現在時刻 とする。
  2. unsafeWallTime と settingsObject の cross-origin isolated capability で coarsen time を呼び出し、その結果を返す。

壁時計 からの moment を使う場合、 ユーザーが時計を進めたり戻したりする状況も設計で考慮してください。

壁時計からのmomentは、 Unixエポックからその momentまでのミリ秒値を Date コンストラクターに渡すか、エポックからナノ秒数を Temporal.Instant コンストラクターに渡せば、JavaScriptで表せます。[Temporal]

このような値表現をコンピュータ間で送信すると ユーザーの時計のズレ（clock skew）が外部に知られてしまい、 追跡ベクトルとなるので避けてください。 代わりに、単調クロックの moment同士の持続時間をやり取りする方式を検討ください。

3.1. 例

DOMイベント発生時刻は以下のように記録できます：

  1. event の timeStamp 属性を this の relevant settings object の 現在の相対タイムスタンプで初期化する。

エラーレポートの経過時間(age)を計算する一例：

  1. report の 生成時刻を settings の 現在の単調時刻で初期化。

後で：

  1. dataを次のキー/値ペアのマップとする：
    age
    report の 生成時刻と、 context の relevant settings object の 現在の単調時刻との差を 最も近い整数に丸めた値
    ...

日をまたぐアトリビューションレポートの有効期限処理例：

  1. source を新しいアトリビューションソース構造体として以下の要素を持たせる：
    ...
    source time
    context の 現在の壁時計時刻
    expiry
    期間文字列の解釈で value["expiry"]を処理

数日後：

  1. もし context の 現在の壁時計時刻 が source の source time + expiry より小さければ、レポートを送信。

4. タイムオリジン（Time Origin）

Unixエポックは、 壁時計（wall clock）上で 1970年1月1日 00:00:00 UTC に対応するモーメントです。

何らかの形で通信できる 各environment settings object群ごとに Unixエポックの推定単調時刻 という 単調クロック（monotonic clock）上の モーメントがあり、 その値は次の手順で初期化されます：

  1. wall time を 壁時計 の 非安全な現在時刻とする。
  2. monotonic time を 単調クロック の 非安全な現在時刻とする。
  3. epoch time を monotonic time - (wall time - Unixエポック) とする。
  4. Unixエポックの推定単調時刻を epoch time で coarsen time を実行した結果に初期化する。
上記の「通信可能性があるsettings object群」の定義は要整理です。 これはfamiliar withに似ていますが Workerも含みます。

パフォーマンス測定は、関係するenvironment settings objectの初期化初期に モーメントからの 持続時間（duration）を報告します。 このモーメントは、そのsettings objectの time origin として格納されます。

time origin タイムスタンプの取得 をglobal（global object）に対して行うには 以下の手順を実行し、持続時間(duration) を返す：

  1. timeOrigin を global の relevant settings object の time originとする。

    Window コンテキストでは、この値は ナビゲーション開始時刻を表します。 Worker や ServiceWorker 内では、 worker起動時刻 となります。[SERVICE-WORKERS]

  2. Unixエポックの推定単調時刻 から timeOrigin までの 持続時間(duration from) を返す。

time origin タイムスタンプの取得で返る値は Unixエポックの global に対する time origin の経過時刻（おおよそ）です。 これはその時刻で実行された Date.now() の値と 一致しない場合があります。なぜなら、この値はシステムやユーザーによる時計調整やクロックスキューの影響を受けない 単調クロックに基づいて記録されているためです。

coarsen time アルゴリズムは、 クロック 上の 非安全なモーメント（timestamp） および オプションの boolean crossOriginIsolatedCapability (デフォルトfalse) を受け取り、次の手順を実行します：
  1. time resolution を 100マイクロ秒、またはより大きい 実装依存値に設定。
  2. もし crossOriginIsolatedCapability がtrueなら、 time resolution を 5マイクロ秒またはそれ以上の 実装依存値に設定。
  3. 実装依存の方法で timestamp をコース化したりジッタを加えたりして、 その分解能が time resolution を超えないようにする。
  4. timestamp を モーメント（moment）として返す。
相対高分解能時刻（relative high resolution time）は、 単調クロック の 非安全なモーメント time と global object global を受け、 以下の手順で 持続時間 を返します：
  1. coarse time を time および global の relevant settings object の cross-origin isolated capability を使って coarsen time を呼びその結果とする。
  2. coarse time と global で relative high resolution coarse time を返す。
relative high resolution coarse time は 単調クロック の モーメント coarseTime と global object global について global の relevant settings object の time origin から coarseTime までの duration from を返します。

現在の高分解能時刻（current high resolution time）は global object current global を受け、 unsafe shared current time と current global で 相対高分解能時刻の結果を返す必要があります。

コース化された共有現在時刻（coarsened shared current time） は オプション引数 boolean crossOriginIsolatedCapability（デフォルト false）を受け、 unsafe shared current time と crossOriginIsolatedCapability で coarsen time を呼んだ結果を返します。

unsafe shared current time は 単調クロック の 非安全な現在時刻を返します。

5. DOMHighResTimeStamp 型定義

DOMHighResTimeStamp 型は、期間（ミリ秒単位）を格納するために使用されます。文脈によっては、時点 を基準にこの期間分経過した後の 時点（例えばタイムオリジンやUnixエポック）を指す場合もあります。 

typedef double DOMHighResTimeStamp;

DOMHighResTimeStamp は、計測可能でかつタイミング攻撃を防止するのに十分な精度を持ったミリ秒単位の時刻を表すべきです。追加の考慮事項については§ 9.1 クロックの分解能を参照してください。

DOMHighResTimeStamp は double 型であり、Unixエポック からある時点までのミリ秒数という、エポック相対の時刻のみを有限の精度で表現できます。2023年の時点の場合、その分解能は約0.2マイクロ秒です。

6. EpochTimeStamp 型定義 

typedef unsigned long long EpochTimeStamp;

`EpochTimeStamp`（以前は `DOMTimeStamp` として知られていた）の利用は非推奨です。可能な限り、代わりにDOMHighResTimeStamp を使用してください。

EpochTimeStamp は、 Unixエポックから特定の時点までの整数ミリ秒数を表します（壁時計基準、うるう秒は除く）。この型を用いる仕様は、ミリ秒数の解釈方法を定義します。

7. Performance インターフェース 

[Exposed=(Window,Worker)]
interface Performance : EventTarget {
    DOMHighResTimeStamp now();
    readonly attribute DOMHighResTimeStamp timeOrigin;
    [Default] object toJSON();
};

7.1. now() メソッド

now() メソッドは必ず、現在の高分解能時間のミリ秒数を返さなければなりません。this の 関連グローバルオブジェクト（期間）を基準にします。

同じ タイムオリジン を持つPerformance オブジェクトで now() メソッドを呼び出したときに返される時刻値は、同じ単調クロックを使用しなければなりません。 now() メソッドから時系列で記録された任意の2つの時刻値の差が負になってはなりません（2つの値が同じタイムオリジンの場合）。

7.2. timeOrigin 属性

timeOrigin 属性は、 タイムオリジン・タイムスタンプを取得 から返された期間（ミリ秒数）を返さなければなりません。その際の 関連グローバルオブジェクトは this です。

Performance.timeOrigin を取得したときに返される時刻値も、 単調クロック（タイムオリジン間で共通利用）に基づき、かつその基準点は [ECMA-262] の time 定義です。（詳細は § 9 セキュリティの考慮事項 を参照）

7.3. toJSON() メソッド

toJSON() の呼び出し時には、[WEBIDL]の default toJSON steps を実行します。

8. WindowOrWorkerGlobalScope mixin への拡張

8.1. performance 属性

インターフェースミックスイン WindowOrWorkerGlobalScope 上の performance 属性は、グローバルオブジェクト（global object）からのパフォーマンス関連属性とメソッドのアクセスを可能にします。 

partial interface mixin WindowOrWorkerGlobalScope {
  [Replaceable] readonly attribute Performance performance;
};

9. セキュリティの考慮事項

このセクションは規範的ではありません。

9.1. クロックの分解能

測定やスケジューリング目的で高精度なタイミング情報へアクセスすることは、多くのアプリケーションで一般的な要件です。 たとえば、ページ上でアニメーションやサウンド、他の動きを協調させるには高分解能の時間へのアクセスが求められ、良好なユーザー体験のために必要です。　 同様に、測定は開発者が重要なコードコンポーネントのパフォーマンスを追跡したり、リグレッション（性能後退）を検出したりなどを可能にします。

しかし、同じ高精度なタイミング情報は、攻撃者が他では可視化・アクセスできないデータを推測・解読するために悪用されることもあります。 たとえば、キャッシュ攻撃、統計的フィンガープリンティング、マイクロアーキテクチャ攻撃は、様々なブラウザやアプリケーション主導の操作における高分解能タイミングデータを悪用してユーザの識別や未知の同一プロセスデータの特定に用いられる恐れがあり、プライバシーやセキュリティ問題となります（詳細は [CACHE-ATTACKS] および [SPECTRE] を参照）。

本仕様はミリ秒より細かい分解能（サブミリ秒）の時刻情報APIを定義します。これは従来の EpochTimeStamp で公開されていたミリ秒分解能よりも高精度です。 しかし、このAPIがなくても攻撃者は実行の繰り返しや統計解析により高分解能の推定値を得られる場合があります。

この新たなAPIが攻撃の精度や速度を有意に高めないようにするため、DOMHighResTimeStamp 型の最小分解能は攻撃を回避できる程度に不正確であるべきです。

必要に応じて、ユーザーエージェントは時間の粗粒化 の処理モデルにおいて time resolution をより大きな値に設定し、アーキテクチャやソフトウェア制約、その他の状況等の理由によるプライバシー・セキュリティ対策を施すことができます。

このような攻撃を緩和するために、ユーザーエージェントは必要と判断したあらゆる緩和手法を講じることができます。こうした対策の運用は、ブラウザの構成やユーザー端末、コンテンツの性質やクロスオリジンデータへのアクセス可否などによって変動します。

その技術は以下を含みうる：

  • 分解能の低下。
  • ジッター（わざと揺らぎを加える）付加。
  • 悪用検知またはAPI呼出しのスロットリング。

このようなタイミングチャネル攻撃を完全に防止することは事実上不可能です。すべての操作を機密情報の値によらず一定時間で必ず実行するか、アプリケーションを時計・タイマー・カウンタなどあらゆる時間関連プリミティブから隔離しなければなりません。しかし、どちらもブラウザ・アプリ開発者の負担や、アプリケーションのパフォーマンスや応答性への悪影響のため実現的ではありません。

クロック分解能は、解決されていない進行中の研究分野であり、すべてのブラウザに当てはまる業界の合意や決定的な推奨はまだありません。議論の経過は Issue 79 を参照してください。

9.2. クロックドリフト

本仕様は、タイムオリジンのゼロ時刻のサブミリ秒分解能APIも定義します。これはアプリケーションに単調クロック の提供と、全ブラウザコンテキスト間の共有を必要とします。単調クロックは物理時間と一致させる必要はありませんが、[ECMA-262] の time 定義に従って設定することが推奨されます。これにより、利用者の新たなフィンガープリント情報の公開を避けるためです （例：この時刻はすでにアプリケーション側で簡単に取得できるため、新たな論理クロック公開は余分な情報を持ち込みうる）。

しかし、上記の仕組みが整っていても、単調クロックは追加のクロックドリフト 解像度を提供する可能性があります。今日、アプリケーションは同一コンテキスト内で 時刻と単調時間値（Date.now()や now()） を複数回タイムスタンプし、それらの間のドリフト（例えば自動またはユーザーによる 時計調整によるもの）を観測できます。timeOrigin 属性があれば、 攻撃者は タイムオリジン（単調クロックにより報告される）と 現在時刻の推定値タイムオリジン（つまり `performance.timeOrigin` と `Date.now() - performance.now()` の差分）の比較を行い、 これらのクロック間のドリフトを長期間にわたって観測できる可能性があります。

実際には、複数ナビゲーションにまたがっても、同じドリフトをアプリケーションは観測し得ます。各コンテキストで論理時刻を保存し、クライアントやサーバーの時刻同期機構によりユーザの時計変更を推定できます。同様に、低レイヤのメカニズム（例：TCPタイムスタンプ）も、複数回アクセスなしにサーバーに高分解能時刻情報を送る可能性があります。したがって、このAPIが提供する情報は、利用者にとって新規または重要なフィンガープリント情報を公開するものではないべきです。

10. プライバシーの考慮事項

このセクションは規範的ではありません。

Document のタイムオリジンの現在の定義は、 当該ドキュメントのオリジンにリクエストが到達する前のクロスオリジンリダイレクト全体の所要時間を公開します。これはクロスオリジン情報の公開となりますが、これによりパフォーマンス指標へ大きな悪影響なしでどのように緩和するかはまだ決まっていません。

議論の経過は Navigation Timing Issue 160 を参照してください。
