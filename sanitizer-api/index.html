1. はじめに

このセクションは規範的ではありません。

ウェブアプリケーションは、クライアント側でHTML文字列を扱う必要があることが多く、 それはクライアントサイドテンプレートソリューションの一部だったり、 ユーザー生成コンテンツのレンダリングの一部だったりします。 安全に扱うことは困難です。 単純に文字列を結合し、Elementの innerHTML に詰め込む素朴な方法は危険であり、 予期せぬ方法でJavaScriptが実行されてしまう可能性があります。

[DOMPURIFY]のようなライブラリは この問題に対処するため、挿入前に文字列を慎重にパースしてサニタイズし、DOMを構築し 許可リストを通じてそのメンバーをフィルタリングします。 しかしこれは脆弱なアプローチであることが判明しています。 ウェブに公開されているパースAPIは、実際に文字列を「本物」のDOMとしてレンダリングする際の ブラウザの挙動と必ずしも合理的に対応しているわけではありません。 さらに、ライブラリはブラウザの挙動が変化するたびに追従する必要があり、 かつて安全だったものが新しいプラットフォームレベルの機能によって 時限爆弾になることもあります。

ブラウザはいつコードを実行するかをかなり把握しています。 ユーザースペースのライブラリよりも、ブラウザ自身に任意の文字列から安全にHTMLをレンダリングする方法を 教えることで、より保守・更新されやすい形で ブラウザのパーサ実装の変化に合わせて対応できます。 本書ではそのためのAPIを提案します。

1.1. 目標

  • 開発者がユーザーコントロールのHTMLを安全に扱い、インジェクション時に 直接スクリプトが実行されるのを防ぐ仕組みを提供することで DOMベースのクロスサイトスクリプティング攻撃のリスクを軽減する。

  • 現在のユーザーエージェントにおいて、そのHTML認識に基づき HTML出力を安全に利用できるようにする。

  • デフォルトの要素と属性のセットを開発者が上書きできるようにする。 特定の要素や属性を追加することで スクリプトガジェット 攻撃を防ぐことができる。

1.2. API概要

Sanitizer APIは、HTMLを含む文字列をパースしてDOMツリーにし、 ユーザーが指定した設定に従ってそのツリーをフィルタリングする機能を提供します。 メソッドは2つのフレーバーで提供されます:

  • 安全・非安全: 「安全」メソッドはスクリプトが実行されるマークアップを生成しません。 つまりXSSに対して安全です。「非安全」メソッドは与えられた内容をパース・フィルタします。 詳しくは § 4 セキュリティに関する考慮事項 を参照。

  • コンテキスト: メソッドはElement およびShadowRoot 上で定義されており、 これらNodeの 子を置換します。これはinnerHTML とほぼ同等です。 また、Document上の 静的メソッドもあり、 ドキュメント全体をパースするものであり DOMParserの parseFromString() とほぼ同等です。

2. フレームワーク

2.1. Sanitizer API

Element インターフェイスは、setHTML() と setHTMLUnsafe() の2つのメソッドを定義します。 どちらも、HTMLマークアップを含むDOMString とオプションの設定を受け取ります。 

partial interface Element {
  [CEReactions] undefined setHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLUnsafeOptions options = {});
  [CEReactions] undefined setHTML(DOMString html, optional SetHTMLOptions options = {});
};
Elementの setHTMLUnsafe(html, options) メソッドの手順:

  1. compliantHTMLを、Get Trusted Type compliant stringアルゴリズム（ TrustedHTML、 thisの 関連グローバルオブジェクト、html、「Element setHTMLUnsafe」、「script」）で得る。

  2. targetを、thisが template 要素であればtemplate contents、そうでなければthisとする。

  3. Set and filter HTML を、target、this、 compliantHTML、options、falseで行う。

Elementの setHTML(html, options) メソッドの手順:

  1. targetを、thisが template 要素であればtemplate contents、そうでなければthisとする。

  2. Set and filter HTML を、target、this、html、 options、trueで行う。 

partial interface ShadowRoot {
  [CEReactions] undefined setHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLUnsafeOptions options = {});
  [CEReactions] undefined setHTML(DOMString html, optional SetHTMLOptions options = {});
};

これらのメソッドはShadowRootにもミラーされています。

ShadowRootの setHTMLUnsafe(html, options) メソッドの手順:

  1. compliantHTMLを、Get Trusted Type compliant stringアルゴリズム（ TrustedHTML、 thisの関連グローバルオブジェクト、html、「ShadowRoot setHTMLUnsafe」、「script」）で得る。

  2. Set and filter HTML を、this、 thisのシャドウホスト（コンテキスト要素として）、 compliantHTML、options、falseで行う。

ShadowRootの setHTML(html, options) メソッドの手順:

  1. Set and filter HTML を、this（ターゲットとして）、this（コンテキスト要素として）、 html、options、trueで行う。

Document インターフェイスには、Document 全体をパースする2つの新しいメソッドが追加されます。 

partial interface Document {
  static Document parseHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLUnsafeOptions options = {});
  static Document parseHTML(DOMString html, optional SetHTMLOptions options = {});
};
parseHTMLUnsafe(html, options) メソッドの手順:

  1. compliantHTMLを、Get Trusted Type compliant stringアルゴリズム（ TrustedHTML、 thisの関連グローバルオブジェクト、html、「Document parseHTMLUnsafe」、「script」）で得る。

  2. documentを新しいDocument（ content typeが"text/html"のもの）とする。

    注: documentはブラウジングコンテキストを持たないため、スクリプト実行は無効です。

  3. documentのallow declarative shadow rootsをtrueに設定する。

  4. Parse HTML from a stringを documentとcompliantHTMLで実行する。

  5. sanitizerを、get a sanitizer instance from optionsにoptionsとfalseを渡して得る。

  6. documentに対してsanitizerとfalseでsanitizeを呼び出す。

  7. documentを返す。

parseHTML(html, options) メソッドの手順:

  1. documentを新しいDocument（ content typeが"text/html"のもの）とする。

    注: documentはブラウジングコンテキストを持たないため、スクリプト実行は無効です。

  2. documentのallow declarative shadow rootsをtrueに設定する。

  3. Parse HTML from a stringを documentとhtmlで実行する。

  4. sanitizerを、get a sanitizer instance from optionsにoptionsとtrueを渡して得る。

  5. documentに対してsanitizerとtrueでsanitizeを呼び出す。

  6. documentを返す。

2.2. SetHTMLオプションと設定オブジェクト

setHTML()系 メソッドはすべてオプション辞書を受け付けます。現時点では、この辞書には1つのメンバーだけが定義されています：

enum SanitizerPresets { "default" };
dictionary SetHTMLOptions {
  (Sanitizer or SanitizerConfig or SanitizerPresets) sanitizer = "default";
};
dictionary SetHTMLUnsafeOptions {
  (Sanitizer or SanitizerConfig or SanitizerPresets) sanitizer = {};
};

Sanitizer 設定オブジェクトはフィルター設定をカプセル化します。 同じ設定は「安全」 と「非安全」両方のメソッドで利用でき、「安全」メソッドは渡された設定に暗黙的に removeUnsafe 操作を行い、設定が渡されなければデフォルト設定が使われます。 意図としては、ページのライフサイクルの早い段階で（1つまたは少数の）設定を構築し、 必要な時にいつでも使えるようにすることです。これにより実装は事前に設定を処理できます。

設定オブジェクトは設定辞書として取得できます。また直接修正も可能です。

[Exposed=Window]
interface Sanitizer {
  constructor(optional (SanitizerConfig or SanitizerPresets) configuration = "default");

  // 設定の取得:
  SanitizerConfig get();

  // Sanitizerのリストやフィールドの変更:
  undefined allowElement(SanitizerElementWithAttributes element);
  undefined removeElement(SanitizerElement element);
  undefined replaceElementWithChildren(SanitizerElement element);
  undefined allowAttribute(SanitizerAttribute attribute);
  undefined removeAttribute(SanitizerAttribute attribute);
  undefined setComments(boolean allow);
  undefined setDataAttributes(boolean allow);

  // スクリプト実行するマークアップの除去（複数リストを変更する場合あり）:
  undefined removeUnsafe();
};

Sanitizer にはconfiguration（SanitizerConfig） が関連付けられています。

constructor(configuration) メソッドの手順:

  1. configurationがSanitizerPresets 文字列の場合：

    1. アサート: configurationがdefaultであること。

    2. configurationを組み込み安全デフォルト設定に設定する。

  2. validをset a configurationで configurationとtrueをthisに渡して得る。

  3. validがfalseなら、TypeErrorを投げる。

get() メソッドの手順は、thisのconfigurationを返すこと。
allowElement(element) メソッドの手順は、 要素を許可すること（elementと thisの configurationを使う）。
removeElement(element) メソッドの手順は 要素の削除をelementと thisの configurationで行う。
replaceElementWithChildren(element) メソッドの手順は 要素を子要素で置換 をelementとthisのconfigurationで行う。
allowAttribute(attribute) メソッドの手順は 属性を許可すること（attributeと thisの configurationを使う）。
removeAttribute(attribute) メソッドの手順は 属性の削除をattributeと thisの configurationで行う。
setComments(allow) メソッドの手順は コメント設定を allowとthisのconfigurationで行う。
setDataAttributes(allow) メソッドの手順は data属性設定を allowとthisの configurationで行う。
removeUnsafe() メソッドの手順は、 thisの configurationをremove unsafe の結果で更新すること。

dictionary SanitizerElementNamespace {
  required DOMString name;
  DOMString? _namespace = "http://www.w3.org/1999/xhtml";

// "elements"で使用
dictionary SanitizerElementNamespaceWithAttributes : SanitizerElementNamespace {

  DOMString? _namespace = null;

  sequence<SanitizerElement> removeElements;

  boolean comments;
  boolean dataAttributes;
};

3. アルゴリズム

HTMLを設定してフィルタするには、Element または DocumentFragment target、Element contextElement、文字列 html、辞書 options、真偽値 safe を受け取る:

  1. safeがtrue かつ contextElementのローカル名が「script」であり、 contextElementの名前空間がHTML名前空間 またはSVG名前空間の場合、returnする。

  2. sanitizerをオプションからサニタイザーインスタンスを取得 (options, safe)で得る。

  3. newChildrenをHTMLフラグメントパースアルゴリズム (contextElement, html, true)の結果とする。

  4. fragmentを新しいDocumentFragment（ ノード文書がcontextElementのノード文書）とする。

  5. newChildrenの各 nodeについてfragmentにappendする。

  6. fragmentに対してsanitizerとsafeでsanitizeを実行する。

  7. target内ですべてをfragmentで置換する。

オプションからサニタイザーインスタンスを取得 は 辞書 options、真偽値 safeを受け取って実行する:

注: このアルゴリズムはSetHTMLOptions と SetHTMLUnsafeOptions の両方で機能し、デフォルト値が異なるだけです。

  1. sanitizerSpecを「default」とする。

  2. options["sanitizer"]が 存在する場合：

    1. sanitizerSpecを options["sanitizer"]に設定する。

  3. アサート: sanitizerSpecはSanitizerインスタンス、 SanitizerPresetsメンバー文字列、または辞書である。

  4. sanitizerSpecが文字列の場合：

    1. アサート: sanitizerSpecが「default」であること。

    2. sanitizerSpecを組み込み安全デフォルト設定に設定する。

  5. アサート: sanitizerSpecはSanitizerインスタンス、 または辞書である。

  6. sanitizerSpecが辞書の場合：

    1. sanitizerを新しいSanitizerインスタンスとする。

    2. setConfigurationResultをset a configuration (sanitizerSpec, not safe, sanitizer)の結果とする。

    3. setConfigurationResultがfalseなら、TypeErrorをthrowする。

    4. sanitizerSpecをsanitizerに設定する。

  7. アサート: sanitizerSpecはSanitizerインスタンスである。

  8. sanitizerSpecをreturnする。

3.1. サニタイズアルゴリズム

メインのsanitize操作は、ParentNode node、Sanitizer sanitizer、真偽値 safeを使い、以下の手順を実行する：

  1. configurationをsanitizerのconfigurationの値とする。

  2. safeがtrueなら、configurationをremove unsafeをconfigurationに対して呼び出した結果に設定する。

  3. sanitize coreを node、configuration、handleJavascriptNavigationUrlsにsafeを設定して呼び出す。

sanitize core操作は、 ParentNode node、SanitizerConfig configuration、真偽値 handleJavascriptNavigationUrlsを使い、 nodeから始まるDOMツリーを反復し、特殊なケース（例：template contents）には再帰する場合がある。手順は以下の通り：

  1. nodeの子それぞれchildについて：

    1. アサート：childはText、 Comment、 Element、 またはDocumentTypeを実装していること。

      注: 現状、このアルゴリズムはHTML パーサ出力にのみ呼ばれ、アサーションは常に成立するはず。DocumentType はparseHTML とparseHTMLUnsafe でのみ現れる。今後他のコンテキストで使われる場合は前提を再検討すること。

    2. childがDocumentTypeを実装していれば continueする。

    3. childがTextを実装していれば continueする。

    4. childがCommentを実装していれば：

      1. configuration["comments"] がtrueでなければ childを削除する。

    5. その他：

      1. elementNameをSanitizerElementNamespace （childのlocal nameとnamespace）とする。

      2. configuration["replaceWithChildrenElements"] がelementNameを含めば：

        1. sanitize coreをchild、 configuration、 handleJavascriptNavigationUrlsで呼び出す。

        2. replace allを childのchildrenに対して child内で実行する。

        3. continueする。

      3. configuration["removeElements"] がelementNameを含む、または configuration["elements"] が空でなくelementNameを含まない場合：

        1. childを削除する。

        2. continueする。

      4. elementNameが«[ "name" → "template", "namespace" → HTML名前空間 ]»と等しい場合：

        1. sanitize coreをchildのtemplate contentsに対して configurationとhandleJavascriptNavigationUrlsで呼び出す。

      5. childがシャドウホストの場合、 sanitize coreをchildのshadow rootに対して configurationとhandleJavascriptNavigationUrlsで呼び出す。

      6. childの属性それぞれattributeについて：

        1. attrNameをSanitizerAttributeNamespace （attributeのlocal nameとnamespace）とする。

        2. configuration["removeAttributes"] がattrNameを含む場合、 attributeをchildから削除する。

        3. configuration["elements"]["removeAttributes"] がattrNameを含む場合、attributeをchildから削除する。

        4. 以下のすべてがfalseなら、attributeをchildから削除する。

          • configuration["attributes"] が存在し、attrNameを含む

          • configuration["elements"]["attributes"] がattrNameを含む

          • "data-"がコード単位の接頭辞であり、 namespaceが nullであり、 configuration["dataAttributes"] がtrue

        5. handleJavascriptNavigationUrlsがtrueの場合：

          1. «[elementName, attrName]»が 組み込みナビゲーションURL属性リストに一致し、 attributeがjavascript: URLを含むなら attributeをchildから削除する。

          2. childのnamespaceが MathML名前空間で、 attrのlocal nameが "href"であり、 attrのnamespaceが nullまたはXLink名前空間で、 attrがjavascript: URLを含むなら attrを削除する。

          3. 組み込みアニメーションURL属性リスト が«[elementName, attrName]»を含み、 attrの値が "href"または"xlink:href"なら attrを削除する。

      7. sanitize coreをchildに対して configurationとhandleJavascriptNavigationUrlsで呼び出す。

注: 現状のブラウザはjavascript: URLを ナビゲーション時のみサポートしています。ナビゲーション自体はXSS脅威ではないため、 javascript: URLへのナビゲーションはブロックしますが、一般的なナビゲーションはブロックしません。

宣言的ナビゲーションには以下のカテゴリがあります：

  1. アンカー要素（HTML・SVG名前空間の<a>）

  2. フォーム要素（form actionによるナビゲーション）

  3. [MathML]では 全要素がアンカーになり得る。

  4. [SVG11]アニメーション。

最初の2つは組み込みナビゲーションURL属性リストでカバーされています。

MathMLはこの仕様で「名前空間ごとのグローバル」ルールが定義できないため、別途記述されています。

SVGアニメーションは 組み込みアニメーションURL属性リストでカバーされます。 ただしSVGアニメーション要素の解釈はアニメーション対象に依存し、サニタイズ時には最終対象が不明なため、 sanitizeアルゴリズムはhref属性のアニメーションをすべてブロックします。

attributeがjavascript: URLを含むか判定するには：

  1. urlを基本URLパーサを attributeの値に対して実行した結果とする。

  2. urlがfailureならfalseを返す。

  3. urlのschemeが "javascript"であるかどうかを返す。

3.2. 設定処理

要素を許可するには、elementとSanitizerConfig configurationを使い、以下を実行する：

  1. elementをelementに対してサニタイザー要素（属性付き）の正規化した結果に設定する。

  2. Removeでelementを configuration["elements"]から削除する。

  3. Appendでelementをconfiguration["elements"]に追加する。

  4. Removeでelementを configuration["removeElements"]から削除する。

  5. Removeでelementを configuration["replaceWithChildrenElements"]から削除する。

注意: allowElementの扱いは他のメソッドより少し複雑です。 要素許可リストには要素ごとの許可・削除属性リストを持てるため、まず対象要素をリストから削除してから追加します。これによりリストは渡された内容でリセットされます（マージや変更はされません）。要素ごとの許可・削除リストは完全に上書きのみ可能です。

注意: Removeは名前と名前空間で一致するため、属性付き要素を追加しても removeElements や replaceWithChildrenElements リストから一致する要素は削除されます。

要素の削除は、elementとSanitizerConfig configurationを使い、以下を実行する：

  1. elementをelementに対してサニタイザー要素の正規化した結果に設定する。

  2. Addで elementをconfiguration["removeElements"]に追加する。

  3. Removeでelementを configuration["elements"]から削除する。

  4. Removeでelementを configuration["replaceWithChildrenElements"]から削除する。

要素を子要素で置換は、elementとSanitizerConfig configurationを使い、以下を実行する：

  1. elementをelementに対してサニタイザー要素の正規化した結果に設定する。

  2. Addで elementをconfiguration["replaceWithChildrenElements"]に追加する。

  3. Removeでelementを configuration["removeElements"]から削除する。

  4. Removeでelementを configuration["elements"]から削除する。

属性を許可するには、attributeとSanitizerConfig configurationを使い、以下を実行する：

  1. attributeをattributeに対してサニタイザー属性の正規化した結果に設定する。

  2. Add でattributeをconfiguration["attributes"]に追加する。

  3. Removeでattributeを configuration["removeAttributes"]から削除する。

属性の削除は、attributeとSanitizerConfig configurationを使い、以下を実行する：

  1. attributeをattributeに対してサニタイザー属性の正規化した結果に設定する。

  2. Addで attributeをconfiguration["removeAttributes"]に追加する。

  3. Removeでattributeを configuration["attributes"]から削除する。

コメント設定はallowとSanitizerConfig configurationを使い、以下を実行する：

  1. configuration["comments"] にallowを設定する。

data属性設定はallowとSanitizerConfig configurationを使い、以下を実行する：

  1. configuration["dataAttributes"] にallowを設定する。

注意: このアルゴリズム名はremove unsafeですが、 この仕様における「unsafe」は ドキュメント挿入時にJavaScriptが実行される内容のみを指し、XSSの機会を除去します。

unsafeの削除はconfigurationに対して以下を行う：

  1. アサート：組み込み安全ベースライン設定は removeElements と removeAttributes キーのみ持ち、elements、 replaceWithChildrenElements、 attributesは持たない。

  2. resultをconfigurationのコピーに設定する。

  3. 組み込み安全ベースライン設定[removeElements]の 各elementについて：

    1. remove an elementで elementとresultを使う。

  4. 組み込み安全ベースライン設定[removeAttributes]の 各attributeについて：

    1. remove an attributeで attributeとresultを使う。

  5. イベントハンドラー属性の各attributeについて：

    1. remove an attributeで attributeとresultを使う。

  6. resultを返す。

設定をセットは、辞書 configuration、真偽値 allowCommentsAndDataAttributes、Sanitizer sanitizerで以下を実行する：

  1. configuration["elements"]の各elementについて：

    1. allow an elementで elementとsanitizerのconfigurationを使う。

  2. configuration["removeElements"]の各elementについて：

    1. remove an elementで elementとsanitizerのconfigurationを使う。

  3. configuration["replaceWithChildrenElements"]の各elementについて：

    1. replace an element with its childrenでelementとsanitizerのconfigurationを使う。

  4. configuration["attributes"]の各attributeについて：

    1. allow an attributeで attributeとsanitizerのconfigurationを使う。

  5. configuration["removeAttributes"]の各attributeについて：

    1. remove an attributeで attributeとsanitizerのconfigurationを使う。

  6. configuration["comments"]が 存在する場合：

    1. set commentsで configuration["comments"]とsanitizerのconfigurationを使う。

    2. そうでなければset commentsで allowCommentsAndDataAttributesとsanitizerのconfigurationを使う。

  7. configuration["dataAttributes"]が 存在する場合：

    1. set data attributesで configuration["dataAttributes"]とsanitizerのconfigurationを使う。

    2. そうでなければset data attributesで allowCommentsAndDataAttributesとsanitizerのconfigurationを使う。

  8. 以下すべてがtrueならtrueを返す：

    • configuration["elements"]のサイズが sanitizer.configuration["elements"]のサイズと等しい。

    • configuration["removeElements"]のサイズが sanitizer.configuration["removeElements"]のサイズと等しい。

    • configuration["replaceWithChildrenElements"]のサイズが sanitizer.configuration["replaceWithChildrenElements"]のサイズと等しい。

    • configuration["attributes"]のサイズが sanitizer.configuration["attributes"]のサイズと等しい。

    • configuration["removeAttributes"]のサイズが sanitizer.configuration["removeAttributes"]のサイズと等しい。

    • configuration["elements"]またはconfiguration["removeElements"]が 存在するか、どちらも存在しないが両方は存在しない。

    • configuration["attributes"]またはconfiguration["removeAttributes"]が 存在するか、どちらも存在しないが両方は存在しない。

注意: 以前の仕様ではconfigの正規化が詳細に定義されていましたが、今は各メソッド定義で扱われます。

注意: この操作はSanitizer の操作メソッドによって定義されます。これらのメソッドは他リストの一致するエントリも削除し、最終ステップのサイズ比較で検出されます。 例：{ allow: ["div", "div"] }はSanitizerの許可リストに1要素のみとなり、最終テストはfalseとなり例外がスローされます。

要素ごとの属性リストや構文エラーの検証は未対応です。

サニタイザー要素（属性付き）の正規化は、SanitizerElementWithAttributes elementに対して以下を行う：

  1. resultをelementに対してサニタイザー要素の正規化した結果に設定する。

  2. elementが辞書の場合：

    1. element["attributes"]の各attributeについて：

      1. attributeに対してサニタイザー属性の正規化した結果を result["attributes"]に追加。

    2. element["removeAttributes"]の各attributeについて：

      1. attributeに対してサニタイザー属性の正規化した結果を result["removeAttributes"]に追加。

  3. resultを返す。

サニタイザー要素の正規化は SanitizerElement elementについて、 サニタイザー名の正規化をelementと デフォルト名前空間としてHTML名前空間で実行し、その結果を返す。
サニタイザー属性の正規化は SanitizerAttribute attributeについて、 サニタイザー名の正規化をattributeと デフォルト名前空間としてnullで実行し、その結果を返す。
サニタイザー名の正規化はnameとデフォルト名前空間defaultNamespaceで以下を実行する：

  1. アサート：nameはDOMString または辞書である。

  2. nameがDOMStringの場合、 «[ "name" → name, "namespace" → defaultNamespace]»を返す。

  3. アサート：nameは辞書かつname["name"]が存在する。

  4. namespaceをname["namespace"]が存在すればそれに、なければdefaultNamespaceに設定。

  5. namespaceが空文字ならnullに設定。

  6. «[
    "name" → name["name"],
    "namespace" → namespace
    ]»を返す。

3.3. 補助アルゴリズム

この仕様で使用される正規化済み element と attribute name のリストについては、 リストメンバーシップ判定は "name" および "namespace" の両方の一致に基づきます：

サニタイザー名リストlistがitemを含むのは、 list内に順序付きマップであるentryがあり、 item["name"]がentry["name"]と等しい かつ item["namespace"]がentry["namespace"]と等しい場合です。
removeは、itemとlist（順序付きマップ）に対し、 removeでlistからitem["name"]がentry["name"]と等しいかつ item["namespace"]がentry["namespace"]と等しいものをすべて削除します。
addは、name（正規化済み）をlist（順序付きマップ）に追加するアルゴリズムです：

  1. もしlistがnameを含むならreturn。

  2. appendでnameをlistに追加。

順序付き集合の等価性は、メンバーの一致によって判定され、順序は考慮しません： 順序付き集合 A と B が等しいのは、AがBの上位集合かつBがAの上位集合のときです。
真偽値 boolのnotは、boolがtrueならfalse、そうでなければtrueを返します。

4. セキュリティに関する考慮事項

Sanitizer APIは、DOMベースのクロスサイトスクリプティング（XSS）を防ぐことを目的としており、与えられたHTMLコンテンツを走査し、設定に従って要素や属性を除去します。指定されたAPIはスクリプト可能なマークアップを残すSanitizerオブジェクトの構築をサポートしてはならず、そうすることは脅威モデル上のバグです。

とはいえ、Sanitizer APIの正しい利用でも防げないセキュリティ問題があり、次のセクションでそのシナリオを説明します。

4.1. サーバーサイドの反射型・保存型XSS

このセクションは規範的ではありません。

Sanitizer APIはDOM内のみで動作し、既存のDocumentFragmentを走査・フィルタする機能を追加します。Sanitizerはサーバーサイドの反射型や保存型XSSには対応しません。

4.2. DOMクラッタリング

このセクションは規範的ではありません。

DOMクラッタリングとは、悪意のあるHTMLがidやname属性を用いて、DOM内のHTML要素のchildrenなどのプロパティを、悪意ある内容で上書きしてアプリケーションを混乱させる攻撃です。

Sanitizer APIはデフォルト状態ではDOMクラッタリング攻撃を防ぎませんが、idやname属性を除去するよう設定できます。

4.3. スクリプトガジェットによるXSS

このセクションは規範的ではありません。

スクリプトガジェットは、攻撃者が有名なJavaScriptライブラリの既存アプリケーションコードを利用し、自身のコードを実行させる手法です。これは無害に見えるコードや一見無害なDOMノードを注入し、フレームワークがその入力に基づきJavaScriptを実行することで達成されます。

Sanitizer APIはこれらの攻撃を防ぐことはできません。ページ制作者は未知の要素を一般に許可しないよう明示的に設定し、またdata-やslot属性、<slot>や<template>などテンプレートやフレームワーク固有のマークアップも明示的に設定する必要があります。 これらの制限は網羅的ではないと考えられるため、ページ制作者はサードパーティライブラリの挙動も十分に調査することが推奨されます。

4.4. 変異型XSS

このセクションは規範的ではありません。

変異型XSS（mXSS）は、HTMLスニペットのパース時に文脈の不一致に基づいて発生する攻撃を指します。特に、パース済みHTMLフラグメントを文字列化し、異なる親要素に挿入して再度パースされた場合、元と異なる解釈になることがあります。例えば、異種コンテンツやタグの入れ子ミスによるパース挙動の変化を利用した攻撃が挙げられます。

Sanitizer APIは文字列をノードツリーへ変換する機能のみを提供します。全てのサニタイザー関数で文脈は暗黙的に与えられます： Element.setHTML()は現在の要素を使い、Document.parseHTML()は新しいドキュメントを作成します。したがってSanitizer APIは変異型XSSの直接的な影響を受けません。

もし開発者がサニタイズ済みノードツリーを.innerHTMLなどで文字列として取得し、再度パースする場合、変異型XSSが発生する可能性があります。このような運用は推奨されません。もしHTMLを文字列として処理・受け渡しする必要がある場合は、その文字列は信頼できないものとみなし、DOMへ挿入する際には必ず再度サニタイズすべきです。つまり、サニタイズ済みであってもシリアライズされたHTMLツリーは再度サニタイズされない限り安全とはみなせません。

mXSSのより詳細な解説は[MXSS]にあります。

  SanitizerConfig get();

  // Modify a Sanitizer’s lists and fields:
  undefined allowElement(SanitizerElementWithAttributes element);
  undefined removeElement(SanitizerElement element);
  undefined replaceElementWithChildren(SanitizerElement element);
  undefined allowAttribute(SanitizerAttribute attribute);

  undefined removeUnsafe();
