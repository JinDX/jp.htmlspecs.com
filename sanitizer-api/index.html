1. はじめに

このセクションは規範的ではありません。

ウェブアプリケーションは、クライアント側でHTML文字列を扱う必要がある場合が多くあります。 例えばクライアント側テンプレートソリューションの一部として、あるいはユーザー生成コンテンツのレンダリングとして等です。安全にこれを行うのは困難です。 文字列を単純に結合して Elementの innerHTML に詰め込むという安易な方法は、 予期しない方法でJavaScriptが実行されるリスクを伴います。

[DOMPURIFY]のようなライブラリは、 挿入前に文字列を注意深くパースしてサニタイズし、DOMを構築し、 許可リストを通してそのメンバーをフィルタリングすることでこの問題に対処しようとします。 しかし、このアプローチは脆弱であることが判明しています。 ウェブ向けに公開されているパースAPIは、 実際にブラウザが「本物の」DOMとして文字列をレンダリングする際の挙動と 必ずしも合理的に一致しません。 さらに、ライブラリ側はブラウザの挙動の変化に常に追随する必要があります。 以前は安全だったものが、プラットフォームレベルの新機能によって 時限爆弾に変わることもあります。

ブラウザは、コードを実行するタイミングをかなり正確に把握しています。 ユーザ空間のライブラリよりも、ブラウザ自身に任意の文字列から安全にHTMLをレンダリングする方法を教えることで、 ブラウザのパーサ実装の変更に合わせて維持・更新される可能性が高い形で、 より良い方法を提供できます。 本文書はそのためのAPIの概要を示します。

1.1. 目的

  • 開発者がユーザーが制御するHTMLを扱う際に、注入時に直接スクリプトが実行されないようにする メカニズムを提供することで、DOMベースのクロスサイトスクリプティング攻撃のリスクを軽減します。

  • 現在のユーザーエージェントにおけるHTMLの理解を考慮して、 HTML出力を安全に利用できるようにします。

  • 開発者がデフォルトの要素や属性のセットを上書きできるようにします。 特定の要素や属性を追加することで スクリプトガジェット 攻撃を防ぐことができます。

1.2. API概要

Sanitizer APIは、HTMLを含む文字列を解析してDOMツリーに変換し、 ユーザーが指定した設定に従ってそのツリーをフィルタリングする機能を提供します。 メソッドは2つの種類があります：

  • 安全・非安全： 「安全」メソッドはスクリプトを実行するマークアップを生成しません。 つまり、XSSから安全なはずです。「非安全」メソッドは、指定された内容をそのまま解析・フィルタします。 詳細は§ 4 セキュリティに関する考慮事項も参照してください。

  • コンテキスト：メソッドはElement および ShadowRoot 上に定義されており、 これらのNodeの 子を置き換えます。 おおむねinnerHTML に類似しています。 Documentにも 静的メソッドがあり、 ドキュメント全体を解析する点で DOMParserの parseFromString() に類似しています。

2. フレームワーク

2.1. Sanitizer API

Element インターフェースは、setHTML() と setHTMLUnsafe() の2つのメソッドを定義します。 どちらもHTMLマークアップを含むDOMString と、オプションの設定を受け取ります。 

partial interface Element {
  [CEReactions] undefined setHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLUnsafeOptions options = {});
  [CEReactions] undefined setHTML(DOMString html, optional SetHTMLOptions options = {});
};

  1. compliantHTMLを、 Get Trusted Type compliant stringアルゴリズムに TrustedHTML、 thisの relevant global object、html、 "Element setHTMLUnsafe"、"script"を渡して呼び出すことで得る。

  2. targetを、 thisが template 要素の場合は template contents、 それ以外の場合はthisとする。

  3. Set and filter HTMLにtarget、this、 compliantHTML、options、falseを渡して呼び出す。

Elementの setHTML(html, options) メソッドの手順は:

  1. targetを、 thisが template 要素の場合は template contents、 それ以外の場合はthisとする。

  2. Set and filter HTMLにtarget、this、html、 options、trueを渡して呼び出す。 

partial interface ShadowRoot {
  [CEReactions] undefined setHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLUnsafeOptions options = {});

  1. compliantHTMLを、 Get Trusted Type compliant stringアルゴリズムに TrustedHTML、 thisの relevant global object、html、 "ShadowRoot setHTMLUnsafe"、"script"を渡して呼び出すことで得る。

  2. Set and filter HTMLに this、 thisの shadow host（コンテキスト要素）、 compliantHTML、options、falseを渡して呼び出す。

ShadowRootの setHTML(html, options) メソッドの手順は:

Document インターフェースは、ドキュメント全体を解析する2つの新しいメソッドを持ちます： 

partial interface Document {
  static Document parseHTMLUnsafe((TrustedHTML or DOMString) html, optional SetHTMLUnsafeOptions options = {});
  static Document parseHTML(DOMString html, optional SetHTMLOptions options = {});
};
parseHTMLUnsafe(html, options) メソッドの手順は:

  1. compliantHTMLを、 Get Trusted Type compliant stringアルゴリズムに TrustedHTML、 thisの relevant global object、html、 "Document parseHTMLUnsafe"、"script"を渡して呼び出すことで得る。

  2. documentを新しい Document（ content typeが"text/html"）として作成する。

    注: documentは ブラウジングコンテキストを持たないため、スクリプトは無効化されます。

  3. documentの allow declarative shadow rootsを trueに設定する。

  4. Parse HTML from a stringに documentとcompliantHTMLを渡して呼び出す。

  5. sanitizerを get a sanitizer instance from options にoptionsとfalseを渡して呼び出すことで得る。

  6. sanitizeを documentにsanitizerとfalseを渡して呼び出す。

  7. documentを返す。

parseHTML(html, options) メソッドの手順は:

  1. documentを新しい Document（ content typeが"text/html"）として作成する。

    注: documentは ブラウジングコンテキストを持たないため、スクリプトは無効化されます。

  2. documentの allow declarative shadow rootsを trueに設定する。

  3. Parse HTML from a stringに documentとhtmlを渡して呼び出す。

  4. sanitizerを get a sanitizer instance from options にoptionsとtrueを渡して呼び出すことで得る。

  5. sanitizeを documentにsanitizerとtrueを渡して呼び出す。

  6. documentを返す。

2.2. SetHTMLオプションと設定オブジェクト

setHTML() 系メソッドはすべて、オプション辞書を受け取ります。 現時点で、この辞書のメンバーは1つだけ定義されています：

enum SanitizerPresets { "default" };
dictionary SetHTMLOptions {
  (Sanitizer or SanitizerConfig or SanitizerPresets) sanitizer = "default";
};
dictionary SetHTMLUnsafeOptions {
  (Sanitizer or SanitizerConfig or SanitizerPresets) sanitizer = {};
};

Sanitizer の設定オブジェクトはフィルター構成をカプセル化します。 同じ構成は「安全」または「非安全」メソッドの両方で利用でき、 「安全」メソッドは渡された設定に対して暗黙的に removeUnsafe 操作を行い、設定が渡されなかった場合はデフォルト構成が使われます。 デフォルトは「安全」と「非安全」メソッドで異なります： 「安全」メソッドはデフォルトで安全になることを目指し、制限的なデフォルトです。一方「非安全」メソッドはデフォルトで制限がありません。 設定の利用意図としては、ページのライフタイムの早い段階で一つ（または少数）の設定を構築しておき、必要なときに再利用することです。これにより、実装側で設定の事前処理が可能となります。

設定オブジェクトは設定辞書として取得することができ、直接変更することも可能です。

[Exposed=Window]
interface Sanitizer {
  constructor(optional (SanitizerConfig or SanitizerPresets) configuration = "default");

  // Query configuration:
  SanitizerConfig get();

  // Modify a Sanitizer’s lists and fields:
  boolean allowElement(SanitizerElementWithAttributes element);
  boolean removeElement(SanitizerElement element);
  boolean replaceElementWithChildren(SanitizerElement element);
  boolean allowAttribute(SanitizerAttribute attribute);
  boolean removeAttribute(SanitizerAttribute attribute);
  boolean setComments(boolean allow);
  boolean setDataAttributes(boolean allow);

  // Remove markup that executes script.
  boolean removeUnsafe();
};

Sanitizer には関連付けられたSanitizerConfig configurationがあります。

constructor(configuration) メソッドの手順は次の通りです：

  1. configurationがSanitizerPresets 文字列である場合：

    1. Assert：configurationがdefault である。

    2. configurationに組み込みの安全なデフォルト構成を設定する。

  2. set a configurationを configurationとtrue、thisで呼び出し、 返り値をvalidとする。

  3. validがfalseなら、TypeErrorを投げる。

get() メソッドの手順はthisのconfigurationの値を返すことです。
allowElement(element) メソッドの手順は要素を許可をelementとthisのconfigurationで呼び出すことです。
removeElement(element) メソッドの手順は 要素を除去をelementとthisのconfigurationで呼び出すことです。
replaceElementWithChildren(element) メソッドの手順は 要素をその子要素に置換を elementとthisのconfigurationで呼び出すことです。
allowAttribute(attribute) メソッドの手順は属性を許可をattributeとthisのconfigurationで呼び出すことです。
removeAttribute(attribute) メソッドの手順は属性を除去をattributeとthisのconfigurationで呼び出すことです。
setComments(allow) メソッドの手順はコメント設定を allowとthisのconfigurationで呼び出すことです。
setDataAttributes(allow) メソッドの手順はdata属性設定をallowとthisのconfigurationで呼び出すことです。
removeUnsafe() メソッドの手順は thisのconfigurationを、remove unsafeを thisのconfigurationに対して呼び出した結果で更新することです。

2.4. 構成の不変条件

構成は開発者が用途に応じて変更することができますし、すべきです。選択肢としては、新しい構成辞書を一から作成する、既存のSanitizerの 構成を修正メソッドで変更する、または既存のget() でSanitizerの 構成を辞書として取得し、辞書を修正して新しいSanitizerを作成することができます。

空の構成は（"unsafe"なメソッド、例えばsetHTMLUnsafeを使った場合）全てを許可します。 "default"構成には組み込みの安全なデフォルト構成が含まれます。 "safe"と"unsafe"のサニタイザーメソッドではデフォルトが異なることに注意してください。

全ての構成辞書が有効とは限りません。有効な構成は冗長（同じ要素を二重に許可するなど）や矛盾（同じ要素を許可と除去両方に指定するなど）を避けます。

有効な構成となるために守るべき条件がいくつかあります：

  • グローバル許可リストと除去リストの混在：

    • elements またはremoveElements のどちらかは存在できますが、両方同時には不可です。 両方が無い場合は、removeElements setを« [] »にするのと同じ意味です。

    • attributes またはremoveAttributes のどちらかは存在できますが、両方同時には不可です。 両方が無い場合は、removeAttributes setを« [] »にするのと同じ意味です。

    • dataAttributes は概念的にattributes 許可リストの拡張です。dataAttributes 属性はattributes リストがある場合のみ許可されます。

  • 異なるグローバルリスト間の重複：

    • elements、 removeElements、 replaceWithChildrenElements 間で重複する要素はありません。

    • attributes とremoveAttributes 間で重複する属性はありません。

  • 同じ要素内での重複：

    • 同じ要素のattributes とremoveAttributes 間で重複はありません。

elements 要素の許可リストは、指定した要素の属性の許可・除去も指定できます。これは[HTML]の構造を模しています。HTMLは グローバル属性と特定要素に適用されるローカル属性の両方を知っています。 グローバル属性とローカル属性は混在可能ですが、ある属性が一方で許可され、他方で禁止されるようなあいまいな構成は通常無効です。

グローバルattributes グローバルremoveAttributes
ローカルattributes 属性がどちらかのリストに一致すれば許可されます。重複は認められません。 属性はローカル許可リストにある場合のみ許可されます。 グローバル除去リストとローカル許可リスト間で重複はありません。 グローバル除去リストはこの要素には機能しませんが、ローカル許可リストがない他の要素には適用されることがあります。
ローカルremoveAttributes 属性がグローバル許可リストにあり、ローカル除去リストに無ければ許可されます。ローカル除去リストはグローバル許可リストの部分集合でなければいけません。 属性がどちらのリストにもない場合に許可されます。 グローバル除去リストとローカル除去リスト間で重複はありません。

多くの場合グローバルリストと各要素ごとのリスト間で重複が許されませんが、グローバル許可リストと各要素の除去リストの組み合わせでは、後者が前者の部分集合でなければならないという非対称性があります。上記の表から重複だけに着目した抜粋は次の通りです：

グローバルattributes グローバルremoveAttributes
ローカルattributes 重複は認められません。 重複は認められません。
ローカルremoveAttributes ローカル除去リストはグローバル許可リストの部分集合でなければいけません。 重複は認められません。

dataAttributes 設定はカスタムデータ属性を許可します。上述の規則はdataAttributes を許可リストとみなせば容易に拡張できます：

グローバルattributes とdataAttributes の両方が設定されている場合
ローカルattributes 全てのカスタムデータ属性が許可されます。どの許可リストにもカスタムデータ属性が記載されてはいけません（重複になるため）。
ローカルremoveAttributes カスタムデータ属性はローカル除去リストに記載されていなければ許可されます。 グローバル許可リストにカスタムデータ属性が記載されてはいけません（重複になるため）。

これらのルールを言葉でまとめると：

  • グローバルリストとローカルリスト間の重複および相互作用：

    • グローバルattributes 許可リストが存在する場合、全要素のローカルリストについて：

      • ローカルattributes 許可リストがある場合、これらのリスト間に重複は認められません。

      • ローカルremoveAttributes 除去リストがある場合、その全ての項目はグローバルattributes 許可リストにも記載されていなければいけません。

      • dataAttributes がtrueの場合、どの許可リストにもカスタムデータ属性が記載されてはいけません。

    • グローバルremoveAttributes 除去リストが存在する場合：

      • ローカルattributes 許可リストがある場合、これらのリスト間に重複は認められません。

      • ローカルremoveAttributes 除去リストがある場合、これらのリスト間に重複は認められません。

      • dataAttributes は存在してはいけません。

SanitizerConfig configが有効であるための条件は、以下のすべてを満たす場合である：

  1. configはelementsキー またはremoveElementsキーのいずれかを持ち、両方は持たないこと。

  2. configはattributesキー またはremoveAttributesキーのいずれかを持ち、両方は持たないこと。

  3. アサート： SanitizerElementNamespaceWithAttributes、 SanitizerElementNamespace、 SanitizerAttributeNamespace のすべての項目が正規化済み（sanitizer要素の正規化または sanitizer属性の正規化を通している）であること。

  4. config[elements]、 config[removeElements]、 config[replaceWithChildrenElements]、 config[attributes]、 または config[removeAttributes]が 存在する場合、 重複を持たないこと。

  5. 両方のconfig[elements]と config[replaceWithChildrenElements]が 存在する場合、 両者の共通部分は 空であること。

  6. 両方のconfig[removeElements]と config[replaceWithChildrenElements]が 存在する場合、 両者の共通部分は 空であること。

  7. config[attributes]が 存在する場合：

    1. config[elements]が 存在する場合：

      1. 任意の elementがconfig[elements]の中に存在する場合：

        1. element[attributes] または element[removeAttributes]が存在する場合、 重複を持たないこと。

        2. config[attributes]と element[attributes]の デフォルト値 « [] » での 交差は 空であること。

        3. element[removeAttributes]の デフォルト値 « [] » は configのattributesの部分集合であること。

        4. dataAttributesが 存在し、 dataAttributesがtrueの場合：

          1. element[attributes] はカスタムデータ属性を含まないこと。

    2. dataAttributesがtrueの場合：

      1. config[attributes]は カスタムデータ属性を含まないこと。

  8. config[removeAttributes]が 存在する場合：

    1. config[elements]が 存在する場合は、 任意の elementがconfig[elements]の中に存在する場合：

      1. config[removeAttributes]と element[attributes]の デフォルト値 « [] » での 交差は 空であること。

      2. config[removeAttributes]と element[removeAttributes]の デフォルト値 « [] » での 交差は 空であること。

    2. config[dataAttributes]は 存在しないこと。

注: 構成の設定 を辞書から行う場合、少し正規化が行われます。特に、許可リストと除去リストが両方とも無い場合は、空の除去リストと解釈されます。 したがって{}自体は有効な構成ではありませんが、 {removeElements:[],removeAttributes:[]}に正規化されるため有効となります。 この正規化ステップは、辞書が無い場合と空の場合を一致させるために選ばれました。つまりsetHTMLUnsafe(txt)と setHTMLUnsafe(txt, {sanitizer: {}})が一貫性を持つようにしています。

3. アルゴリズム

HTMLを設定・フィルタするには、Element またはDocumentFragment であるtarget、Element であるcontextElement、文字列のhtml、辞書のoptions、真偽値のsafeを受け取る：

  1. safeがtrueで、contextElementのローカル名が"script"、 かつcontextElementの名前空間がHTML名前空間 またはSVG名前空間である場合、returnする。

  2. sanitizerをget a sanitizer instance from options にoptionsとsafeを渡して得る。

  3. newChildrenをHTMLフラグメントパースアルゴリズム にcontextElement、html、trueを渡して得る。

  4. fragmentを新しいDocumentFragment （ノード文書はcontextElementのノード文書）として作成する。

  5. 各node（newChildrenの要素）について、 appendでfragmentに追加する。

  6. sanitizeを fragmentにsanitizerとsafeで実行する。

  7. Replace allで target内をfragmentで置換する。

optionsからサニタイザーインスタンスを取得する には 辞書のoptionsと真偽値のsafeを受け取る：

注: このアルゴリズムはSetHTMLOptions と SetHTMLUnsafeOptions 両方に対応しています。違いはデフォルトのみです。

  1. sanitizerSpecを"default"に設定する。

  2. もしoptions["sanitizer"] 存在するなら：

  1. アサート：name は DOMString または dictionary のいずれかである。

  2. もし name が DOMString なら、«[ "name" → name, "namespace" → defaultNamespace]» を返す。

  3. アサート：name は dictionary であり、 name["name"] が 存在する。

  4. namespace を name["namespace"] が 存在すればその値、 そうでなければ defaultNamespace にする。

  5. もし namespace が空文字列なら、null に設定する。

  6. «[
    "name" → name["name"],
    "namespace" → namespace
    ]» を返す。

3.5. 補助アルゴリズム

この仕様で使用される、正規化済みの element と attribute name のリストにおける要素の所属判定は、"name" と "namespace" の両方が一致するかどうかで行われる：

A Sanitizer name list に含まれる item は、listの中に順序付けられたマップであるentryが存在し、かつ item["name"] が entry["name"] と等しく、item["namespace"] が entry["namespace"] と等しい場合に成立する。
To 削除する an item from a list that is an ordered map, listから、 item["name"] が等しく、item["namespace"] が等しいすべてのentryを削除する。
To 追加する a name to a list, where name is canonicalized and list is an ordered map:

  1. もしlistがnameを含んでいれば、何もしない。

  2. Append name を list に追加する。

順序付き集合の等価性は、順序を考慮せずメンバーが等しいことである： 順序付き集合 A と B は、 等しい とされるのは、AがBの上位集合であり、 かつBがAの上位集合である場合である。
順序付きマップは キーと 値の タプルの並びである。 順序付きマップの等価性は、このタプル並びを順序付き集合として扱った場合の等価性である。 順序付きマップ A と B は、 等しい とされるのは、Aのエントリから成る 順序付き集合と、 Bのエントリから成る 順序付き集合が 等しい場合である。
リスト listは、重複がある とされるのは、任意の itemがlistに存在し、かつ item["name"]がentry["name"]であり、 item["namespace"]がentry["namespace"]である entryがlist内に2つ以上存在する場合である。
リスト listから 重複を除去する には、

  1. resultを « [] » とする。

  2. 全ての entry について、add entryをresultに追加する。

  3. resultを返す。

2つのリスト A と B の SanitizerElement の共通部分は、 集合の共通部分と同じだが、 集合のエントリは canonicalize a sanitizer nameが事前に適用されている：

  1. set Aを « [] » とする。

  2. set Bを « [] » とする。

  3. 全ての entry of A について、appendで canonicalize a sanitizer nameを entryに対して行い、set Aに追加する。

  4. 全ての entry of B について、appendで canonicalize a sanitizer nameを entryに対して行い、set Bに追加する。

  5. set Aとset Bの共通部分を返す。

boolean bool の 否定を判定するには、 boolがtrueならfalseを返し、それ以外ならtrueを返す。
コメントは、 アルゴリズム内の特定の箇所に対して説明文を含むものである。

4. セキュリティに関する考慮事項

Sanitizer APIは、供給されたHTMLコンテンツを走査し、構成に従って要素や属性を除去することで、DOMベースのクロスサイトスクリプティング（XSS）を防ぐことを目的としています。指定されたAPIは、スクリプト可能なマークアップを残すSanitizerオブジェクトの構築をサポートしてはならず、そうすることは脅威モデル上のバグとなります。

とはいえ、Sanitizer APIの正しい使用では保護できないセキュリティ上の問題もあり、これらのシナリオは以下のセクションで説明します。

4.1. サーバーサイド反射型・保存型XSS

このセクションは規範的ではありません。

Sanitizer APIはDOM内のみで動作し、既存のDocumentFragmentを走査・フィルタする機能を追加します。Sanitizerはサーバーサイドの反射型または保存型XSSには対応しません。

4.2. DOMクラッタリング

このセクションは規範的ではありません。

DOMクラッタリングとは、悪意のあるHTMLがidやname属性を使って要素に名前を付けることで、DOM内のHTML要素のchildrenなどのプロパティが悪意ある内容によって上書きされ、アプリケーションを混乱させる攻撃を指します。

Sanitizer APIはデフォルト状態ではDOMクラッタリング攻撃を防ぎませんが、idやname属性を除去するように構成することが可能です。

4.3. スクリプトガジェットによるXSS

このセクションは規範的ではありません。

スクリプトガジェットとは、攻撃者が有名なJavaScriptライブラリの既存のアプリケーションコードを利用して自身のコードを実行させる手法です。これは、無害に見えるコードや一見無害なDOMノードを注入し、それをフレームワークが解析・解釈してJavaScriptの実行を行う場合によく見られます。

Sanitizer APIはこれらの攻撃を防ぐことはできませんが、ページ著者が未知の要素を一般的に明示的に許可する必要があり、さらにテンプレートやフレームワーク固有のコードによく使われる未知の属性や要素（例：data-属性、slot属性、<slot>要素や<template>要素など）についても明示的に構成する必要があります。これらの制限は網羅的ではないと考えられるため、ページ著者にはサードパーティライブラリの挙動を十分に確認することが推奨されます。

4.4. 変異型XSS

このセクションは規範的ではありません。

変異型XSS（mXSS）とは、HTMLスニペットを正しいコンテキストなしでパースした際に発生するパーサコンテキストの不一致を利用した攻撃です。特に、パースされたHTMLフラグメントを文字列として直列化し、その文字列を異なる親要素に挿入した場合、同じようにパース・解釈される保証はありません。こうした攻撃の例としては、外部コンテンツや入れ子が間違ったタグのパース挙動の変化を利用するものがあります。

Sanitizer APIは、文字列をノードツリーへ変換する関数のみを提供します。すべてのSanitizer関数が暗黙的にコンテキストを指定します：Element.setHTML()は現在の要素を使用し、Document.parseHTML()は新しいドキュメントを作成します。そのためSanitizer APIは直接的に変異型XSSの影響を受けません。

開発者が例えば.innerHTMLなどによってサニタイズされたノードツリーを文字列として取得し、それを再度パースする場合、変異型XSSが発生する可能性があります。このような利用は推奨されません。HTMLを文字列として処理・渡す必要がある場合でも、その文字列は信頼できないものとみなし、DOMに挿入する際は再度サニタイズする必要があります。言い換えれば、サニタイズ後に直列化したHTMLツリーはもはや「サニタイズ済み」とみなすことはできません。

mXSSのより詳しい解説は、[MXSS]に記載されています。

  SanitizerConfig get();
