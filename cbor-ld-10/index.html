要約

CBORは、コンパクトなバイナリデータのシリアライズおよびメッセージング形式です。この 仕様は、Linked DataをシリアライズするためのCBORベースの形式であるCBOR-LD 1.0を定義します。 このエンコーディングは、現在数億ものシステムにデプロイされている既存のJSON-LDエコシステムを 活用し、Linked Dataの効率的なエンコーディング方式を求める人々にコンパクトな シリアライズ形式を提供するよう設計されています。セマンティック圧縮方式を利用することで、 汎用的な圧縮方式よりも60%を超えて優れた圧縮率を実現できます。この形式は 主に、ストレージや帯域幅に制約のあるプログラミング環境でLinked Dataを使用する方法を提供し、 相互運用可能なセマンティックなワイヤレベルプロトコルを構築し、CBORベースのストレージエンジンで Linked Dataを効率的に保存することを目的としています。

          1.3 設計目標と根拠

          この節は非規範的です。

          CBOR-LDは、次の設計目標を満たします:

          単純さ
          既存のJSON-LD実装があれば、実装は容易であるべきです。
          効率的なストレージ
          エンコーディング処理は、積極的にコンパクトなLinked Dataバイナリ 形式を生成するべきです。
          汎用アルゴリズム
          エンコーディングアルゴリズムは汎用化されていなければなりません。
          セマンティック圧縮
          エンコーディング形式は、Linked Data URL（用語および 値）の圧縮を最大化するべきです。ここに注目することで、アルゴリズムが汎用圧縮アルゴリズムよりも 優れた圧縮率を達成できるようになります。
          生のバイナリ
          Baseエンコードされたバイナリ値、およびその他の圧縮可能なデータ型は、 汎用性を犠牲にせずに可能な場合、Baseエンコード形式から生のバイナリ形式に 変換されるべきです。

          同様に、次のものは非目標です。

          • 理論的な純粋性。
          • CBOR-LD専用の新しい圧縮/展開アルゴリズムを作成すること。
          • 地球上のあらゆる用語のための大規模な集中型レジストリ。
          • JSON-LDからCBOR-LDへ、そして元に戻すラウンドトリップのために、 大きな仕様変更や複雑な実装を必要とするデータ形式。

          この仕様に取り組む中で、次の落とし穴が特定されました:

          • ハッシュ化は高速だと広く考えられていますが、小さな文字列集合に対する文字列 比較よりは遅くなります。これは、ハッシュ化するには 文字列全体を消費しなければならないためです。文字列比較を行う場合は、 バイト単位の比較が失敗した瞬間に中断できます。つまり、 2つの値を処理する場合、文字列比較は処理を中断できますが、ハッシュ化は できません。多くのグローバルに衝突耐性のあるハッシュは256ビットですが、32バイトの 値は大きく、多くの場合、衝突耐性を犠牲にすることなくより良くできます （blake2b-8のようなハッシュアルゴリズムではこれはできません）。
          • JSON-LDデータの前処理を必要とするものは、実用になりません -- コストが高く、その意味でデータにセマンティクスをエンコードしていることになるためです。 これは場合によってはより良い圧縮につながることがありますが、許容できる 投資対効果を提供せずにプロセッサーの複雑さを増大させます。
          • スコープ付きコンテキストは、より汎用化された解決策において最も困難な側面です。 アルゴリズムがスコープを深く理解する必要があるほど、 問題は組合せ的になり、その結果、可能な圧縮値が増え、 圧縮サイズに悪影響を及ぼします。コツは、既存の JSON-LD処理アルゴリズムの決定性を活用し、JSON-LD処理に影響しない場合にのみ 近道を取ることです。

          2. 基本概念

          この節は非規範的です。

          大まかに言うと、CBOR-LDは、追加の圧縮のために 次の仕組みを利用できる、JSON-LD向けのコンパクトなバイナリシリアライズです:

          • セマンティック圧縮: JSON-LD 文書で使用される JSON-LD コンテキストを解析することで、 JSON-LD 用語から整数へのマッピングを動的に作成します。これらのマッピングは、 可逆的な方法で作成されます。
          • 型付き値圧縮（コーデック）: データ型に関するアプリオリな理解に基づいて、 型付き値を可逆的な方法で動的に圧縮します（例: URL から共通の 接頭辞文字列を削除します）。
          • 型付き値圧縮（レジストリ辞書）: ユースケース固有の型付き値圧縮に使用するため、 CBOR-LD レジストリエントリに一覧表示される静的辞書を作成します。

          コーデックは、型付き値を汎用的な方法で圧縮するための基本的なプリミティブです。セマンティック圧縮は JSON-LD用語を圧縮するためのものです。レジストリ辞書は、ユースケース固有の方法で型付き値を圧縮する ためのものです。これらはいずれも任意です — CBOR-LDは、 これらの圧縮戦略のいずれか、すべて、またはいずれも使用せずに利用できます。まとめると、特定のユースケースで 使用されるこれらの戦略の集合は、処理モデルとして知られます。

          3. セマンティック圧縮

          この節は非規範的です。

          セマンティック圧縮は、コンパクトなCBOR-LDペイロードを作成するための強力なツールです。中核となる考え方は、 外部JSON-LDコンテキストオブジェクトの情報内容を用いてJSON-LD用語を圧縮することです。 これらの外部コンテキストオブジェクトは、ペイロードの作成者と消費者の双方が利用できるため、 これは可逆的な方法で行われます。

          一般的なセマンティック圧縮処理では、JSON-LD Documentを取得し、 次を行います:

          1. JSON-LDを圧縮できるかどうかを判断する（URI経由でコンテキストを参照するJSON-LD文書内の @context値を見つける。埋め込みコンテキスト値は圧縮できない）。
          2. JSON-LDコンテキストを処理し、CBOR-LD term-codec map — 圧縮可能な すべての用語のリストを構築する。リストを（Unicodeコードポイント順に）ソートする。 各用語にバイト値を関連付け、0から開始して順に増やす。
          3. JSON-LD DocumentをCBOR-LDにエンコードする。これは、各キーを term-codec map内の用語に関連付けられたバイト値で置き換えることで構成される。
          4. 元のJSON-LDが必要になった場合、CBOR-LDからJSON-LDへのデコードは、 各CBORバイト値をterm-codec map内の関連付けられたキーで置き換えることで構成される。

          4. CBOR-LD用CBORタグ

          CBOR-LDペイロードは、バイナリレベルでCBOR-LDであると識別できる必要があります。CBORは本来、 これを「タグ」機能によってサポートしています — これは、グローバルレジストリを介して ペイロードの残りを記述する、バイナリデータ内のヘッダー値です。

          CBORタグ0xCB1D（タグ値51997）は、CBOR-LDで使用するために IANA CBORタグレジストリに登録されています。このタグ値の直後に続くデータは、 圧縮ペイロードを作成するために使用されたレジストリエントリーを識別します。

          CBOR-LDペイロードは、タグ0xCB1Dでタグ付けされた項目が 2要素の配列であり、最初の要素が major type 0 integerで MUSTあるように構造化されていなければなりません。 この整数はCBOR-LD Registry Entry IDです。バイナリ構造は tag([registryEntryId, payload])です。

          その後、CBOR-LD Registry Entry IDの値を使用して、CBOR-LD Registry内の CBOR-LD Registry Entryを検索します。

          4.1 CBOR-LD Registry

          単一のCBORタグ値を使用しながら、消費に必要な異なる 圧縮テーブル素材を必要とするCBOR-LDの可能なユースケースを無制限に拡張できるように、 次を定義します。

          CBOR-LD Registryは、CBOR-LDペイロードの消費者が展開に必要なterm-codec mapを再構築するために 必要な情報を提供するグローバルなリストです。CBOR-LD Registry Entryには、次が含まれます:

          • Registry Entry ID: 正の整数。
          • Use Case: このエントリーが使用されるCBOR-LDペイロードの種類。
          • typeTables: このレジストリエントリーで使用されるレジストリ辞書を含む配列。
          • processingModel: このレジストリエントリーで使用される処理モデル。処理 モデルは 次を指定します:
            • 動的なJSON-LD用語圧縮にセマンティック圧縮を使用するかどうか、
            • 動的な型付き値圧縮に使用されるコーデック。
            default処理モデルはセマンティック圧縮を使用し、この文書で 指定されているコーデックを含みます。
          • provisional: エントリーが暫定的かどうかを示すブール値。暫定エントリーは 変更または削除される場合があります。

          文字列"callerProvidedTable"はtypeTablesに現れることがあり、このユース ケースでは、グローバルに定義されていない Type Tableが必要であることを示します。

          5. アルゴリズム

          この節では、JSON-LDをCBOR-LDへ変換し、その逆も行うために必要なアルゴリズムを規定します。 この節のアルゴリズムの大部分はセマンティック圧縮機能に関連しており、CBOR-LDペイロードが セマンティック圧縮を使用しない場合のCBOR-LDとJSON-LDの間の変換では使用されません。

          5.1 エンコードとデコード

          5.1.1 JSON-LDから CBOR-LDへのエンコードアルゴリズム

          このアルゴリズムは、マップtypeTable、整数registryEntryId、および JSON-LD 文書jsonldDocumentを入力として受け取り、16進文字列 cborldBytesを返します。

          1. registryEntryIdを 5.6.1 CBOR タグ構造取得アルゴリズムに渡した結果をprefixに設定する。
          2. stateを空のマップに設定する。
          3. registryEntryIdに関連付けられたレジストリエントリーがセマンティック 圧縮を必要とする場合:
            1. state.strategyを"compression"に設定する。
            2. state.typeTableをtypeTableに設定する。
            3. state.registryEntryIdをregistryEntryIdに設定する。
            4. stateを 5.2.1 変換アルゴリズムを初期化するに渡した結果に設定する。
            5. stateと jsonldDocumentを inputDocumentとして5.2.2 文書変換アルゴリズムに渡した結果を outputに設定する。
            6. suffixをoutputのCBORエンコーディングに設定する。
          4. そうでない場合、suffixをjsonldDocumentのCBORエンコーディングに設定する。
          5. prefixの前にsuffixを付加したものの16進エンコーディングを cborldBytesに設定する。
          6. cborldBytesを返す。

          5.1.2 CBOR-LDから JSON-LDへのデコードアルゴリズム

          このアルゴリズムは、CBOR-LDペイロードcborldBytesを受け取り、JSON-LD文書 jsonldDocumentを返します。

            1. state.strategy を "decompression" に設定する。
            2. registryEntryId に関連付けられた CBOR-LD Varint Registry Entry 内の typeTables 配列にある各エントリ type: map について、そのエントリを state.typeTable に追加し、state.reverseTypeTable 内の type の値を inverseMap に設定する。ここで inverseMap は、対応関係を反転した map である。
            3. state を 5.2.1 変換初期化アルゴリズムに渡した結果に state を設定する。
            4. suffix をバイト列からマップへ復号した結果に input を 設定する。
            5. state と input を inputDocuments として 5.2.2 文書変換アルゴリズムに渡した結果に jsonldDocument を設定する。
          5. そうでなければ、cborldBytes を解析した結果に jsonldDocument を設定する。
          6. jsonldDocument を返す。

          5.2 変換アルゴリズム

          この節のアルゴリズムは、データ形式間で入力を抽象的に変換する「converter」の振る舞いを 説明します。この節で後述する"compression"および"decompression"戦略のような「strategy」と 組み合わせて使用すると、これらのアルゴリズムは具象データ形式間の変換としてインスタンス化できます。 "compression"戦略はJSON-LDからCBOR-LDへ変換し、"decompression" 戦略はCBOR-LDからJSON-LDへ変換します。

          このアルゴリズムはマップstateを受け取り、返します。

            1. outputを空のマップに設定する。

          5.2.3 汎用変換 アルゴリズム

          このアルゴリズムは、マップinput、output、state、および activeContext を入力として受け取り、マップstateおよびoutputを含むマップを返します。

          1. state.strategyが"compression"に設定されている場合:

          3. state.strategyが"compression"に設定されている場合、 state、activeContext、input、および outputを渡して 5.2.5.4 圧縮用オブジェクト型取得アルゴリズムを実行した結果である resultについて、stateを result.stateに、objectTypesをresult.objectTypesに設定する。

          5. activeContextと objectTypesを 5.3.4 型スコープ付きコンテキスト適用アルゴリズムに渡した結果を activeContextに設定する。
          6. state.strategyが"compression"に設定されている場合、 state、 input、およびactiveContextを渡して 5.2.5.3 圧縮用入力エントリー取得アルゴリズムを実行した結果である resultについて、stateをresult.stateに、 termEntriesをresult.termEntriesに設定する。
          7. そうでない場合、state、input、output、および activeContextを渡して 5.2.6.3 展開用入力エントリー取得アルゴリズムを 実行した結果であるresultについて、stateをresult.stateに、 outputを result.outputに、termEntriesをresult.termEntriesに設定する。

            3. pluralをtermInfo.pluralの値に設定し、 termTypeを termInfo.def内の@typeの値に設定する。

            6. values内のunconvertedValueごとに:

            7. pluralがtrueに設定されている場合、 outputValuesを outputsに設定する。そうでない場合、outputValuesを outputsの最初の要素に設定する。

          9. resultを空のマップに設定する。
          10. result.stateをstateに、result.outputを outputに設定する。
          11. resultを返す。

          5.2.4 値変換アルゴリズム

          4. outputが定義されている場合、stateとoutputを含む マップであるresultを返す。
          5. valueが配列の場合:
            1. outputsを空の配列に設定する。
            2. valueの各elementについて:
              1. activeContext、state、 termInfo、termType、およびelementを valueとして渡し、5.2.4 値変換アルゴリズムを 実行した結果をresultとする。stateを result.stateに設定し、 result.outputを outputsに追加する。

          5.2.5 圧縮戦略 アルゴリズム

          5.2.6 展開 戦略アルゴリズム

          5.2.6.2 展開用値変換アルゴリズム

          5.2.6.4 展開用オブジェクト型取得アルゴリズム

              1. keyが奇数の場合、pluralをtrueに設定する。そうでない場合、 pluralをfalseに設定する。

          5.3 アクティブコンテキスト処理

          この節のアルゴリズムは、圧縮または展開の任意の時点で、JSON-LD 文書に関連付けられたコンテキスト文書のどの構成要素が使用されているかを 判断する方法を説明します。これらのアルゴリズムには、CBOR-LDで埋め込みコンテキスト、 型スコープ付きコンテキスト、およびプロパティスコープ付きコンテキストを適用する方法が含まれます。 これは、この仕様で後に定義されるContext Loadingアルゴリズムとは対照的です。Context Loadingアルゴリズムは、CBOR-LD 圧縮技法の中核である、用語から整数へのマッピングを構築する方法を説明します。 Active Context ProcessingアルゴリズムとContext Loadingアルゴリズムを合わせることで、CBOR-LDとの間で変換する際に JSON-LDコンテキスト文書をどのように処理すべきかを規定します。

          5.3.1 アクティブ コンテキスト初期化アルゴリズム

          このアルゴリズムは、マップpreviousActiveContextおよびtermMapを受け取り、 マップactiveContextを返します。これは使用中のアクティブコンテキストを更新し、 '@type'のすべてのエイリアスを見つけます。

          1. activeContextを新しいマップに設定する。
          2. activeContext.previousActiveContextをpreviousActiveContextに設定する。
          3. activeContext.termMapをtermMapに設定する。
          4. activeContext.typeTermsを配列['@type']に設定する。
          5. termMap内の[term, def]ごとに:
            1. def内の"@id"の値が"@type"である場合、termを activeContext.typeTermsに追加する。
          6. activeContextを返す。

          5.3.2 埋め込み コンテキスト適用アルゴリズム

          このアルゴリズムは、マップstate、activeContext、およびinputを 入力として受け取り、 マップstateおよびactiveContextを含むマップresultを返します。

          1. state、 activeContext.termMapを activeTermMapとして、またinput内の'@context'の値を contextsとして 5.3.5 用語 マップ更新アルゴリズムに渡した結果をtermMapUpdateResultに設定する。
          2. stateをtermMapUpdateResult.stateに設定する。
          3. termMapをtermMapUpdateResult.activeTermMapに設定する。
          4. termMapとactiveContextを previousActiveContextとして渡し、 5.3.1 アクティブコンテキスト初期化アルゴリズムを実行した結果を newActiveContextに設定する。
          5. resultを新しいマップに設定し、result.activeContextを newActiveContextに、 result.stateをstateに設定する。
          6. resultを返す。

          5.3.3 プロパティスコープ付きコンテキスト適用アルゴリズム

          5.3.5 用語マップ更新アルゴリズム

          このアルゴリズムは、マップstate、activeTermMap、およびマップまたは配列 contexts、ならびにブール値typeScope とpropertyScopeを入力として受け取ります。これらはいずれも、指定されない場合は falseを既定値とします。これはマップstate およびactiveTermMapを返します。

          1. contextsが配列でない場合、contextsを、以前のcontextsの値を 唯一の要素として持つ配列に設定する。
          2. allowProtectedOverrideをpropertyScopeの値に設定する。
          3. propagateDefaultをtypeScopeの値の否定に設定する。
          4. contexts内のcontextIdentifierごとに:
            1. stateおよび contextIdentifierを渡して5.4.2 コンテキスト読み込みアルゴリズムを 実行した結果をloadResultに設定する。
            2. entryをloadResult.entryに、 contextをentry.contextに、state をloadResult.stateに設定する。
            3. @propagateがcontext内に現れる場合、propagate をcontext内の@propagateの値に設定する。 そうでない場合、propagateをpropagateDefaultの値に設定する。
            4. newTermMapを空のマップに設定する。entry.termMap内の [key, value]ごとに:
              1. valueの内容を新しいマップ newValueに浅くコピーし、propagate をnewValueに追加する。
              2. newTermMap内のkeyの値を newValueに設定する。
            5. activeTermMap内の[term, activeDef]ごとに:
              1. defを newTermMap内のtermの値とする。
              2. defが定義されている場合:

                  1. allowProtectedOverrideが falseに設定され、かつdefが activeDefと同一でない場合、 エラーERR_PROTECTED_TERM_REDEFINITIONを投げる。

              3. そうでなく、termがcontext内に現れる場合、 newTermMap内のtermの値を activeDefからのすべての値を含むマップに設定する。
            6. activeTermMapの値をnewTermMapの値に設定する。
          5. resultを空のマップに設定する。
          6. result.stateをstateに、 result.activeTermMapをactiveTermMapに設定する。
          7. resultを返す。

          5.5.3 URL コーデック

          URL コーデックは、値の接頭辞に基づいて URL 値を圧縮します。 この接頭辞は、URL 接頭辞 テーブル内で一致する最長のエントリです。URL は配列としてエンコードされ、最初の整数が URL の接頭辞を識別します。コーデックはまず、値が既知の JSON-LD 用語識別子と一致するかどうかを確認し、一致する場合は単純な整数としてエンコードします。それ以外の場合は、 値の接頭辞に基づいて、以下の接頭辞固有のサブコーデックのいずれかに 委譲します。

          5.5.3.1 URL 接頭辞テーブル

          URL 接頭辞テーブルは、URL 接頭辞をそれに 関連付けられた圧縮整数識別子にマッピングします。URL コーデックを使用する実装は、以下のテーブルを使用しなければなりません。URL 接頭辞逆引きテーブルは、この テーブルを反転したものです（すなわち、整数から URL 接頭辞文字列へのマップです）。

          URL 接頭辞 整数識別子
          http:// 1
          https:// 2
          urn:uuid: 3
          data: 4
          did:v1:nym: 1024
          did:key: 1025
          5.5.3.2 URL エンコーダーを作成する

          このアルゴリズムは、マップ state、マップ termInfo、 および文字列 valueToEncode を受け取り、エンコーダーオブジェクト または undefined を返します。

          1. valueToEncode が文字列でない場合は、 ERR_UNSUPPORTED_JSON_TYPE エラーをスローします。
          2. termId を、 state.termToId 内で valueToEncode を検索した結果に設定します。
          3. termId が整数である場合、5.5.3.3 URL をエンコードするを介して 呼び出されたときに、termId を メジャータイプ 0（符号なし整数）の CBOR 項目としてエンコードするエンコーダーを返します。
          4. valueToEncode を URL として解析することを試みます。 解析に失敗した場合（すなわち、値が有効な URL でない場合）は、 undefined を返します。
          5. prefix を、URL 接頭辞 テーブル内のエントリのうち、 valueToEncode が prefix で始まり、 prefix に続く valueToEncode の残りの部分に 「:」（U+003A COLON）文字が含まれないものに設定します。そのような エントリが存在しない場合は、undefined を返します（これは、この URL を 圧縮できないことを意味します）。
          6. prefix が「https://」または 「http://」である場合、 valueToEncode を渡して 5.5.3.6.1 HTTP URL エンコーダーを作成するを実行した結果を返します。
          7. prefix が「urn:uuid:」である場合、 valueToEncode を渡して 5.5.3.7.1 UUID URN エンコーダーを作成するを実行した結果を返します。
          8. prefix が「data:」である場合、 valueToEncode を渡して 5.5.3.8.1 データ URL エンコーダーを作成するを実行した結果を返します。
          9. prefix が「did:v1:nym:」または 「did:key:」である場合、 valueToEncode を渡して 5.5.3.9.1 Base58 DID URL エンコーダーを作成するを実行した結果を返します。
          10. undefined を返します（これは、この URL を圧縮できないことを意味します）。
          5.5.3.3 URL をエンコードする

          このアルゴリズムは、 5.5.3.2 URL エンコーダーを作成するによって生成されたエンコーダーオブジェクトを受け取り、CBOR バイナリデータを返します。

          1. エンコーダーが用語 ID の一致用に作成された場合（ 5.5.3.2 URL エンコーダーを作成するのステップ 3）、メジャータイプ 0（符号なし整数）オブジェクトとして termId を CBOR エンコードしたものを返します。
          2. それ以外の場合は、適切な接頭辞固有のサブエンコーダーの encode メソッドを呼び出し、結果として得られる CBOR バイナリデータを返します。
          5.5.3.4 URL デコーダーを作成する

          このアルゴリズムは、マップ state、マップ termInfo、 および値 valueToDecode を受け取り、デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode が文字列である場合、undefined を返します （これは、値がすでに圧縮されていない URL 文字列であり、そのまま渡されることを意味します）。
          2. valueToDecode が配列である場合:
            1. prefixId を valueToDecode の最初の要素に設定します。
            2. prefix を、URL 接頭辞 逆引きテーブル内で prefixId に対応する値に設定します。
            3. prefix が「http://」または 「https://」である場合、valueToDecode を渡して 5.5.3.6.3 HTTP URL デコーダーを作成するを実行した結果を decoder に設定します。
            4. それ以外で、prefix が「urn:uuid:」である場合、 valueToDecode を渡して 5.5.3.7.3 UUID URN デコーダーを作成するを実行した結果を decoder に設定します。
            5. それ以外で、prefix が「data:」である場合、 valueToDecode を渡して 5.5.3.8.3 データ URL デコーダーを作成するを実行した結果を decoder に設定します。
            6. それ以外で、prefix が「did:v1:nym:」または 「did:key:」である場合、valueToDecode を渡して 5.5.3.9.4 Base58 DID URL デコーダーを作成するを実行した結果を decoder に設定します。
            7. prefix が上記のいずれにも一致しなかった場合（ prefixId に対応するエントリが URL 接頭辞逆引きテーブルに 存在しない場合を含みます）、または decoder が undefined である場合（これは、サブデコーダーが valueToDecode の形状を拒否したことを意味します）、 ERR_UNKNOWN_COMPRESSED_VALUE エラーをスローします。
            8. decoder を返します。
          3. valueToDecode が整数である場合、term を state.idToTerm 内で valueToDecode を検索した結果に設定し、 5.5.3.5 URL をデコードするの呼び出しが term を返すデコーダーを返します。
          4. undefined を返します。
          5.5.3.5 URL をデコードする

          このアルゴリズムは、 5.5.3.4 URL デコーダーを作成するによって生成されたデコーダーオブジェクトを受け取り、文字列値を返します。

          1. デコーダーが整数の用語 ID 値用に作成された場合（ 5.5.3.4 URL デコーダーを作成するのステップ 3）、関連付けられた term 文字列を返します。
          2. それ以外の場合は、適切な接頭辞固有のサブデコーダーの decode メソッドを呼び出し、結果として得られる文字列を返します。
          5.5.3.6 HTTP URL コーデック

          HTTP URL コーデックは、接頭辞が http:// または https:// である値を処理します。エンコードされた HTTP URL は、 2 要素の CBOR 配列 [prefixId, remainder] として表されます。ここで prefixId は URL 接頭辞テーブルの整数であり、 remainder は、URL から接頭辞を取り除いた部分をテキスト文字列として表したものです。

          5.5.3.6.1 HTTP URL エンコーダーを作成する

          このアルゴリズムは、文字列 value を受け取り、 エンコーダーオブジェクトまたは undefined を返します。

          1. value が "https://" で始まる場合、 prefix を "https://" に設定します。
          2. それ以外で、value が "http://" で始まる場合、 prefix を "http://" に設定します。
          3. それ以外の場合は、undefined を返します。
          4. value と prefix を含む エンコーダーオブジェクトを返します。
          5.5.3.6.2 HTTP URL をエンコードする

          このアルゴリズムは、 5.5.3.6.1 HTTP URL エンコーダーを作成するから得られたエンコーダーオブジェクトを受け取り、CBOR バイナリデータを返します。

          1. prefixId を、URL 接頭辞テーブル内の encoderData.prefix の値に設定します。
          2. rest を、encoderData.value の encoderData.prefix の後から始まる部分文字列に設定します。
          3. 2 要素の配列 [prefixId, rest] を CBOR エンコードして返します。 これは、メジャータイプ 0（符号なし整数）の項目に続いてメジャータイプ 3 （テキスト文字列）の項目を含むメジャータイプ 4（配列）です。
          5.5.3.6.3 HTTP URL デコーダーを作成する

          このアルゴリズムは、CBOR 配列 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode がちょうど 2 つの要素を持ち、2 番目の 要素がテキスト文字列である場合:
            1. prefix を、URL 接頭辞 逆引きテーブル内の valueToDecode[0] の値に設定します。
            2. prefix を含むデコーダーオブジェクトを返します。
          2. それ以外の場合は、undefined を返します。
          5.5.3.6.4 HTTP URL をデコードする

          このアルゴリズムは、 5.5.3.6.3 HTTP URL デコーダーを作成するから得られたデコーダーオブジェクトと CBOR 配列 valueToDecode を受け取り、文字列を返します。

          1. decoderData.prefix と valueToDecode[1] を連結したものを返します。
          5.5.3.7 UUID URN コーデック

          UUID URN コーデックは、接頭辞が urn:uuid: である値を処理します。エンコードされた UUID URN は、 2 要素の CBOR 配列 [3, uuidValue] です。ここで uuidValue は、 UUID をバイナリ形式で表す 16 バイトのメジャータイプ 2（バイト文字列） （UUID 文字列がすべて小文字の場合）、または UUID 値をそのまま含む メジャータイプ 3（テキスト文字列）（大文字と小文字の区別をバイナリ形式経由で 往復変換できない場合）のいずれかです。

          5.5.3.7.1 UUID URN エンコーダーを作成する

          このアルゴリズムは、文字列 value を受け取り、エンコーダー オブジェクトまたは undefined を返します。

          1. value が "urn:uuid:" で始まらない場合、 undefined を返します。
          2. value を含むエンコーダーオブジェクトを返します。
          5.5.3.7.2 UUID URN をエンコードする

          このアルゴリズムは、エンコーダーオブジェクトを受け取り、CBOR バイナリデータを返します。

          1. prefixId を 3（ URL 接頭辞テーブル内の "urn:uuid:" のエントリ）に設定します。
          2. rest を、encoderData.value の "urn:uuid:" の後から始まる部分文字列に設定します。
          3. rest に大文字が含まれない場合（すなわち、その小文字形式と 等しい場合）:
            1. uuidBytes を、[RFC4122] に従って rest を解析して得られる UUID の 16 バイトのバイナリ表現に設定します。 rest を有効な UUID として解析できない場合は、 エラーがスローされ、エンコードは失敗します。
            2. 2 要素の配列 [3, uuidBytes] を CBOR エンコードして返します。 これは、メジャータイプ 0（符号なし整数）の項目に続いて、長さ 16 の メジャータイプ 2（バイト文字列）の項目を含むメジャータイプ 4（配列）です。
          4. それ以外の場合は、2 要素の配列 [3, rest] を CBOR エンコードして返します。 これは、メジャータイプ 0（符号なし整数）の項目に続いてメジャータイプ 3 （テキスト文字列）の項目を含むメジャータイプ 4（配列）です。
          5.5.3.7.3 UUID URN デコーダーを作成する

          このアルゴリズムは、CBOR 配列 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode がちょうど 2 つの要素を持ち、2 番目の 要素がバイト文字列またはテキスト文字列である場合、デコーダーオブジェクトを返します。
          2. それ以外の場合は、undefined を返します。
          5.5.3.7.4 UUID URN をデコードする

          このアルゴリズムは、デコーダーオブジェクトと CBOR 配列 valueToDecode を受け取り、文字列を返します。

          1. valueToDecode[1] がテキスト文字列である場合、uuid を valueToDecode[1] に設定します。
          2. それ以外の場合、uuid を、[RFC4122] に従った 16 バイトのバイナリ値 valueToDecode[1] の UUID 文字列表現（小文字で、ハイフン区切りの形式）に設定します。
          3. "urn:uuid:" と uuid を連結したものを返します。
          5.5.3.8 データ URL コーデック

          データ URL コーデックは、接頭辞が data: である値（[RFC2397] で定義されるデータ URL）を処理します。 Base64 でエンコードされたデータ URL （data:<mediatype>;base64,<data> に一致するもの）は、 3 要素の CBOR 配列 [4, mediatype, dataBytes] としてエンコードされます。Base64 でないデータ URL は、 2 要素の CBOR 配列 [4, rest] としてエンコードされます。ここで rest は、URL の "data:" より後の部分です。

          5.5.3.8.1 データ URL エンコーダーを作成する

          このアルゴリズムは、文字列 value を受け取り、エンコーダー オブジェクトまたは undefined を返します。

          1. value が "data:" で始まらない場合、 undefined を返します。
          2. value を正規表現 ^data:(?<mediatype>.*);base64,(?<data>.*)$ と照合することを試みます。ここで各 .* は貪欲です （すなわち、文字列「;base64,」が複数回出現する場合、 mediatype グループは最後の出現位置まで拡張されます）。
          3. 照合に成功した場合:
            1. data を、data キャプチャグループの値に設定します。
            2. 往復変換の完全性を検証します。data を Base64 からデコードし、 Base64 に再エンコードします。結果が data と等しい場合、 value を含み、base64 が true に設定されたエンコーダーオブジェクトを返します。
          4. value を含み、base64 が false に設定されたエンコーダーオブジェクトを返します。
          5.5.3.8.2 データ URL をエンコードする

          このアルゴリズムは、 5.5.3.8.1 データ URL エンコーダーを作成するから得られたエンコーダーオブジェクトを受け取り、CBOR バイナリデータを返します。

          1. prefixId を 4（ URL 接頭辞テーブル内の "data:" のエントリ）に設定します。
          2. encoderData.base64 が true である場合:
            1. encoderData.value を解析して、mediatype （"data:" と ";base64," の間の部分文字列） および dataBytes（Base64 データ部分をバイナリデコードしたもの） を抽出します。
            2. 3 要素の配列 [prefixId, mediatype, dataBytes] を CBOR エンコードして返します。これは、メジャータイプ 0 の整数、 メジャータイプ 3 のテキスト文字列、およびメジャータイプ 2 のバイト文字列を含む メジャータイプ 4（配列）です。
          3. それ以外の場合:
            1. rest を、encoderData.value の "data:" の後から始まる部分文字列に設定します。
            2. 2 要素の配列 [prefixId, rest] を CBOR エンコードして返します。 これは、メジャータイプ 0 の整数とメジャータイプ 3 のテキスト文字列を含む メジャータイプ 4（配列）です。
          5.5.3.8.3 データ URL デコーダーを作成する

          このアルゴリズムは、CBOR 配列 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode がちょうど 3 つの要素を持ち、2 番目の 要素がテキスト文字列で、3 番目の要素がバイト文字列である場合、 デコーダーオブジェクトを返します（Base64 モード）。
          2. それ以外で、valueToDecode がちょうど 2 つの要素を持ち、 2 番目の要素がテキスト文字列である場合、デコーダーオブジェクトを返します （非 Base64 モード）。
          3. それ以外の場合は、undefined を返します。
          5.5.3.8.4 データ URL をデコードする

          このアルゴリズムは、デコーダーオブジェクトと CBOR 配列 valueToDecode を受け取り、文字列を返します。

          1. valueToDecode が 3 つの要素を持つ場合（Base64 モード）:
            1. mediatype を valueToDecode[1] に設定します。
            2. encodedData を、バイト文字列 valueToDecode[2] の Base64 エンコードに設定します。
            3. "data:"、mediatype、 ";base64,"、および encodedData を 連結したものを返します。
          2. それ以外の場合（非 Base64 モード）は、"data:" と valueToDecode[1] を連結したものを返します。
          5.5.3.9 Base58 DID URL コーデック

          Base58 DID URL コーデックは、接頭辞が did:v1:nym: または did:key: であり、 オーソリティ（および任意のフラグメント）コンポーネントが通常、 multibase の base58btc でエンコードされた文字列（文字 'z' で始まるもの）である DID URL を処理します。圧縮された DID URL は、 2 要素または 3 要素の CBOR 配列 [prefixId, authority] または [prefixId, authority, fragment] としてエンコードされます。コンポーネントが 'z' で始まり、有効な base58btc である場合は、生のバイト文字列として 格納されます。それ以外の場合は、テキスト文字列として格納されます。

          注記
          このコーデックで使用される URL 接頭辞テーブルのエントリは、 did:v1:nym: → 1024 および did:key: → 1025 です。
          5.5.3.9.1 Base58 DID URL エンコーダーを作成する

          このアルゴリズムは、文字列 value を受け取り、 エンコーダーオブジェクトまたは undefined を返します。

          1. 接頭辞が "did:" で始まる URL 接頭辞テーブル内の各 [prefix, prefixId] ペアについて:
            1. value が prefix で始まる場合、 value、prefix、および prefixId を含むエンコーダーオブジェクトを返します。
          2. undefined を返します。
          5.5.3.9.2 Base58 DID URL をエンコードする

          このアルゴリズムは、 5.5.3.9.1 Base58 DID URL エンコーダーを作成するから得られたエンコーダーオブジェクトを受け取り、 CBOR バイナリデータを返します。

          1. suffix を、encoderData.value の encoderData.prefix の後から始まる部分文字列に設定します。
          2. suffix を '#' の最初の出現位置で分割して、 authority と、存在する場合は fragment を生成します。
          3. authorityEncoded を、 authority に Multibase Base58 から CBOR トークンへの変換アルゴリズムを適用した結果に設定します。
          4. entries を [encoderData.prefixId, authorityEncoded] で初期化します。
          5. fragment が定義されている場合:
            1. fragmentEncoded を、 fragment に Multibase Base58 から CBOR トークンへの変換アルゴリズムを適用した結果に設定します。
            2. fragmentEncoded を entries に追加します。
          6. entries をメジャータイプ 4（配列）として CBOR エンコードして返します。
          5.5.3.9.3 Multibase Base58 から CBOR トークンへの変換

          Multibase Base58 から CBOR トークンへの変換アルゴリズムは、 文字列 str を受け取り、値が base58btc multibase 文字列である場合は メジャータイプ 2（バイト文字列）の CBOR トークンを、それ以外の場合は メジャータイプ 3（テキスト文字列）の CBOR トークンを返します。

          1. str が 'z' で始まる場合:
            1. decoded を、str の base58btc の内容 （すなわち str[1:]）をデコードした結果に設定します。
            2. デコードに成功した場合、decoded に対する メジャータイプ 2（バイト文字列）の CBOR トークンを返します。
          2. str に対するメジャータイプ 3（テキスト文字列）の CBOR トークンを返します。
          5.5.3.9.4 Base58 DID URL デコーダーを作成する

          このアルゴリズムは、CBOR 配列 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode がちょうど 2 つまたは 3 つの要素を持つ場合:
            1. prefix を、URL 接頭辞 逆引きテーブル内の valueToDecode[0] の値に設定します。
            2. prefix を含むデコーダーオブジェクトを返します。
          2. それ以外の場合は、undefined を返します。
          5.5.3.9.5 Base58 DID URL をデコードする

          このアルゴリズムは、 5.5.3.9.4 Base58 DID URL デコーダーを作成するから得られたデコーダーオブジェクトと CBOR 配列 valueToDecode を受け取り、文字列を返します。

          1. url を decoderData.prefix に設定します。
          2. valueToDecode[1] がバイト文字列である場合、 'z' に続けて valueToDecode[1] の base58btc エンコードを url に追加します。
          3. それ以外の場合、valueToDecode[1] を url に追加します。
          4. valueToDecode が 2 つを超える要素を持つ場合:
            1. valueToDecode[2] がバイト文字列である場合、 "#z" に続けて valueToDecode[2] の base58btc エンコードを url に追加します。
            2. それ以外の場合、"#" に続けて valueToDecode[2] を url に追加します。
          5. url を返します。

          5.5.4 XSD 日付コーデック

          XSD 日付コーデックは、型が http://www.w3.org/2001/XMLSchema#date である値を圧縮します。日付値は、 Unix エポック（1970-01-01T00:00:00Z）からの秒数を表す整数として表現され、 負でない値には CBOR メジャータイプ 0（符号なし整数）、 エポックより前の日付にはメジャータイプ 1（負の整数）としてエンコードされます。 コーデックは、この表現を介して正確に往復変換できる日付文字列のみを 圧縮します。それ以外の場合、元の文字列はそのまま保持されます。

          注記
          エポックからの秒数は、UTC の午前 0 時の日付 （すなわち、時点 <date>T00:00:00Z）から計算されます。タイムゾーン オフセットまたは時刻コンポーネントがあると往復変換が曖昧になるため、 時刻コンポーネント（'T'）を含む値は、このコーデックでは圧縮されません。
          5.5.4.1 XSD 日付 エンコーダーを作成する

          このアルゴリズムは、文字列 value を受け取り、エンコーダー オブジェクトまたは undefined を返します。

          1. value が文字 'T'（時刻コンポーネントを示す）を 含む場合、undefined を返します（この値は代わりに XSD DateTime コーデックによって圧縮されます）。
          2. parsed を、value を ISO 8601 日付文字列として 解析した結果に設定します。この結果は Unix エポックからのミリ秒数として表されます。
          3. parsed が有効な数値でない場合、 undefined を返します。
          4. value と parsed を含む エンコーダーオブジェクトを返します。
          5.5.4.2 XSD 日付をエンコードする

          このアルゴリズムは、 5.5.4.1 XSD 日付 エンコーダーを作成するから得られたエンコーダーオブジェクトを受け取り、CBOR バイナリデータを返します。

          1. secondsSinceEpoch を floor(encoderData.parsed / 1000) に設定します。
          2. reconstituted を、Unix エポックから secondsSinceEpoch × 1000 ミリ秒後の時点を表す ISO 8601 文字列に設定します。
          3. expectedDate を、 reconstituted 内の 'T' の最初の出現位置より前の 部分文字列に設定します。
          4. encoderData.value が expectedDate と等しくない場合、圧縮は非可逆になります。 encoderData.value をメジャータイプ 3（テキスト文字列） オブジェクトとして CBOR エンコードして返します。
          5. secondsSinceEpoch が 0 以上である場合、 secondsSinceEpoch をメジャータイプ 0（符号なし整数） オブジェクトとして CBOR エンコードして返します。
          6. それ以外の場合、secondsSinceEpoch を メジャータイプ 1（負の整数）オブジェクトとして CBOR エンコードして返します。
          5.5.4.3 XSD 日付 デコーダーを作成する

          このアルゴリズムは、値 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode が整数（正、負、または 0）である場合、デコーダーオブジェクトを返します。
          2. それ以外の場合は、undefined を返します。
          5.5.4.4 XSD 日付をデコードする

          このアルゴリズムは、デコーダーオブジェクトと整数値 value を受け取り、文字列を返します。

          1. dateString を、Unix エポックから value * 1000 ミリ秒後の時点を表す ISO 8601 文字列に設定します。
          2. dateString 内の 'T' の最初の出現位置より前の 部分文字列を返します。

          5.5.5 XSD DateTime コーデック

          XSD DateTime コーデックは、型が http://www.w3.org/2001/XMLSchema#dateTime である値を圧縮します。秒精度の 日時値は、Unix エポックからの秒数を表す単一の整数 （メジャータイプ 0 または 1）としてエンコードされます。秒未満の精度を持つ日時値は、 2 要素の CBOR 配列 [seconds, milliseconds] としてエンコードされ、どちらもメジャータイプ 0 の整数です。XSD 日付コーデックと同様に、値は圧縮が可逆である場合 （元の文字列がエンコードを介して正確に往復変換される場合）にのみ圧縮されます。

          5.5.5.1 XSD DateTime エンコーダーを作成する

          このアルゴリズムは、文字列 value を受け取り、エンコーダー オブジェクトまたは undefined を返します。

          1. value が文字 'T' を含まない場合、 undefined を返します（この値は代わりに XSD 日付コーデックによって圧縮されます）。
          2. parsed を、value を ISO 8601 日時文字列として 解析した結果に設定します。この結果は Unix エポックからのミリ秒数として表されます。
          3. parsed が有効な数値でない場合、 undefined を返します。
          4. value と parsed を含む エンコーダーオブジェクトを返します。
          5.5.5.2 XSD DateTime をエンコードする

          このアルゴリズムは、 5.5.5.1 XSD DateTime エンコーダーを作成するから得られたエンコーダーオブジェクトを受け取り、CBOR バイナリ データを返します。

          1. secondsSinceEpoch を floor(encoderData.parsed / 1000) に設定します。
          2. encoderData.value が文字 '.' を含まない場合（ミリ秒コンポーネントがない場合）:
            1. expectedDate を、Unix エポックから secondsSinceEpoch * 1000 ミリ秒後の時点を表す ISO 8601 文字列で、部分文字列 ".000Z" を "Z" に置き換えたものに設定します。
            2. encoderData.value が expectedDate と等しくない場合、圧縮は非可逆になります。 encoderData.value をメジャータイプ 3 （テキスト文字列）オブジェクトとして CBOR エンコードして返します。
            3. secondsSinceEpoch が 0 以上である場合、 secondsSinceEpoch をメジャータイプ 0（符号なし整数） オブジェクトとして CBOR エンコードして返します。
            4. それ以外の場合、secondsSinceEpoch を メジャータイプ 1（負の整数）オブジェクトとして CBOR エンコードして返します。
          3. それ以外の場合（ミリ秒コンポーネントが存在する場合）:
            1. millisecondIndex を、encoderData.value 内の 最初の '.' のインデックスに設定します。
            2. milliseconds を、 encoderData.value の位置 millisecondIndex + 1 から始まる数字の整数値に設定します。
            3. expectedDate を、Unix エポックから secondsSinceEpoch * 1000 + milliseconds ミリ秒後の時点を表す ISO 8601 文字列に設定します。
            4. encoderData.value が expectedDate と等しくない場合、圧縮は非可逆になります。 encoderData.value をメジャータイプ 3 （テキスト文字列）オブジェクトとして CBOR エンコードして返します。
            5. 2 要素の配列 [secondsSinceEpoch, milliseconds] を CBOR エンコードして返します。これは、2 つのメジャータイプ 0 （符号なし整数）の項目を含むメジャータイプ 4（配列）です。
          5.5.5.3 XSD DateTime デコーダーを作成する

          このアルゴリズムは、値 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode が整数である場合、デコーダーオブジェクトを返します。
          2. valueToDecode がちょうど 2 つの要素を持つ配列で、 両方の要素が数値である場合、デコーダーオブジェクトを返します。
          3. それ以外の場合は、undefined を返します。
          5.5.5.4 XSD DateTime をデコードする

          このアルゴリズムは、デコーダーオブジェクトと値 value （整数または 2 要素の整数配列）を受け取り、文字列を返します。

          1. value が整数である場合:
            1. instantMs を value * 1000 に設定します。
            2. dateString を、Unix エポックから instantMs ミリ秒後の時点を表す ISO 8601 文字列に設定します。
            3. dateString 内の部分文字列 ".000Z" を "Z" に置き換えたものを返します。
          2. それ以外の場合（value が 2 要素の配列である場合）:
            1. instantMs を value[0] * 1000 + value[1] に設定します。
            2. instantMs を返します。

          5.5.6 Multibase コーデック

          Multibase コーデックは、型が https://w3id.org/security#multibase である値を圧縮します。multibase でエンコードされた 文字列は、最初のバイトが Multibase 接頭辞バイトであり、残りのバイトが デコードされたバイナリペイロードである生のバイト文字列に変換されます。 これにより、基数エンコード文字のオーバーヘッドなしで、バイナリペイロードを コンパクトに格納できます。

          次の multibase エンコードがサポートされています:

          Multibase 接頭辞文字 Multibase 接頭辞バイト エンコード
          z 0x7a Base58btc
          u 0x75 Base64url（パディングなし）
          M 0x4d Base64pad
          5.5.6.1 Multibase エンコーダーを作成する

          このアルゴリズムは、値 value を受け取り、エンコーダー オブジェクトまたは undefined を返します。

          1. value が文字列でない場合、undefined を返します。
          2. value の最初の文字が 'z'、 'u'、または 'M' である場合、 value を含むエンコーダーオブジェクトを返します。
          3. それ以外の場合は、undefined を返します（サポートされていない multibase エンコード）。
          5.5.6.2 Multibase をエンコードする

          このアルゴリズムは、 5.5.6.1 Multibase エンコーダーを作成するから得られたエンコーダーオブジェクトを受け取り、CBOR バイナリデータを返します。

          1. prefixChar を encoderData.value の最初の文字に設定します。
          2. prefixByte を、prefixChar に対応する Multibase 接頭辞バイト に設定します。
          3. content を、 encoderData.value の最初の文字の後から始まる部分文字列に設定します。
          4. prefixChar が 'z' である場合、 suffixBytes を content の base58btc デコードに設定します。
          5. それ以外で、prefixChar が 'u' である場合、 suffixBytes を content の base64url デコードに設定します。
          6. それ以外の場合（prefixChar が 'M'）、 suffixBytes を content の base64pad デコードに設定します。
          7. bytes を長さ 1 + length(suffixBytes) のバイト配列に設定し、最初のバイトを prefixByte に、残りのバイトを suffixBytes に設定します。
          8. bytes をメジャータイプ 2（バイト文字列） オブジェクトとして CBOR エンコードして返します。
          5.5.6.3 Multibase デコーダーを作成する

          このアルゴリズムは、値 valueToDecode を受け取り、 デコーダーオブジェクトまたは undefined を返します。

          1. valueToDecode がバイト文字列でない場合、 undefined を返します。
          2. valueToDecode の最初のバイトが 0x7a、 0x75、または 0x4d である場合、 デコーダーオブジェクトを返します。
          3. それ以外の場合は、undefined を返します（サポートされていない multibase 接頭辞バイト）。
          5.5.6.4 Multibase をデコードする

          このアルゴリズムは、デコーダーオブジェクトとバイト文字列 value を受け取り、文字列を返します。

          1. prefixByte を value の最初のバイトに設定します。
          2. suffix を value の残りのバイト （すなわち、最初のバイトより後のすべてのバイト）に設定します。
          3. prefixByte が 0x7a（'z'）である場合:
            1. "z" と suffix の base58btc エンコードを 連結したものを返します。
          4. prefixByte が 0x75（'u'）である場合:
            1. "u" と suffix の base64url エンコード （パディングなし）を連結したものを返します。
          5. prefixByte が 0x4d（'M'）である場合:
            1. "M" と suffix の base64pad エンコードを 連結したものを返します。
          6. それ以外の場合は、value を返します。

          5.6 CBORタグ処理

          5.6.1 CBORタグ 構造取得アルゴリズム

          このアルゴリズムは、整数registryEntryIdを入力として受け取り、バイト文字列 prefixを返します。
          1. registryEntryBytesをregistryEntryIDのCBORエンコーディングに設定する。
          2. prefixを、registryEntryBytesをバイト文字列 0xD9CB1D82の末尾に追加した結果に設定する。
          3. prefixを返す。

          5.6.2 レジストリーエントリーID 取得アルゴリズム

          このアルゴリズムは、エンコードされたCBOR-LDペイロードcborldBytesを入力として受け取り、 デコードされる主データであるsuffix、およびsuffixを展開するために使用すべき registryEntryId値を返します。

          1. cborldbytes上のCBORタグが0xCB1D（タグ値51997）でない場合、 ERR_NON_CBOR_LD_TAGエラーを投げる。
          2. タグ付き項目が、最初の要素がCBOR整数である2要素の配列でない場合、 ERR_INVALID_PAYLOAD_STRUCTUREエラーを投げる。
          3. registryEntryIdの値を、タグ付き配列のインデックス0にある整数の値に設定する。
          4. suffixの値を、配列の2番目の要素の値に設定する。
          5. resultを空のマップに設定する。
          6. result.suffixをsuffixの値に、 result.registryEntryIdを registryEntryIdの値に設定する。
          7. resultを返す。
