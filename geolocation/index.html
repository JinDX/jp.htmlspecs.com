1. はじめに

このセクションは規範的ではありません。

ジオロケーションは、実装をホストするデバイスにのみ関連付けられた位置情報への高水準インターフェースを定義します。位置情報の一般的な取得元には、全地球測位システム（GPS）やIPアドレス、RFID、WiFiやBluetoothのMACアドレス、GSM/CDMAのセルID、またはユーザー入力などネットワーク信号から推定される位置が含まれます。API自体は基盤となる位置情報の取得元に依存しません。また、APIがデバイスの実際の位置を返す保証はありません。

エンドユーザーが権限を付与した場合、 ジオロケーションは以下を提供します：

  • 緯度、経度、高度、速度、進行方向などの位置データ、および取得した位置データの精度や取得時刻をGeolocationPositionインターフェース経由で提供します。
  • getCurrentPosition()メソッドによる「ワンショット」位置更新や、watchPosition()メソッドによるホストデバイスの位置が大きく変化した際の更新受信に対応します。
  • PositionOptionsのmaximumAgeを使うことで、指定した時間より新しいキャッシュされた位置（最後の位置のみキャッシュされる）を要求できます。
  • GeolocationPositionErrorとして、位置取得時に発生したエラーをアプリケーションが受け取る方法を提供します。
  • enableHighAccuracy経由で「高精度」位置データの要求をサポートしますが、リクエストはユーザーエージェントによって無視される場合があります。

1.1 対象範囲

このセクションは規範的ではありません。

この仕様は、ホストデバイスに関連付けられた地理的位置情報を取得するためのスクリプティングAPIの提供に限定されます。地理的位置情報は、世界測地系座標 [WGS84] で提供されます。マークアップ言語の提供や、地理的位置を識別するURLの新規スキーム定義は含みません。

2. 例

このセクションは規範的ではありません。

このAPIは「ワンショット」位置リクエストと繰り返しの位置更新の両方を可能にするよう設計されています。以下に一般的な利用例を示します。

2.1 現在位置の取得

このセクションは規範的ではありません。

ユーザーの現在位置をリクエストします。ユーザーが許可すれば、位置オブジェクトが返されます。

例 1: ワンショット位置リクエスト 
navigator.geolocation.getCurrentPosition(position => {
  const { latitude, longitude } = position.coords;
  // 緯度・経度中心の地図を表示します。
});

3.1 ユーザーの同意

このセクションは規範的ではありません。

ジオロケーションは強力な機能であり、位置情報をWebアプリケーションと共有する前に、エンドユーザーの明示的な許可が必要です。この要件は、権限の確認手順によって規範的に強制されており、getCurrentPosition()やwatchPosition()メソッドが依存しています。

エンドユーザーは通常、ユーザーインターフェースを通じて 明示的な許可 を与えますが、そこではユーザーが選択できる様々な許可 有効期間 が提示されることが一般的です。 有効期間 の選択肢はユーザーエージェントによって異なりますが、通常は時間ベース（例：「1日」）、またはブラウザを閉じるまで、あるいは許可を無期限に与える選択肢が提示される場合もあります。 許可の 有効期間 は、ユーザーエージェントが許可を 付与 する期間を決定し、その期間が終了すると許可は自動的にデフォルトの 許可状態 に戻され、エンドユーザーは次回使用時に新たな選択を求められます。

許可の有効期間の細かさはユーザーエージェントによって異なりますが、本仕様では、ユーザーエージェントがデフォルトで単一のブラウジングセッションに期間を限定することを推奨します（規範的な要件は3.4 API利用権限の確認を参照）。

7.1 enableHighAccuracy メンバー

enableHighAccuracy メンバーは、 アプリケーションができるだけ正確な位置情報を受け取りたいというヒントを実装に伝えます。 このメンバーの目的は、アプリケーションが高精度のジオロケーション修正を必要としないことを実装に知らせ、 実装が多くの電力を消費する位置情報プロバイダー（例：GPS）の使用を避けてもよいようにすることです。

注意: enableHighAccuracyに関する注意

enableHighAccuracyメンバーは応答速度の低下や消費電力の増加につながる場合があります。 ユーザーがこの機能を無効にしたり、デバイスがフラグ未指定時以上の精度を提供できない場合もあります。

7.2 timeout メンバー

timeoutメンバーは、位置取得が失効するまでの最大時間（ミリ秒単位）を表します。

注意: タイムアウトの計算タイミング

ドキュメントが可視になるまでや、API利用権限の取得にかかる時間は timeout メンバーの計算には含まれません。 timeout メンバーは位置取得が開始されてからのみ適用されます。

注意: 即時キャンセル

options.timeout の値が0の場合は即座に失敗することがあります。

D. 変更履歴

このセクションは非規範的です。

2022年にW3C勧告として公開されて以来、Geolocationには以下の規範的変更が加えられました。

  • 修正: エミュレートされたエラーコードのみが使用されることを明確化 (#187)
  • 追加: 位置情報のエミュレーションをサポート (#183)
  • 修正: 停止中はNaNではなくnullを使用 (#173)
  • 修正: データ型と処理方法を定義するための取得アルゴリズムの更新
  • 追加: 精度の単位を定義 (#162)
  • 修正: 非セキュアコンテキストのチェック (#157)
  • 追加: GeolocationCoordinatesとGeolocationPosに.toJSON()を公開
  • 修正: 緯度・経度の単位と測地系の参照を明確化

2021年の最初の公開作業草案から2022年のW3C勧告の公開までに、Geolocationには以下の規範的変更が加えられました。

  • "Queue a task"／"in parallel"の使い方を修正 (#118)
  • 権限の有効期間を提案 (#108)
  • OSレベルの権限変更への対応 (#109)
  • DOMTimeStampからEpochTimeStampへ変更 (#104)
  • watchPosition()エラー時に0を返す (#100)
  • ドキュメントが完全にアクティブでない場合にエラーでコールバック (#97)
  • 完全にアクティブでないドキュメントを適切に処理 (#90)
  • "geolocation task queue"の追加 (#87)

2016年の第二版公開以降、この仕様には以下の実質的な変更が加えられました。

  • Request a positionは、ドキュメントが可視である場合、または可視になった場合のみ進行するようになりました。
  • acquire a positionの一部としてキャッシュの仕組みを明確化：最後の位置のみがキャッシュされ、いつでも破棄可能です。
  • 権限の付与とUI要件の処理はPermissions仕様に依存するようになりました。
  • errorCallbackがnullable（null可能）になりました。
  • APIはPermissions Policyで制御可能となり、APIの公開範囲が制限されます。
  • callbacksは"EventHandler"オブジェクト（.handleEvent()メソッドを持つオブジェクト）としてではなく、IDLのコールバック関数としてのみ扱われるようになりました。
  • APIはセキュアコンテキスト（HTTPSのみ）でのみ公開されるようになりました。
  • インターフェイスはもはや[WebIDL]のレガシー[NoInterfaceObject]を使用せず、Geolocationおよび他のインターフェイスはグローバルスコープに存在します。また、インターフェイス名はNavigatorGeolocation*からGeolocation*に変更されました。

変更の全リストについてはコミット履歴を参照してください。
