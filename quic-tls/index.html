1. はじめに

この文書は、QUIC [QUIC-TRANSPORT] がTLS [TLS13]を使用してどのように保護されるかを説明します。¶

TLS 1.3は、以前のバージョンと比べて、接続確立における重要な レイテンシ改善を提供します。パケット損失がない場合、ほとんどの新しい接続は 1往復以内に確立され、保護できます。同じクライアントとサーバー間の 後続の接続では、クライアントは多くの場合、アプリケーションデータを即座に 送信できます。つまり、ゼロ往復セットアップを使用します。¶

この文書は、TLSがQUICのセキュリティコンポーネントとして どのように機能するかを説明します。¶

2. 表記規約

この文書におけるキーワード「MUST」、「MUST NOT」、 「REQUIRED」、「SHALL」、「SHALL NOT」、「SHOULD」、 「SHOULD NOT」、「RECOMMENDED」、「NOT RECOMMENDED」、「MAY」、および「OPTIONAL」は、 ここに示すようにすべて大文字で現れる場合に限り、BCP 14 [RFC2119] [RFC8174] に記述されているとおりに解釈されます。¶

この文書は、[QUIC-TRANSPORT]で確立された用語を使用します。¶

簡潔にするため、TLSという略語はTLS 1.3を指すために使用されますが、 より新しいバージョンも使用できます。セクション4.2を参照してください。¶

2.1. TLSの概要

TLSは、信頼できない媒体（たとえばインターネット）上で 通信手段を確立する方法を2つのエンドポイントに提供します。 TLSはピアの認証を可能にし、エンドポイントが交換するメッセージに 機密性と完全性の保護を提供します。¶

内部的に、TLSは階層化されたプロトコルであり、その構造は 図1に示されています。¶

各コンテンツ層メッセージ（例：ハンドシェイク、アラート、 アプリケーションデータ）は、レコード層によって型付きTLSレコードの列として 搬送されます。レコードは個別に暗号学的に保護され、その後、順序付けと 確実な配送を提供する信頼性のあるトランスポート（通常はTCP）上で送信されます。¶

TLSの認証済み鍵交換は、クライアントとサーバーという2つの エンドポイント間で行われます。クライアントが交換を開始し、サーバーが応答します。 鍵交換が正常に完了すると、クライアントとサーバーの両方が秘密を共有します。 TLSは、事前共有鍵（PSK）と、有限体または楕円曲線上の Diffie-Hellman（(EC)DHE）鍵交換の両方をサポートします。PSKはEarly Data（0-RTT）の基礎です。後者は、(EC)DHE鍵が破棄されたときに 前方秘匿性（FS）を提供します。2つのモードを組み合わせて、PSKを認証に 使用しつつ前方秘匿性を提供することもできます。¶

TLSハンドシェイクの完了後、クライアントはサーバーの識別情報を 学習し認証します。また、サーバーは任意でクライアントの識別情報を 学習し認証できます。TLSは、サーバーとクライアントの両方について、X.509 [RFC5280]証明書に基づく認証をサポートします。 （再開の場合のように）PSK鍵交換が使用される場合、PSKの知識が ピアの認証として機能します。¶

TLS鍵交換は攻撃者による改ざんに耐性があり、参加する いずれのピアにも制御できない共有秘密を生成します。¶

TLSは、QUICにとって関心のある2つの基本的なハンドシェイクモードを 提供します。¶

  • 完全な1-RTTハンドシェイク。クライアントは 1往復後にアプリケーションデータを送信でき、サーバーはクライアントから 最初のハンドシェイクメッセージを受信した直後に応答します。¶
  • 0-RTTハンドシェイク。クライアントは サーバーについて以前に学習した情報を使用して、アプリケーションデータを 即座に送信します。このアプリケーションデータは攻撃者によって再生される 可能性があるため、0-RTTは、再生された場合に望ましくない影響を引き起こす 可能性のある動作を開始する命令の搬送には適していません。¶

0-RTTアプリケーションデータを伴う単純化したTLSハンドシェイクを 図2に示します。¶

図2では、QUICで使用されない EndOfEarlyDataメッセージを省略しています。 セクション8.3を参照してください。同様に、ChangeCipherSpecも KeyUpdateメッセージもQUICでは使用されません。ChangeCipherSpecはTLS 1.3では 冗長です。セクション8.4を参照してください。 QUICには独自の鍵更新機構があります。セクション6を参照してください。¶

データはいくつかの暗号化レベルを使用して保護されます。¶

  • Initial鍵¶
  • Early data（0-RTT）鍵¶
  • Handshake鍵¶
  • Application data（1-RTT）鍵¶

アプリケーションデータは、early dataレベルとapplication data レベルにのみ現れます。ハンドシェイクおよびアラートメッセージは任意のレベルに現れ得ます。¶

0-RTTハンドシェイクは、クライアントとサーバーが以前に通信している場合に 使用できます。1-RTTハンドシェイクでは、クライアントは、サーバーから送信された すべてのハンドシェイクメッセージを受信するまで、保護されたアプリケーションデータを 送信できません。¶

3. プロトコルの概要

QUIC [QUIC-TRANSPORT]は、 パケットの機密性および完全性保護の責任を担います。そのために、TLS ハンドシェイク [TLS13]から導出された鍵を使用しますが、 （TCPの場合のように）TLSレコードをQUIC上で搬送するのではなく、TLSハンドシェイクおよび アラートメッセージをQUICトランスポート上で直接搬送します。QUICトランスポートは、 図3に示すように、TLSレコード層の責任を引き継ぎます。¶

QUICはまた、セキュリティと性能にとって重要なパラメータの 認証およびネゴシエーションをTLSに依存します。¶

厳密な階層化ではなく、これら2つのプロトコルは協調します。 QUICはTLSハンドシェイクを使用し、TLSはQUICが提供する信頼性、 順序付き配送、およびレコード層を使用します。¶

高いレベルでは、TLSコンポーネントとQUICコンポーネントの間には 2つの主な相互作用があります。¶

  • TLSコンポーネントはQUICコンポーネントを介して メッセージを送受信し、QUICはTLSに信頼性のあるストリーム抽象を提供します。¶
  • TLSコンポーネントはQUICコンポーネントに一連の更新を 提供します。これには、(a) インストールする新しいパケット保護鍵、および (b) ハンドシェイク完了、サーバー証明書などの状態変更が含まれます。¶

図4は、これらの相互作用をより詳細に示し、 QUICパケット保護を特に明示しています。¶

TCP上のTLSとは異なり、データを送信したいQUICアプリケーションは、 TLS Application Dataレコードを使用して送信しません。代わりに、QUIC STREAM フレームまたは他のフレームタイプとして送信し、それらはQUICパケットで搬送されます。¶

4. TLSメッセージの搬送

QUICはTLSハンドシェイクデータをCRYPTOフレームで搬送します。 各フレームは、オフセットと長さによって識別される連続したハンドシェイクデータの ブロックで構成されます。これらのフレームはQUICパケットにパッケージ化され、 現在の暗号化レベルで暗号化されます。TCP上のTLSと同様に、TLSハンドシェイク データがQUICに渡されると、それを信頼性をもって配送するのはQUICの責任です。 TLSによって生成される各データ片は、TLSが現在使用している鍵のセットに 関連付けられます。QUICがそのデータを再送する必要がある場合、TLSがすでに 新しい鍵に更新されていても、同じ鍵を使用MUSTします。¶

各暗号化レベルはパケット番号空間に対応します。使用される パケット番号空間によって、フレームの意味論が決まります。一部のフレームは、 異なるパケット番号空間では禁止されています。[QUIC-TRANSPORT]のセクション12.5を参照してください。¶

パケットはワイヤ上で並べ替えられる可能性があるため、QUICは 表1に示すように、特定のパケットを保護するために どの鍵が使用されたかを示すためにパケットタイプを使用します。異なるタイプの パケットを送信する必要がある場合、エンドポイントは、それらを同じUDPデータグラムで 送信するために結合パケットを使用SHOULDします。¶

4.1. TLSへのインターフェース

図4に示すように、QUICから TLSへのインターフェースは4つの主要な機能で構成されます。¶

  • ハンドシェイクメッセージの送受信¶
  • 再開されたセッションから保存済みの トランスポート状態およびアプリケーション状態を処理し、0-RTTデータを 生成または受け入れることが有効かどうかを判断すること¶
  • 鍵の再生成（送信と受信の両方）¶
  • ハンドシェイク状態の更新¶

TLSを構成するために追加の機能が必要になる場合があります。 特に、QUICとTLSは、証明書検証 [RFC5280]などの ピア資格情報の検証をどちらが担当するかについて合意する必要があります。¶

4.1.1. ハンドシェイク完了

この文書では、TLSスタックがハンドシェイク完了を 報告したときに、TLSハンドシェイクは完了したと見なされます。これは、 TLSスタックがFinishedメッセージを送信し、ピアのFinishedメッセージを 検証したときに発生します。ピアのFinishedメッセージを検証することで、 以前のハンドシェイクメッセージが変更されていないことの保証が エンドポイントに提供されます。ハンドシェイクは両方のエンドポイントで 同時に完了するわけではないことに注意してください。したがって、 ハンドシェイクの完了に基づく要件は、対象となるエンドポイントの視点に依存します。¶

4.1.2. ハンドシェイク 確認済み

この文書では、サーバーにおいてTLSハンドシェイクは、 ハンドシェイクが完了したときに確認済みと見なされます。サーバーは、 ハンドシェイクが完了したらただちにHANDSHAKE_DONEフレームを送信MUSTします。 クライアントでは、HANDSHAKE_DONEフレームを受信したときに、 ハンドシェイクは確認済みと見なされます。¶

さらに、クライアントは、1-RTTパケットに対する 確認応答を受信したときにハンドシェイクを確認済みと見なしてMAYいます。 これは、1-RTT鍵で送信された最小のパケット番号を記録し、 受信した1-RTT ACKフレームのLargest Acknowledgedフィールドと比較することで 実装できます。後者が前者以上になると、ハンドシェイクは確認済みです。¶

4.1.3. ハンドシェイク メッセージの送受信

ハンドシェイクを進行させるため、TLSはハンドシェイク メッセージを送受信できることに依存します。このインターフェースには2つの基本機能があります。 1つはQUICがハンドシェイクメッセージを要求する機能であり、もう1つはQUICが ハンドシェイクメッセージを構成するバイトを提供する機能です。¶

ハンドシェイクを開始する前に、QUICは搬送したい トランスポートパラメータ（セクション8.2を参照）をTLSに提供します。¶

QUICクライアントは、TLSにTLSハンドシェイクバイトを 要求することでTLSを開始します。クライアントは最初のパケットを送信する前に ハンドシェイクバイトを取得します。QUICサーバーは、クライアントの ハンドシェイクバイトをTLSに提供することで処理を開始します。¶

エンドポイントのTLSスタックはいつでも、現在の 送信用暗号化レベルと受信用暗号化レベルを持ちます。TLS暗号化レベルは、 データを保護するために使用されるQUICパケットタイプと鍵を決定します。¶

各暗号化レベルは異なるバイト列に関連付けられており、 CRYPTOフレームでピアに信頼性をもって送信されます。TLSが送信すべき ハンドシェイクバイトを提供すると、それらは現在の暗号化レベルの ハンドシェイクバイトに追加されます。その暗号化レベルは、結果として生成される CRYPTOフレームがどのタイプのパケットに入れられるかを決定します。 表1を参照してください。¶

4つの暗号化レベルが使用され、Initial、0-RTT、 Handshake、および1-RTTパケット用の鍵を生成します。CRYPTOフレームは、 0-RTTレベルを除いた3つのレベルだけで搬送されます。これら4つのレベルは、 3つのパケット番号空間に対応します。InitialおよびHandshake暗号化パケットは それぞれ独自の別個の空間を使用し、0-RTTおよび1-RTTパケットは application dataパケット番号空間を使用します。¶

QUICは、TLSハンドシェイクレコードの保護されていない 内容をCRYPTOフレームの内容として取り込みます。TLSレコード保護はQUICでは使用されません。 QUICはCRYPTOフレームをQUICパケットに組み立て、それらをQUICパケット保護を 使用して保護します。¶

QUIC CRYPTOフレームはTLSハンドシェイクメッセージだけを 搬送します。TLSアラートはQUIC CONNECTION_CLOSEエラーコードに変換されます。 セクション4.8を参照してください。 TLSアプリケーションデータおよびその他のコンテンツタイプは、どの暗号化レベルでも QUICによって搬送できません。それらがTLSスタックから受信された場合はエラーです。¶

エンドポイントがネットワークからCRYPTOフレームを含む QUICパケットを受信した場合、次のように処理します。¶

  • パケットが現在のTLS受信用暗号化レベルを 使用している場合、通常どおりデータを入力フローに順序付けます。STREAMフレームと同様に、 オフセットはデータ列内の適切な位置を見つけるために使用されます。この処理の結果として 新しいデータが利用可能になった場合、それは順番にTLSへ配送されます。¶
  • パケットが以前にインストールされた 暗号化レベルからのものである場合、そのフローで以前に受信したデータの末尾を 超えて延びるデータを含んではMUST NOTいけません。実装は、この要件への 違反をPROTOCOL_VIOLATION型の接続エラーとして扱わMUSTなければなりません。¶
  • パケットが新しい暗号化レベルからの ものである場合、TLSによる後続処理のために保存されます。TLSがこの暗号化レベルからの 受信へ移行すると、保存されたデータをTLSに提供できます。TLSがより高い暗号化レベルの 鍵を提供したとき、TLSがまだ消費していない前の暗号化レベルからのデータがある場合、 これはPROTOCOL_VIOLATION型の接続エラーとして扱わMUSTなければなりません。¶

TLSに新しいデータが提供されるたびに、新しい ハンドシェイクバイトがTLSから要求されます。受信したハンドシェイクメッセージが 不完全であるか、送信するデータがない場合、TLSはバイトを提供しないことがあります。¶

CRYPTOフレームの内容は、TLSによって逐次処理されるか、 完全なメッセージまたはフライトが利用可能になるまでバッファされることがあります。 TLSは、順番に到着したハンドシェイクバイトのバッファリングを担当します。QUICは、 順序外に到着した、またはまだ準備ができていない暗号化レベルのハンドシェイクバイトの バッファリングを担当します。QUICはCRYPTOフレームに対するフロー制御手段を提供しません。 [QUIC-TRANSPORT]のセクション7.5を参照してください。¶

TLSハンドシェイクが完了すると、TLSが送信する必要のある 最終的なハンドシェイクバイトとともに、これがQUICに示されます。この段階で、 ピアがハンドシェイク中に広告したトランスポートパラメータは認証されます。 セクション8.2を参照してください。¶

ハンドシェイクが完了すると、TLSは受動的になります。 TLSは依然としてピアからデータを受信し、対応して応答できますが、 アプリケーションまたはQUICから明示的に要求されない限り、それ以上データを 送信する必要はありません。データを送信する理由の1つは、サーバーが 追加または更新されたセッションチケットをクライアントに提供したい場合です。¶

ハンドシェイクが完了すると、QUICはCRYPTOストリームで 到着するデータだけをTLSに提供する必要があります。ハンドシェイク中に使用されるのと 同じ方法で、受信データを提供した後にTLSから新しいデータが要求されます。¶

4.1.4. 暗号化レベルの 変更

特定の暗号化レベルの鍵がTLSで利用可能になると、 TLSはその暗号化レベルの読み取り鍵または書き込み鍵が利用可能であることを QUICに示します。¶

新しい鍵が利用可能になることは、常にTLSに入力を 提供した結果です。TLSは、（クライアントによって）初期化された後、または 新しいハンドシェイクデータが提供されたときにのみ新しい鍵を提供します。¶

ただし、TLS実装は一部の処理を非同期に行う場合があります。 特に、証明書の検証処理には時間がかかることがあります。TLS処理の完了を 待っている間、エンドポイントは、まだ利用可能でない鍵を使用して処理される 可能性のある受信パケットをバッファSHOULDします。これらのパケットは、TLSによって鍵が 提供されると処理できます。エンドポイントは、この間に処理可能なパケットに 応答し続けるSHOULDです。¶

入力を処理した後、TLSはハンドシェイクバイト、 新しい暗号化レベルの鍵、またはその両方を生成することがあります。¶

新しい暗号化レベルが利用可能になると、TLSは QUICに3つの項目を提供します。¶

  • 秘密¶
  • 関連データ付き認証暗号 （AEAD）関数¶
  • 鍵導出関数（KDF）¶

これらの値はTLSがネゴシエートする値に基づいており、 QUICがパケット保護鍵およびヘッダー保護鍵を生成するために使用されます。 セクション5および セクション5.4を参照してください。¶

0-RTTが可能な場合、クライアントがTLS ClientHello メッセージを送信した後、またはサーバーがそのメッセージを受信した後に準備完了となります。 QUICクライアントに最初のハンドシェイクバイトを提供した後、TLSスタックは 0-RTT鍵への変更を通知することがあります。サーバー側では、ClientHello メッセージを含むハンドシェイクバイトを受信した後、TLSサーバーは0-RTT鍵が 利用可能であることを通知することがあります。¶

TLSは一度に1つの暗号化レベルのみを使用しますが、 QUICは複数のレベルを使用することがあります。たとえば、Finishedメッセージ （Handshake暗号化レベルのCRYPTOフレームを使用）を送信した後、 エンドポイントはSTREAMデータ（1-RTT暗号化）を送信できます。Finished メッセージが失われた場合、エンドポイントは失われたメッセージを再送するために Handshake暗号化レベルを使用します。パケットの並べ替えまたは損失により、 QUICは複数の暗号化レベルのパケットを処理する必要が生じることがあります。 ハンドシェイク中、これはTLSが現在使用している暗号化レベルより高いレベルおよび 低いレベルのパケットを処理する可能性があることを意味します。¶

特に、サーバー実装は、0-RTT暗号化レベルと同時に Handshake暗号化レベルのパケットを読み取れる必要があります。クライアントは、 Handshake鍵で保護されたACKフレームを0-RTTデータと交互に送信する可能性があり、 サーバーは失われたHandshakeパケットを検出するために、それらの確認応答を 処理する必要があります。¶

QUICはまた、TLS実装では通常利用できない可能性がある 鍵にもアクセスする必要があります。たとえば、クライアントは、その暗号化レベルで CRYPTOフレームを送信する準備が整う前に、Handshakeパケットを確認応答する 必要があるかもしれません。したがって、TLSは自らの使用のために鍵を生成する前に、 QUICに鍵を提供する必要があります。¶

4.1.5. TLSインターフェース 概要

図5は、 クライアントとサーバーの両方について、QUICとTLSの間の交換を要約します。 実線の矢印はハンドシェイクデータを搬送するパケットを示し、破線の矢印は アプリケーションデータを送信できる場所を示します。各矢印には、その送信に 使用される暗号化レベルが付けられています。¶

図5は、 単一のメッセージ「フライト」を形成する複数のパケットが個別に処理される 様子を示し、どの受信メッセージが異なる動作を引き起こすかを示しています。 これは、異なる暗号化レベルのハンドシェイクメッセージを取得するための 複数の「Get Handshake」呼び出しを示します。新しいハンドシェイクメッセージは、 受信パケットが処理された後に要求されます。¶

図5は、 単純なハンドシェイク交換の1つの可能な構造を示しています。 正確な処理は、エンドポイント実装の構造およびパケットの到着順序によって 異なります。実装は、異なる数の操作を使用したり、それらを別の順序で 実行したりすることがあります。¶

4.2. TLSバージョン

この文書は、TLS 1.3 [TLS13]がQUICでどのように使用されるかを説明します。¶

実際には、TLSハンドシェイクは使用するTLSのバージョンを ネゴシエートします。両方のエンドポイントがそのバージョンをサポートしている場合、 TLS 1.3より新しいTLSバージョンがネゴシエートされる可能性があります。QUICが 使用するTLS 1.3の機能がより新しいバージョンでサポートされている限り、これは許容されます。¶

クライアントは、1.3より古いTLSバージョンを提示しては MUST NOTいけません。不適切に構成されたTLS実装は、TLS 1.2または それより古い別のTLSバージョンをネゴシエートする可能性があります。エンドポイントは、 1.3より古いTLSバージョンがネゴシエートされた場合、接続を終了MUSTしなければなりません。¶

4.3. ClientHelloサイズ

クライアントからの最初のInitialパケットには、最初の 暗号学的ハンドシェイクメッセージの開始部分または全体が含まれます。TLSではこれは ClientHelloです。サーバーは、新しい着信QUIC接続を受け入れるかどうかを 判断するために、ClientHello全体を解析する必要がある場合があります（たとえば、 Server Name Identification（SNI）やApplication-Layer Protocol Negotiation （ALPN）などの拡張にアクセスするため）。ClientHelloが複数のInitialパケットに またがる場合、そのようなサーバーは最初に受信した断片をバッファする必要があり、 クライアントのアドレスがまだ検証されていない場合、過剰なリソースを消費する可能性があります。 これを避けるため、サーバーはRetry機能（[QUIC-TRANSPORT]のセクション8.1を参照）を使用して、 検証済みアドレスを持つクライアントからの部分的なClientHelloメッセージだけを バッファしてMAYいます。¶

QUICパケットおよびフレーミングは、ClientHelloメッセージに 少なくとも36バイトのオーバーヘッドを追加します。クライアントがSource Connection IDフィールドをゼロバイトより長く選択した場合、このオーバーヘッドは 増加します。オーバーヘッドには、トークンまたは8バイトを超えるDestination Connection IDも含まれていません。これらはいずれも、サーバーがRetryパケットを 送信する場合に必要になることがあります。¶

典型的なTLS ClientHelloは、1200バイトのパケットに容易に 収まります。しかし、QUICによって追加されるオーバーヘッドに加え、この制限を 超えさせる可能性のある変数がいくつかあります。大きなセッションチケット、 複数または大きな鍵共有、サポートされる暗号、署名アルゴリズム、バージョン、 QUICトランスポートパラメータ、およびその他のネゴシエート可能なパラメータと 拡張の長い一覧により、このメッセージが大きくなる可能性があります。¶

サーバーにとって、接続IDおよびトークンに加え、TLSセッション チケットのサイズは、クライアントが効率的に接続できる能力に影響を与えることがあります。 これらの値のサイズを最小限に抑えることで、クライアントがそれらを使用しながら ClientHelloメッセージ全体を最初のInitialパケットに収められる可能性が高まります。¶

TLS実装は、ClientHelloがInitialパケットを搬送する データグラムに対するQUICの要件を満たすほど大きいことを保証する必要はありません。 [QUIC-TRANSPORT]のセクション 14.1を参照してください。QUIC実装は、PADDINGフレームまたは パケット結合を使用して、データグラムが十分な大きさになるようにします。¶

4.4. ピア認証

認証の要件は、使用されているアプリケーションプロトコルに 依存します。TLSはサーバー認証を提供し、サーバーがクライアント認証を 要求することを許可します。¶

クライアントはサーバーの識別情報を認証MUSTしなければなりません。 これは通常、サーバーの識別情報が証明書に含まれていること、およびその証明書が 信頼された主体によって発行されていることの検証を伴います（例として [RFC2818]を参照）。¶

注：サーバーが認証のために証明書を提供する場合、 証明書チェーンのサイズが大量のバイトを消費する可能性があります。QUICでは、 クライアントアドレスを検証する前にサーバーが受信1バイトごとに3バイトまでしか 送信できないため、証明書チェーンのサイズを制御することは性能にとって重要です。 [QUIC-TRANSPORT]のセクション8.1を参照してください。 証明書チェーンのサイズは、名前または拡張の数を制限すること、ECDSAのような 小さい公開鍵表現を持つ鍵を使用すること、または証明書圧縮 [COMPRESS]を使用することで管理できます。¶

サーバーは、ハンドシェイク中にクライアントが認証することを 要求してMAYいます。サーバーは、要求されたときにクライアントが認証できない場合、 接続を拒否してMAYいます。 クライアント認証の要件は、アプリケーションプロトコルおよび展開によって異なります。¶

サーバーは、QUICが提供する多重化により、クライアントが 証明書要求をそれを引き起こしたアプリケーションレベルのイベントと関連付けられないため、 （[TLS13]のセクション 4.6.2で定義される）ハンドシェイク後クライアント認証を使用しては MUST NOTいけません（[HTTP2-TLS13]を参照）。より具体的には、 サーバーはハンドシェイク後TLS CertificateRequestメッセージを送信しては MUST NOTいけません。また、クライアントはそのようなメッセージの受信を PROTOCOL_VIOLATION型の接続エラーとして扱わMUSTなければなりません。¶

4.5. セッション再開

QUICはTLS 1.3のセッション再開機能を使用できます。 これは、ハンドシェイク完了後にCRYPTOフレームでNewSessionTicketメッセージを 搬送することで行われます。セッション再開は0-RTTを提供するために使用でき、 0-RTTが無効な場合にも使用できます。¶

セッション再開を使用するエンドポイントは、再開接続を 作成するときに、現在の接続に関する一部の情報を記憶する必要がある場合があります。 TLSは一部の情報を保持することを要求します。[TLS13]のセクション 4.6.1を参照してください。QUIC自体は、0-RTTも使用される場合を除き、 接続再開時に状態が保持されることに依存しません。[QUIC-TRANSPORT]のセクション7.4.1および セクション4.6.1を参照してください。QUICを使用する アプリケーションプロトコルは、再開された接続間で保持される状態に依存することがあります。¶

クライアントは、再開に必要な任意の状態をセッションチケットと ともに保存できます。サーバーは、状態を搬送するためにセッションチケットを 使用できます。¶

セッション再開により、サーバーは元の接続での活動を 再開された接続と関連付けることができ、これはクライアントにとって プライバシー上の問題となる可能性があります。クライアントは、この相関を避けるために 再開を有効にしないことを選択できます。クライアントは、サーバー以外の主体が 接続を相関できるようになるため、チケットを再利用すべきではSHOULD NOTありません。 [TLS13]の付録 C.4を参照してください。¶

4.6. 0-RTT

QUICの0-RTT機能により、クライアントはハンドシェイクが 完了する前にアプリケーションデータを送信できます。これは、以前の接続から ネゴシエート済みパラメータを再利用することで可能になります。これを有効にするため、 0-RTTは、クライアントが重要なパラメータを記憶し、サーバーが同じ情報を 回復できるTLSセッションチケットをサーバーに提供することに依存します。¶

この情報には、[TLS13]によって 管理されるTLS状態、QUICトランスポートパラメータ、選択されたアプリケーションプロトコル、 およびアプリケーションプロトコルが必要とする可能性のある任意の情報を決定する パラメータが含まれます。セクション4.6.3を参照してください。 この情報は、0-RTTパケットとその内容がどのように形成されるかを決定します。¶

同じ情報が両方のエンドポイントで利用可能であることを 確実にするため、0-RTTを確立するために使用されるすべての情報は同じ接続から 取得されます。エンドポイントは、0-RTTの送信または処理を変更する可能性のある 情報を選択的に無視することはできません。¶

[TLS13]は、元の接続から 0-RTTを使用しようとするまでの時間に7日間の制限を設けています。0-RTTの使用には、 他にも制約があり、特にリプレイ攻撃への潜在的な露出によるものがあります。 セクション9.2を参照してください。¶

4.6.1. 0-RTTの有効化

NewSessionTicketメッセージ内のTLS early_data拡張は、 （max_early_data_sizeパラメータ内で）サーバーが受け入れる意思のあるTLS 0-RTTデータ量を伝えるように定義されています。QUICはTLS early dataを使用しません。 QUICはearly dataを搬送するために0-RTTパケットを使用します。したがって、 max_early_data_sizeパラメータは、サーバーがQUIC 0-RTTデータを受け入れる 意思があることを示す番兵値0xffffffffを保持するように転用されます。 サーバーが0-RTTデータを受け入れないことを示すには、early_data拡張は NewSessionTicketから省略されます。クライアントがQUIC 0-RTTで送信できる データ量は、サーバーによって提供されるinitial_max_dataトランスポートパラメータによって 制御されます。¶

サーバーは、max_early_data_sizeフィールドが 0xffffffff以外の値に設定されたearly_data拡張を送信してはMUST NOTいけません。 クライアントは、early_data拡張に他の値を含むNewSessionTicketを受信した場合、 それをPROTOCOL_VIOLATION型の接続エラーとして扱わMUSTなければなりません。¶

0-RTTパケットを送信したいクライアントは、 後続のハンドシェイクのClientHelloメッセージでearly_data拡張を使用します。 [TLS13]のセクション4.2.10を参照してください。 その後、0-RTTパケットでアプリケーションデータを送信します。¶

0-RTTを試みるクライアントは、サーバーがNEW_TOKEN フレームを送信していた場合、アドレス検証トークンも提供することがあります。 [QUIC-TRANSPORT]のセクション 8.1を参照してください。¶

4.6.2. 0-RTTの 受け入れと拒否

サーバーは、EncryptedExtensionsでearly_data拡張を 送信することで0-RTTを受け入れます。[TLS13]のセクション4.2.10を参照してください。 その後、サーバーは受信した0-RTTパケットを処理し確認応答します。¶

サーバーは、early_data拡張なしでEncryptedExtensionsを 送信することで0-RTTを拒否します。サーバーがTLS HelloRetryRequestを送信する場合、 常に0-RTTを拒否します。0-RTTを拒否する場合、サーバーは、たとえ処理できたとしても、 いかなる0-RTTパケットも処理してはMUST NOTいけません。0-RTTが拒否された場合、 クライアントは、その状態を検出できるなら、0-RTTパケットに対する確認応答の受信を PROTOCOL_VIOLATION型の接続エラーとして扱うSHOULDです。¶

0-RTTが拒否された場合、クライアントが想定していた すべての接続特性が不正確である可能性があります。これには、アプリケーションプロトコル、 トランスポートパラメータ、および任意のアプリケーション構成の選択が含まれます。 したがって、クライアントは、それらのストリームに結び付いたアプリケーション状態を含め、 すべてのストリームの状態をリセットMUSTしなければなりません。¶

クライアントは、RetryまたはVersion Negotiation パケットを受信した場合、0-RTTを再試行してMAYいます。これらのパケットは 0-RTTの拒否を意味しません。¶

4.6.3. 0-RTT設定の 検証

サーバーがearly_data拡張を含むClientHelloを受信した場合、 クライアントからの0-RTTデータを受け入れるか拒否するかを判断する必要があります。 この判断の一部はTLSスタックによって行われます（たとえば、再開される暗号スイートが ClientHelloに含まれていることの確認。[TLS13]のセクション4.2.10を参照）。 TLSスタックに0-RTTデータを拒否する理由がない場合でも、QUICスタックまたは QUICを使用するアプリケーションプロトコルは、再開セッションに関連付けられた トランスポートまたはアプリケーションの構成がサーバーの現在の構成と互換性がないため、 0-RTTデータを拒否することがあります。¶

QUICは、追加のトランスポート状態が0-RTTセッション チケットに関連付けられることを要求します。これを実装する一般的な方法の1つは、 ステートレスセッションチケットを使用し、この状態をセッションチケットに保存することです。 QUICを使用するアプリケーションプロトコルには、状態の関連付けまたは保存に関して 同様の要件がある場合があります。この関連付けられた状態は、0-RTTデータを拒否する 必要があるかどうかを判断するために使用されます。たとえば、HTTP/3設定 [QUIC-HTTP]は、クライアントからの 0-RTTデータがどのように解釈されるかを決定します。QUICを使用する他の アプリケーションは、0-RTTデータを受け入れるか拒否するかを判断するための 異なる要件を持つことがあります。¶

4.7. HelloRetryRequest

HelloRetryRequestメッセージ（[TLS13]のセクション 4.1.4を参照）は、クライアントに新しい情報、 たとえば鍵共有の提供を要求したり、クライアントの何らかの特性を検証したりするために 使用できます。QUICの観点からは、HelloRetryRequestはInitialパケットで搬送される 他の暗号学的ハンドシェイクメッセージと区別されません。原理的には、この機能を アドレス検証に使用することも可能ですが、QUIC実装は代わりにRetry機能を使用する SHOULDです。[QUIC-TRANSPORT]のセクション8.1を 参照してください。¶

4.8. TLSエラー

TLSでエラーが発生した場合、[TLS13]のセクション6で 定義されている適切なアラートが生成されます。¶

TLSアラートはQUIC接続エラーに変換されます。AlertDescription 値に0x0100が加算され、CRYPTO_ERROR用に予約された範囲のQUICエラーコードが 生成されます。[QUIC-TRANSPORT]のセクション20.1を参照してください。結果の値は、 0x1c型のQUIC CONNECTION_CLOSEフレームで送信されます。¶

QUICは「fatal」アラートレベルだけを伝達できます。 TLS 1.3では、「warning」レベルの既存の用途は接続終了を通知することだけです。 [TLS13]のセクション 6.1を参照してください。QUICは接続終了のための代替メカニズムを提供し、 TLS接続はエラーが発生した場合にのみ閉じられるため、QUICエンドポイントはTLSからの 任意のアラートを「fatal」レベルであるかのように扱わMUSTなければなりません。¶

QUICは、特定のエラーコードの代わりに汎用コードを使用することを 許可します。[QUIC-TRANSPORT]のセクション11を 参照してください。TLSアラートの場合、これには任意のアラートを handshake_failure（QUICでは0x0128）のような汎用アラートに置き換えることが 含まれます。エンドポイントは、機密情報を露出する可能性を避けるために 汎用エラーコードを使用してMAYいます。¶

4.9. 未使用鍵の破棄

QUICが新しい暗号化レベルへの移行を完了した後、 以前の暗号化レベルのパケット保護鍵は破棄できます。これは、ハンドシェイク中に 数回発生し、鍵が更新される場合にも発生します。セクション6を参照してください。¶

パケット保護鍵は、新しい鍵が利用可能になったときに ただちに破棄されるわけではありません。下位の暗号化レベルのパケットにCRYPTOフレームが 含まれている場合、そのデータを再送するフレームは同じ暗号化レベルで送信され MUSTます。 同様に、エンドポイントは、確認応答されるパケットと同じ暗号化レベルで パケットの確認応答を生成します。したがって、新しい暗号化レベルの鍵が利用可能になった後も、 下位の暗号化レベルの鍵が短時間必要になる可能性があります。¶

エンドポイントは、ピアからその暗号化レベルのすべての 暗号学的ハンドシェイクメッセージを受信し、ピアも同じことを完了していない限り、 特定の暗号化レベルの鍵を破棄できません。これを判断するための異なる方法が、 Initial鍵（セクション4.9.1）および Handshake鍵（セクション4.9.2）について 提供されています。これらの方法は、ピアが必要なすべての確認応答を受信していない 可能性があるため、その暗号化レベルでパケットが受信または送信されることを 防ぐものではありません。¶

エンドポイントが古い鍵を保持する場合でも、新しいデータは 現在利用可能な最も高い暗号化レベルで送信されMUSTます。ACKフレームおよび CRYPTOフレーム内のデータの再送だけが以前の暗号化レベルで送信されます。 これらのパケットにはPADDINGフレームも含めてMAYいます。¶

4.9.1. Initial鍵の 破棄

Initialシークレット（セクション5.2）で保護されたパケットは 認証されていません。つまり、攻撃者が接続を妨害する意図でパケットを偽装する 可能性があります。これらの攻撃を制限するため、Initialパケット保護鍵は 他の鍵よりも積極的に破棄されます。¶

Handshakeパケットが正常に使用されることは、 これ以上Initialパケットを交換する必要がないことを示します。これらの鍵は、 InitialパケットからすべてのCRYPTOフレームを受信した後にのみ生成できるためです。 したがって、クライアントは最初にHandshakeパケットを送信したときにInitial鍵を 破棄MUSTし、サーバーはHandshakeパケットを最初に正常処理したときに Initial鍵を破棄MUSTしなければなりません。エンドポイントはこの時点以降、 Initialパケットを送信してはMUST NOTいけません。¶

これにより、Initial暗号化レベルの損失回復状態を 放棄し、未解決のInitialパケットを無視することになります。¶

4.9.2. Handshake鍵の 破棄

エンドポイントは、TLSハンドシェイクが確認済み （セクション4.1.2）になったときに、 そのHandshake鍵を破棄MUSTしなければなりません。¶

4.9.3. 0-RTT鍵の 破棄

0-RTTパケットと1-RTTパケットは同じパケット番号空間を 共有し、クライアントは1-RTTパケットを送信した後に0-RTTパケットを送信しません （セクション5.6）。¶

したがって、クライアントは、1-RTT鍵をインストールしたら、 その時点以降は用途がないため、0-RTT鍵をただちに破棄するSHOULDです。¶

さらに、サーバーは1-RTTパケットを受信したらすぐに 0-RTT鍵を破棄してMAYいます。ただし、パケットの並べ替えにより、0-RTTパケットが 1-RTTパケットの後に到着する可能性があります。サーバーは、1-RTT鍵でその内容を 再送する必要なしに、並べ替えられたパケットを復号できるようにするため、 0-RTT鍵を一時的に保持してMAYいます。1-RTTパケットを受信した後、 サーバーは短時間のうちに0-RTT鍵を破棄MUSTしなければなりません。 RECOMMENDEDされる期間は、Probe Timeout （PTO、[QUIC-RECOVERY]を参照）の3倍です。サーバーは、 すべての0-RTTパケットを受信したと判断した場合、より早く0-RTT鍵を破棄して MAYいます。これは欠落しているパケット番号を追跡することで実行できます。¶

5. パケット保護

TCP上のTLSと同様に、QUICはTLSによってネゴシエートされた AEADアルゴリズム [AEAD] を使用し、TLS ハンドシェイクから導出された鍵でパケットを保護します。¶

QUICパケットは、そのタイプに応じて異なる保護を持ちます。¶

  • Version Negotiationパケットには暗号学的保護がありません。¶
  • RetryパケットはAEAD_AES_128_GCMを使用して、偶発的な 変更に対する保護を提供し、有効なRetryを生成できる主体を制限します。 セクション5.8を参照してください。¶
  • Initialパケットは、クライアントが送信した最初のInitial パケットのDestination Connection IDフィールドから導出された鍵で AEAD_AES_128_GCMを使用します。セクション5.2を 参照してください。¶
  • その他すべてのパケットは、TLSによってネゴシエートされた 鍵とアルゴリズムを使用して、機密性と完全性のための強力な暗号学的保護を持ちます。¶

このセクションでは、Handshakeパケット、0-RTTパケット、および 1-RTTパケットにパケット保護を適用する方法を説明します。同じパケット保護処理は Initialパケットにも適用されます。ただし、Initialパケットに使用される鍵を 判別することは容易であるため、これらのパケットは機密性または完全性の保護を 持つとは見なされません。Retryパケットは固定鍵を使用するため、同様に 機密性および完全性の保護を欠きます。¶

5.1. パケット保護鍵

QUICは、TLSがレコード保護鍵を導出するのと同じ方法で パケット保護鍵を導出します。¶

各暗号化レベルには、各方向に送信されるパケットを保護するための 個別の秘密値があります。これらのトラフィックシークレットはTLSによって導出され （セクション7.1 of [TLS13]を参照）、Initial 暗号化レベルを除くすべての暗号化レベルでQUICによって使用されます。 Initial暗号化レベルの秘密は、セクション5.2で 説明されているように、クライアントの初期Destination Connection IDに 基づいて計算されます。¶

パケット保護に使用される鍵は、TLSが提供するKDFを使用して TLSシークレットから計算されます。TLS 1.3では、 セクション 7.1 of [TLS13]で説明される HKDF-Expand-Label関数が、ネゴシエートされた暗号スイートのハッシュ関数とともに 使用されます。QUICにおけるHKDF-Expand-Labelのすべての使用では、長さ0のContextを 使用します。¶

文字列を使用して記述されるラベルは、引用符や末尾のNUL バイトなしで、ASCII [ASCII]を使用して バイトとしてエンコードされることに注意してください。¶

TLSの他のバージョンは、QUICで使用されるために同様の 関数を提供MUSTします。¶

現在の暗号化レベルの秘密とラベル「quic key」は、AEAD鍵を 生成するためにKDFへの入力となります。ラベル「quic iv」はInitialization Vector（IV）を導出するために使用されます。セクション5.3を 参照してください。ヘッダー保護鍵は「quic hp」ラベルを使用します。 セクション5.4を参照してください。これらのラベルを 使用することで、QUICとTLSの間で鍵の分離が提供されます。セクション9.6を 参照してください。¶

「quic key」と「quic hp」はどちらも鍵を生成するために 使用されるため、これらのラベルとともにHKDF-Expand-Labelへ提供されるLengthは、 AEADまたはヘッダー保護アルゴリズムにおける鍵のサイズによって決定されます。 「quic iv」とともに提供されるLengthは、AEAD nonceの最小長、またはそれより大きい 場合は8バイトです。[AEAD]を 参照してください。¶

Initialシークレットに使用されるKDFは、常にTLS 1.3の HKDF-Expand-Label関数です。セクション5.2を 参照してください。¶

5.2. Initialシークレット

Initialパケットはパケット保護処理を適用しますが、クライアントの 最初のInitialパケットのDestination Connection IDフィールドから導出された秘密を 使用します。¶

この秘密は、0x38762cf7f55934b34d179ae6a4c80cadccbb7f0aを saltとし、Destination Connection IDフィールドを入力鍵材料（IKM）として、 HKDF-Extract（セクション2.2 of [HKDF]を参照）を使用することで 決定されます。これにより、送信と受信のための2つの別個の秘密を導出するために 使用される中間疑似ランダム鍵（PRK）が生成されます。¶

クライアントがInitialパケットを構築するために使用する秘密は、 PRKとラベル「client in」をTLS [TLS13]のHKDF-Expand-Label関数への入力として 使用し、32バイトの秘密を生成します。サーバーによって構築されるパケットは、 ラベル「server in」で同じ処理を使用します。Initialシークレットおよび鍵を 導出するときのHKDFのハッシュ関数はSHA-256 [SHA]です。¶

この処理を疑似コードで示すと次のようになります。¶ 

initial_salt = 0x38762cf7f55934b34d179ae6a4c80cadccbb7f0a
initial_secret = HKDF-Extract(initial_salt,
                              client_dst_connection_id)

client_initial_secret = HKDF-Expand-Label(initial_secret,
                                          "client in", "",
                                          Hash.length)
server_initial_secret = HKDF-Expand-Label(initial_secret,
                                          "server in", "",
                                          Hash.length)
¶

HKDF-Expand-Labelで使用される接続IDは、クライアントが送信した Initialパケット内のDestination Connection IDです。これは、クライアントがRetry パケットを受信した後にInitialパケットを作成する場合を除き、ランダムに選択された 値になります。その場合、Destination Connection IDはサーバーによって選択されます。¶

QUICの将来のバージョンは、新しいsalt値を生成する SHOULDです。これにより、QUICの各バージョンで鍵が異なることが 保証されます。これは、QUICの1つのバージョンだけを認識するミドルボックスが、 将来のバージョンのパケット内容を閲覧または変更することを防ぎます。¶

TLS 1.3で定義されたHKDF-Expand-Label関数は、提示されたTLS バージョンにTLS 1.3が含まれていない場合でも、Initialパケットに使用 MUSTされます。¶

後続のInitialパケットを構築するために使用される秘密は、 サーバーが選択した接続ID値を使用するためにRetryパケットを送信したときに 変化します。クライアントがサーバーからのInitialパケットに応答して使用する Destination Connection IDを変更しても、秘密は変化しません。¶

注：Destination Connection IDフィールドは、サーバーが 長さ0のSource Connection IDフィールドを持つRetryパケットを送信する場合の 長さ0を含め、最大20バイトまでの任意の長さにできます。Retry後、Initial鍵は、 サーバーがそのパケットを受信したという保証をクライアントに提供しないため、 クライアントはサーバーアドレスを検証するためにRetryパケットを含む交換に 依存する必要があります。セクション8.1 of [QUIC-TRANSPORT]を参照してください。¶

付録AにはInitialパケットの サンプルが含まれています。¶

5.3. AEADの使用

QUICパケット保護に使用されるAuthenticated Encryption with Associated Data（AEAD）関数（[AEAD]を 参照）は、TLS接続で使用するためにネゴシエートされたAEADです。たとえば、 TLSがTLS_AES_128_GCM_SHA256暗号スイートを使用している場合、 AEAD_AES_128_GCM関数が使用されます。¶

QUICは、TLS_AES_128_CCM_8_SHA256を除き、 [TLS13]で定義される任意の暗号スイートを 使用できます。暗号スイートに対するヘッダー保護方式が定義されていない限り、 暗号スイートをネゴシエートしてはMUST NOTいけません。この文書は、 TLS_AES_128_CCM_8_SHA256を除く[TLS13]で定義されたすべての暗号スイートについて、 ヘッダー保護方式を定義します。これらの暗号スイートは16バイトの認証タグを 持ち、入力より16バイト大きい出力を生成します。¶

エンドポイントは、自身がサポートしていない暗号スイートを 提示するClientHelloを拒否してはMUST NOTいけません。そうでなければ、 新しい暗号スイートを展開できなくなります。これはTLS_AES_128_CCM_8_SHA256にも 適用されます。¶

パケットを構築するとき、AEAD関数はヘッダー保護を適用する前に 適用されます。セクション5.4を参照してください。 保護されていないパケットヘッダーは関連データ（A）の一部です。パケットを処理するとき、 エンドポイントは最初にヘッダー保護を解除します。¶

パケットの鍵とIVは、セクション5.1で説明されるように計算されます。 nonce Nは、パケット保護IVとパケット番号を組み合わせることで形成されます。 再構築されたQUICパケット番号の62ビットは、ネットワークバイト順で、IVのサイズまで ゼロで左詰めされます。パディングされたパケット番号とIVの排他的論理和が AEAD nonceを形成します。¶

AEADの関連データAは、短いヘッダーまたは長いヘッダーの 最初のバイトから始まり、保護されていないパケット番号を含むところまでの QUICヘッダーの内容です。¶

AEADへの入力平文Pは、[QUIC-TRANSPORT]で 説明されるQUICパケットのペイロードです。¶

AEADの出力暗号文Cは、Pの代わりに送信されます。¶

一部のAEAD関数には、同じ鍵とIVの下で暗号化できるパケット数に 制限があります。セクション6.6を参照してください。 これはパケット番号の上限より低い場合があります。エンドポイントは、使用中のAEADに 設定された任意の制限を超える前に、鍵更新（セクション6）を開始MUSTします。¶

5.4. ヘッダー保護

QUICパケットヘッダーの一部、特にPacket Numberフィールドは、 パケット保護鍵およびIVとは別に導出された鍵を使用して保護されます。 「quic hp」ラベルを使用して導出された鍵は、経路上の要素に公開されない フィールドに対して機密性保護を提供するために使用されます。¶

この保護は、最初のバイトの下位ビットとPacket Numberフィールドに 適用されます。長いヘッダーを持つパケットでは最初のバイトの下位4ビットが 保護され、短いヘッダーを持つパケットでは最初のバイトの下位5ビットが保護されます。 どちらのヘッダー形式でも、これは予約ビットとPacket Number Lengthフィールドを 覆います。短いヘッダーのパケットではKey Phaseビットも保護されます。¶

同じヘッダー保護鍵が接続期間中使用され、鍵更新後も値は 変化しません（セクション6を参照）。これにより、 ヘッダー保護を使用してKey Phaseを保護できます。¶

この処理はRetryパケットまたはVersion Negotiationパケットには 適用されません。これらは保護されたペイロードや、この処理によって保護される フィールドを含まないためです。¶

5.4.1. ヘッダー保護の 適用

ヘッダー保護は、パケット保護が適用された後に適用されます （セクション5.3を参照）。 パケットの暗号文がサンプリングされ、暗号化アルゴリズムへの入力として使用されます。 使用されるアルゴリズムはネゴシエートされたAEADに依存します。¶

このアルゴリズムの出力は5バイトのマスクであり、排他的論理和を 使用して保護対象のヘッダーフィールドに適用されます。パケットの最初のバイトの 下位ビットは最初のマスクバイトの下位ビットでマスクされ、パケット番号は 残りのバイトでマスクされます。短いパケット番号エンコーディングにより生じる可能性のある 未使用のマスクバイトは使用されません。¶

図6は、ヘッダー保護を 適用するためのサンプルアルゴリズムを示しています。ヘッダー保護の解除は、 パケット番号長（pn_length）を決定する順序だけが異なります（ここでは「^」が 排他的論理和を表すために使用されています）。¶

具体的なヘッダー保護関数は、選択された暗号スイートに 基づいて定義されます。セクション5.4.3および セクション5.4.4を参照してください。¶

図7は、長いヘッダーの パケット（Initial）と短いヘッダーのパケット（1-RTT）の例を示します。 図7は、各ヘッダー内でヘッダー保護の対象と なるフィールド、およびサンプリングされる保護済みパケットペイロードの部分を 示します。¶

TLS暗号スイートをQUICで使用できるようにする前に、 その暗号スイートで使用されるAEADに対するヘッダー保護アルゴリズムが指定され MUSTます。この文書は、AEAD_AES_128_GCM、AEAD_AES_128_CCM、 AEAD_AES_256_GCM（これらのAES AEADはすべて[AEAD]で 定義されています）、およびAEAD_CHACHA20_POLY1305 （[CHACHA]で定義）に対する アルゴリズムを定義します。TLSが暗号スイートを選択する前は、AES ヘッダー保護が使用されます（セクション5.4.3）。 これはAEAD_AES_128_GCMパケット保護に対応します。¶

5.4.2. ヘッダー保護 サンプル

ヘッダー保護アルゴリズムは、ヘッダー保護鍵と、パケットの Payloadフィールドからの暗号文サンプルの両方を使用します。¶

常に同じバイト数がサンプリングされますが、受信エンドポイントによる 保護の解除を考慮する必要があります。受信エンドポイントはPacket Numberフィールドの 長さを知らないためです。暗号文のサンプルは、Packet Numberフィールドの開始から 4バイト後のオフセットから取得されます。つまり、ヘッダー保護のために パケット暗号文をサンプリングする際、Packet Numberフィールドは4バイト長 （可能な最大エンコード長）であると仮定されます。¶

エンドポイントは、完全なサンプルを含むのに十分な長さがない パケットを破棄MUSTします。¶

サンプリングに十分なデータが利用可能であることを保証するため、 パケットは、エンコードされたパケット番号と保護済みペイロードの合計長が、 ヘッダー保護に必要なサンプルより少なくとも4バイト長くなるように パディングされます。[TLS13]で 定義される暗号スイート（この文書でヘッダー保護方式が定義されていない TLS_AES_128_CCM_8_SHA256を除く）は、16バイトの拡張と16バイトのヘッダー保護 サンプルを持ちます。そのため、パケット番号が1バイトでエンコードされる場合は、 保護されていないペイロード内に少なくとも3バイトのフレームが必要となり、 2バイトのパケット番号エンコーディングの場合は2バイトのフレームが必要になります。¶

サンプリングされた暗号文は、次の疑似コードによって 決定できます。¶

# pn_offset is the start of the Packet Number field.
sample_offset = pn_offset + 4

sample = packet[sample_offset..sample_offset+sample_length]
¶

短いヘッダーのパケットのパケット番号オフセットは、 次のように計算できます。¶

pn_offset = 1 + len(connection_id)
¶

また、長いヘッダーのパケットのパケット番号オフセットは、 次のように計算できます。¶

pn_offset = 7 + len(destination_connection_id) +
                len(source_connection_id) +
                len(payload_length)
if packet_type == Initial:
    pn_offset += len(token_length) +
                 len(token)
¶

たとえば、短いヘッダー、8バイトの接続ID、および AEAD_AES_128_GCMで保護されたパケットの場合、サンプルはバイト13から28まで （ゼロ始まりのインデックスを使用）を含みます。¶

同じUDPデータグラムに複数のQUICパケットが含まれる ことがあります。各パケットは個別に処理されます。¶

5.4.3. AESベースの ヘッダー保護

このセクションでは、AEAD_AES_128_GCM、 AEAD_AES_128_CCM、およびAEAD_AES_256_GCMのパケット保護アルゴリズムを 定義します。AEAD_AES_128_GCMおよびAEAD_AES_128_CCMは、Electronic Codebook（ECB）モードの128ビットAESを使用します。AEAD_AES_256_GCMは ECBモードの256ビットAESを使用します。AESは[AES]で定義されています。¶

このアルゴリズムは、パケット暗号文から16バイトを サンプリングします。この値はAES-ECBへの入力として使用されます。疑似コードでは、 ヘッダー保護関数は次のように定義されます。¶

header_protection(hp_key, sample):
  mask = AES-ECB(hp_key, sample)
¶

5.4.4. ChaCha20ベースの ヘッダー保護

AEAD_CHACHA20_POLY1305が使用されている場合、ヘッダー保護は セクション 2.4 of [CHACHA]で定義される 生のChaCha20関数を使用します。これは、256ビット鍵と、パケット保護出力から サンプリングされた16バイトを使用します。¶

サンプリングされた暗号文の最初の4バイトはブロックカウンターです。 ChaCha20実装は、バイト列の代わりに32ビット整数を受け取ることがあります。 その場合、バイト列はリトルエンディアン値として解釈されます。¶

残りの12バイトはnonceとして使用されます。ChaCha20実装は、 バイト列の代わりに3つの32ビット整数の配列を受け取ることがあります。 その場合、nonceバイトは32ビットリトルエンディアン整数の列として 解釈されます。¶

暗号化マスクは、5個のゼロバイトを保護するためにChaCha20を 呼び出すことで生成されます。疑似コードでは、ヘッダー保護関数は次のように 定義されます。¶

header_protection(hp_key, sample):
  counter = sample[0..3]
  nonce = sample[4..15]
  mask = ChaCha20(hp_key, counter, nonce, {0,0,0,0,0})
¶

5.5. 保護された パケットの受信

エンドポイントが特定のパケット番号を持つパケットを正常に 受信したら、同じパケット番号空間でより大きいパケット番号を持つすべてのパケットを、 同じ鍵、または鍵更新がある場合は後続のパケット保護鍵のいずれかで正常に 保護解除できないなら、破棄MUSTします。セクション6を 参照してください。同様に、鍵更新を引き起こすように見えるが正常に保護解除できない パケットは破棄MUSTされます。¶

パケットの保護解除の失敗は、必ずしもピアのプロトコルエラーや 攻撃の存在を示すわけではありません。QUICで使用される切り詰められたパケット番号 エンコーディングにより、パケットが大幅に遅延した場合、パケット番号が誤って デコードされることがあります。¶

5.6. 0-RTT鍵の使用

0-RTT鍵が利用可能な場合（セクション 4.6.1を参照）、リプレイ保護がないため、プロトコルに対するリプレイ攻撃を 避けるには、その使用に制限が必要です。¶

[QUIC-TRANSPORT]で定義されたフレームのうち、STREAM、 RESET_STREAM、STOP_SENDING、およびCONNECTION_CLOSEフレームは、アプリケーションデータを 搬送するため、0-RTTで使用するには潜在的に安全ではありません。0-RTTで受信された アプリケーションデータにより、サーバー上のアプリケーションがデータを1回だけでなく 複数回処理する可能性があります。再生されたアプリケーションデータの処理の結果として サーバーが追加の動作を行うと、望ましくない結果を招く可能性があります。したがって、 クライアントは、使用中のアプリケーションによって明示的に要求されない限り、 アプリケーションデータに0-RTTを使用してはMUST NOTいけません。¶

QUICを使用するアプリケーションプロトコルは、0-RTTの許容される 使用を定義するプロファイルを含めMUSTます。そうでなければ、0-RTTは アプリケーションデータを搬送しないQUICフレームを搬送するためにのみ使用できます。 たとえば、HTTPのプロファイルは[HTTP-REPLAY]で 説明され、HTTP/3で使用されます。セクション10.9 of [QUIC-HTTP]を参照してください。¶

パケットの再生によって追加の接続試行が発生する可能性はありますが、 アプリケーションデータを搬送しない再生フレームを処理する影響は、影響を受ける接続の 状態変更に限定されます。再生されたパケットを使用してTLSハンドシェイクを正常に 完了することはできません。¶

クライアントは、TLSハンドシェイクの完了前に送信するデータに対して 追加の制限を適用したい場合がMAYあります。¶

クライアントは、それ以外では0-RTT鍵を1-RTT鍵と同等に扱いますが、 0-RTT鍵では特定のフレームを送信できません。セクション 12.5 of [QUIC-TRANSPORT]を 参照してください。¶

クライアントは、自身の0-RTTデータがサーバーに受け入れられた ことを示す表示を受信した場合、サーバーのすべてのハンドシェイクメッセージを受信するまで 0-RTTデータを送信できます。クライアントは、0-RTTデータが拒否されたことを示す 表示を受信した場合、0-RTTデータの送信を停止するSHOULDです。¶

サーバーは、パケットを保護するために0-RTT鍵を使用しては MUST NOTいけません。サーバーは、0-RTTパケットへの確認応答を 保護するために1-RTT鍵を使用します。クライアントは、受信した0-RTTパケットの復号を 試みてはMUST NOTいけません。代わりに、それらを破棄 MUSTします。¶

クライアントは1-RTT鍵をインストールしたら、それ以上0-RTT パケットを送信してはMUST NOTいけません。¶

注：0-RTTデータはサーバーが受信するにつれて確認応答できますが、 0-RTTデータの確認応答を含むパケットは、TLSハンドシェイクが完了するまで クライアントによってパケット保護を解除できません。パケット保護を解除するために 必要な1-RTT鍵は、クライアントがすべてのサーバーハンドシェイクメッセージを 受信するまで導出できません。¶

5.7. 順序外の保護された パケットの受信

並べ替えや損失により、最終的なTLSハンドシェイクメッセージを 受信する前に、保護されたパケットがエンドポイントによって受信されることがあります。 クライアントはサーバーからの1-RTTパケットを復号できませんが、サーバーはクライアントからの 1-RTTパケットを復号できます。いずれの役割のエンドポイントも、ハンドシェイクを 完了する前にピアからの1-RTTパケットを復号してはMUST NOTいけません。¶

サーバーは、クライアントから最初のハンドシェイクメッセージを 受信した後に1-RTT鍵を利用できますが、クライアント状態に関する保証を欠いています。¶

  • サーバーが事前共有鍵を使用することを選択し、 クライアントの事前共有鍵バインダーを検証した場合を除き、クライアントは 認証されていません。セクション 4.2.11 of [TLS13]を参照してください。¶
  • サーバーがRetryパケットまたはその他の手段で クライアントのアドレスを検証していない限り、クライアントは生存性を 実証していません。セクション 8.1 of [QUIC-TRANSPORT]を 参照してください。¶
  • サーバーが応答する任意の受信済み0-RTTデータは、 リプレイ攻撃によるものである可能性があります。¶

したがって、ハンドシェイクが完了する前のサーバーによる1-RTT鍵の 使用は、データ送信に限定されます。サーバーは、TLSハンドシェイクが完了する前に、 受信した1-RTT保護パケットを処理してはMUST NOTいけません。 確認応答を送信することは、パケット内のすべてのフレームが処理されたことを示すため、 サーバーはTLSハンドシェイクが完了するまで1-RTTパケットへの確認応答を送信できません。 1-RTT鍵で保護された受信パケットは保存され、ハンドシェイク完了後に後で復号され、 使用されてMAYいます。¶

注：TLS実装は、ハンドシェイク完了前にすべての1-RTT シークレットを提供することがあります。QUIC実装が1-RTT読み取り鍵を持っている場合でも、 それらの鍵はハンドシェイク完了前に使用してはなりません。¶

サーバーがクライアントFinishedメッセージを待つという要件は、 そのメッセージが配送されることへの依存関係を生み出します。クライアントは、 いずれかのHandshakeパケットが確認応答されるまで、Finishedメッセージを搬送する CRYPTOフレームのコピーを含むHandshakeパケットと結合して1-RTTパケットを送信することで、 これが意味する潜在的なヘッドオブラインブロッキングを避けることができます。これにより、 サーバーはそれらのパケットを即座に処理できます。¶

サーバーは、TLS ClientHelloを受信する前に0-RTT鍵で保護された パケットを受信する可能性があります。サーバーは、ClientHelloを受信することを見込んで、 後で復号するためにこれらのパケットを保持してMAYいます。¶

クライアントは一般に、ハンドシェイクが完了するのと同時に 1-RTT鍵を受け取ります。1-RTTシークレットを持っている場合でも、クライアントは TLSハンドシェイクが完了する前に、受信した1-RTT保護パケットを処理しては MUST NOTいけません。¶

5.8. Retryパケットの完全性

Retryパケット（セクション 17.2.5 of [QUIC-TRANSPORT]を参照）は、 Retry Integrity Tagを搬送し、これは2つの性質を提供します。ネットワークによって 偶発的に破損したパケットを破棄できること、およびInitialパケットを観測した主体だけが 有効なRetryパケットを送信できることです。¶

Retry Integrity Tagは128ビットのフィールドであり、次の入力を 使用するAEAD_AES_128_GCM [AEAD]の出力として 計算されます。¶

  • 秘密鍵Kは、 0xbe0c690b9f66575a1d766b54e368c84eと等しい128ビットです。¶
  • nonce Nは、 0x461599d35d632bf2239825bbと等しい96ビットです。¶
  • 平文Pは空です。¶
  • 関連データAは、図8に 示すRetry Pseudo-Packetの内容です。¶

秘密鍵とnonceは、0xd9c9943e6101fd200021506bcc02814c73030f25c79d71ce876eca876e6fca8eを 秘密として使用し、ラベル「quic key」および「quic iv」 （セクション5.1）を指定して HKDF-Expand-Labelを呼び出すことで導出された値です。¶

Retry Pseudo-Packetはワイヤ上では送信されません。これは、 送信されたRetryパケットからRetry Integrity Tagを取り除き、次の2つのフィールドを 前置することで計算されます。¶

ODCID Length:

ODCID Lengthフィールドは、後続するOriginal Destination Connection IDフィールドのバイト単位の長さを含み、8ビット符号なし 整数としてエンコードされます。¶

Original Destination Connection ID:

Original Destination Connection IDは、このRetryが応答する InitialパケットのDestination Connection IDの値を含みます。このフィールドの 長さはODCID Lengthで与えられます。このフィールドの存在により、有効なRetry パケットはInitialパケットを観測した主体によってのみ送信できることが保証されます。¶

6. 鍵更新

ハンドシェイクが確認済みになると（セクション 4.1.2を参照）、エンドポイントは鍵更新を開始してMAYいます。¶

Key Phaseビットは、パケットを保護するためにどのパケット保護鍵が 使用されるかを示します。Key Phaseビットは、最初の1-RTTパケットの鍵セットでは 初期値0に設定され、その後の各鍵更新を通知するために反転されます。¶

Key Phaseビットにより、受信者は変更を引き起こした最初のパケットを 受信する必要なく、鍵材料の変更を検出できます。変更されたKey Phaseビットに気付いた エンドポイントは、鍵を更新し、変更された値を含むパケットを復号します。¶

鍵更新を開始すると、両方のエンドポイントが鍵を更新します。これは、 エンドポイントが独立して鍵を更新できるTLSとは異なります。¶

この機構は、1-RTT暗号化鍵を使用して送信されるKeyUpdateメッセージに 依存するTLSの鍵更新機構を置き換えます。エンドポイントはTLS KeyUpdateメッセージを 送信してはMUST NOTいけません。エンドポイントは、TLS KeyUpdate メッセージの受信を、unexpected_messageのfatal TLSアラートと等価な0x010a型の 接続エラーとして扱わMUSTなければなりません。 セクション4.8を参照してください。¶

図9は、使用されていた初期の鍵セット （@Mで識別）が更新された鍵（@Nで識別）に置き換えられる鍵更新処理を示します。 Key Phaseビットの値は角括弧[]で示されています。¶

6.1. 鍵更新の開始

エンドポイントは、パケット保護のために別個の読み取りシークレットと 書き込みシークレットを維持します。エンドポイントは、自身のパケット保護書き込み シークレットを更新し、それを使用して新しいパケットを保護することで鍵更新を開始します。 エンドポイントは、セクション7.2 of [TLS13]で行われるように、 既存の書き込みシークレットから新しい書き込みシークレットを作成します。これは、 TLSが提供するKDF関数をラベル「quic ku」とともに使用します。対応する鍵とIVは、 セクション5.1で定義されるように、その秘密から 作成されます。ヘッダー保護鍵は更新されません。¶

たとえば、TLS 1.3で書き込み鍵を更新するには、 HKDF-Expand-Labelを次のように使用します。¶

secret_<n+1> = HKDF-Expand-Label(secret_<n>, "quic ku",
                                 "", Hash.length)
¶

エンドポイントはKey Phaseビットの値を反転し、更新された鍵と IVを使用して以後のすべてのパケットを保護します。¶

エンドポイントは、ハンドシェイクを確認済みにする前に鍵更新を 開始してはMUST NOTいけません（セクション4.1.2）。 エンドポイントは、現在のkey phaseの鍵で保護して送信したパケットに対する確認応答を 受信していない限り、後続の鍵更新を開始してはMUST NOTいけません。これにより、別の鍵更新を開始できる前に 両方のピアで鍵が利用可能であることが保証されます。これは、各key phaseで送信された 最小のパケット番号と、1-RTT空間で確認応答された最大のパケット番号を追跡することで 実装できます。後者が前者以上になると、別の鍵更新を開始できます。¶

注：1-RTTパケット以外のパケットの鍵は更新されません。 それらの鍵はTLSハンドシェイク状態からのみ導出されます。¶

鍵更新を開始するエンドポイントは、パケット受信用に使用する鍵も 更新します。これらの鍵は、更新後にピアが送信するパケットを処理するために必要になります。¶

エンドポイントは、新しい鍵を使用して送信されたパケットを 正常に保護解除するまで古い鍵を保持MUSTします。エンドポイントは、 新しい鍵を使用して送信されたパケットを保護解除した後もしばらくの間、古い鍵を保持する SHOULDです。古い鍵を早く破棄しすぎると、遅延したパケットが 破棄される可能性があります。パケットの破棄はピアによってパケット損失として解釈され、 性能に悪影響を及ぼす可能性があります。¶

6.2. 鍵更新への応答

ピアは、現在のkey phase内のパケットに対する確認応答を 受信した後、鍵更新を開始することを許可されます。エンドポイントは、自身が最後に 送信したパケットを保護するために使用した値とは異なるkey phaseを持つパケットを 処理するとき、鍵更新を検出します。このパケットを処理するため、エンドポイントは 次のパケット保護鍵とIVを使用します。これらの鍵の生成に関する考慮事項については、 セクション6.3を参照してください。¶

パケットが次の鍵とIVを使用して正常に処理された場合、ピアは 鍵更新を開始しています。エンドポイントは、応答として、セクション6.1で説明されているように、対応するkey phaseへ送信鍵を 更新MUSTします。送信鍵は、更新された鍵で受信されたパケットに対する 確認応答を送信する前に更新されMUSTます。鍵更新を引き起こしたパケットを、 更新された鍵で保護されたパケットで確認応答することにより、エンドポイントは鍵更新が 完了したことを通知します。¶

エンドポイントは、通常のパケット送信動作に従って、パケットまたは 確認応答の送信を遅らせることができます。鍵更新に応答してただちにパケットを生成する 必要はありません。エンドポイントが次に送信するパケットは更新された鍵を使用します。 確認応答を含む次のパケットにより、鍵更新が完了します。エンドポイントが、鍵更新を 開始したパケットへの確認応答を含む更新済み鍵のパケットを送信する前に、2回目の更新を 検出した場合、それはピアが確認を待たずに鍵を2回更新したことを示します。エンドポイントは、 そのような連続した鍵更新をKEY_UPDATE_ERROR型の接続エラーとして扱って MAYいます。¶

エンドポイントが、古い鍵で保護されたパケットに含まれる確認応答を 受信し、その確認応答されたパケットのいずれかが新しい鍵で保護されていた場合、 それをKEY_UPDATE_ERROR型の接続エラーとして扱ってMAYいます。 これは、ピアが鍵更新を開始するパケットを受信し確認応答したが、応答として鍵を 更新していないことを示します。¶

6.3. 受信用鍵生成の タイミング

見かけ上の鍵更新に応答するエンドポイントは、Key Phaseビットが 無効であったことを示す可能性のあるタイミングサイドチャネル信号を生成しては MUST NOTいけません（セクション9.5を参照）。 エンドポイントは、鍵更新がまだ許可されていない場合、破棄された鍵の代わりに ランダム化されたパケット保護鍵を使用できます。ランダム化された鍵を使用することで、 パケット保護を解除しようとしてもタイミング変動が生じず、無効なKey Phaseビットを 持つパケットが拒否されます。¶

受信パケット用の新しいパケット保護鍵を作成する処理は、 鍵更新が発生したことを明らかにする可能性があります。エンドポイントは、パケット処理の 一部として新しい鍵を生成してMAYいますが、これは攻撃者が鍵更新の 発生時期を知り、Key Phaseビットの値を漏らすために利用できるタイミング信号を 作成します。¶

エンドポイントは通常、現在および次の受信パケット保護鍵を 利用可能にしておくことが期待されます。鍵更新が完了した後の短期間、PTOまで、 エンドポイントは次の受信パケット保護鍵セットの生成を延期してMAYいます。 これにより、エンドポイントは2組の受信鍵だけを保持できます。セクション 6.5を参照してください。¶

一度生成されたら、次のパケット保護鍵セットは、受信されたパケットが 後で破棄された場合でも保持されるSHOULDです。見かけ上の鍵更新を含むパケットは容易に 偽造でき、鍵更新の処理自体は大きな労力を必要としませんが、この処理を誘発することは 攻撃者によってDoSに使用される可能性があります。¶

この理由により、エンドポイントは受信パケット用に2組の パケット保護鍵、すなわち現在の鍵と次の鍵を保持できなければMUSTなりません。 これらに加えて前の鍵を保持すると性能が向上する可能性がありますが、これは必須ではありません。¶

6.4. 更新された鍵による 送信

エンドポイントは、古い鍵で保護されたパケットを送信することは ありません。現在の鍵だけが使用されます。パケット保護に使用される鍵は、新しい鍵へ 切り替えた直後に破棄できます。¶

より大きいパケット番号を持つパケットは、より小さいパケット番号を 持つパケットと同じ、またはそれより新しいパケット保護鍵で保護され MUSTます。より小さいパケット番号を持つパケットに新しい鍵が 使用されていた場合に、古い鍵で保護解除に成功したエンドポイントは、これを KEY_UPDATE_ERROR型の接続エラーとして扱わMUSTなければなりません。¶

6.5. 異なる鍵による受信

鍵更新中の受信パケットについては、ネットワークによって遅延した場合、 古い鍵で保護されたパケットが到着することがあります。古いパケット保護鍵を保持しておくことで、 これらのパケットを正常に処理できます。¶

次のkey phaseの鍵で保護されたパケットは、前のkey phaseの 鍵で保護されたパケットと同じKey Phase値を使用するため、古い鍵で保護されたパケットを 処理するには、両者を区別する必要があります。これはパケット番号を使用して行えます。 復元されたパケット番号が現在のkey phaseの任意のパケット番号より小さい場合、 以前のパケット保護鍵を使用します。復元されたパケット番号が現在のkey phaseの任意の パケット番号より大きい場合、次のパケット保護鍵を使用する必要があります。¶

以前、現在、および次のパケット保護鍵の間で選択する処理が、 どの鍵がパケット保護を解除するために使用されたかを明らかにする可能性のある タイミングサイドチャネルを露出しないように、いくらかの注意が必要です。 詳細についてはセクション9.5を参照してください。¶

あるいは、エンドポイントは2組のパケット保護鍵だけを保持し、 ネットワーク内での並べ替えを考慮するのに十分な時間が経過した後、以前の鍵を 次の鍵と入れ替えることができます。この場合、Key Phaseビットだけを使用して 鍵を選択できます。¶

エンドポイントは、次の受信鍵セットを現在の鍵へ昇格させた後、 その後続のパケット保護鍵セットを作成するまで、おおよそProbe Timeout （PTO。[QUIC-RECOVERY]を参照）の 期間を許容してMAYいます。これらの更新された鍵は、その時点で 以前の鍵を置き換えてMAYいます。PTOは主観的な測定値である （つまり、ピアがRTTについて異なる見方を持つ可能性がある）という注意点はありますが、 この時間は、並べ替えられたパケットが、たとえ確認応答されていたとしてもピアによって 損失と宣言されるのに十分長く、かつピアがさらなる鍵更新を開始できる程度に短いと 期待されます。¶

エンドポイントは、ピアが古い鍵を保持している期間中に鍵更新を 開始するパケットをピアが復号できない可能性を考慮する必要があります。エンドポイントは、 以前の鍵更新が受信されたことを確認する確認応答を受信した後、鍵更新を開始するまで PTOの3倍待つSHOULDです。十分な時間を許容しないと、パケットが 破棄される可能性があります。¶

エンドポイントは、新しい鍵を使用して保護されたパケットを 受信した後、PTOの3倍を超えて古い読み取り鍵を保持しないSHOULDです。 この期間の後、古い読み取り鍵と対応する秘密は破棄されるSHOULDです。¶

6.6. AEAD使用の制限

この文書は、QUICを使用する通信の機密性と完全性を攻撃する際に、 過度の使用が攻撃者に不釣り合いな優位を与えないようにするため、AEADアルゴリズムの 使用制限を設定します。¶

TLS 1.3で定義される使用制限は、機密性に対する攻撃からの 保護のために存在し、AEAD保護の成功した適用に適用されます。認証暗号における 完全性保護も、パケット偽造の試行回数を制限することに依存します。TLSは、 任意のレコードが認証チェックに失敗した後に接続を閉じることでこれを実現します。 これに対してQUICは、認証できない任意のパケットを無視し、複数の偽造試行を 許容します。¶

QUICは、AEADの機密性制限と完全性制限を別々に考慮します。 機密性制限は、特定の鍵で暗号化されるパケット数に適用されます。完全性制限は、 特定の接続内で復号されるパケット数に適用されます。各AEADアルゴリズムについて これらの制限を強制する詳細を以下に示します。¶

エンドポイントは、各鍵セットについて暗号化されたパケット数を カウントMUSTします。同じ鍵で暗号化されたパケットの総数が、 選択されたAEADの機密性制限を超えた場合、エンドポイントはそれらの鍵の使用を停止 MUSTします。エンドポイントは、選択されたAEADで許容される 機密性制限より多くの保護済みパケットを送信する前に、鍵更新を開始 MUSTします。鍵更新が不可能であるか、完全性制限に達した場合、 エンドポイントは接続の使用を停止MUSTし、パケットの受信に対する 応答としてステートレスリセットのみを送信します。鍵更新が不可能になる状態に達する前に、 AEAD_LIMIT_REACHED型の接続エラーで接続をただちに閉じることが RECOMMENDEDされます。¶

AEAD_AES_128_GCMおよびAEAD_AES_256_GCMの場合、機密性制限は 223個の暗号化パケットです。付録B.1を 参照してください。AEAD_CHACHA20_POLY1305の場合、機密性制限は可能なパケット数 （262）より大きいため、無視できます。AEAD_AES_128_CCMの場合、 機密性制限は221.5個の暗号化パケットです。 付録B.2を参照してください。制限を適用することで、 攻撃者が使用中のAEADをランダム置換と区別できる確率が低減されます。 [AEBounds]、 [ROBUST]、および [GCM-MU]を参照してください。¶

送信パケットをカウントすることに加え、エンドポイントは接続の 存続期間中に認証に失敗した受信パケット数をカウントMUSTします。接続内で、 すべての鍵にわたり、認証に失敗した受信パケットの総数が、選択されたAEADの 完全性制限を超えた場合、エンドポイントはAEAD_LIMIT_REACHED型の接続エラーで 接続をただちに閉じ、それ以上パケットを処理してはMUSTいけません。¶

AEAD_AES_128_GCMおよびAEAD_AES_256_GCMの場合、完全性制限は 252個の無効なパケットです。付録B.1を 参照してください。AEAD_CHACHA20_POLY1305の場合、完全性制限は 236個の無効なパケットです。[AEBounds]を参照してください。AEAD_AES_128_CCMの場合、 完全性制限は221.5個の無効なパケットです。 付録B.2を参照してください。この制限を適用することで、 攻撃者がパケットの偽造に成功する確率が低減されます。 [AEBounds]、 [ROBUST]、および [GCM-MU]を参照してください。¶

パケットサイズを制限するエンドポイントは、より高い機密性制限と 完全性制限を使用してMAYいます。詳細については付録Bを 参照してください。¶

将来の分析および仕様は、AEADに対する機密性または完全性の制限を 緩和してMAYいます。¶

QUICで使用するために指定される任意のTLS暗号スイートは、 機密性および完全性の余裕を保つ関連AEAD関数の使用制限を定義 MUSTします。つまり、認証できるパケット数、および認証に 失敗し得るパケット数について制限が指定されMUSTます。値の根拠となる 分析への参照、およびその分析で使用された仮定を提供することで、制限をさまざまな 使用条件に適応させることができます。¶

7. Initialメッセージのセキュリティ

Initialパケットは秘密鍵で保護されていないため、攻撃者による 改ざんの可能性があります。QUICは、パケットを読み取ることができない攻撃者に 対する保護を提供しますが、攻撃者がパケットを観測して注入できる攻撃に対して 追加の保護を提供しようとはしません。TLSメッセージ自体を変更するなどの 一部の改ざんは検出可能ですが、ACKを変更するなどの一部の改ざんは検出できません。¶

たとえば、攻撃者はACKフレームを含むパケットを注入して、 パケットが受信されていないように見せかけたり、接続状態について誤った印象を 作り出したりすることができます（たとえば、ACK Delayを変更することによって）。 そのようなパケットによって、正当なパケットが重複として破棄される可能性があることに 注意してください。実装は、Initialパケットに含まれるデータのうち、他の方法で 認証されていないものに依存する際には注意を払うSHOULDです。¶

攻撃者がHandshakeパケットで搬送されるデータを改ざんすることも 可能ですが、その種の改ざんにはTLSハンドシェイクメッセージの変更が必要であるため、 そのような改ざんはTLSハンドシェイクを失敗させます。¶

8. TLSハンドシェイクに対するQUIC固有の調整

TLSハンドシェイクのいくつかの側面は、QUICで使用される場合に異なります。¶

QUICはTLSから追加の機能も必要とします。暗号パラメータの ネゴシエーションに加えて、TLSハンドシェイクはQUICトランスポートパラメータの 値を搬送し認証します。¶

8.1. プロトコルネゴシエーション

QUICは、暗号学的ハンドシェイクが認証済みプロトコル ネゴシエーションを提供することを要求します。TLSはApplication-Layer Protocol Negotiation [ALPN]を使用してアプリケーションプロトコルを選択します。 アプリケーションプロトコルについて合意するために別の機構が使用されない限り、 エンドポイントはこの目的のためにALPNを使用MUSTします。¶

ALPNを使用する場合、アプリケーションプロトコルがネゴシエート されない場合、エンドポイントはno_application_protocol TLSアラート（QUICエラー コード0x0178。セクション4.8を参照）を伴って、 接続（[QUIC-TRANSPORT]のセクション10.2を参照）をただちに 閉じMUSTます。 [ALPN]はサーバーがこのアラートを使用することだけを 指定していますが、QUICクライアントはALPNネゴシエーションが失敗したときに 接続を終了するためにエラー0x0178を使用MUSTします。¶

アプリケーションプロトコルは、その上で動作できるQUIC バージョンを制限してMAYいます。 サーバーは、クライアントが選択したQUICバージョンと互換性のある アプリケーションプロトコルを選択MUSTします。サーバーは、互換性のある アプリケーションプロトコルを選択できないことを、0x0178 （no_application_protocol）型の接続エラーとして扱わMUSTなければなりません。 同様に、クライアントは、サーバーによる互換性のないアプリケーションプロトコルの選択を 0x0178型の接続エラーとして扱わMUSTなければなりません。¶

8.2. QUICトランスポート パラメータ拡張

QUICトランスポートパラメータはTLS拡張で搬送されます。 QUICの異なるバージョンは、トランスポート構成をネゴシエートするための 異なる方法を定義することがあります。¶

TLSハンドシェイクにトランスポートパラメータを含めることで、 これらの値に対する完全性保護が提供されます。¶

   enum {
      quic_transport_parameters(0x39), (65535)
   } ExtensionType;
¶

quic_transport_parameters拡張のextension_dataフィールドは、 使用中のQUICバージョンによって定義される値を含みます。¶

quic_transport_parameters拡張は、ハンドシェイク中の ClientHelloおよびEncryptedExtensionsメッセージで搬送されます。エンドポイントは quic_transport_parameters拡張を送信MUSTします。 quic_transport_parameters拡張を含まないClientHelloまたはEncryptedExtensions メッセージを受信したエンドポイントは、0x016d型（fatal TLS missing_extension アラートと等価。セクション4.8を参照）の エラーで接続を閉じMUSTます。¶

トランスポートパラメータは、ハンドシェイク完了前に利用可能に なります。サーバーはこれらの値をハンドシェイク完了より前に使用することがあります。 ただし、トランスポートパラメータの値はハンドシェイクが完了するまで認証されないため、 これらのパラメータのいかなる使用も、その真正性に依存することはできません。 トランスポートパラメータへのいかなる改ざんも、ハンドシェイクを失敗させます。¶

エンドポイントは、QUICを使用しないTLS接続 （[TLS13]で定義されるTCP上のTLSの使用など）で この拡張を送信してはMUST NOTいけません。この拡張をサポートする 実装は、トランスポートがQUICでないときにこの拡張を受信した場合、 fatal unsupported_extensionアラートを送信MUSTします。¶

quic_transport_parameters拡張をネゴシエートすると、 EndOfEarlyDataが削除されます。セクション8.3を参照してください。¶

8.3. EndOfEarlyDataメッセージの削除

TLS EndOfEarlyDataメッセージはQUICでは使用されません。 QUICは、0-RTTデータの終了を示したり、Handshake鍵への変更を通知したりするために このメッセージに依存しません。¶

クライアントはEndOfEarlyDataメッセージを送信しては MUST NOTいけません。サーバーは、0-RTTパケット内のCRYPTOフレームの 受信をPROTOCOL_VIOLATION型の接続エラーとして扱わMUSTなければなりません。¶

その結果、EndOfEarlyDataはTLSハンドシェイクトランスクリプトに 現れません。¶

8.4. TLSミドルボックス 互換モードの禁止

[TLS13]の付録D.4は、 一部のミドルボックスのバグに対する回避策として、TLS 1.3ハンドシェイクへの 変更を説明しています。TLS 1.3ミドルボックス互換モードでは、ClientHelloおよび ServerHelloのlegacy_session_idフィールドを32バイトの値に設定し、その後 change_cipher_specレコードを送信します。フィールドとレコードはいずれも意味上の 内容を持たず、無視されます。¶

このモードは、TCP上のTLSに干渉するミドルボックスにのみ適用されるため、 QUICでは用途がありません。QUICはchange_cipher_specレコードを搬送する手段も 提供しません。クライアントはTLS 1.3互換モードの使用を要求しては MUST NOTいけません。サーバーは、空でないlegacy_session_id フィールドを持つTLS ClientHelloの受信をPROTOCOL_VIOLATION型の接続エラーとして 扱うSHOULDです。¶

9. セキュリティに関する考慮事項

TLSに適用されるすべてのセキュリティ上の考慮事項は、QUICにおける TLSの使用にも適用されます。[TLS13]全体と その付録を読むことが、QUICのセキュリティ特性を理解する最良の方法です。¶

このセクションでは、TLS統合に固有のより重要なセキュリティ面の 一部を要約します。ただし、文書の残りの部分にもセキュリティに関連する詳細が 多数あります。¶

9.1. セッションのリンク可能性

TLSセッションチケットを使用すると、サーバーおよび場合によっては 他の主体が、同じクライアントによって行われた接続を相関できます。詳細については セクション4.5を参照してください。¶

9.2. 0-RTTによる リプレイ攻撃

[TLS13]のセクション8で 説明されているように、TLS early dataの使用にはリプレイ攻撃への露出が伴います。 QUICにおける0-RTTの使用も同様にリプレイ攻撃に対して脆弱です。¶

エンドポイントは、[TLS13]で 説明されるリプレイ保護を実装し使用MUSTしますが、 これらの保護が不完全であることは認識されています。したがって、リプレイのリスクに対する 追加の考慮が必要です。¶

QUICは、それが搬送する可能性のあるアプリケーションプロトコル情報を 介する場合を除き、リプレイ攻撃に対して脆弱ではありません。 [QUIC-TRANSPORT]で定義される フレームタイプに基づくQUICプロトコル状態の管理は、リプレイに対して脆弱ではありません。 QUICフレームの処理は冪等であり、フレームが再生、並べ替え、または失われても 無効な接続状態を生じることはありません。QUIC接続は、QUICが担う アプリケーションプロトコルによって生じるものを除き、接続の存続期間を超えて続く 影響を生成しません。¶

TLSセッションチケットとアドレス検証トークンは、接続間で QUIC構成情報を搬送するため、具体的には、サーバーが接続確立およびアドレス検証で 使用される状態を効率的に回復できるようにするために使用されます。これらは、 エンドポイント間でアプリケーション意味論を伝達するために使用してはMUST NOTいけません。 クライアントは、それらを不透明な値として扱わMUSTなければなりません。 これらのトークンが再利用される可能性があるため、リプレイに対するより強力な保護が 必要です。¶

接続で0-RTTを受け入れるサーバーは、0-RTTなしで接続を受け入れる 場合より高いコストを負担します。これには、より高い処理コストおよび計算コストが 含まれます。サーバーは、0-RTTを受け入れる際に、リプレイの確率と関連するすべての コストを考慮する必要があります。¶

最終的に、0-RTTによるリプレイ攻撃のリスクを管理する責任は アプリケーションプロトコルにあります。QUICを使用するアプリケーションプロトコルは、 そのプロトコルが0-RTTをどのように使用するか、およびリプレイ攻撃から保護するために 採用される対策を説明MUSTします。リプレイリスクの分析では、 アプリケーション意味論を搬送するすべてのQUICプロトコル機能を考慮する必要があります。¶

0-RTTを完全に無効にすることは、リプレイ攻撃に対する最も効果的な 防御です。¶

QUIC拡張は、リプレイ攻撃がその動作にどのように影響するかを 説明するか、0-RTTでの拡張の使用を禁止するかのいずれかを行わMUSTなければなりません。 アプリケーションプロトコルは、アプリケーション意味論を搬送する拡張の0-RTTでの使用を 禁止するか、リプレイ緩和戦略を提供するかのいずれかを行わMUSTなければなりません。¶

9.3. パケット反射攻撃の 軽減

サーバーから大量のハンドシェイクメッセージを生じさせる小さな ClientHelloは、パケット反射攻撃において、攻撃者が生成するトラフィックを 増幅するために使用される可能性があります。¶

QUICはこの攻撃に対して3つの防御を含みます。第一に、ClientHelloを 含むパケットは最小サイズまでパディングされMUSTます。第二に、 検証されていない送信元アドレスに応答する場合、サーバーは受信したバイト数の 3倍を超えるバイト数を送信することを禁止されます（[QUIC-TRANSPORT]のセクション8.1を 参照）。最後に、Handshakeパケットの確認応答は認証されるため、盲目的な攻撃者は それらを偽造できません。これらの防御を組み合わせることで、増幅の程度が制限されます。¶

9.4. ヘッダー保護の 分析

[NAN]は、 「Hide Nonce」（HN）変換と呼ばれる、nonceプライバシーを提供する 認証暗号アルゴリズムを分析しています。この文書における一般的なヘッダー保護構成は、 それらのアルゴリズムの1つ（HN1）です。ヘッダー保護はパケット保護AEADの後に 適用され、AEAD出力からバイト集合（sample）をサンプリングし、 次のように擬似ランダム関数（PRF）を使用してヘッダーフィールドを暗号化します。¶

protected_field = field XOR PRF(hp_key, sample)
¶

この文書のヘッダー保護の変種は、汎用PRFの代わりに 擬似ランダム置換（PRP）を使用します。ただし、すべてのPRPはPRFでもあるため [IMC]、 これらの変種はHN1構成から逸脱しません。¶

hp_keyはパケット保護鍵と異なるため、ヘッダー保護は [NAN]で定義されるAE2セキュリティを達成し、 したがって保護されたパケットヘッダーであるfieldのプライバシーを保証します。 この構成に基づく将来のヘッダー保護の変種は、同等のセキュリティ保証を確保するために PRFを使用MUSTします。¶

同じ鍵と暗号文サンプルを複数回使用すると、ヘッダー保護を 損なうリスクがあります。同じ鍵と暗号文サンプルで2つの異なるヘッダーを保護すると、 保護されたフィールドの排他的論理和が明らかになります。AEADがPRFとして動作すると仮定すると、 Lビットがサンプリングされる場合、2つの暗号文サンプルが同一になる確率は 2-L/2、すなわち誕生日限界に近づきます。この文書で説明される アルゴリズムでは、その確率は264分の1です。¶

攻撃者がパケットヘッダーを変更することを防ぐため、ヘッダーは パケット保護を使用して推移的に認証されます。パケットヘッダー全体は認証済み 追加データの一部です。偽造または変更された保護フィールドは、パケット保護が解除された後に のみ検出できます。¶

9.5. ヘッダー保護のタイミング サイドチャネル

攻撃者は、パケット番号またはKey Phaseの値を推測し、 エンドポイントにタイミングサイドチャネルを通じて推測を確認させる可能性があります。 同様に、パケット番号長に対する推測が試され、露出する可能性があります。パケットの受信者が パケット保護の解除を試みずに重複するパケット番号を持つパケットを破棄する場合、 そのパケット番号が受信済みパケットと一致することをタイミングサイドチャネルを通じて 明らかにする可能性があります。認証がサイドチャネルを持たないためには、 ヘッダー保護の解除、パケット番号の回復、およびパケット保護の解除という処理全体が、 タイミングおよびその他のサイドチャネルなしで一体として適用されMUSTます。¶

パケットの送信では、パケットペイロードおよびパケット番号の 構築と保護は、パケット番号またはそのエンコードされたサイズを明らかにする サイドチャネルを持たないものでなければMUSTなりません。¶

鍵更新中、新しい鍵の生成に要する時間が、鍵更新が発生したことを タイミングサイドチャネルを通じて明らかにする可能性があります。あるいは、攻撃者が パケットを注入する場合、このサイドチャネルは注入されたパケット上のKey Phaseの値を 明らかにする可能性があります。鍵更新を受信した後、エンドポイントは セクション6.3で説明されるように、次の 受信パケット保護鍵セットを生成して保存するSHOULDです。鍵更新を受信する前に新しい鍵を生成することで、 パケットの受信によってKey Phaseの値を漏らすタイミング信号が生じることを防ぎます。¶

これは、この鍵生成をパケット処理中に行わないことに依存し、 エンドポイントが受信用に3組のパケット保護鍵、すなわち前のkey phase、現在の key phase、および次のkey phaseの鍵を維持することを必要とする場合があります。 エンドポイントは、代わりに、古い鍵を破棄するまで次の受信パケット保護鍵の生成を 延期し、任意の時点で保持する必要のある受信鍵を2組だけにすることを選択できます。¶

9.6. 鍵の多様性

TLSを使用する際には、TLSの中心的な鍵スケジュールが使用されます。 TLSハンドシェイクメッセージが秘密の計算に組み込まれる結果として、 QUICトランスポートパラメータ拡張を含めることで、ハンドシェイク鍵および 1-RTT鍵が、TCP上でTLSを実行しているサーバーによって生成される可能性のある鍵と 同じにならないことが保証されます。クロスプロトコル鍵同期の可能性を避けるため、 鍵分離を改善する追加の手段が提供されます。¶

QUICパケット保護鍵およびIVは、TLSの同等の鍵とは異なるラベルを 使用して導出されます。¶

この分離を維持するため、QUICの新しいバージョンは、 パケット保護鍵およびIV、さらにヘッダー保護鍵の鍵導出のために新しいラベルを 定義するSHOULDです。このバージョンのQUICは文字列「quic」を使用します。 他のバージョンでは、その文字列の代わりにバージョン固有のラベルを使用できます。¶

Initialシークレットは、ネゴシエートされたQUICバージョンに固有の 鍵を使用します。QUICの新しいバージョンは、Initialシークレットの計算に使用される 新しいsalt値を定義するSHOULDです。¶

9.7. ランダム性

QUICは、接続IDなどのプロトコル値について直接的に、またTLSを 介して推移的に、エンドポイントが安全な乱数を生成できることに依存します。 安全な乱数生成に関する指針については[RFC4086]を 参照してください。¶

この例では、TLSはアプリケーション書き込みシークレットを生成し、 サーバーはそこからHKDF-Expand-Labelを使用して4つの値、すなわち鍵、IV、 ヘッダー保護鍵、および鍵が更新された後に使用される秘密（この最後の値はこの例では これ以上使用されません）を生成します。¶ 

secret
    = 9ac312a7f877468ebe69422748ad00a1
      5443f18203a07d6060f688f30f21632b

key = HKDF-Expand-Label(secret, "quic key", "", 32)
    = c6d98ff3441c3fe1b2182094f69caa2e
      d4b716b65488960a7a984979fb23e1c8

iv  = HKDF-Expand-Label(secret, "quic iv", "", 12)
    = e0459b3474bdd0e44a41c144

hp  = HKDF-Expand-Label(secret, "quic hp", "", 32)
    = 25a282b9e82f06f21f488917a4fc8f1b
      73573685608597d0efcb076b0ab7a7a4

ku  = HKDF-Expand-Label(secret, "quic ku", "", 32)
    = 1223504755036d556342ee9361d25342
      1a826c9ecdf3c7148684b36b714881f9
¶

次は、空のDestination Connection IDを持つ最小パケットを 保護する手順を示しています。このパケットは単一のPINGフレーム（すなわち、 0x01だけのペイロード）を含み、パケット番号654360564を持ちます。この例では、 長さ3のパケット番号（すなわち49140がエンコードされる）を使用することで、 パケットのペイロードをパディングする必要を避けています。パケット番号がより少ない バイトでエンコードされる場合、PADDINGフレームが必要になります。¶

pn                 = 654360564 (decimal)
nonce              = e0459b3474bdd0e46d417eb0
unprotected header = 4200bff4
payload plaintext  = 01
payload ciphertext = 655e5cd55c41f69080575d7999c25a5bfb
¶

結果として得られる暗号文は可能な最小サイズです。ヘッダー保護用の サンプルを生成するために1バイトがスキップされます。¶

sample = 5e5cd55c41f69080575d7999c25a5bfb
mask   = aefefe7d03
header = 4cfe4189
¶

packet = 4cfe4189655e5cd55c41f69080575d7999c25a5bfb

付録 B. AEADアルゴリズム分析

このセクションは、AEAD_AES_128_GCM、AEAD_AES_128_CCM、および AEAD_AES_256_GCMのAEADアルゴリズム制限を導出する際に使用された分析を文書化します。 以下の分析では、乗算（*）、除算（/）、累乗（^）の記号、および優先順位を 確立するための括弧を使用します。次の記号も使用されます。¶

t:

認証タグのサイズ（ビット単位）。これらの暗号では、tは 128です。¶

n:

ブロック関数のサイズ（ビット単位）。これらの暗号では、nは 128です。¶

k:

鍵のサイズ（ビット単位）。これはAEAD_AES_128_GCMおよび AEAD_AES_128_CCMでは128、AEAD_AES_256_GCMでは256です。¶

l:

各パケット内のブロック数（以下を参照）。¶

q:

エンドポイントによって作成され保護される真正なパケット数。 この値は、鍵を更新する前に保護できるパケット数の上限です。¶

v:

エンドポイントが受け入れる偽造パケット数。この値は、 エンドポイントが鍵を更新する前に拒否できる偽造パケット数の上限です。¶

o:

攻撃者が行うオフライン理想暗号問い合わせの量。¶

以下の分析は、各メッセージの生成に関与するブロック操作数の カウントに依存します。この分析は、最大211（l = 27）および 216（l = 212）サイズのパケットについて行われます。211のサイズは一般的な 展開パターンに一致する制限であると期待されます。一方、216はQUICパケットの 最大可能サイズです。パケットサイズを厳密に制限するエンドポイントだけが、より小さい パケットサイズを使用して導出される、より大きな機密性および完全性の制限を使用できます。¶

AEAD_AES_128_GCMおよびAEAD_AES_256_GCMの場合、メッセージ長（l）は、 ブロック単位の関連データの長さに、ブロック単位の平文の長さを加えたものです。¶

AEAD_AES_128_CCMの場合、ブロック暗号操作の総数は、 関連データのブロック単位の長さ、暗号文のブロック単位の長さ、平文のブロック単位の 長さ、および1を加えた合計です。この分析では、これはパケット長のブロック数の 2倍の値に単純化されます（つまり、211バイトに制限されたパケットでは 2l = 28、それ以外では2l = 213）。 この単純化は、パケットがすべての関連データと暗号文を含むことに基づいています。 これにより、パケットごとの操作数が1から3ブロック過大評価されます。¶

B.1. AEAD_AES_128_GCMおよびAEAD_AES_256_GCMの使用制限の分析

[GCM-MU]は、TLS 1.3および QUICで使用されるAEAD_AES_128_GCMおよびAEAD_AES_256_GCMに対する具体的な 境界を規定しています。このセクションは、いくつかの単純化した仮定を使用して この分析を文書化します。¶

  • 攻撃者が偽造試行で使用する暗号文ブロック数は v * lによって制限されます。これは偽造試行回数に各パケットのサイズ （ブロック単位）を掛けたものです。¶
  • 攻撃者が行うオフライン作業の量は、分析における 他の要因を支配しません。¶

[GCM-MU]の境界は、 [AEBounds]で使用されるものよりも 厳密で完全であり、[TLS13]で説明されるものより 大きな制限を可能にします。¶

B.1.1. 機密性の上限

機密性について、[GCM-MU]の定理（4.3）は、nonceを繰り返さない 単一ユーザーについて、攻撃者が得る実際のAEADアルゴリズムとランダムなAEAD アルゴリズムとの識別優位を決定する支配的な項が次のとおりであることを示しています。¶

2 * (q * l)^2 / 2^n
¶

目標優位を2-57とすると、これは次の関係を 生じます。¶

q <= 2^35 / l
¶

したがって、211バイトを超えるパケットを送信しない エンドポイントは、攻撃者に目標値2-57より有意な優位を 得させることなく、単一の接続で228個を超えるパケットを 保護できません。パケットサイズを216まで許容するエンドポイントの 制限は、代わりに223です。¶

B.1.2. 完全性の上限

完全性について、[GCM-MU]の定理（4.3）は、攻撃者がパケットの 偽造に成功する際に得る優位が、次を超えないことを示しています。¶

(1 / 2^(8 * n)) + ((2 * v) / 2^(2 * n))
        + ((2 * o * v) / 2^(k + n)) + (n * (v + (v * l)) / 2^k)
¶

目標は、この優位を2-57に制限することです。 AEAD_AES_128_GCMでは、この不等式の第4項が残りを支配するため、結果に 大きな影響を与えることなく他の項を除去できます。これにより、次の近似が 得られます。¶

v <= 2^64 / l
¶

211バイトを超えるパケットから保護を解除しようと しないエンドポイントは、最大257個のパケットから保護の解除を 試みることができます。処理されるパケットサイズを制限しないエンドポイントは、 最大252個のパケットから保護の解除を試みることができます。¶

AEAD_AES_256_GCMでは同じ項が支配しますが、kの値が大きいため 次の近似が得られます。¶ 

v <= 2^192 / l
¶

これはAEAD_AES_128_GCMの制限より実質的に大きいものです。 ただし、この文書は、どちらの制限も十分に大きいため、両方の関数に同じ制限を 適用することを推奨します。¶

B.2. AEAD_AES_128_CCMの使用制限の分析

TLS [TLS13]および [AEBounds]は、AEAD_AES_128_CCMの 使用制限を指定していません。ただし、QUICで使用される任意のAEADは、 機密性と完全性の両方が維持されることを保証する使用制限を必要とします。 このセクションはその分析を文書化します。¶

[CCM-ANALYSIS]は この分析の基礎として使用されます。その分析の結果は、[TLS13]で 選択されたものに基づく使用制限を導出するために使用されます。¶

機密性について、[CCM-ANALYSIS]の定理2は、攻撃者が理想的な 擬似ランダム置換（PRP）に対して得る識別優位が次を超えないことを示しています。¶

(2l * q)^2 / 2^n
¶

[CCM-ANALYSIS]の定理1における完全性制限は、同じ メッセージ数に対して、攻撃者に厳密に高い優位を提供します。機密性優位と 完全性優位の目標は同じであるため、定理1だけを考慮する必要があります。¶

定理1は、攻撃者が理想的なPRPに対して得る優位が次を 超えないことを示しています。¶

v / 2^t + (2l * (v + q))^2 / 2^n
¶

tとnはいずれも128であるため、第1項は 第2項に比べて無視できるほど小さく、結果に大きな影響を与えることなく その項を除去できます。¶

これにより、暗号化試行と復号試行の両方を、機密性のみの定理に よって生成されるものと同じ制限で組み合わせる関係が得られます。目標優位を 2-57とすると、これは次のようになります。¶

v + q <= 2^34.5 / l
¶

q = vと設定することで、機密性と完全性の両方の 制限値を生成できます。したがって、パケットを211バイトに制限する エンドポイントは、機密性と完全性の両方の制限として226.5個の パケットを持ちます。パケットサイズを制限しないエンドポイントの制限は 221.5です。¶
