1. 序論

この節は非規範的です。

現代のウェブアプリケーションは、多くの場合、完全なページナビゲーションを 行わずに、ユーザーインタラクションに応じてコンテンツを動的に更新します。 構造的なDOM変更、コンテンツフル・ペイント、履歴状態の変更など、 これらのインタラクションによって開始される効果は、歴史的に、正しい ユーザー操作へ測定し帰属させることが困難でした。

典型的なシングルページアプリケーションのパターンを考えてみます。ユーザーが 商品リンクをクリックすると、 clickイベントハンドラーが起動します。このハンドラーは、商品の詳細を取得するためにネットワークのfetch()を開始します。 応答が到着すると、コールバックが実行され、新しいコンテンツをDOMへ動的に注入し、 History APIまたはNavigation APIを使用してURLを更新します。これはユーザーには ナビゲーションのように見えますが、Largest Contentful Paint (LCP)（[LARGEST-CONTENTFUL-PAINT]）のような既存の指標は 初回ページ読み込みのみを測定し、Interaction to Next Paint (INP)はクリック自体の即時の 視覚的フィードバックのみを測定するため、その後の重要なレンダリングと「ソフト」 ナビゲーションは捕捉されません。

この仕様は、[EVENT-TIMING] APIを活用してインタラクションを定義し、 [ASYNC-CONTEXT] 提案を使用して、非同期タスク境界を越えた因果関係を追跡します。これは、 ブラウザーが「ソフトナビゲーション」を含むこれらの効果を識別し報告する方法を定義し、 [PAINT-TIMING]および[LARGEST-CONTENTFUL-PAINT]と統合することで、レンダリング変更を パフォーマンス タイムラインへ帰属させます。

さらに、この仕様は[CONTAINER-TIMING]提案と統合します。ユーザー インタラクションがDOM変更を引き起こすと、それらの変更されたノードは新しい「コンテナルート」として指定されます。 それらのサブツリー内で後続するコンテンツフル・ペイントは、それぞれのルートへ帰属され、 そのルートは元のインタラクションまで追跡されます。これにより、初回イベントディスパッチのはるか後に発生する、 リッチで動的なページ更新を効率的かつ正確に測定できます。

2. Performance Timeline のスライシング

この節は非規範的である。

性能指標（イベントタイミング、レイアウトシフト、リソース読み込み、または 長いアニメーションフレームなど）を、それらの起点となるナビゲーションに帰属させるため、 performance timeline は、navigationId 属性を PerformanceEntry 上で用いて セグメントに分割される。

時間にまたがる遷移の正確なスライス点を決定することは複雑である。一貫した 帰属を確保するため、この仕様は、URL の変更があるたびに timeline を ただちにスライスするのではなく、遷移を ソフトナビゲーション が performance timeline に発行される準備が整うまで遅延させるモデルを採用する。

設計上のトレードオフ、帰属規則、およびエッジケースの詳細な議論については、 [PERFORMANCE-TIMELINE] issue の コメントを参照のこと。

2.1. ナビゲーション ID

navigation id は、 global object の存続期間内において各ナビゲーション（ハードおよび ソフトの両方）に割り当てられる一意の識別子である。

各 Window は、以下を持つ。

  • current navigation id。これは 64-bit unsigned integer であり、 初期値は Window の initial navigation id value に設定される。

  • navigation count。これは 64-bit unsigned integer であり、初期値は 0 である。

  • initial navigation id value。これは 64-bit unsigned integer であり、 初期値は 100 から 10000 までのランダムな整数に設定される。

  • navigation id increment。これは 64-bit unsigned integer であり、 初期値はユーザーエージェントにより選択される小さい正の整数に設定される。

注: initial navigation id value と navigation id increment は、 navigationId 値を計算するために用いられる（ increment the current navigation id を参照）。 これにより、開発者がそれをナビゲーションの正確な回数を数えるために利用したり、 それが 0 から始まると仮定したりすることを抑止する。 ユーザーエージェントは、異なる Window オブジェクト間で共有されるグローバルなナビゲーション値を用いないことが期待される。これは クロスオリジン漏えいを防ぐためである。

Window window が与えられたとき、 increment the current navigation id するには、次を実行する。

  1. window の navigation count を、window の navigation count に 1 を加えた値に設定する。

  2. newId を、window の initial navigation id value に、 （window の navigation count と window の navigation id increment の積）を加えた値とする。

  3. window の current navigation id を newId に設定する。

2.2. PerformanceEntry 拡張

[Exposed=(Window,Worker)]
partial interface PerformanceEntry {
    readonly attribute unsigned long long navigationId;
};
各 PerformanceEntry は、関連付けられた navigation id を持つ。 これは 64-bit unsigned integer であり、初期値は 0 である。

navigationId 属性の getter は、this の navigation id を返さなければならない。

2.3. Performance Timeline との統合

queue a PerformanceEntry において、手順 1（entry の初期化）の後に、 以下の手順を追加する。

  1. newEntry の navigation id が 0 である場合:

    1. global が Window オブジェクトである場合:

      1. newEntry の navigation id を、 global の current navigation id に設定する。

3. インタラクション基盤

3.1. Interaction Context の導入

ソフトナビゲーションの検出は、タスクの因果関係を追跡し、特定の操作（例: DOM ノードの追加）が 特定のユーザーインタラクションによって引き起こされたことを観測できる能力に依存する。

この仕様は、この伝播を処理するために TC39 の [ASYNC-CONTEXT] 提案を活用する。すべての 新しいユーザーインタラクション（Event Timing によって定義されるもの）または関連するナビゲーションイベントは、 新しい InteractionContext を作成する。この context は、隠された内部専用の AsyncContext.Variable （[[ActiveInteractionContext]] と表記される）に格納される。

Web プラットフォームの AsyncContext との統合により、この変数は非同期の継続 （例: setTimeout、fetch、await）に自動的に 付加され、ブラウザーが 後続の効果を元のインタラクションに帰属させることを可能にする。

スクリプトの伝播に加えて、この仕様は、DOM を変更するユーザーインタラクションがどのように container roots（[CONTAINER-TIMING] 経由）を確立するかを定義する。 これらの roots により、 contentful paints のような後続のレンダリング効果を、現在アクティブな非同期スクリプトが 存在しない場合でも、起点となった interaction context まで追跡できる。

3.2. InteractionContext 構造体

InteractionContext は、単一のインタラクションからソフト ナビゲーションを検出するために必要なデータを保持するために用いられる struct である。これは以下の items を持つ。

  • id。初期状態では未設定 - この context に関連付けられた interaction id。

  • context document。Document - この context に関連付けられた Document。

  • start time。初期状態では未設定 - インタラクションの開始時刻を表す。

  • navigation type。初期状態では未設定 - ナビゲーションの種類 （例: "push", "replace"）を表す。

  • first URL value。初期状態では未設定 - 最初の 変更時点での URL の値を表す。

  • first URL update timestamp。 初期状態では未設定 - このインタラクション中における最初の URL 変更の時刻を表す。

  • first scroll timestamp。初期状態では未設定 - このインタラクション中における最初の scroll イベントの時刻を表す。

  • first input timestamp。初期状態では未設定 - このインタラクション中における最初の input イベントの時刻を表す。

  • first contentful paint。初期値は null - このインタラクションに対する最初の InteractionContentfulPaint entry。

  • largest contentful paint。初期値は null - これまでのこのインタラクションに対する最大の InteractionContentfulPaint entry。

  • last URL value。初期状態では未設定 - このインタラクション中における 直近の URL 変更の値を表す。

  • emitted フラグ。初期値は false - そのインタラクションにより soft navigation entry が発行されたことを示す。

注: この仕様の将来のバージョンでは、 navigation のより完全な履歴を提供するために、interaction context 中に発生する すべての URL 更新を追跡する可能性がある。last URL value は、 現在 PerformanceSoftNavigation の name で公開されるのが first URL value のみであっても、効果を interaction の最終状態へ正確に帰属させることを確保するため、内部的に追跡される。

3.3. 基盤アルゴリズム

Document document が与えられたとき、 get the current interaction context するには、次を実行する。

  1. context を、内部の AsyncContext.Variable [[ActiveInteractionContext]] の値とする。

  2. context が null でなく、かつ context の context document が document と等しくない場合、null を返す。

  3. context を返す。

Document document、interaction id、および Event event が与えられたとき、 get the context for an interaction するには、次を実行する。

  1. document の interaction id to interaction context[interaction id] が exists する場合、 document の interaction id to interaction context[interaction id] を返す。

  2. interaction context を新しい InteractionContext とする。

  3. interaction context の id を interaction id に設定する。

  4. interaction context の context document を document に設定する。

  5. interaction context の start time を event の startTime に設定する。

  6. document の interaction id to interaction context[interaction id] を interaction context に Set する。

  7. interaction context を返す。

Document document と Event event が与えられたとき、 update the interaction contexts for an event するには、次を実行する。

  1. timestamp を、document の relevant global object が与えられたときの current high resolution time とする。

  2. is scroll を、event の type が "scroll" であれば true、そうでなければ false とする。

  3. is input を、event の type が has dispatched input event を引き起こす event type（[EVENT-TIMING] で定義されるもの）であれば true、そうでなければ false とする。

  4. is scroll が false であり、かつ is input が false である場合、戻る。

  5. document の interaction id to interaction context の values の各 interaction context について、次を実行する。

    1. is scroll が true であり、かつ interaction context の first scroll timestamp が未設定である場合:

      1. interaction context の first scroll timestamp を timestamp に設定する。

    2. is input が true であり、かつ interaction context の first input timestamp が未設定である場合:

      1. interaction context の first input timestamp を timestamp に設定する。

null または PerformanceEventTiming オブジェクト timingEntry が与えられたとき、 set the current interaction context for event dispatch するには、次を実行する。

  1. timingEntry が null である場合、null を返す。

  2. event を、timingEntry の associated event とする。

  3. interaction id を、timingEntry の interactionId とする。

  4. interaction id が null または 0 である場合、null を返す。

  5. document を、event の relevant global object の associated Document とする。

  6. document と event が与えられたものとして、 Update the interaction contexts for an event を行う。

  7. interaction context を、 document、interaction id、および event が与えられたものとして geting the context for an interaction を行った結果とする。

  8. 内部の AsyncContext.Variable [[ActiveInteractionContext]] の値を interaction context に設定する。

  9. interaction context を返す。

unset the current interaction context after event dispatch するには、次を実行する。

  1. 内部の AsyncContext.Variable [[ActiveInteractionContext]] の値を null に設定する。

4. インタラクションの Contentful Paint

4.1. InteractionContentfulPaint インターフェイス

[Exposed=Window]
interface InteractionContentfulPaint : PerformanceEntry {
    readonly attribute LargestContentfulPaint largestContentfulPaint;
    readonly attribute unsigned long long interactionId;

    object toJSON();
};

InteractionContentfulPaint includes PaintTimingMixin;

各 InteractionContentfulPaint は、以下を持つ。

  • 関連付けられた context。これは InteractionContext である。

  • 関連付けられた largest contentful paint。これは LargestContentfulPaint entry である。

  • 関連付けられた paint timing info。

largestContentfulPaint 属性の getter は、this の largest contentful paint を返さなければならない。

interactionId 属性の getter は、this の context の id を返さなければならない。

name 属性の getter は空文字列を返さなければならない。

entryType 属性の getter は "interaction-contentful-paint" を返さなければならない。

startTime 属性の getter は、this の context の start time を返さなければならない。

duration 属性の getter は、this の paint timing info に対する default paint timestamp と、this の context の start time との差を返さなければならない。

注: entry は、関連付けられた InteractionContext から 静的なインタラクションプロパティ（interactionId や startTime など）を動的に照会する一方で、 その視覚的更新の正確な状態をこれらの指標が反映するように、 作成時に paint の paint timing info と候補の LargestContentfulPaint entry の静的なスナップショットを取得する。

4.2. Interaction Contentful Paint アルゴリズム

Window window、InteractionContext interaction context、LargestContentfulPaint lcpEntry、および paint timing info paintTimingInfo が与えられたとき、 create an interaction contentful paint entry するには、次を実行する。

  1. entry を、window の realm 内の新しい InteractionContentfulPaint オブジェクトとする。

  2. entry の context を interaction context に設定する。

  3. entry の largest contentful paint を lcpEntry に設定する。

  4. entry に関連付けられた paint timing info を paintTimingInfo に設定する。

  5. entry を返す。

LargestContentfulPaint lcpEntry、Document document、および InteractionContext interaction context が与えられたとき、 process a new attributed contentful paint candidate するには、次を実行する。

  1. interaction context の first scroll timestamp が設定済みであり、かつ lcpEntry の renderTime が interaction context の first scroll timestamp より大きい場合、戻る。

  2. interaction context の first input timestamp が設定済みであり、かつ lcpEntry の renderTime が interaction context の first input timestamp より大きい場合、戻る。

  3. global を、document の relevant global object とする。

  4. paint timing info を、lcpEntry に関連付けられた paint timing info とする。

  5. icpEntry を、 global、interaction context、lcpEntry、および paint timing info が与えられたものとして create an interaction contentful paint entry を行った結果とする。

  6. icpEntry を Queue する。

  7. icpEntry を global の performance entry buffer に追加する。

  8. interaction context の first contentful paint が null である場合:

    1. interaction context の first contentful paint を icpEntry に設定する。

  9. interaction context の largest contentful paint を icpEntry に設定する。

注: InteractionContentfulPaint entries は、 インタラクションが最終的に soft navigation になるかどうかとは独立して、検出された時点で performance timeline に発行される。これにより、開発者はすべてのインタラクションについて レンダリング更新を監視できる。

5. ソフトナビゲーション

5.1. PerformanceSoftNavigation インターフェイス

[Exposed=Window]
interface PerformanceSoftNavigation : PerformanceEntry {
    readonly attribute DOMString navigationType;
    readonly attribute unsigned long long interactionId;
    InteractionContentfulPaint? getLargestInteractionContentfulPaint();
};

PerformanceSoftNavigation includes PaintTimingMixin;

各 PerformanceSoftNavigation は、以下を持つ。

  • 関連付けられた context。これは InteractionContext である。

  • 関連付けられた paint timing info。

navigationType 属性の getter は、this の context の navigation type を返さなければならない。

interactionId 属性の getter は、this の context の id を返さなければならない。

getLargestInteractionContentfulPaint() method は、this の context の largest contentful paint を返さなければならない。

name 属性の getter は、this の context の first URL value を返さなければならない。

entryType 属性の getter は "soft-navigation" を返さなければならない。

startTime 属性の getter は、this の context の start time を返さなければならない。

duration 属性の getter は、発行時における this の presentationTime と、this の context の start time との差を返さなければならない。

注: 実際には、getLargestInteractionContentfulPaint() は非 null の entry を返すことが期待される。確認済みの soft navigation は、その発行を トリガーするために、少なくとも 1 つの検出済み interaction contentful paint を必要とするためである。

しかし、将来の反復では、最初の paint より前、URL 変更の直後に soft navigation が committed と見なされうるかどうかを評価する可能性がある。そのようなモデルでは、この method は 発行時に null を返す可能性がある。

このタイミングに関する主な設計上の考慮事項は、URL 変更から最初の paint までの間に発生する 他の timeline entries（例: LayoutShift、PerformanceResourceTiming、LongAnimationFrameTiming など）の帰属である。多くのサイトは、 たとえば fetch 要求を開始した直後に新しい URL を commit するが、その間も現在のページは 以前の navigation 状態のままである。一貫した timeline slicing を確保するため、この仕様は、 最初の paint が遷移を確認するまで、そのような entries すべてを以前の navigation 識別子に帰属させる。

5.2. ソフトナビゲーションアルゴリズム

soft navigation とは、 次の条件を満たす same-document navigation である。

  • InteractionContext がアクティブである間に、 same-document URL change が発生する。

  • 同じ InteractionContext に帰属される contentful paint が発生する。

Document document と InteractionContext interaction context が与えられたとき、 evaluate soft navigation emission するには、次を実行する。

  1. interaction context の emitted が true である場合、戻る。

  2. document の active soft navigation candidate が interaction context でない場合、戻る。

  3. interaction context の first URL value が未設定である場合、戻る。

  4. interaction context の first contentful paint が null である場合、 戻る。

  5. window を、document の relevant global object とする。

  6. entry を、window と interaction context が与えられたものとして create a soft navigation entry を行った結果とする。

  7. window と entry が与えられたものとして Emit a soft navigation entry を行う。

  8. interaction context の emitted を true に設定する。

Window window と InteractionContext interaction context が与えられたとき、 create a soft navigation entry するには、次を実行する。

  1. entry を、window の realm 内の新しい PerformanceSoftNavigation オブジェクトとする。

  2. entry の context を interaction context に設定する。

  3. first paint を、interaction context の first contentful paint とする。

  4. first paint が null でない場合:

    1. entry に関連付けられた paint timing info を、first paint に 関連付けられた paint timing info に設定する。

  5. entry を返す。

Window window と PerformanceSoftNavigation entry が与えられたとき、 emit a soft navigation entry するには、次を実行する。

  1. window が与えられたものとして、Increment the current navigation id を行う。

  2. entry を Queue する。

  3. entry を window の performance entry buffer に追加する。

注: この navigationId は、この PerformanceSoftNavigation entry について、performance timeline へ queue される一部として自動的に設定され、 他の PerformanceEntry 型の挙動と一致する。

Document document、string url、 および string navigation type が与えられたとき、 process a same document commit するには、次を実行する。

  1. url が document の url と等しい場合、戻る。

  2. interaction context を、document が与えられたものとして getting the current interaction context を行った結果とする。

  3. interaction context が null である場合、戻る。

  4. interaction context の last URL value を url に設定する。

  5. interaction context の first URL update timestamp が 未設定である場合:

    1. interaction context の first URL update timestamp を、document の relevant global object が与えられたときの current high resolution time に設定する。

    2. interaction context の first URL value を url に設定する。

    3. interaction context の navigation type を navigation type に設定する。

  6. document の active soft navigation candidate を interaction context に設定する。

  7. document と interaction context が与えられたものとして Evaluate soft navigation emission を行う。

6. 仕様との統合

6.1. HTML との統合

6.1.1. Document

各 document は、interaction id to interaction context を持つ。これは map であり、 初期状態では空である。

各 document は、active soft navigation candidate を持つ。これは InteractionContext または null であり、初期値は null である。

6.1.2. History

update document for history step application において、5.5.1（documentsEntryChanged が true であり、かつ documentIsNew が false である場合）の前に、process a same document commit を、Document、 target entry の url、および "traverse" が与えられたものとして行う。
shared history push/replace steps （[HTML] で定義されるもの）において、 URL が更新された後、process a same document commit を、 Document、新しい URL、および操作種別（"push" または "replace" のいずれか）が与えられたものとして行う。

6.1.3. Node

各 node は、associated interaction context を持ち、初期値は null である。

6.1.4. ハードナビゲーション

新しい Window が作成されるとき（例: "hard" navigation 中）、その current navigation id は、§ 2.1 ナビゲーション ID で指定されるように初期化される。

[NAVIGATION-TIMING] において、初期ナビゲーション用の PerformanceNavigationTiming entry を作成する際、ユーザーエージェントは、その navigationId を Window の current navigation id に設定しなければならない。

6.2. Event Timing との統合

この仕様は、[EVENT-TIMING] における interactions の定義を拡張し、 現代的な Web アプリケーションおよび Single Page Applications に関連する追加の event type を含める。

以下の event type は、interaction （[EVENT-TIMING] で定義されるもの）の一部と見なされる。

  • navigate

  • popstate

  • hashchange

これらのイベントがユーザーインタラクションの結果として dispatch されるとき、ユーザーエージェントは、 document の relevant global object に対して getting the next interactionId することにより取得される一意の interaction id をそれらに割り当てなければならない。あるイベントが、すでに 割り当て済みの interaction id を持つ以前の interaction によってトリガーされた場合、ユーザーエージェントは同じ識別子を再利用するべきである。

注:これらのイベントは、ユーザー操作からの起点イベント （例: ブラウザー UI の戻るボタンをクリックすること）である場合にのみ、primary interactions として報告される。ほとんどの他のシナリオ、たとえば click handler が手動で history を操作する場合などでは、後続の popstate または navigate イベントは 独立したユーザーインタラクションとは見なされない。

これらのプログラムによりトリガーされたイベントには isTrusted フラグが設定されていない可能性があるが、それらは [ASYNC-CONTEXT] によって、元の interaction へ 正しく帰属される。history API および navigation API は、起点となるタスクの非同期継続として扱われるためである。

注: navigate、popstate、および hashchange イベントは、 soft navigations を追跡し interactionId を割り当てるための内部シグナルとして 用いられる。この仕様は、これらの event type を PerformanceEventTiming entries として performance timeline へ公開することを要求せず、その判断を [EVENT-TIMING] 仕様に委ねる。

注: [EVENT-TIMING] 仕様は、 processingStart および processingEnd hooks を明示的に定義し、 この統合のために十分早い段階で interactionId の割り当てが行われることを 確保するよう更新されることが期待される。現在、interactionId の割り当ては通常、 event processing の終了時まで遅延される。

この仕様では、 安全なグローバルカウンターを確保するため、interactionId に unsigned long long を用いる一方で、 [EVENT-TIMING] は現在、それを unsigned long として定義している。この不一致は、 両仕様の将来のバージョンで解決されることが期待される。

Window window に対して、 get the next interactionId するには、次を実行する。

  1. window の interaction count を、window の interaction count に 1 を加えた値に設定する。

  2. window の initial interactionId value に、 （window の interaction count と window の interactionId increment の積）を加えた値を返す。

[EVENT-TIMING] で定義される、DOM 仕様の event dispatch アルゴリズムへの変更において、次を行う。

  1. timingEntry を initialize and record event timing processing start の結果に設定する手順の直後に、以下の手順を実行する。

    1. timingEntry が与えられたものとして、Set the current interaction context for event dispatch を行う。

  2. record event timing processing end を呼び出す手順の 直前に、以下の手順を実行する。

    1. Unset the current interaction context after event dispatch を行う。

6.3. Largest Contentful Paint (LCP)統合

この仕様は、ペイントをインタラクションへ帰属させるため、[LARGEST-CONTENTFUL-PAINT]アルゴリズムへフックします。

report largest contentful paintにおいて、 candidatesを、list of painted images とlist of painted textの和集合とします:

  1. contextToNewLargestCandidateを新しいmapとします。

  2. candidates内の各candidateについて:

    1. elementをcandidateのelementとします。

    2. elementがnullである場合、continueします。

    3. interaction contextを、elementのassociated interaction contextとします。

    4. interaction contextがnullである場合、continueします。

    5. sizeをcandidateのsizeとします。

    6. interaction contextのlargest contentful paintがnullでなく、かつ sizeがinteraction contextのlargest contentful paintのlargestContentfulPaintの size以下である場合、 continueします。

    7. contextToNewLargestCandidate[interaction context]が未設定であるか、 sizeが contextToNewLargestCandidate[interaction context]のsizeより大きい場合:

      1. contextToNewLargestCandidate[interaction context]を candidateに設定します。

  3. contextToNewLargestCandidate内の各interaction context → newLcpCandidateについて:

    1. newLcpEntryを、newLcpCandidateを表すLargestContentfulPaint エントリーとします。

    2. newLcpEntry、 document、および interaction contextが与えられたものとして、Process a new attributed contentful paint candidateします。

    3. documentおよびinteraction contextが与えられたものとして、Evaluate soft navigation emissionします。

注: § 6.4 Container Timing統合の節は、DOMへの変更が、影響を受ける要素について previously reported paintsをリセットする方法を定義し、 それらが[LARGEST-CONTENTFUL-PAINT]アルゴリズムによって再評価され、現在のインタラクションへ正しく 帰属されることを保証します。

注: [LARGEST-CONTENTFUL-PAINT] アルゴリズムは、関連付けられたインタラクションコンテキストを持つすべてのペイントを無視することが期待されます。これらは、この仕様の § 4.1 InteractionContentfulPaintインターフェイスで扱われるためです。

6.4. Container Timing統合

[CONTAINER-TIMING] APIは、共通のDOM 祖先によってレンダリング効果をグループ化する仕組みを提供します。この仕様は、その仕組みと統合し、 レンダリング変更をユーザーインタラクションへ帰属させます。

node insertにおいて、またはnodeが 次のいずれかの方法で変更されるとき:

  • 要素のclassまたはstyle 属性が変更される。

  • リソース属性（imgまたはvideo要素上のsrcなど）が変更される。

次のステップを実行します:

  1. documentを、そのnodeのnode documentとします。

  2. interaction contextを、 documentが与えられたものとしてgetting the current interaction contextした結果とします。

  3. interaction contextがnullでない場合:

    1. そのnodeのassociated interaction context をinteraction contextに設定します。

    2. そのnodeを（[CONTAINER-TIMING]で定義される）container rootとして指定します。

    3. そのnodeおよびその各子孫について、それらをDocumentの （[PAINT-TIMING]で定義される） previously reported paintsから削除します。

注: 変更されたサブツリーを previously reported paintsの集合から削除することにより、ユーザー エージェントは、これらの要素に対する次のコンテンツフル・ペイントが再検出され、新しい InteractionContextへ正しく帰属されることを保証します。

注: "node insert"への参照は、 新しい要素を文書構造へ追加するすべての操作（例: appendChild、innerHTMLの更新など）を包含する高水準の 記述です。これらを、文書構造の変更を追跡する内部実装フックへ対応付けることは、 ユーザーエージェントに委ねられます。

注: ユーザーエージェントは、コンテナルートの疎な ツリーを維持することが推奨されます。変更されたnodeが、同じ interaction contextに関連付けられた既存のcontainer rootの子孫である場合、 ユーザーエージェントは、新しい冗長なrootを作成しないことを選択できます。

さらに、大規模なDOM変更中の高価なメインスレッド作業を避けるため、ユーザーエージェントは、 子孫をpreviously reported paintsから削除する処理を最適化することが 推奨されます。 即時の網羅的な走査の代わりに、ユーザーエージェントは変更されたrootを"dirty"としてマークし、 後続のツリー走査中（例: レイアウトまたはペイント中）にこの状態を遅延して伝播できます。 表示されないことが分かっているサブツリー、またはcontent-visibilityやCSS containmentのような既存の仕組みによってスキップされるサブツリーも、このリセット処理中にスキップできます。

7. セキュリティとプライバシーの考慮事項

ソフトナビゲーションをパフォーマンスタイムラインへ公開すること自体には、 セキュリティおよびプライバシーへの影響はありません。しかし、検出されたソフト ナビゲーションの結果として各種ペイントタイミングエントリーをリセットすることは、 特に訪問済みリンクが パーティション化される前には、影響を持つ可能性があります。したがって、 :visitedキャッシュをパーティション化せずにそのようなペイント操作を公開することは、 問題となるペイント操作を慎重に分析し、それらがオリジンをまたいでユーザーの履歴を 公開しないことを確認した後にのみ行う必要があります。

8. 付録: 重複するインタラクションと競合状態

この節は非規範的である。

Web アプリケーションは、複数のユーザーインタラクションを短時間に連続して処理することが多い。この仕様は、 そのような重なり合うインタラクションを、以下のモデルによって扱う。

  • コンテキストの独立性: 各インタラクションは、自身の InteractionContext を独立して管理する。 複数の context が同時に「進行中」になり得て、それぞれが自身の URL 変更および contentful paints を追跡する。

  • アクティブ候補の単一性: 多数のインタラクションがアクティブになり得る一方で、Document は、任意の時点で 1 つの active soft navigation candidate だけを認識する。

  • プリエンプション: interaction context は、最初の same-document URL 変更をトリガーした瞬間に active soft navigation candidate になる。後続のインタラクション変更が発生した場合、それは前のものを横取りし、 新しい candidate になる。

  • 発行検証: PerformanceSoftNavigation は、発行 条件（URL 変更 + contentful paint）が満たされた時点で、active soft navigation candidate である interaction context に対してのみ発行される。これにより、soft navigation の報告が document の現在の視覚状態および navigation 状態と一貫したままになる。

  • 持続的な帰属: あるインタラクションが soft navigation candidate として横取りされた場合でも、 それがアクティブであり続ける限り、自身の InteractionContentfulPaint entries の帰属および報告を継続する。これにより、それ自体は 「navigations」と見なされない、並行したレンダリング更新を正確に測定できる。
