概要

支援技術は、障害のある人々に適切な情報を伝えるために、文書の構造および期待される挙動に関する セマンティック情報を必要とする。この仕様は、ウェブグラフィックスに固有のコアWAI-ARIA 1.1 [WAI-ARIA-1.1] モジュールとしてのロール を定義する。これらのセマンティクスにより、作者はグラフィックスのアクセシビリティを改善するために、 支援技術に対してグラフィックの論理構造を表現できる。これにより支援技術は、セマンティックな ナビゲーションを可能にし、スタイルおよび対話機能を適応させ、利用者に最適な体験を提供できる。 これらの機能は、HTML [HTML52] および SVG [SVG2] によって定義されるグラフィックスおよび文書構造要素を補完する。

この文書は、WAI-ARIA 概要で説明される WAI-ARIA スイートの一部である。

    1. 導入

    この節は非規範的である。

    WAI-ARIA は、ウェブコンテンツおよびアプリケーションの アクセシビリティと相互運用性を改善するためのフレームワークを提供する技術仕様である。これは、 ウェブブラウザーがウェブコンテンツ内のアクセシビリティセマンティクスを、プラットフォーム固有の アクセシビリティ API に対応付けられるようにする。これにより、作者がプラットフォーム依存性を含めることを 要求されることなく、ウェブコンテンツはネイティブプラットフォームアプリケーションと同様に、 プラットフォーム支援技術と相互運用できるようになる。

    この仕様は、グラフィックスをサポートするように設計された WAI-ARIA [WAI-ARIA-1.1] のモジュール型拡張である。この 仕様の目的には次が含まれる:

    • WAI-ARIA を拡張し、チャート、グラフ、 地図、技術図面および科学図などの構造化グラフィックスをサポートするためのセマンティックな 拡張を生成する。これは、Scalable Vector Graphics と同様に、HTML 5 Canvas、ならびに HTML およびその他のマークアップ言語の CSS スタイリングで 生成されるグラフィックスにも適用可能である。
    • WAI-ARIA のモジュール化および拡張のための 新しいガバナンスモデルに合わせる。
    • 支援技術をサポートし、セマンティックなナビゲーション、代替スタイリング、および対話性サポートを 可能にする構造的セマンティクス拡張を提供する。
    • SVG 2 および ARIA 1.1 と調和して動作し、 すべての主要ブラウザーにおいて、WAI-ARIA および HTML 5.2 と同等の、一貫して動作するアクセシビリティ インフラストラクチャを得る。

    この仕様は、すべての構造化グラフィックスまたは図で使用される中核ロールを定義する。 これは、図形やキャンバスなどのグラフィカルなマークアップ要素を説明するために使用できるデフォルトロールを 確立する。 代替テキストを提供し、要素間の関係を示すための WAI-ARIA 属性と 組み合わせることで、多くの図および図解に注釈を付けるためのフレームワークを提供する。 将来の作業では、このフレームワークを拡張し、チャートや地図などのデータが豊富なグラフィックスに対して、 より詳細な注釈を可能にする。

    WAI-ARIA のより詳細な説明については、 WAI-ARIA 導入および、それが Rich Internet Application Accessibility にどのように適用されるかを参照のこと。

    1.1 対象読者

    この仕様は、グラフィックス用の WAI-ARIA モジュールを定義し、グラフィックス固有の 要素ロールで構成される。 これはいくつかの読者層に影響する:

    • WAI-ARIA およびグラフィックス WAI-ARIA 機能を含むコンテンツを処理する ユーザー エージェント;
    • 障害のあるユーザーに特化した読書体験を提供する支援 技術;
    • ウェブグラフィックスの作者;
    • 作者が適合するグラフィックスを作成するのを支援するオーサリングツール; および
    • WAI-ARIA およびこの WAI-ARIA Graphics モジュールの適切な使用を 検証する適合性チェッカー。

    各適合要件は、それが適用される読者層を示す。

    1.2 ユーザーエージェントサポート

    このモジュールは、WAI-ARIA [WAI-ARIA-1.1] で定義される一般的な ユーザーエージェントサポート原則に従う。 ここで定義されるロールは、アクセシビリティ API に 公開される情報以外については、ユーザーエージェントの挙動に変更を要求しない。 しかし、ここで定義されるセマンティクスは、ユーザーエージェントが読者に提示される一般的なユーザー インターフェイスを強化できる能力を提供する。 たとえば、ユーザーエージェントは、グラフィカルな環境に適した代替キーボードナビゲーションを 提供したり、ユーザーが大きな文書からグラフィックのコピーを抽出できるようにしたりしてもよい。

    1.3 WAI-ARIA とホスト言語の共進化

    WAI-ARIA Graphics モジュールは、 WAI-ARIA [WAI-ARIA-1.1] で定義される WAI-ARIA とホスト言語の共進化の モデルに従う。これは、HTML [HTML52]、SVG [SVG2] および EPUB などのサポート言語で セマンティクスを拡張すること、または ARIA のサポートを明示的に含まないその他のマークアップベースの 言語においてアクセシビリティ強化技術として使用されることを意図している。 WAI-ARIA ロールは、作者がスタイルおよびスクリプトを通じて新しい型の オブジェクトを作成する場合、または文書の意味ではなく視覚的外観を記述するマークアップ言語を使用する場合に、 支援技術に対してセマンティクスを明確にする。

    マークアップ言語がこれらのセマンティクスの一部をネイティブに提供する場合があるものの、 WAI-ARIA Graphics モジュールによって提供される セマンティクスには持続的な必要性があると予想される。 一部のホスト言語は、ユーザーインターフェイス以外の機能のセマンティクスを作成するために存在する。たとえば、 SVG は、グラフィカルオブジェクトの生成の背後にある セマンティクスを表現するものであり、それらのオブジェクトが表す可能性のあるユーザーインターフェイス コンポーネントのセマンティクスではない。このようなホスト言語は、設計上、この仕様のすべての機能に対応付けられる ネイティブセマンティクスを提供しない場合がある。これらのホスト言語では、WAI-ARIA Graphics モジュールを、セマンティック情報を追加するための長期的なアプローチとして採用できる。

    1.4 オーサリング慣行

    1.4.1 オーサリングツール

    WAI-ARIA および Graphics WAI-ARIA のロール、状態およびプロパティの定義における 多くの要件は、コードを検証するために使用される他の品質管理プロセスと同様に、開発プロセス中に 自動的にチェックできる。グラフィックスを作成している作者を支援するために、これらのツールは DOM からの Graphics WAI-ARIA ロールの セマンティック構造を、この仕様で定義されるものと比較し、作者にエラーを通知したり、単にその構造を強制する テンプレートを作成したりできる。

    1.4.2 テスト慣行および ツール

    対話型コンテンツのアクセシビリティは、静的チェックだけでは確認できない。対話型コンテンツの 開発者は、ウィジェットおよびアプリケーションへの デバイス非依存アクセスをテストし、ユーザー操作中のすべてのコンテンツおよび変更に対する アクセシビリティ API アクセスを検証すべきである。

    1.5 支援技術

    アクセシビリティセマンティクスへのプログラム的アクセスは、支援技術にとって不可欠である。詳細については、 WAI-ARIA [WAI-ARIA-1.1] の 支援技術の節を参照のこと。

    特にグラフィックスロールについては、支援技術の2つのカテゴリが特に関連するが、 それぞれ異なるニーズを持つ:

    • スクリーンリーダー、点字ディスプレイ、テキスト専用ディスプレイまたはプリンターなどのテキストベースの提示。 これらの技術は、複雑なグラフィックを、意味のある構造およびコンポーネント間の関係を保持しつつ、 セマンティックなテキスト説明に置き換える必要がある。
    • 色調整済みディスプレイ、画面拡大鏡、大きな印刷文書、またはグラフィックサポート付きのエンボスプリンター などの代替グラフィカル提示。これらの技術は、主に装飾的なグラフィカル機能と、コンテンツの意味を 伝えるために不可欠なグラフィカル機能とを区別する必要がある。

    ロールの説明は、そのロールを持つ要素のどの機能がセマンティックに重要と見なされ、可能な限り読者に 伝えられるべきかを示唆する。

    2. 適合性

    この仕様の主要コンテンツは規範的であり、適合性主張に影響する要件を定義する。導入資料、付録、 「非規範的」と示された節およびその下位節、図、例、および注記は参考情報 （非規範的）である。非規範的資料は、ガイドラインの解釈を助ける助言情報を提供するが、 適合性主張に影響する要件を作成しない。

    規範的な節は、この仕様に適合する実装のためにユーザーエージェントが従わなければならない要件を提供する。 この文書におけるキーワード MUST、MUST NOT、REQUIRED、SHALL、SHALL NOT、SHOULD、RECOMMENDED、MAY、および OPTIONAL は、この文書では Keywords for use in RFCs to indicate requirement levels [RFC2119] に記述されるように 解釈される。RFC-2119 キーワードは大文字で書式化され、class="rfc2119" を持つ要素に 含まれる。上記のキーワードが使用されているが、この書式を共有しない場合、それらは RFC 2119 の意味での 形式的情報を伝えず、単なる説明、すなわち参考情報である。この仕様では、可能な限りそのような用法は避けられている。

    規範的な節は、作者、ユーザーエージェントおよび支援技術がこの仕様に適合する実装のために MUST 従わなければならない要件を提供する。

    非規範的（参考情報）な節は、この仕様を理解するのに有用な情報を提供する。そのような節には推奨慣行の例が 含まれる場合があるが、この仕様に適合するために、そのような推奨に従うことは要求されない。

    3. 重要な用語

    この節は非規範的である。

    一部の用語はその場で定義されるが、次の定義はこの文書全体で使用される。

    アクセシビリティ API

    オペレーティングシステムおよびその他のプラットフォームは、オブジェクトおよびイベントに関する情報を支援技術へ公開するインターフェイスの集合を提供する。支援技術は これらのインターフェイスを使用して、それらのウィジェットに関する情報を取得し、それらと対話する。アクセシビリティ API の例には、Microsoft Active Accessibility [MSAA]、Microsoft User Interface Automation [UI-AUTOMATION]、MSAA with UIA Express [UIA-EXPRESS]、 Mac OS X Accessibility Protocol [AXAPI]、Linux/Unix Accessibility Toolkit [ATK] および Assistive Technology Service Provider Interface [AT-SPI]、ならびに IAccessible2 [IAccessible2] がある。

    支援技術

    次のようなハードウェアおよび/またはソフトウェア:

    • ユーザーエージェントによって提供されるサービスに依存して Web コンテンツを取得およびレンダリングする
    • API の使用を通じて、ユーザーエージェントまたはウェブコンテンツ自体と連携する、および
    • 障害のある人々によるウェブコンテンツとのユーザー対話を容易にするために、ユーザーエージェントが提供するものを 超えるサービスを提供する

    この定義は、他の文書で使用される定義とは異なる場合がある。

    この文書の文脈で重要な支援技術の例には、次が含まれる:

    • レンダリングされたテキストおよび画像の視覚的可読性を拡大し改善するために使用される画面拡大鏡;
    • 合成音声またはリフレッシュ可能な点字ディスプレイを通じて情報を伝えるために最もよく使用される スクリーンリーダー;
    • テキストを合成音声に変換するために使用されるテキスト読み上げソフトウェア;
    • 音声による制御およびディクテーションを可能にするために使用される音声認識ソフトウェア;
    • キーボードをシミュレートするために使用される代替入力技術（ヘッドポインター、オンスクリーンキーボード、 単一スイッチ、および呼気/吸気デバイスを含む）;
    • マウスによるポイントおよびクリックをシミュレートするために使用される代替ポインティングデバイス。
    属性

    この仕様では、属性はマークアップ言語における用法と同様に使用される。属性は、 要素によって表されるオブジェクトの 状態およびプロパティに関する情報を提供するために、 要素に追加される構造的特徴である。

    クラス

    類似した特性を共有するインスタンスオブジェクトの集合。

    要素

    この仕様では、要素はマークアップ言語における用法と同様に使用される。要素は、オブジェクトのデータプロファイルを含む、マークアップ言語内の 構造要素である。

    イベント

    計算システム内の他のオブジェクトに対し、オブジェクトの 状態における個別の変化を伝達するために使用される プログラム的メッセージ。ウェブページへのユーザー入力は、相互作用を記述し、文書オブジェクトの 状態変化の通知を提供できる抽象イベントを通じて仲介されることが多い。一部のプログラミング言語では、 イベントはより一般的に通知として知られている。

    参考情報

    情報提供を目的として提供され、適合性のために要求されないコンテンツ。適合性のために要求される コンテンツは規範的と呼ばれる。

    規範的

    適合性のために要求される。対照的に、参考情報または「非規範的」と識別されるコンテンツは、 適合性のために要求されない。

    オブジェクト

    ユーザーインターフェイスの文脈では、知覚されるユーザー体験における項目であり、マークアップ言語では 1つ以上の要素によって表され、ユーザーエージェントによってレンダリングされる。

    プログラミングの文脈では、類似するオブジェクトの一般的な特性を定義する1つ以上のクラスおよびインターフェイスのインスタンス化である。アクセシビリティ API 内のオブジェクトは、1つ以上の DOM オブジェクトを表す場合がある。アクセシビリティ API は、DOM インターフェイスとは異なる定義済みインターフェイスを持つ。
    オントロジー

    クラスの特性、およびそれらが互いにどのように関係するかについての記述。

    プロパティ

    特定のオブジェクトの性質に不可欠な 属性、またはそのオブジェクトに関連付けられたデータ値を表す属性。 プロパティの変更は、オブジェクトの意味または提示に大きな影響を与える場合がある。特定のプロパティ （たとえば、aria-multiline）は、 状態よりも変更されにくいが、変更頻度の違いは規則ではないことに注意。 aria-activedescendant、aria-valuenow、および aria-valuetext などの少数のプロパティは、頻繁に変更されることが 期待される。状態とプロパティの明確化を参照。

    ロール

    型の主要な指標。 このセマンティックな関連付けにより、ツールは、その型の他のオブジェクトについてのユーザーの 期待と一貫した方法で、オブジェクトとの対話を提示およびサポートできる。

    セマンティクス

    人間が理解するものの意味を、コンピューターがオブジェクトの表現、たとえば要素や属性を処理でき、さまざまな人間がそのオブジェクトについて相互に一貫した理解に達する 方法で確実に表現できるように定義したもの。

    状態

    状態は、ユーザー操作または自動化されたプロセスに応じて変化する可能性のあるオブジェクトの 特性を表す動的なプロパティである。状態はオブジェクトの 本質的な性質には影響しないが、オブジェクトまたはユーザー対話の可能性に関連付けられたデータを表す。 状態とプロパティの明確化を参照。

    分類体系

    さまざまなクラスの特性が互いにどのように関係するかの 階層的定義であり、その中でクラスは階層内のスーパークラスのプロパティを継承する。分類体系は、 オントロジーの形式的定義の一部を構成できる。

    ユーザーエージェント

    Web コンテンツを取得し、レンダリングし、エンドユーザーとの対話を容易にする任意のソフトウェア。 この定義は、他の文書で使用される定義とは異なる場合がある。

    ウィジェット

    ユーザーが対話できる個別のユーザーインターフェイスオブジェクト。 ウィジェットは、1つの値または操作を持つ単純なオブジェクト（例: チェックボックスおよびメニュー項目）から、 多くの管理されたサブオブジェクトを含む複雑なオブジェクト（例: ツリーおよびグリッド）まで及ぶ。

    4. グラフィックスロール

    この節は、 WAI-ARIA ロール 分類体系への追加を定義し、 すべてのロールの特性およびプロパティを説明する。 このモジュールによって提供されるフィールドの説明については、ARIA Roles を参照。

    作者には、支援技術に提示される内容に影響を与え、ロールおよびプロパティの使用を通じて ナビゲーションに影響を与える能力が与えられる。 これには、要素をセマンティックな重要性がないものとしてマークする能力が含まれる。 グラフィックスでは、すべての要素を提示しナビゲートすることが、 グラフィックを理解し使用することをより難しくする多くの場合がある。

    作者は、 ロール none または presentation を割り当てることにより、 要素を文書のセマンティック表現 （アクセシビリティツリー）から除外するようにマークしてもよい。 このロールを持つ要素は、その子またはテキストコンテンツが親要素に直接含まれているかのように、 支援技術によって透過的に扱われるべきである。 さらに、 img や graphics-symbol などの特定のロールは、 親要素に割り当てられると、すべての子 DOM 構造がアクセシビリティツリーから省略されることになる。 これは、ロール特性表の "Children Presentational" 値によって示される。 最後に、グラフィックス言語のネイティブセマンティクスも、 セマンティックデータが付与されていない DOM 構造をデフォルトで無視する場合がある。 SVG については、これは SVG Accessibility API Mappings 仕様 [SVG-AAM-1.0] で定義される。

    すべての場合において、提示用と見なされるためには、要素は対話型であってはならず、 アクセシブルなプロパティまたは代替テキストを割り当てられていてはならない。 none または presentation のロールは、 対話型要素、または WAI-ARIA の状態およびプロパティを持つ要素では無視される。

    4.1 ロールの定義

    以下は、この仕様で定義される WAI-ARIA ロールのアルファベット順リストである。 それらは通常、文書およびリッチインターネットアプリケーション内のグラフィックスに注釈を付けるために、 WAI-ARIA で定義される他のロールと組み合わせて使用される [WAI-ARIA-1.1]。

    graphics-document
    コンテンツの視覚的外観またはレイアウトが意味を伝える document の一種。
    graphics-object
    セマンティックな意味を持つ個別のオブジェクトまたはサブコンポーネントを表す graphics-document の節。 グラフィカルオブジェクトは、それ自体に入れ子になったサブコンポーネントを持つ場合がある。
    graphics-symbol
    単純な意味またはカテゴリを伝えるために使用されるグラフィカルオブジェクトであり、 特定の視覚的外観よりも意味の方が重要であるもの。 これはチャートまたは地図など、より大きな構造化グラフィックのコンポーネントであってもよい。 記号自体はアトミックなオブジェクトであり、 子は提示用である。

    graphics-document (role)

    コンテンツの視覚的外観またはレイアウトが意味を伝える document の一種。

    他の document 型と同様に、 graphics-document ロールは、関連情報を含むページ領域のルート要素に適用される。 そこでは、ユーザーの主要な対話モードは、アプリケーションを制御することではなく、 文書を閲覧することであることが期待される。 このロールを持つ要素は、 文書ファイルのルート要素であっても、 その内部にある入れ子構造のルートであってもよい。

    graphics-document は、次のように類似するロールと区別できる:

    • 他の文書と比べて、graphics-document は、その視覚的（通常は二次元の）表現の セマンティックな重要性によって区別される。 ユーザーエージェントおよび支援技術は、 グラフィックのナビゲーションをサポートするときにこれを考慮に入れる SHOULD。 文書を再フォーマットまたは再スタイルするアクセシビリティ技術は、 コンテンツのセマンティックロールおよび関係と一貫する方法を除き、 graphics-document のレイアウトを変更する SHOULD NOT。

    • img と比べて、graphics-document は、そのコンテンツの構造化された性質によって区別される。 その子要素はセマンティックな意味を持つ場合があり、 リンクまたはその他の対話型ウィジェットを含む場合がある。

    • graphics-object と比べて、 graphics-document は自己完結している。 その意味は周囲のコンテンツから切り離されても保持される。 graphics-document ロールを持つ要素は、 子コンテンツの解釈のための範囲および文脈を定義する。

    一般に、作者は チャート、地図、図、技術図面、青写真および説明用グラフィックスなどの 構造化グラフィックスに対して graphics-document ロールを使用する SHOULD。 ただし、単一の大きなグラフィックが、意味を損なうことなく安全に再配置できる 個別の領域を持つ場合、 それらの各領域は個別の graphics-document である SHOULD。 それらをまとめるために、代替ロール（figure など）を使用してもよい。 1つの graphics-document は、別のものの内部に入れ子にすることもできる。 たとえば、地図に埋め込まれた棒グラフまたはチャートパネルの行列は、 graphics-document のロールを持つべきである。 入れ子の文書はカプセル化を提供する。 内側および外側のグラフィックスのコンポーネント間のナビゲーションは明示的であるべきである。

    注記

    ARIA 1.0 仕様に基づくユーザーエージェントおよび支援技術をサポートするために、 作者は document ロールをフォールバック値として含めたい場合がある。 形式は role="graphics-document document" である。

    将来の仕様は、特別なセマンティック構造を持つ特定の種類のグラフィカル文書について、 より具体的なロールを定義する場合がある。それらのより具体的なロールは、 graphics-document のサブクラスになる。

    graphics-object (role)

    セマンティックな意味を持つ個別のオブジェクトまたはサブコンポーネントを表す graphics-document の節。 グラフィカルオブジェクトは、それ自体に入れ子になったサブコンポーネントを持つ場合がある。

    接続されていないオブジェクトの集合を表すコンテナー要素には、代わりに group または list ロールを与えるべきである。 セマンティックな意味を持たず、祖先によって提供されるセマンティック文脈を変更しない グループ化要素 （たとえば、スタイリングまたはレイアウトのためだけに使用される div または SVG g ） にはロールを与える SHOULD NOT。 ロールがないことは、 none または presentation ロールで明示的に示してもよい。

    graphics-document とは異なり、 graphics-object は自己完結している必要はなく、 ナビゲーションのための新しい文脈を確立しない。 しかし、ユーザーエージェントおよび支援技術は、 ユーザー、特に非視覚ユーザーが、 入れ子になったオブジェクト構造を、 入れ子リストと同様に階層的な方法でナビゲートする方法を提供する SHOULD。

    注記

    ARIA 1.0 仕様に基づくユーザーエージェントおよび支援技術をサポートするために、 作者は group ロールをフォールバック値として含めたい場合がある。 形式は role="graphics-object group" である。

    4.2 グラフィックス用のその他のロール

    ARIA 1.1 [WAI-ARIA-1.1] で定義される次の中核 ARIA ロールも、 グラフィックスに注釈を付けるために関連する:

    • img （画像）は、分割できない全体として知覚される単一のグラフィックを定義する。 graphics-document とは異なり、 画像はナビゲート可能または対話型の子コンテンツを持つことはできない。 graphics-symbol とは異なり、 画像は非視覚ユーザーにその意味を完全に伝えるために、 詳細なテキスト説明を必要とする場合がある。
    • figure は、包含する文書の主要な一部であるが通常の読書順序の外側にある コンテンツ（グラフィックスを含む）のためのコンテナー要素を定義する。 figure は多くの場合、img または graphics-document ロールを持つ 1つ以上の要素を含むが、 テキストキャプション、クレジット、またはその他の関連コンテンツを含む場合もある。

    次の例は、文書内での img、figure、および graphics-document の適切な使用を示す。

    例 6

    HTML 5 文書内では、インラインの SVG がアトミックな img を表す場合がある。 グラフィックがテキストの自然な読書フローの一部を形成する場合、 最初の例のように、このロールで十分である: 

    <p>A repeating SVG gradient is defined using the
    <code>spreadMethod</code> attribute.  
    A value of <code>repeat</code> causes the color stops
    to repeat in the same order, from beginning to end:</p>
<svg class="inline-example" role="img">
    <title>A repeating linear gradient</title>
    <desc>The gradient starts dark, slowly shifting to light, 
        then quickly dark again.  This pattern repeats four
        times left to right, each time brightening across a large
        region and then getting dark within a short space.
    </desc>
    <linearGradient id="repeat" x2="25%" spreadMethod="repeat">
        <stop offset="0"     stop-color="black" />
        <stop offset="0.8" stop-color="white" />
        <stop offset="1"     stop-color="black" />
    </linearGradient>
    <rect width="100%" height="100%" fill="url(#repeat)" />
</svg>

    次の例は同じコンテンツを提供しているが、 ここでは段落テキストの流れから独立して再配置できる figure として構造化されている。 

    <figure id="fig1" role="figure region">
    <svg id="fig1A" class="nested-figure" role="figure"
         aria-labelledby="fig1-caption, fig1A-caption">
        <!-- markup omitted-->
    </svg>
    <svg id="fig1B" class="nested-figure" role="figure"
         aria-labelledby="fig1-caption, fig1B-caption">
        <linearGradient id="reflect" spreadMethod="reflect"
                        xlink:href="#repeat" />
        <text id="fig1B-caption" class="caption" dy="1em"
              >b) spreadMethod="reflect"</text>
        <rect role="img"
              y="25%" width="100%" height="75%" fill="url(#reflect)" >
            <title>A reflecting linear gradient</title>
            <desc>The gradient again starts dark, slowly shifting to light, 
              then quickly dark again.  However, the next repeat shifts
              quickly to the light, then slowly back dark.  
              The original pattern is then repeated, followed by the reflected
              version again.
            </desc>
        </rect>
    </svg>
    <figcaption id="fig1-caption"
                >Figure 1: Repeating SVG gradients</figcaption>
</figure>

    最後に、最後のバージョンは graphics-document を使用してグラフィック上に複雑な注釈を含めており、 それは引き続き figure 要素内に含まれている。 グラフィックの2つの節は graphics-object 要素であり、 注釈は個別のラベルおよび説明を持つ。

        <desc>

            starting dark, slowly shifting to light, 
            then quickly dark again.
            This cycle covers one-quarter of the width of the graphic,
            starting on the left.
            The two gradients differ in their repeat cycles.
        </desc>
        <defs>
            <!-- markup omitted -->
        </defs>
        <g role="graphics-object" aria-labelledby="repeat-label">
            <!-- markup omitted -->
        </g>            
        <g role="graphics-object" aria-labelledby="reflect-label">
            <desc>

                the graphic.
                Each cycle of the gradient alternates direction,
                left-to-right then right-to-left,
                as do the arrows that emphasize the pattern.
            </desc>
            <rect y="50%" width="100%" height="50%" fill="url(#reflect)" />
            </rect>
            <use y="70%" xlink:href="#gradient-vector" >
                <title>1st cycle</title>
                <desc>left-to-right arrow, starting from x="0"</desc>

                 xlink:href="#gradient-vector" >

                <desc>right-to-left arrow, ending at x="25%"</desc>
            </use>
            <use y="70%" x="50%" xlink:href="#gradient-vector" >

    5. 状態およびプロパティ

    WAI-ARIA は、さまざまな オペレーティングシステムプラットフォーム上のプラットフォームアクセシビリティ API をサポートするために使用される、 アクセシビリティの状態およびプロパティの集合を提供する。支援技術は、公開されたユーザーエージェント DOM を通じて、またはプラットフォームアクセシビリティ API へのマッピングを通じて、 この情報にアクセスしてもよい。ロールと組み合わせることで、ユーザーエージェントは、 任意の時点でユーザーに伝えるためのユーザーインターフェイス情報を支援技術に提供できる。 状態またはプロパティの変更は、支援技術への通知をもたらし、 それにより変更が発生したことをユーザーに知らせることができる。
