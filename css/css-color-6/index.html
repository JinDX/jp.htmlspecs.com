1. はじめに

この節は規範的ではありません。

このモジュールは、2つの新しい <color> 関数を追加します: contrast-color() と color-layers() です。

2. 対比色の計算: contrast-color() 関数

contrast-color() 関数記法は、 指定された背景色または前景色に対して 十分なコントラストを持つ色を、 手作業で計算することなく特定します。

その構文は次のとおりです:

<contrast-color()> = contrast-color(
  [ [ <color> && [ tbd-fg | tbd-bg ] && <target-contrast>? ] |
    [ <color> && [ tbd-fg | tbd-bg ] && <target-contrast>, <color># ] ] )
<target-contrast> = <wcag2>
テスト
  • contrast-color-001.html （ライブテスト） （ソース）
  • color-computed-contrast-color-function.html （ライブ テスト） （ソース）
  • color-invalid-contrast-color-function.html （ライブ テスト） （ソース）
  • color-valid-contrast-color-function.html （ライブ テスト） （ソース）

唯一の必須引数は、コントラストの計算対象となる 基準色です。 これは通常（ただし必ずしもそうとは限りません）背景色です。

任意の <color> 値のリストは、 色候補を表します。 候補が指定されていない場合、既定の候補 white, black が使用されます。

tbd-fg および tbd-bg キーワードは、 コントラストを計算する際の 基準色の役割を示します: 候補背景色のリストに対するテキストとして（text）、 または 候補テキスト色のリストに対する背景としてです。 （色の組の有効なコントラストは、その使用方法に依存します。）

基準 色が前景か背景かを指定するキーワードは TBD です。 [Issue #7359]

<target-contrast> 引数は、使用するコントラスト アルゴリズムを指定します。 色候補が指定されていない場合、 <target-contrast> は省略できます。 その場合、UA が選択したアルゴリズムが使用されます。

<target-contrast> 関数記法への引数は、 目標 コントラストレベルを示します。 目標コントラストレベルを持つ複数のコントラストアルゴリズムを指定でき、 その場合、それらの要件は同時に適用されます。 複数のアルゴリズムを指定する contrast-color() 関数は、 各アルゴリズムについて 目標コントラストレベルを指定しなければならず、 そうでない場合は 無効です。

目標 コントラストレベルが省略された場合、 最も高いコントラストを持つ色候補が返されます。 それ以外の場合、 返される色は、そのレベルを満たすか上回る最初の色候補であり、 どれも該当しない場合は white または black が既定で選ばれます。

2.1. 採用される色の決定

2.1.1. 目標コントラストがある場合

候補色は、 リストの最初の色から始めて順にテストされ、 自動的に追加される white, black で終わります。 基準 色に対して、指定されたコントラストレベルを最初に満たす色が採用されます。

どの候補色も合格しない場合、最初に指定されたアルゴリズムに従って、 white または black のうちコントラストが高い方が採用されます。 両者のコントラストが同じ場合は、white が採用されます。 

contrast-color(wheat tbd-bg wcag2(aa), bisque, darkgoldenrod, olive, sienna, darkgreen, maroon)

計算は次のとおりです:

  • wheat (#f5deb3) は背景であり、相対輝度は 0.749 です

  • bisque (#ffe4c4) の相対輝度は 0.807、コントラスト比は 1.073 です

  • darkgoldenrod (#b8860b) の相対 輝度は 0.273、コントラスト比は 2.477 です

  • olive (#808000) の相対輝度は 0.200、 コントラスト比は 3.193 です

  • sienna (#a0522d) の相対輝度は 0.137、コントラスト比は 4.274 です

  • darkgreen (#006400 ) の相対 輝度は 0.091、コントラスト比は 5.662 です

  • maroon (#800000) の相対輝度は 0.046、コントラスト比は 8.333 です

リスト内で、望ましいコントラスト比 4.5 を満たす最初の色は darkgreen です。

contrast-color(wheat tbd-bg wcag2(5.8), bisque, darkgoldenrod, olive, sienna, darkgreen, maroon)

計算は次のとおりです:

  • 相対輝度とコントラスト比は前の例と同じです。

リスト内で、望ましいコントラスト比 5.8 を満たす最初の色は maroon です。

contrast-color(wheat tbd-bg wcag2(AA), bisque, darkgoldenrod, olive)

計算は次のとおりです:

  • 相対輝度とコントラスト比は前の例と同じです。

リスト内のどの色も、望ましいコントラスト比 4.5 を満たしません。 wheat という 基準色に対する white のコントラストは 1.314 であり、4.5（AA）より小さいため、 目標コントラストを満たしません。 したがって、 black が返されます。これは 15.982 > 4.5 のコントラストを持ちます。

2.1.2. 目標コントラストが指定されていない場合

候補色は、 リストの最初の色から順にテストされます。 これまでにテストされたすべての色のうち、基準色に対するコントラストが最も高い色は 暫定的な採用色です。

リストの末尾に到達すると、現在の 暫定的な採用色 が全体の採用色になります。 したがって、リスト内の2つの色が同じコントラストを持つ場合は、 先に出現した方が採用されます。

--myAccent: #b22222;
color: contrast-color(wheat tbd-bg wcag2, tan, sienna, var(--myAccent), #d2691e)

計算は次のとおりです:

  • wheat (#f5deb3) は背景であり、相対 輝度は 0.749 です

  • tan (#d2b48c) の相対輝度は 0.482、 コントラスト比は 1.501 です

  • sienna (#a0522d) の相対輝度は 0.137、コントラスト比は 4.273 です

  • --myAccent (#b22222) の相対輝度は 0.107、コントラスト比は 5.081 です

  • #d2691e の相対輝度は 0.305、 コントラスト比は 2.249 です

最も高いコントラスト比は 5.081 なので、var(--myAccent) が採用されます

foo {
  --bg: hsl(200 50% 80%);
  --purple-in-hsl: hsl(300 100% 25%);
  color: contrast-color(var(--bg) tbd-fg wcag2, hsl(200 83% 23%), purple, var(--purple-in-hsl));
}

計算は次のとおりです:

  • --bg は rgb(179 213 230) であり、 相対輝度は 0.628835 です

  • hsl(200 83% 23%) は rgb(10 75 107) であり、相対輝度は 0.061575、コントラスト比は 6.08409 です

  • purple は rgb(128 0 128) であり、相対 輝度は 0.061487、コントラスト比は 6.08889 です

  • --purple-in-hsl も rgb(128 0 128) であり、相対輝度は 0.061487、コントラスト比は 6.08889 です。これは purple のコントラストより大きくないため、purple が採用されます。

ここで計算された値は、早期に低い精度へ丸めると誤った結果になることを示すため、 有効数字6桁で示されています（0.061575 は 0.061487 に非常に近く、 6.08409 は 6.08889 に非常に近い）。

2.2. コントラストアルゴリズム

現在は WCAG 2.1 のみがサポートされていますが、 このアルゴリズムには問題があることが知られており、 特に暗い背景で顕著です。 このモジュールの将来の改訂では、追加のコントラストアルゴリズムが導入される可能性があります。

2.2.1. WCAG 2.1: wcag2 キーワードと wcag2() 関数

wcag2 キーワードと wcag2() 関数記法は、 [WCAG21] の 輝度コントラストアルゴリズムを使用することを示します。 その構文は次のとおりです: 

<wcag2> =  wcag2 | wcag2([<number> | [ aa | aaa ] && large? ])

その 目標コントラスト レベルのキーワードは次のように対応します:

  • aa は 4.5 に計算されます

  • aa large（または large aa）は 3 に計算されます

  • aaa は 7 に計算されます

  • aaa large（または large aaa）は 4.5 に計算されます

色の組のコントラストを求めるには、 まずそれぞれの CIE 輝度（Y）を D65 白色点に対して相対的に計算します。 次に WCAG 2.1 のコントラストを計算します: contrast = (Yl + 0.05) / (Yd + 0.05) ここで、Yd はその組における暗い方の色の輝度であり、 Yl は明るい方の色の輝度です。 係数 0.05 は、観察時のフレアによる輝度寄与を表します。

基準 色が次であるとします 
color(display-p3 0.38 0.11 0.05)

一方、リスト内の最初の 色候補は次です

yellow

計算は次のとおりです:

  • color(display-p3 0.38 0.11 0.05) は color(xyz 0.06191 0.03568 0.00463) であるため、相対輝度は 0.03568 です

  • yellow は rgb(100% 100% 0%) であり、これは color(xyz 0.76998 0.92781 0.13853) なので、相対輝度は 0.92781 です

  • コントラストは (0.92781 + 0.05) / (0.03568 + 0.05) = 11.4123 です

2.2.2. 半透明色のコントラスト

この節は、入力が不透明色であることを必要とするコントラストアルゴリズムにのみ適用されます。

色の組のコントラストを計算する際:

  1. 背景色はまず、不透明な背景色を求めるため、 単純なアルファ合成（Compositing 1 § 5.1 Simple alpha compositing を参照）を用いて 不透明なキャンバス色の上に合成されます。

    #7358 この計算に適した不透明なキャンバス色を 見つける。 候補は次のとおりです:

    • alpha = 1 の前景色

    • canvas システム色

    • 使用されるアルゴリズムに対してコントラストを最小化するように計算された、その他の色

  2. 次に、前景色はこの不透明な背景色の上に合成され、 不透明な前景色が求められます。

3. 複数の色のレイヤー化: color-layers() 関数

color-layers() 関数記法は、 任意の <blend-mode> の後に、 2つ以上の <color> レイヤーのリストを取ります。

その構文は次のとおりです:

<color-layers()> = color-layers([ <blend-mode>, ]? <color># )

4. <color> 値の解決

4.1. contrast-color() 値の解決

すべての <color> パラメーターが、 それぞれの色空間における対応する色へ解決される場合、 算出値は採用された色であり、 CSS Color 4 § 15. Resolving <color> Values に従って解決されます。 それ以外の場合（関数内で currentColor が使用された場合）、 算出値は、各コンポーネントが 算出された contrast-color() 関数であり、 これにより子要素への継承が保持されます。

例えば、次の値が与えられた場合 
contrast-color(rgb(179 213 230) tbd-bg wcag2(AA), cadetblue, hsl(200 83% 23%))

cadetblue とのコントラストは 1.97 である一方、 hsl(200 83% 23%) とのコントラストは 6.09 であり、 AA の値である 4.5 を超えます。 したがって、解決された hsl 関数の算出値を持ちます: 

rgb(10 75 107)

例えば、現在色の値として rgb(179 213 230) が与えられている場合、 次の値 

contrast-color(currentColor tbd-bg wcag2(AA), hsl(200 83% 23%), purple)

は次の算出値を持ちます

contrast-color(currentColor tbd-bg wcag2(4.5), rgb(10 75 107), rgb(128 0 128))

4.2. color-layers() 値の解決

すべての <color> パラメーターが、 それぞれの色空間における対応する色へ解決される場合、 算出値は合成された色であり、 CSS Color 4 § 15. Resolving <color> Values に従って解決されます。 それ以外の場合（関数内で currentColor が使用された場合）、 算出値は、明示的な <blend-mode> を持ち、 各コンポーネントが 算出された color-layers() 関数であり、 これにより子要素への継承が保持されます。

9. 変更点

9.1. Colors 5 からの変更点

新しい contrast-color() 関数により、 指定された色との WCAG 2.1 コントラストに基づいて、 色のリストから1つを選択できます。 新しい color-layers() 関数は、 指定された blend mode を使用して、2つ以上の色を合成した結果を返します。

    オーサリングツールは、汎用 CSS 文法および このモジュール内の各機能の個別の文法に従って構文的に正しい スタイルシートを書き、かつこのモジュールで説明されるスタイルシートの 他のすべての適合要件を満たす場合、 この仕様に適合します。

    部分的な実装

    作者が前方互換の構文解析規則を利用して フォールバック値を割り当てられるようにするため、CSS レンダラーは、 使用可能なレベルのサポートを持たない任意の at-rule、プロパティ、 プロパティ値、キーワード、およびその他の構文構造を 無効として扱い（かつ適切に 無視し）なければなりません。特に、ユーザーエージェントは、 単一の複数値プロパティ宣言の中で、サポートされていないコンポーネント値を 選択的に無視し、サポートされている値を尊重してはなりません。 いずれかの値が無効と見なされる場合（サポートされていない値はそう扱われる必要があります）、 CSS は宣言全体を無視することを要求します。

    不安定な機能および独自機能の実装

    将来の安定した CSS 機能との衝突を避けるため、 CSSWG は、CSS の不安定な機能および 独自拡張の実装について、 ベストプラクティスに従うことを推奨します。

    非実験的な実装

    仕様が Candidate Recommendation 段階に達すると、 非実験的な実装が可能になり、実装者は 仕様に従って正しく実装されていることを実証できる CR レベルの任意の機能について、接頭辞なしの実装を リリースするべきです。

    CSS の相互運用性を実装間で確立し維持するため、 CSS Working Group は、非実験的な CSS レンダラーに対し、 任意の CSS 機能の接頭辞なし実装をリリースする前に、 実装報告書（および必要に応じて、その実装報告書に使用した テストケース）を W3C に提出することを求めます。W3C に提出された テストケースは、CSS Working Group によるレビューと修正の対象となります。

    テストケースおよび実装報告書の提出に関する詳細情報は、 CSS Working Group のウェブサイト http://www.w3.org/Style/CSS/Test/ で確認できます。 質問は public-css-testsuite@w3.org メーリングリストに送ってください。
