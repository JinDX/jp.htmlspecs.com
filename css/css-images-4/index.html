概要

このモジュールは、<image>型および置換要素に関連するCSSレベル4の機能を含んでいます。 これはCSSレベル2[CSS2]および前バージョン[css-images-3]の機能を含み、拡張しています。 "CSS Images Module Level 3"[css-images-3]と比較した主な拡張は、<image>型に対する複数の追加、例えばimage()記法、element() 記法、円錐グラデーションなどです。

CSSは、構造化文書（HTMLやXMLなど）の画面、紙などでのレンダリングを記述する言語です。

1. 導入

このセクションは規範的ではありません。

このモジュールは、2D画像を表現するための追加方法を導入します。 例えば、色のフォールバック付きURLや、 円錐グラデーション、 または文書内の別の要素のレンダリングなどです。

1.1. 値の定義

この仕様は、CSSプロパティ定義の慣例（[CSS2]）に従い、値定義構文（[CSS-VALUES-3]）を使用しています。 本仕様で定義されていない値型は、CSS Values & Units [CSS-VALUES-3]で定義されています。 他のCSSモジュールとの組み合わせにより、これらの値型の定義が拡張される場合があります。

プロパティごとに定義されている値に加え、 本仕様で定義されているすべてのプロパティは、 CSS全域キーワードもプロパティ値として受け入れます。 可読性のため、明示的に繰り返して記載していません。

2. 2D画像値：<image>型

<image>値型は2D画像を表します。url参照、image記法、 またはgradient記法のいずれかです。 構文は以下の通りです：

<image> = <url> | <image()> | <image-set()> | <cross-fade()> | <element()> | <gradient>

<image>は多くのCSSプロパティで使用できます。 例：background-image、list-style-image、cursorプロパティ（[CSS2]）。 これらのプロパティ値の<url>コンポーネントを置き換えます。

場合によっては画像が無効になることがあります。 例：<url>が有効な画像形式でないリソースを指している場合や、画像の読み込みに失敗した場合です。 無効な画像は、 透明の単色画像として、自然寸法なしで描画されます。 ただし、無効な画像は 文脈によってはエラーハンドリング処理を引き起こすことがあります。 例えば、無効な画像がlist-style-imageの場合、noneとして扱われ、 list-style-typeが代わりに描画されます。[CSS2]

画像が読み込み中の場合、 読み込み中画像となります。読み込み中画像は 無効な画像ではありませんが、類似の挙動となります： 透明の単色画像として、自然寸法なしで描画され、 フォールバック描画が可能な文脈ではそれが表示されますが、 フォールバックリソースの読み込みは行いません。 もしくは、読み込み中画像が既に読み込まれている画像を置き換える場合 （文書やスタイルシートの変更によるなど）でUAがこの情報を追跡している場合、 読み込み中画像の代わりに既に読み込まれた画像を表示し続けることがあります。

部分的に読み込まれた画像（自然寸法が判明しているが画像データが完全に読み込まれていない画像）は、 読み込み中画像として扱うか、部分データで描画された読み込み済み画像として扱う場合があります。 例えば、UAはインターレースGIFの最初のパスのピクセルデータが読み込まれた時点で表示したり、 画像ヘッダー（サイズ情報を含む）が解析された時点で表示したり、 画像全体が読み込まれるまで待ってから使用することもあります。

算出済み <image> 値は、 指定値に対し、<url>、<color>、 <length>が算出されたものとなります。

2.1. 画像ファイル形式

UAは、<image>値から参照される際、 <image>が有効なすべてのプロパティにおいて 少なくとも以下の画像ファイル形式をサポートしなければなりません：

  • PNG（[PNG]で定義）
  • SVG（[SVG11]で定義）、 セキュア静的モードを使用 （[SVG-INTEGRATION]参照）
  • UAがアニメーション対応<image>をサポートする場合、 SVG（[SVG11]で定義）、 セキュアアニメーションモードを使用 （[SVG-INTEGRATION]参照）

UAは他のファイル形式もサポートしてもよいです。

2.4. 解像度・タイプネゴシエーション：image-set()記法

ユーザーのデバイスに最適な画像解像度を提供することは難しい課題です。 理想的には、画像は閲覧されるデバイスと同じ解像度であるべきですが、 ユーザーごとに異なる場合があります。 しかし、どの画像を送信するかの判断には他の要因も影響します。 例えば、ユーザーが低速なモバイル回線の場合、 高解像度画像が読み込まれるまで待つよりも低解像度画像を受け取りたい場合があります。 image-set()関数では、著者はこれらの問題をほとんど無視でき、 複数の解像度の画像を提供するだけで、 UAが状況に応じて最適なものを選択します。

この解決策は解像度がファイルサイズの代理であると仮定していますが、 ベクター画像の多解像度セットや、ベクター画像とラスター画像の混在（例：アイコン）には適切に対応しません。 例えば、高解像度にはベクターを使い、 低解像度にはピクセル最適化されたビットマップ、 低帯域には同じベクター（高解像度でもファイルサイズが小さいため）を使うなど。

image-set()の構文は次の通りです：

<image-set()> = image-set( <image-set-option># )
<image-set-option> = [ <image> | <string> ]
                     [ <resolution> || type(<string>) ]

HTMLのpicture要素の機能と同様に、"w"と"h"の寸法を追加すべきです。

<string>はimage-set()内で <url>を表します。

image-set()関数は自身の中に直接または間接的に （他の<image>型の引数として） ネストできません。

<image-set-option>は、image-set()関数が表す可能性のある画像を定義します。3つの部分から構成されます：

  • 画像参照（必須）。 これはURLでも、 CSS生成画像でも（例：linear-gradient()など）可能です。

  • <resolution>（任意）。 UAがどの<image-set-option>を選択するか判断する際に使われます。 画像参照がラスター画像の場合は、その画像の自然解像度も指定し、 他のデータソースによる自然解像度を上書きします。

    指定がなければ、 選択時は1xとして扱われます。 また、画像の自然解像度も1xがデフォルトですが、 他のデータソースが自然解像度を提供する場合はそちらが優先されます。

  • type( <string> )関数（任意）は、 <string>で画像のMIMEタイプを指定します。

    <string>が 有効なMIMEタイプ文字列として解析された際、 無効だったり、サポートされていない画像形式だった場合、 <image-set-option>は有効なオプションを定義しません。 （このことはimage-set()関数自体の有効性には影響しません。）

    画像自体には影響しません。 例えば、<image-set-option>でurl("picture.png") 1x type("image/jpeg") のように指定しても有効であり、選択されればPNG画像が表示されます（JPEG指定でも）。

    指定がなければ、<image-set-option>には影響しません。

image-set()関数は1つ以上の<image-set-option>のリストを含み、 その中から1つだけ選択してその画像を表します：

  1. まず、<image-set-option>のうち type()値で 未知または非対応MIMEタイプを指定しているものをリストから除外します。

  2. 次に、リスト内で前のオプションと同じ<resolution>を持つ <image-set-option>を除外します。

  3. 残った<image-set-option>の中から、 UA固有の基準（ディスプレイ解像度や通信速度など）に基づいてどれを読み込むか選択します。

  4. その後、image-set()関数は選択された <image>（および<image-set-option>）を表します。

UAは、<image-set-option>の選択基準がページの寿命中に十分変化した場合、 その都度image-set()で使うオプションを変更してもよいです。

この例はimage-set()を使い3種類の画像を提供する方法を示します： 「通常」バージョン、 「高解像度」バージョン、 そして高品質印刷用の超高解像度バージョン （プリンタは非常に高い解像度を持つことがあるため）： 
background-image: image-set( "foo.png" 1x,
                             "foo-2x.png" 2x,
                             "foo-print.png" 600dpi );
この例はtype()関数の使用例です。 新しい高品質フォーマットと古くから広く使われているフォーマットの両方で 同じ画像の複数バージョンを提供します： 
background-image: image-set( "foo.avif" type("image/avif"),
                             "foo.jpg" type("image/jpeg") );

AVIF画像が先頭にある点に注意してください。 両画像の解像度が同じ（指定がないため1xでデフォルト）なので、 JPEG画像は2番目に来ているため、 AVIF対応UAでは自動的に除外されます。

古いUAでは、 AVIF画像は無視され （UAが"image/avif"ファイル非対応と認識するため）、 代わりにJPEGが選ばれます。

ラスター画像はベクター画像やCSS生成画像と混在できます。

例えばこのコードでは、 微細なディテールを持つ高解像度画像は対応するディスプレイで使い、 低解像度環境では普通のCSSlinear-gradient()が使われます：

background-image: image-set( linear-gradient(cornflowerblue, white) 1x,
                             url("detailed-gradient.png") 3x );

2.5. 画像のフォールバックと注釈：image()記法

image() 関数では、著者は以下のことができます：

  • media fragmentsを使って画像の一部を切り抜く

  • 単色を画像として使う

  • 指定したURLの画像がダウンロードまたはデコードできない場合に、単色画像にフォールバックする

  • 画像のメタデータに指定された向きを自動的に尊重する

image() 記法の定義は次の通りです：

image() = image( <image-tags>? [ <image-src>? , <color>? ]! )
<image-tags> = [ ltr | rtl ]
<image-src> = [ <url> | <string> ]

<string>はimage()で使用される場合、<url>を表します。 CSSでのURLと同様に、 相対URLは絶対URLに解決されます （Values & Units [CSS-VALUES-3]で記述されている通り）、 指定されたimage()値が算出される時に解決されます。

画像にEXIFなどのメタデータで向き（オリエンテーション）が指定されている場合、 UAはその画像をメタデータの指示通り正しく回転または反転して表示しなければなりません。

2.5.1. 画像フォールバック

URLと<color> の両方がimage()で指定されている場合、 URLが無効な画像または読み込み中画像を表す時は、 image()関数はURLが全く指定されていないかのように描画されます。 § 2.5.3 単色画像で指定された通りの単色画像が生成されます。

URLのみが指定されている場合（<color>なし） かつそれが無効な画像または読み込み中画像を表す時は、 image()関数も同様に扱われます。

フォールバックカラーを使えば、画像の読み込みに失敗してもテキストが読みやすいままにできます。 例えば、次のレガシーコードは画像が矩形かつ透明でなければ問題なく動作します： 
body      { color: black; background: white; }
p.special { color: white; background: url("dark.png") black; }

画像が読み込まれなかった場合でも、 背景色が残るので白いテキストが読みやすいままです。 ただし、画像に透明部分がある場合、 黒色がその下に見えてしまい、 それはおそらく望ましくありません。 image() 関数はこの点を改善します： 

body      { color: black; background: white; }
p.special { color: white; background: image("dark.png", black); }

今度は、画像が読み込まれれば黒は全く表示されません。 何らかの理由で画像が失敗した場合は黒が現れ、 白いテキストが白い背景の上に表示されることを防いでくれます。

2.5.2. 画像フラグメント

image()で指定されたURLがリソースの一部（例：メディアフラグメント識別子の利用）を表す場合、その部分はコンテキストから切り出され、独立した画像として利用されます。

例えば、次の画像とCSSがあるとします： 

background-image: image('sprites.svg#xywh=40,0,20,20')

...この場合、要素の背景は画像の(40px,0px)から始まり、幅・高さ20pxの部分、すなわち1/4だけ塗られた円になります。

CSSの将来互換なパース規則を活用して画像スライスのフォールバックを提供できるよう、image()記法をサポートする実装は、画像指定時に必ずxywh=#,#,#,#形式のメディアフラグメント識別子をサポートしなければなりません（image()による指定）。[MEDIA-FRAGS]

画像フラグメントはurl()記法でも利用できます。 ただし、メディアフラグメント記法を理解できないレガシーUAはフラグメントを無視し、画像全体を表示します。

image()記法はUAにメディアフラグメントのサポートを要求するため、画像フラグメントURLを使う際にCSSの将来互換なパース規則でフォールバックを提供できます： 

background-image: url('swirl.png'); /* 古いUA */
background-image: image('sprites.png#xywh=10,30,60,20'); /* 新しいUA */

URLが実装で理解できないフラグメント識別子構文や、その画像タイプに有効でない構文を使っている場合、そのURLは無効な画像として扱われます。

注: このエラー処理はimage()限定であり、URLの定義には含まれていません（レガシー互換のため）。

2.5.3. 単色画像

image() 関数が<color> のみ（URL指定なし）で指定された場合、その色の単色画像として扱われ、自然寸法は持ちません。

例えば、背景画像を「色付け」する簡単な方法として、他の画像の上に半透明色を重ねて表示できます： 
background-image: image(rgba(0,0,255,.5)), url("bg-image.png");

background-colorではこれはできません。生成される単色は必ず全ての背景画像の下層になるためです。

2.5.4. 双方向対応画像

任意の<image-src>sを記述する前に、作者は画像の方向性を指定できます。これはHTML要素にdir属性を追加するのと似ています。 方向性付き画像が、逆方向のdirectionを持つ要素に使用された場合、画像はインライン方向で反転しなければなりません （例えばインライン方向がX軸なら、scaleX(-1)で変形されたかのように）。

注: この宣言がなければ、画像は方向性を持たず、周囲要素のdirectionには影響されません。

リストは、コンテンツ側を指す矢印画像を箇条書きに使うことがあります。 リストがLTR・RTL両方のテキストを含む場合、箇条書きは左または右に来ます。 一方の向きに合わせて作られた画像は、他方では逆に外側を向いてしまいます。 これを修正するには、次のようなコードを使います： 
<ul style="list-style-image: image(ltr 'arrow.png');">
  <li dir='ltr'>My bullet is on the left!</li>
  <li dir='rtl'>MY BULLET IS ON THE RIGHT!</li>

2.6. 画像の合成：cross-fade()記法

画像の遷移時、CSSは開始画像・終了画像を合成した中間画像を明示的に参照する方法を必要とします。 これはcross-fade()関数によって実現されます。 この関数は合成する2つの画像と、遷移の進行度を指定します。

注: 作者はcross-fade() 関数を単色画像で色付けしたり、放射グラデーションと合成して特定領域を強調表示するなど、様々な画像操作にも利用できます。

cross-fade()の構文は以下の通りです：

cross-fade() = cross-fade( <cf-image># )
<cf-image> = <percentage [0,100]>? && [ <image> | <color> ]

この関数は、1つ以上の画像を合成して生成された画像を表します。

<percentage>は、各画像を他の画像とブレンドした際にどれだけ残すかを示します。 <percentage>は0%から100%までの値でなければならず、 それ以外の値は無効です。

パーセンテージが省略されている場合、 指定されたすべてのパーセンテージを合計し100%から差し引きます。 結果が0%未満の場合は0%に切り捨てられ、 省略された画像に均等に割り当てられます（算出値時）。

この処理は算出値には反映されませんが、 すべての引数の合計が100%を超える場合、 サイズ・描画の詳細では合計が100%になるように再スケールされます。

逆に合計が100%未満の場合は、 サイズ・描画の詳細で追加のtransparent引数があるかのように扱われ、 合計が100%になるまで残りの値が割り当てられます。

<color> が指定された場合、 これは「自動」寸法の単色画像となります （結果画像のサイズ計算には全く影響しません。詳細は下記のサイズ指定を参照してください）。

2.6.1. cross-fade() サイズ指定

cross-fade()で表される画像の寸法は、 関数の<image>引数の寸法の加重平均となります。 <color>引数は影響しません。 計算方法は以下の通りです：

cross-fade()の自然寸法を決定する手順：

  1. imagesを空のリストとする。

  2. cross-fade()関数の各argumentで <image>値を持つものについて：

    1. itemを幅・高さ・パーセンテージからなるタプルとする。

    2. オブジェクトサイズネゴシエーションアルゴリズムを <image>に対して cross-fade()が現れる文脈に応じて実行し、 itemの幅・高さを結果の具体的オブジェクトサイズの値に設定する。

    3. itemのパーセンテージをargumentのパーセンテージ値に設定する。

  3. imagesが空の場合、 自然寸法なしで返す。

  4. percentage sumをimages内のitemsの すべてのパーセンテージの合計とする。

  5. images内の各itemについて、 itemのパーセンテージをpercentage sumで割り、 その値をitemのパーセンテージに設定する。

    アサート：images内のパーセンテージの合計が100%となる。

  6. final widthとfinal heightを0pxとする。

  7. images内の各itemについて、 itemの幅×パーセンテージをfinal widthに加算、 itemの高さ×パーセンテージをfinal heightに加算。

  8. final widthを自然幅として、 final heightを自然高さとして返す。

2.6.2. cross-fade() 描画

cross-fade()で表される画像は、 引数の加重平均で計算されます。計算方法は以下の通りです：

cross-fade()の外観を決定する手順：

  1. imagesを空のリストとする。

  2. sizeを幅・高さからなるタプルとし、 cross-fade()関数の具体的オブジェクトサイズ（ cross-fade()の自然寸法を利用）で初期化する。

  3. cross-fade()関数の各argumentについて：

    1. itemを画像とパーセンテージからなるタプルとする。

    2. argumentが<image>の場合、 sizeの幅・高さにリスケールしitemの画像に設定する。 それ以外は<color>の場合 itemの画像にsize寸法の単色画像を設定する。

    3. itemのパーセンテージをargumentのパーセンテージ値に設定する。

  4. percentage sumをimages内のitemsの すべてのパーセンテージの合計とする。

  5. percentage sumが100%未満の場合、 size寸法の透明黒単色画像と残りパーセンテージを持つタプルをimagesに追加する。 逆に100%を超える場合は、 images内の各itemについて パーセンテージをpercentage sumで割り、値を更新する。

  6. final imageをsize寸法の画像とし、 各ピクセルはimages内の各item画像の対応するピクセルの加重線形平均値とする。 （色チャネルとアルファチャネル両方を平均する。） この計算では各ピクセルの色はプリマルチプライドsRGBでなければならない。

    この操作の詳細

    これはN-way Porter-Duff dissolve操作の適用です。 Wikipediaによるとdissolveは確率操作であり、 結果ピクセルは各元画像の対応ピクセルから重みで独立してランダム選択されます。 しかしピクセルが十分小さくなれば、プリマルチプライド色空間での平均になります。

    特に `cross-fade(white 50%, transparent 50%)` は 半透明の白単色画像を生成します。 （プリマルチプライドでない空間で不透明白と透明黒を平均すると半透明グレーになるが、 それとは異なります。）

    プリマルチプライド変換は精度損失を伴い、 グラフィックライブラリがこの操作をネイティブでサポートするとは限りませんが、 所定の効果が得られるならどんな方法でも構いません。

    例えばピクセルごとのアルファでパーセンテージを再配分し、 非プリマルチプライド空間で色チャネル平均を取ることもできます。 例：cross-fade(rgb(255 0 0 / 1) 40%, rgb(0 255 0 / .5) 20%, rgb(0 0 255 / 0) 40%)を描画する場合、 1/.5/0アルファで再配分すると40%/10%/0%（正規化で80%/20%/0%）になり、 そのまま生の色チャネル値の平均を取ることでrgb(204 51 0 / .5)画像になります。 （アルファチャネルの平均は元のパーセンテージで計算します。）

  7. final imageを返す。

2.6.3. 複雑なcross-fade()の簡素化

WG決議により、画像の「等価性」概念を定義し、 算出値時に「同じ」画像を合成してパーセンテージを合算する。

WG決議により、 算出値時に直接ネストされたcross-fade()を単純化し、パーセンテージを展開・フラット化する。 例：cross-fade(A 10%, cross-fade(B 30%, C 70%) 90%)はcross-fade(A 10%, B 27%, C 63%)になる。

2.7.1. ペイントソース

ホスト言語によっては、要素がpaint source（描画元）を提供する場合があります。 paint sourceは固有の見た目を持ち、レイアウトや描画なしで具体的オブジェクトサイズを取得できるため、 描画されていなくても画像として利用できます。

HTMLでは、img、 video、 canvas 要素がpaint sourceを提供します。

SVGでは、paint serverを提供する要素がpaint sourceとなります。注: SVG1.1では、 <linearGradient>、<radialGradient>、 <pattern> 要素がpaint sourceを提供します。 これらは仕様で記載されたとおり描画され、 座標系は次のように定義されます：

objectBoundingBox
座標系の原点は描画先の具体的オブジェクトサイズの矩形の左上で、 幅・高さも同じです。 1ユーザー座標は、その具体的オブジェクトサイズの幅・高さとなります。
userSpaceOnUse
座標系の原点は描画先の具体的オブジェクトサイズの矩形の左上で、 幅・高さも同じです。ユーザー座標はCSSのpx単位と同じ大きさです。

2.7.2. 文書外ソースの利用: ElementSources インターフェース

element()関数は通常は文書内の要素を選択しますが、 paint source（描画元）を提供する要素は必ずしも文書内にある必要はありません。 例えば、HTMLのcanvas要素は スクリプトのみで作成・管理・描画でき、 文書に挿入する必要がありません。

必要なのは、その要素への参照方法だけです。 IDセレクタでは文書外の要素を選択できませんが、 elementSourcesマップオブジェクトが提供されます。 

partial namespace CSS {
  [SameObject] readonly attribute any elementSources;
};

elementSourcesマップ内でキーが文字列、 値がpaint source（描画元）を提供するオブジェクトの場合、 それらはelement()関数から利用可能です。

element()が<id-selector>を使った場合、 ID値（先頭の#を除く）をまずelementSourcesマップで検索します：

  • 見つかった場合、そのオブジェクトがpaint sourceを提供すれば element()関数はそのpaint sourceを表します。
  • 見つかったがpaint sourceを提供しない場合は、 element()関数は無効な画像を表します。
  • IDがマップに見つからない場合は、通常通り文書内を検索します。

このIDセレクタの再利用はMozの挙動と一致します。 文法先頭にcustom-identを置くと構文空間を多く消費するため避けていますが、 別案として言語定義キーワード＋<custom-ident> のように始めてもよいかもしれません。 （例：element(external fancy)など） 名前の提案歓迎です。

例えば外部canvasで凝ったアニメーション背景を作ることもできます： 
<script>
var bg = document.createElement('canvas');
bg.height = 200;
bg.width = 1000;
drawFancyBackground(bg);
CSS.elementSources.set('fancy', bg);
</script>
<style>
h1 {
  background-image: element(#fancy);
}
</style>

"fancy" キャンバスに描画・アニメーションすることで、 すべてのH1要素の背景が自動的に同期して更新されます。

elementSourcesマップは 文書より先にIDセレクタで参照されるため、 文書内に#fancyがあっても 背景は必ずelementSourcesの値から取得されます。

2.7.3. 循環検出

element()関数は、例えば自分自身を背景画像として使うなど、意味のない循環関係を生じる場合があります。 しかし、依存グラフを管理し、一般的な循環検出アルゴリズムを使用することで、これらの関係は容易かつ確実に検出・解決できます。

依存グラフは以下のような辺を持ちます：

  • すべての要素はその子要素に依存する
  • ある要素Aがelement()関数を使って要素Bを参照するプロパティを持つ場合、AはBに依存する
  • ホスト言語が他の要素の描画を参照する方法を定義している場合、参照する要素は参照される要素に依存する。 例えば、SVGでは<use>要素は参照した要素に依存します。

グラフに循環が存在する場合、循環に関与したすべてのelement()関数は無効な画像となります。

3. グラデーション

グラデーションは、1つの色から別の色へ滑らかに変化する画像です。 これは背景画像やボタンなどの微妙な陰影付けなど、さまざまな用途で一般的に使われます。 このセクションで説明されているグラデーション関数により、作者は簡潔な構文でそのような画像を指定でき、 UAはページ描画時に自動的に画像を生成します。 <gradient>の構文は以下の通りです： 

<gradient> = [
  <linear-gradient()> | <repeating-linear-gradient()> |
  <radial-gradient()> | <repeating-radial-gradient()> |
  <conic-gradient()>  | <repeating-conic-gradient()> ]

この仕様で定義されている他の<image>型と同様に、 グラデーションは画像を受け付ける任意のプロパティで利用可能です。 例：

  • background: linear-gradient(white, gray);
  • list-style-image: radial-gradient(circle, #006, #00a 90%, #0000af 100%, white 100%)

グラデーションは、具体的オブジェクトサイズの寸法を持つボックスに描画されます。 このボックスはグラデーションボックスと呼ばれます。 ただし、グラデーション自体には自然寸法はありません。

例えば、グラデーションを背景として使用した場合、 デフォルトではグラデーションは要素のパディングボックスのサイズのグラデーションボックスに描画されます。 background-sizeが100px 200pxなど明示的に指定されている場合、 グラデーションボックスは幅100px、高さ200pxになります。 同様に、グラデーションをlist-style-imageとして使用した場合、 ボックスは1em四方となり、それがそのプロパティのデフォルトオブジェクトサイズです。

グラデーションは、開始点と終了点、グラデーション線（グラデーションの種類によっては、線、光線、あるいは螺旋）を定義し、 そこに色を指定します。 色はその線上で滑らかに補間され、各グラデーション型がグラデーション線の色を使って実際のグラデーションの生成方法を定めています。

3.1. 線形グラデーション：linear-gradient() 記法

このレベルでは、<color-interpolation-method>引数が追加されました。 linear-gradient()やrepeating-linear-gradient()に指定でき、 グラデーション線上の色補間時に使用する色空間と経路を示します。 詳細はCSS Color 4 §  12. 色補間を参照してください。 

linear-gradient() = linear-gradient(
  [ [ <angle> | to <side-or-corner> ] || <color-interpolation-method> ]? ,
  <color-stop-list>
)
<side-or-corner> = [left | right] || [top | bottom]

3.1.1. 色空間による補間効果：例

このセクションは規範的ではありません。

色空間による補間効果は大きく異なる場合があります。

この例では、同じ色のペア #f01 と #081 の線形グラデーションが 3種類の色空間で描画されています。 中央のグラデーションはガンマ補正されたsRGBを使用しており、 CSS Images 3で唯一選択できた方法でした。 結果は中央部分が明らかに暗すぎます。 上部のグラデーションはCIE Labを使用しており、 より知覚的に均一な結果を得られます。 下部のグラデーションはOklabを使っており、 この場合CIE Labとほぼ同じ結果になります。
  • linear-gradient(in lab to right, #F01, #081)
  • linear-gradient(in srgb to right, #F01, #081)
  • linear-gradient(in Oklab to right, #F01, #081)

この例では、同じ2色（whiteと#01E）の線形グラデーションを3つの異なる色空間で描画しています。 真ん中のグラデーションはガンマ補正されたsRGBを使用しており、中央部分がやはり暗すぎて、やや彩度が落ち、少し紫がかっています。 上のグラデーションはCIE Labを使用し、暗すぎる中央部分を回避しますが、紫の色味が強くなります。 下のグラデーションはOklabを使用しており、知覚的により均一で、紫がかりがまったくありません。
  • linear-gradient(in lab to right, white, #01E)
  • linear-gradient(in srgb to right, white, #01E)
  • linear-gradient(in Oklab to right, white, #01E)

この例では、同じ2色（#44Cと#795）の線形グラデーションを3つの異なる色空間で描画しています。 これにより、CIE Labの色相の非線形性がすべての青系の色に影響することが分かります。これは、彩度の高い主青から白へのグラデーションだけでなく他の場合にも当てはまります。 真ん中のグラデーションはガンマ補正されたsRGBを使用しており、中央部分がやはり暗すぎて、少し紫がかっています。 上のグラデーションはCIE Labを使用し、暗すぎる中央部分を回避しますが、紫の色味が強くなります。 下のグラデーションはOklabを使用しており、やはり知覚的により均一で、紫がかりがまったくありません。
  • linear-gradient(in lab to right, #44C, #795)
  • linear-gradient(in srgb to right, #44C, #795)
  • linear-gradient(in Oklab to right, #44C, #795)

グラデーション補間において、矩形色空間ではなく極座標色空間を選ぶことで、 カラーストップの色相が大きく異なる場合でも脱彩度を避けることができます。 極座標色空間で補間すると、彩度（クロマ）を維持しやすくなりますが、 中間色がガマット外になることもあり、 その場合はガマットマッピングによってガマット内に戻されます。

この例では、同じ2色（#A37と#595）の線形グラデーションを、2つの極座標空間を含む5つの異なる色空間で描画しています。 上から順に、CIE LCH、CIE Lab、sRGB、Oklab、Oklchです。

矩形空間では中間色が灰色がかりますが、極座標空間の中間色は曲線的かつ彩度維持された経路を辿ります。

  • linear-gradient(in lch to right, #A37, #595)
  • linear-gradient(in lab to right, #A37, #595)
  • linear-gradient(in srgb to right, #A37, #595)
  • linear-gradient(in Oklab to right, #A37, #595)
  • linear-gradient(in oklch to right, #A37, #595)

3.2. 放射状グラデーション：radial-gradient() 記法

このレベルでは、<color-interpolation-method>引数が radial-gradient()およびrepeating-radial-gradient()に追加されました。 これはグラデーションライン上で色を補間する際に使用する色空間と補間経路を指定します。 詳細はCSS Color 4 § 12. 色補間を参照してください。 

radial-gradient() = radial-gradient(
  [ [ [ <rg-ending-shape> || <rg-size> ]? [ at <position> ]? ] || <color-interpolation-method>]? ,
  <color-stop-list>
)
この例では、同じ2色 color(display-p3 0.918 0.2 0.161) と #081 の放射状グラデーションを3つの異なる色空間で描画しています。 なお、カラーストップはすべて同じ色空間である必要はありません。 真ん中のグラデーションはガンマ補正されたsRGBを使用しており、中央部分が明らかに暗すぎます。 上のグラデーションはCIE Labを使用しており、より知覚的に均一な結果となります。 下のグラデーションはOklabを使用しており、ほぼCIE Labと同じ結果になります。
  • radial-gradient(in lab farthest-side at left bottom, color(display-p3 0.918 0.2 0.161), #081)
  • radial-gradient(in srgb farthest-side at left bottom, color(display-p3 0.918 0.2 0.161), #081)
  • radial-gradient(in Oklab farthest-side at left bottom, color(display-p3 0.918 0.2 0.161), #081)

3.3. 円錐グラデーション：conic-gradient() 記法

円錐グラデーションは、円の中心を指定することから始まります。 放射状グラデーションと似ていますが、円錐グラデーションのカラーストップは中心から伸びる線上ではなく、 円周上に配置されるため、中心を回転するにつれて色が滑らかに変化します。 （放射状グラデーションは中心から外側へ進むにつれて色が変化します。）

円錐グラデーションは、回転角度、グラデーションの中心、そしてカラーストップのリストを指定することで定義されます。 線形や放射状グラデーションが<length>でカラーストップの位置を指定するのに対し、 円錐グラデーションのカラーストップは<angle>で指定します。 中心から全方向に線（レイ）が引かれ、それぞれの線の色はグラデーションライン上で交差する位置の色となります。

注: このグラデーションは「conic」または「conical」と呼ばれることがあります。 これは、カラーストップが片側で大きく明るくされている場合、上から見た円錐のようなパターンができるためです。 文脈によっては「angle」グラデーションと呼ぶこともあり、 線の回転角度を変化させて生成されるためです。

この例では、conic-gradient(at 25% 30%, white, black 60%)がどのように描画されるかを視覚的に示しています。 カラーストップ位置は必ず角度になるため、at 25% 30%の効果はグラデーションの2次元平行移動のみであり、 グラデーションの描画方法自体には影響しません。

3.3.1. conic-gradient()構文

円錐グラデーションの構文は以下の通りです：

conic-gradient() = conic-gradient(
  [ [ [ from <angle> ]? [ at <position> ]? ] || <color-interpolation-method> ]? ,
  <angular-color-stop-list>
)

各引数の定義は以下の通りです：

<angle>
グラデーション全体をこの角度分回転します。 省略した場合、デフォルトは0degです。 単位識別子は<angle>がゼロの場合省略可能です。
<position>
グラデーションの中心を決定します。 <position>型（background-positionでも使用）については [CSS-VALUES-3]で定義されており、 center-pointがオブジェクト領域、gradient boxがポジショニング領域として解決されます。 省略した場合、デフォルトはcenterになります。

通常、円錐グラデーションでは 0degで急激な色の変化が起きるのは望ましくなく、最初と最後のカラーストップを同じ色にすることで回避されます。 これを自動で実現するキーワードがあれば便利かもしれません。

グラデーションのクリッピング用の半径（内半径・外半径）は有用でしょうか？ 有用なら、カラーストップ位置に長さ指定もサポートできそうです（専用半径があるため）。

3.3.2. カラーストップの配置

カラーストップは、グラデーション中心周りを円状に曲がる グラデーションライン上に配置されます。 0%と100%の位置はどちらも0degです。 線形グラデーションと同様に、0degはページの上方向を指し、 角度が増えると円周上を時計回りに進んでいきます。

注: グラデーションラインを螺旋状（スパイラル）と考えると分かりやすいかもしれません。 実際に描画されるのは0degから360degまでの区間だけなので、範囲外の角度で「重なり」が発生する混乱も避けられます。

カラーストップは0%より前や100%より後の位置にも配置できますが、 それらの領域は直接描画には使われません。 しかし、その位置のカラーストップは描画領域内のカラーストップの色に補間や繰り返しを通じて影響を与えます（繰り返しグラデーション参照）。 例えば、conic-gradient(red -50%, yellow 150%)は 0deg（#f50）で赤みがかったオレンジ色から始まり、 360deg（#fa0）でオレンジがかった黄色に変化する円錐グラデーションを生成します。

グラデーションの任意の点の色は、まずグラデーション中心からその点を通る一意なレイを見つけ、 そのレイがグラデーションラインと交差する位置での色がその点の色となります。

3.3.3. 円錐グラデーションの例

以下のconic-gradient()の例は、特に指定がない限り、幅300px・高さ200pxのボックスに適用されるものとします。

下記は同じ基本的な円錐グラデーションを指定する様々な方法です： 
background: conic-gradient(#f06, gold);
background: conic-gradient(at 50% 50%, #f06, gold);
background: conic-gradient(from 0deg, #f06, gold);
background: conic-gradient(from 0deg at center, #f06, gold);
background: conic-gradient(#f06 0%, gold 100%);
background: conic-gradient(#f06 0deg, gold 1turn);

下記は同じ基本的な円錐グラデーションを指定する様々な方法です。 これは、角度が[0deg, 360deg)の範囲外のカラーストップが直接描画されなくても、 描画されたグラデーション部分の色に影響を与えることを示しています。 
background: conic-gradient(white -50%, black 150%);
background: conic-gradient(white -180deg, black 540deg);
background: conic-gradient(hsl(0,0%,75%), hsl(0,0%,25%));

下記は同じ回転した円錐グラデーションを指定する2つの異なる方法です。1つは回転角あり、1つはなし： 
background: conic-gradient(from 45deg, white, black, white);
background: conic-gradient(hsl(0,0%,75%), white 45deg, black 225deg, hsl(0,0%,75%));

すべてのカラーストップを回転角分だけオフセットするだけでは、全く異なるグラデーションとなり、意図したものにはなりません：

background: conic-gradient(white 45deg, black 225deg, white 405deg);

円錐グラデーションに放射状グラデーションを重ねて、 色相・彩度ホイールを描画した例： 
background: radial-gradient(gray, transparent),
            conic-gradient(red, magenta, blue, aqua, lime, yellow, red);
border-radius: 50%;
width: 200px; height: 200px;

円錐グラデーションを使ってシンプルな円グラフを描画した例。 0degのカラーストップの位置は、直前のカラーストップの位置と同じに修正されます。 その結果、異なる色のカラーストップ間には極小（不可視）の遷移となり、実質的に単色のセグメントが生成されます。 
background: conic-gradient(yellowgreen 40%, gold 0deg 75%, #f06 0deg);
border-radius: 50%;
width: 200px; height: 200px;

3.4. 繰り返しグラデーション：repeating-linear-gradient()、repeating-radial-gradient()、およびrepeating-conic-gradient()記法

linear-gradient()、radial-gradient()、conic-gradient()に加え、 この仕様ではrepeating-linear-gradient()、repeating-radial-gradient()、 repeating-conic-gradient()値も定義しています。 これらの記法は、先に定義したそれぞれの非繰り返し型と同じ値を取り、同じように解釈されます。

基本的な繰り返し円錐グラデーション: 
background: repeating-conic-gradient(gold, #f06 20deg);

カラーストップを急激に切り替えて繰り返すことで、スターバースト型の背景が作れます: 
background: repeating-conic-gradient(
                hsla(0,0%,100%,.2) 0deg 15deg,
                hsla(0,0%,100%,0) 0deg 30deg
            ) #0ac;

急激な切り替えの繰り返しカラーストップを使って市松模様を作成する例: 
background: repeating-conic-gradient(black 0deg 25%, white 0deg 50%);
background-size: 60px 60px;

同じ市松模様は非繰り返しの円錐グラデーションでも作成できます:

background: conic-gradient(black 25%, white 0deg 50%, black 0deg 75%, white 0deg);
background-size: 60px 60px;

3.5. グラデーション色の定義

グラデーションの色はカラーストップ（<color> と、グラデーションライン上の対応する位置）、 およびカラー遷移ヒント（2つのカラーストップ間の色遷移の中間点を表す位置） により指定し、 それらがグラデーションライン上に配置され、 ライン上の各点の色を定義します。 （各グラデーション関数が グラデーションラインの形状と長さ、 および開始点と終了点を定義します；詳細は上記参照。）

グラデーションフィールド全体の色は、グラデーション関数で指定されたように グラデーションライン上の特定の点に結び付けることで決定されます。 UAはグラデーション色を若干「ディザ」する（個々のピクセルをグラデーションライン上の近傍色でランダムに交互表示する） ことで、より滑らかなグラデーションにすることもできます。

3.5.1. カラーストップリスト

カラーストップと遷移ヒントは、 カラーストップリストとして指定されます。 これは、2つ以上のカラーストップに任意の遷移ヒントを挟んだリストです： 

<color-stop-list> =
  <linear-color-stop> , [ <linear-color-hint>? , <linear-color-stop> ]#
<linear-color-stop> = <color> && <color-stop-length>?
<linear-color-hint> = <length-percentage>
<color-stop-length> = <length-percentage>{1,2}

<angular-color-stop-list> =
  <angular-color-stop> , [ <angular-color-hint>? , <angular-color-stop> ]#
<angular-color-stop> = <color> && <color-stop-angle>?
<angular-color-hint> = <angle-percentage>
<color-stop-angle> = <angle-percentage>{1,2}

<color-stop> = <color-stop-length> | <color-stop-angle>
注意：<color-stop-list>と<angular-color-stop-list>は、構造が完全に同一ですが、 停止位置やヒントの指定に<length>を使うか <angle>を使うかだけが異なります。

鉄道図（railroad diagram）で表現すると、両者とも次のパターンになります：

カラーストップに位置が2つ指定されている場合、 それは同じ色で2つのカラーストップ（それぞれの位置に1つずつ）を指定したのと同等です。2箇所に位置を指定することで、色を2回繰り返さずともグラデーション内に単色の「ストライプ」を簡単に作れます。

パーセンテージは、グラデーションラインの始点から終点までの長さを基準に解決されます。 0%は始点、100%は終点となります。 長さ指定はグラデーションライン上で始点から終点方向に測定します。

カラーストップやトランジションヒントの位置は 通常は始点から終点の間に置かれますが、 必ずしもそうである必要はありません。 グラデーションラインは両方向に無限に伸びており、 任意の位置を指定できます。

カラーストップの位置が省略された場合、 自動的に位置が割り当てられます。 カラーストップリストの最初や最後のカラーストップには グラデーションラインの始点または終点が割り当てられます（それぞれ）。 それ以外の場合は、両隣のストップの中間位置が割り当てられます。 連続して位置指定がないストップがある場合は、 両隣の位置付きストップの間を等間隔に分割して配置されます。 詳細は§ 3.5.3 カラーストップの「修正」参照。

3.5.2. グラデーションラインの着色

各カラーストップの位置では、 グラデーションラインはそのカラーストップの色となります。 最初のカラーストップより前では、 グラデーションラインは最初のカラーストップの色となり、 最後のカラーストップより後では、 グラデーションラインは最後のカラーストップの色となります。 2つのカラーストップの間では、 グラデーションラインの色は2つのカラーストップの色で補間されます。 補間は指定された色空間で、プリマルチプライドアルファを使って行われます（CSS Color 4 § 12.3 アルファ補間参照）。 グラデーション関数で<color-interpolation-method>が指定されていない場合、 デフォルトの補間色空間が使われます（[css-color-4]参照）。

デフォルトでは、 この補間は線形です。2つのカラーストップ間の25%、50%、75%の位置では、 それぞれ2つの色の25%、50%、75%の混合色になります。

ただし、2つのカラーストップの間にトランジションヒントが指定されている場合、 補間は非線形となり、ヒントによって制御されます：

  1. 2つのカラーストップ間の距離に対するトランジションヒントの位置をパーセンテージ（0〜1の数値）で決定します。 0はヒントが最初のカラーストップの位置にあること、1は2番目のカラーストップの位置にあることを示します。 このパーセンテージをHとします。
  2. 任意の2つのカラーストップ間の点について、 その点の位置を同様にパーセンテージ（P）で求めます。
  3. その点での色の重み付けCは、 PlogH(.5)で計算します。
  4. その点の色は、2つのカラーストップの色を線形混合したものになり、 (1 - C)が最初のストップ、Cが2番目のストップの重みです。

注: トランジションヒントは「中間色」（2つの周囲カラーストップの50%混合色）をどこに配置するかを指定します。 ヒントが2つのカラーストップのちょうど中間にある場合、上記の補間アルゴリズムは通常の線形補間と同じになります。 ヒントが他の位置にある場合は、周囲カラーストップ間で滑らかな指数曲線となり、 「中間色」がヒント位置に正確に現れます。

下記はトランジションヒントなし（上）と、 赤と青のカラーストップ間にヒントを指定したもの（下）の線形グラデーションの例です。

上 - トランジションヒントなし（デフォルトの中間ヒント）:

background: linear-gradient(to right, red 0%, blue 100%);

下 - トランジションヒントあり:

background: linear-gradient(to right, red 0%, 25%, blue 100%);

複数のカラーストップが同じ位置の場合、 リストで先に指定されたものから後に指定されたものへの無限小の遷移となり、 その位置で色が急に切り替わります（滑らかなグラデーションにはなりません）。

3.5.3. カラーストップの「修正」

各使用済み位置のカラーストップを解決する際、以下の手順を順番に適用します：

  1. 最初のカラーストップに位置が指定されていない場合は、その位置を0%に設定します。 最後のカラーストップに位置が指定されていない場合は、その位置を100%に設定します。
  2. あるカラーストップやトランジションヒントの位置が、 それ以前のカラーストップやトランジションヒントより小さい場合は、 それ以前で最も大きく指定された位置と同じ位置に設定します。
  3. まだ位置が指定されていないカラーストップがある場合は、 位置のない隣接するカラーストップの連なりごとに、 直前・直後の位置付きカラーストップの間を等間隔で割り当てます。

これらのルール適用後は、 すべてのカラーストップとトランジションヒントに 明確な位置と色が設定され、昇順になります。

注: px, em, %など異なる単位を混在させる場合は注意してください。 これによりカラーストップが意図せず前のストップより手前に配置されることがあります。 例えば、 background-image: linear-gradient(yellow 100px, blue 50%) の場合、背景エリアが少なくとも200px以上ある間は修正が発生しません。 しかし、150pxしかない場合、blueカラーストップの位置は75pxとなり、 yellowカラーストップより手前となってしまい、100pxに修正されます。 また、このようなカラーストップの相対順序はレイアウトが完了するまで決まらないため、 アニメーションやトランジションで滑らかに補間されません。

下記は複数ペアのグラデーション例です。 各ペアの後者は、前者に上記の「修正」ルールを手動で適用したものです。 どちらも描画結果は同じになります。各矢印内の数字は、どの修正ステップが適用されたかを示します。 
1. linear-gradient(red, white 20%, blue)
   =1=>
   linear-gradient(red 0%, white 20%, blue 100%)

2. linear-gradient(red 40%, white, black, blue)
   =1,3=>
   linear-gradient(red 40%, white 60%, black 80%, blue 100%)

3. linear-gradient(red -50%, white, blue)
   =1,3=>
   linear-gradient(red -50%, white 25%, blue 100%)

4. linear-gradient(red -50px, white, blue)
   =1,3=>
   linear-gradient(red -50px, white calc(-25px + 50%), blue 100%)

5. linear-gradient(red 20px, white 0px, blue 40px)
   =2=>

4. 1Dイメージ値：<image-1D>型とstripes()記法

<image>値が2次元（2D）画像を表すのに対し、 <color>は0次元（0D）画像 （どちらの軸にも変化しない）と考えることができます。 一方、文脈によっては、抽象的で方向性のない1軸上に色を指定する 1次元（1D）画像が必要になる場合があります。 これはpaint line（塗り線）上に色を指定します。 <image-1D>型は、このような1D画像を表し、 stripes()の関数記法などが含まれます： 

<image-1D> = <stripes()>
<stripes()> = stripes( <color-stripe># )
<color-stripe> = <color> && [ <length-percentage> | <flex> ]?

stripes() 関数は、1D画像を、カンマ区切りの色付きストライプのリストとして定義します。 各ストライプは、指定された順序でpaint line上に端から端まで並びます。

<color-stripe>は、指定された<color> と太さの単色ストライプを定義します。 太さが省略された場合、デフォルトは1frです。 太さの値の解釈は以下の通りです：

<percentage [0,100]>
パーセンテージの太さはtotal widthに対する比率です。 0%～100%（含む）の値のみ有効です。
<length [0,∞]>
負の長さは無効です。
<flex>
<flex>は、関数内で他の非<flex>エントリの太さをtotal widthから引いて（結果は0未満なら0）、 残り幅に対して合計<flex>値比で分配されます。 合計値が1fr未満の場合は、残り幅に合計値を掛けて分配します。

total widthは、stripes()関数が使われる文脈で定義されます。 ストライプの合計がtotal widthより小さい場合、 paint lineの残りの長さは透明黒で塗りつぶされ、 最後にtransparent引数が指定されたかのように扱われます。 合計が大きい場合は、total widthを超えたストライプや部分は切り捨てられます。

例えば、stripes(red 1fr, green 2fr, blue 100px)をtotal widthが400pxで使うと、100pxの青、残り300pxを赤1:緑2の比率で分けて100px赤、200px緑となります。 これは、青100pxを400pxから引いた残り300pxに対して、赤1シェア・緑2シェアで分配します。

一方、stripes(red .1fr, green .2fr, blue 100px)をtotal width400pxで使うと、 赤30px・緑60px・青100px・透明210pxとなります。 残り300pxに合計<flex>値.3を掛けて90pxとし、 それを1:2で分配します。

（これは、flexレイアウトがライン上の小さい<flex>合計値を扱う方法と似ており、 <flex>値が0に近づいても滑らかで連続的な挙動になることを保証します。）

この関数の算出値は、 各ストライプごとに算出色と、<flex>値または算出済み<length-percentage>値で表される順序付きリストとなります。

5. CSSにおける画像とオブジェクトのサイズ指定

5.1. オブジェクトのサイズ指定: object-fit プロパティ

名前: object-fit
値: fill | none | [contain | cover] || scale-down
初期値: fill
適用対象: 置換要素
継承: no
パーセンテージ: n/a
算出値: 指定キーワード
正規順序: 文法どおり
アニメーションタイプ: 離散

object-fit プロパティは、置換要素の内容が使用される高さ・幅で確立されたボックスにどのように収まるかを指定します。

fill
置換コンテンツは要素の内容ボックス全体に収まるようにサイズ指定されます。 オブジェクトの実際のオブジェクトサイズは要素の使用幅・高さと同じになります。
none
置換コンテンツは要素の内容ボックス内にリサイズされません。 オブジェクトの実際のオブジェクトサイズは、サイズ指定なしでデフォルトサイズアルゴリズムを使い、 デフォルトオブジェクトサイズは置換要素の使用幅・高さと同じになります。
contain
置換コンテンツはアスペクト比を維持しつつ、要素の内容ボックス内に収まるようにサイズ指定されます。 実際のオブジェクトサイズは、要素の使用幅・高さに対してcontain制約を解決したサイズになります。

scale-downフラグが使われた場合、内容はnoneまたはcontainを指定した場合のうち、 より小さい実際のオブジェクトサイズになるようにサイズ指定されます。

注: noneとcontainはどちらもコンテンツの自然なアスペクト比を尊重するため、「小さい」の意味は明確です。

cover
置換コンテンツはアスペクト比を維持しつつ、要素の内容ボックス全体を満たすようにサイズ指定されます。 実際のオブジェクトサイズは、要素の使用幅・高さに対してcover制約を解決したサイズになります。

scale-downフラグが使われた場合、内容はnoneまたはcoverを指定した場合のうち、 より小さい実際のオブジェクトサイズになるようにサイズ指定されます。

注: noneとcoverはどちらもコンテンツの自然なアスペクト比を尊重するため、「小さい」の意味は明確です。

scale-down
contain scale-downと同等。

コンテンツが置換要素の内容ボックスを完全に埋めない場合、未充填部分には置換要素の背景が表示されます。 置換要素は常にコンテンツを内容ボックス内にクリップするため、コンテンツがはみ出すことはありません。 内容ボックス内でオブジェクトの位置を指定するにはobject-positionプロパティを参照してください。

注: object-fit プロパティは[SMIL10]のfit属性や、 <meetOrSlice>パラメータ、 preserveAspectRatio属性（[SVG11]）の意味と似ています。

注: object size negotiationアルゴリズムに従い、 実際のオブジェクトサイズ（またはこの場合、コンテンツサイズ）はオブジェクト自身を直接スケーリングするものではありません。 これは、表示キャンバスのサイズ情報としてオブジェクトに渡されるのみです。 そのサイズにどのように描画するかは画像フォーマットごとに異なります。 特にラスター画像は常に指定されたサイズにスケーリングされますが、SVGは指定サイズを「SVGビューポート」として使い、 ルート<svg>要素の属性値によって描画方法が決まります。

6. 画像処理

6.1. 画像解像度の上書き: image-resolution プロパティ

画像解像度は、1単位長さあたりの画像ピクセル数として定義されます（例：ピクセル毎インチ）。 一部の画像フォーマットは画像の解像度情報を記録できます。 この情報は、フォーマット処理時に画像の実際サイズを決定する際に役立つことがあります。 しかし、この情報が誤っている場合もあり、その場合は無視すべきです。 デフォルトでは、CSSは1画像ピクセル＝1CSS px単位とみなしますが、 image-resolutionプロパティで別の解像度を使用できます。

名前: image-resolution
値: [ from-image || <resolution> ] && snap?
初期値: 1dppx
適用対象: 全ての要素
継承: yes
パーセンテージ: n/a
算出値: 指定キーワードや<resolution>（snapによる調整の可能性あり、下記参照）
正規順序: 文法どおり
アニメーションタイプ: 離散

image-set() 記法は画像の本来の解像度を変更できますが、 理想的にはこのプロパティを設定しなくても自動的に反映されるべきです。 どう対応するのが最善でしょうか？ 初期値をauto（「CSSで別指定がなければ1dppx」）にする？ image-resolutionはCSSで他で解像度が指定された画像には効果なしとする？ もしくはimage-set()は常に1dppx画像を生成するように文言修正する？

image-resolutionプロパティは、要素内または上に使われるすべてのラスター画像の推奨解像度を指定します。 これはコンテンツ画像（例：置換要素や生成コンテンツ）および装飾画像（background-imageなど）双方に影響します。 画像の推奨解像度は、画像の自然寸法を決定するために使用されます。 値の意味は以下の通りです：

<resolution>
推奨解像度を明示的に指定します。 この場合の「dot」は1画像ピクセルに対応します。
from-image
画像の推奨解像度は画像フォーマットで指定されているもの（本来の解像度）となります。 画像自身が解像度を持たない場合は、明示的指定があればそれを使い、なければデフォルトは1dppx。

注: CSS Images 3 § 2.1.2 画像メタデータに使用できるメタデータへの制約があります。

snap
"snap"キーワードが指定された場合、 算出<resolution>（あれば）は、 1画像ピクセルが整数個のデバイスピクセルに対応する最も近い値に丸められます。 解像度が画像から取られた場合も、使用される本来の解像度は同様に調整されます。

SVGなどのベクターフォーマットは本来の解像度を持たないため、このプロパティはベクター画像には効果がありません。

プリンタはコンピューター画面より遥かに高い解像度を持つため、 画面上で綺麗でも印刷すると画像が粗く見える場合があります。 image-resolutionプロパティを使えば、 高解像度画像を文書に埋め込んで適切なサイズを維持でき、 画面でも紙でも美しく表示できます： 
img.high-res {
  image-resolution: 300dpi;
}

この設定で、300dpiで幅5インチの画像は実際に5in幅で表示されます。 設定しない場合は、画像幅が15000ピクセルなのでCSSデフォルトの96ピクセル/インチで約15.6インチ幅になります。

一部画像フォーマットは画像データに解像度を埋め込めます。 このルールはUAが画像自身の解像度を使い、なければCSS px単位で1画像ピクセルを使うよう指定します。 
img { image-resolution: from-image }

以下の2つのルールはどちらもUAが画像自身の解像度を使うよう指定しますが、 解像度がない場合はデフォルトの1dppxではなく300dpiが使われます。

img { image-resolution: from-image 300dpi }
img { image-resolution: 300dpi from-image }
このルールでは画像解像度が300dpiに設定されます（画像内の値は無視される）。 
img { image-resolution: 300dpi }

一方このルールは、画面解像度が96dpiの場合 画像は288dpi（3画像ピクセル＝1デバイスピクセル）で描画されます：

img { image-resolution: 300dpi snap; }

snapキーワードは画像から解像度を取る場合にも使えます：

img { image-resolution: snap from-image; }

画像が300dpiと宣言している場合、上記の状況では288dpi（3画像ピクセル＝1デバイスピクセル）で表示され、 72dpiと宣言している場合は96dpi（1画像ピクセル＝1デバイスピクセル）で表示されます。

7. 補間

このセクションでは、本仕様で定義された新しい値型の補間方法について説明します。 CSSトランジションやCSSアニメーションなどのモジュールで利用されます。

下記のアルゴリズムが単に2つの値を「補間する」や「遷移する」とだけ述べている場合は、 トランジション仕様で記載された方法で補間してください。 それ以外の場合、補間詳細で変数 t が参照されることがあります。 これは0%から100%までの数値で、遷移の進捗を表します。 トランジションの継続時間、経過時間、使用中のタイミング関数に基づいて値が設定されます。 例えば、線形タイミング関数で1秒の継続時間の場合、0.3秒経過時点では t は 30% となります。

7.1. <image> の補間

すべての画像は補間可能ですが、 一部の特殊な画像型（グラデーションなど）は独自の補間規則を持っています。 一般的には、画像は start image のサイズにスケーリングされ、 遷移中は end image のサイズへ変化しつつクロスフェードされます。

具体的には、 補間の各時点で画像は cross-fade( (100% - t) start image, end image) に等しくなります。

画像なしとの補間の特別扱い （例: "background-image: url(foo);" から "background-image: none;" への補間）。

7.2. cross-fade() の補間

cross-fade() の3つの構成要素は独立して補間されます。 このため、ネストされた cross-fade() 記法になる場合があります。

7.3. <gradient> の補間

このセクションは要レビュー・改善。 特に linear-gradient() の扱いが不完全だと思われる― グラデーションラインの「長さ」（0%から100%までの距離）を開始・終了位置間で明示的に補間すべきで、 途中で伸びて縮むようなアニメーションにならないようにしたい。

グラデーション画像はCSSトランジションやアニメーションで直接補間でき、 別のグラデーションへの滑らかなアニメーションが可能です。 補間可能なグラデーションにはいくつか制約があります：

  1. 開始・終了両方のグラデーションは同じ関数で表現されていなければならない。 （例：linear-gradient() から linear-gradient() への遷移はOKだが、 linear-gradient() から radial-gradient()や repeating-linear-gradient()は不可。)

  2. 開始・終了両方のグラデーションは同じ数の<color-stop>を持つこと。 この目的では、すべてのrepeatingグラデーションは無限個のカラーストップを持つとみなされ、互いに一致します。

  3. どちらのグラデーションも<length> と<percentage>のカラーストップを混在させてはならない。

両方のグラデーションが上記制約をすべて満たす場合、下記の方法で補間すること。 3番目のみ満たさない場合は、50%で急激に遷移させること（将来の仕様で別途指定がない限り）。 1つ目または2つ目を満たさない場合は、一般画像同様 cross-fade()で補間すること。

注: 50%での急激な遷移は、コンテンツがクロスフェードに依存しないようにするためであり、 将来的にこの場合の賢い補間規則を追加できるようにするためです。

  1. 開始・終了両グラデーションを明示形に変換：

    線形グラデーションの場合:

    • 方向が<angle>で指定されていれば、すでに明示形。

    • それ以外の場合は、 同等の描画を得られる<angle>（[0deg, 360deg)）に変換する。

      開始・終了両方のグラデーションがキーワード指定で、 それぞれが対応する角度の絶対差が180degを超える場合は、 小さい方の角度に360deg加算する。例：「to left」(270deg) から「to top」(0deg) の遷移は、 期待通り四分の一回転（時計回り）となり、 三分の四回転（反時計回り）にはならないようにするため。

    放射グラデーションの場合:

    • サイズが2つの<length>または<percentage>で指定されていれば明示形。

    • それ以外の場合は、等価な終端形状になる2つの<length>に変換する。 <rg-ending-shape>が circleなら、 ellipseに変換する。

  2. グラデーションの各構成要素とカラーストップを独立して補間する。 線形グラデーションでは、構成要素は角度のみ。 放射グラデーションでは、中心の水平・垂直位置と軸の長さ。

  3. カラーストップの補間では、まず開始グラデーションの各カラーストップを 終了グラデーションの同じインデックスのカラーストップに対応付ける。 repeatingグラデーションの場合は、開始・終了グラデーションの最初の指定カラーストップが同じインデックスとなり、 他は適切にインデックス付けされる（リストを必要に応じて繰り返し・シフトする）。 その後、各ペアのカラーストップについて位置と色を独立して補間する。

7.4. stripes() の補間

グラデーションと同様に、 2つのstripes()は補間可能で、 ある画像から別の画像へのスムーズなアニメーションが可能です。 stripes() の補間にはいくつか制約があります：

  1. 開始・終了両方の画像は同じ数の<color-stripe>を持つこと。

  2. どちらの画像も<length>、 <percentage>、 <flex> のストライプを混在させてはならない。

両画像が両方の制約を満たす場合、下記の方法で補間すること。 2番目のみ満たさない場合は、50%で急激に遷移（将来の仕様で別途指定がない限り）。 1つ目を満たさない場合は、一般画像同様 cross-fade()で補間すること。

注: 50%での急激な遷移は、コンテンツがクロスフェードに依存しないようにするためであり、 将来的にこの場合の賢い補間規則を追加できるようにするためです。

  1. 画像の各構成要素とストライプを独立して補間する。

  2. ストライプの補間では、開始画像の各ストライプを 終了画像の同じインデックスのストライプに対応付ける。 各ペアのストライプについて、太さと 色を独立して補間する。

8. 直列化

このセクションでは、本仕様で導入されたすべての新しいプロパティや値型の直列化方法について説明します。 CSSオブジェクトモデル [CSSOM] とのインターフェース目的です。

このモジュールで定義された任意の関数を直列化する場合は、 その個別の文法に従い、文法が記述された順序で直列化し、 意味が変わらない範囲で構成要素は省略し、 スペース区切りのトークンを単一スペースで結合し、 直列化されたカンマごとに単一スペースを付与してください。

cross-fade()の場合は、 常に<percentage>を直列化してください。

例えば、次のようにグラデーションを指定した場合： 
Linear-Gradient( to bottom, red 0%,yellow,black 100px)

次のように直列化されなければなりません：

linear-gradient(rgb(255, 0, 0), rgb(255, 255, 0), rgb(0, 0, 0) 100px)

スタイルシート

本仕様に適合するスタイルシートとは、 このモジュールで定義された構文を用いた全ての文が、汎用CSS文法および本モジュールで定義された各機能の個別文法に従って有効である場合です。

本仕様に適合するレンダラーとは、 スタイルシートを対応する仕様で定義されたように解釈することに加え、 本仕様で定義されたすべての機能を正しくパースし、文書を適切にレンダリングすることです。ただし、UAがデバイスの制限により文書を正しくレンダリングできない場合でも、それは非適合とはなりません。（例えば、UAはモノクロモニター上で色をレンダリングする必要はありません。）

本仕様に適合するオーサリングツールとは、 汎用CSS文法および本モジュールの各機能の個別文法に従って構文的に正しいスタイルシートを書き出し、本モジュールで説明されているスタイルシートのその他の適合要件をすべて満たす場合です。

部分的な実装

著者が将来互換性のあるパース規則を利用してフォールバック値を指定できるように、CSSレンダラーは、利用可能なサポートレベルがないat規則・プロパティ・プロパティ値・キーワード・その他構文構造を無効として扱い、適切に無視しなければなりません。特に、ユーザーエージェントは、サポートされない値を選択的に無視しつつ、1つの複数値プロパティ宣言内でサポートされた値のみを適用することはしてはなりません。いずれかの値が無効（サポート外値は必ず無効）とみなされる場合、CSSではその宣言全体が無視されます。
