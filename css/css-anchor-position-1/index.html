概要

この仕様は、アンカーポジショニングを定義します。 これは、配置された要素が、ページ上の1つ以上の「アンカー要素」を基準にしてサイズと位置を決定できる仕組みです。

CSSは、構造化文書（HTMLやXMLなど）の表示方法を画面や紙などで記述するための言語です。

1. はじめに

CSSの絶対位置指定は、作成者がボックスをページ上の任意の場所に配置できるようにします。 この際、他のボックスのレイアウトは（包含ブロック以外は）考慮されません。 この柔軟性は非常に便利ですが、同時に制限にもなります——多くの場合、他のボックスを基準に配置したいことがあります。アンカーポジショニング （position-anchorプロパティやposition-areaプロパティ、 またはanchor 関数のanchor()やanchor-size()によって） これを実現できます。 絶対位置指定ボックスをページ上の他の1つ以上のボックス（アンカー参照）に「アンカー」し、 さらに重なりやオーバーフローを避けるために複数の配置候補を試すこともできます。

例えば、作成者はツールチップをターゲット要素の中央上に配置したいと考えるかもしれません。 ただし、ツールチップが画面外にはみ出す場合は、ターゲット要素の下に配置するべきです。 これは次のCSSで実現できます: 
.anchor {
  anchor-name: --tooltip;
}
.tooltip {
  /* Fixposは包含ブロックとの関係を気にしなくてよいことを意味します;
     ツールチップはDOMのどこにでも存在できます。 */
  position: fixed;

  /* アンカー動作はデフォルトで--tooltipアンカーを参照します。 */
  position-anchor: --tooltip;

  /* ツールチップの下端をアンカーの上端に揃える;
     また、水平方向の書字モードではツールチップとアンカーが中央揃えになります。 */
  position-area: block-start;

  /* ウィンドウからはみ出す場合、自動的に上下を入れ替えて
     ツールチップの上端をアンカーの下端に揃えます。 */
  position-try: flip-block;

  /* 幅が広くなりすぎないようにする */
  max-inline-size: 20em;
}

1.1. 値の定義

この仕様はCSSプロパティ定義の慣例（[CSS2]）および値定義構文（[CSS-VALUES-3]）に従います。 この仕様で定義されていない値型は、CSS Values & Units [CSS-VALUES-3]で定義されています。 他のCSSモジュールとの組み合わせにより、これらの値型の定義が拡張される場合があります。

各プロパティの定義に記載されている値に加え、 この仕様で定義される全プロパティは CSS全体キーワードも値として使用できます。 可読性のため、ここでは明示的に繰り返していません。

CSSにおけるセレクタマッチ以外のほとんどの操作と同様に、 この仕様の機能はフラット化された要素ツリー上で動作します。

2. アンカーの決定

2.1. アンカーの作成: anchor-nameプロパティ

名前: anchor-name
値: none | <dashed-ident>#
初期値: none
適用対象: 主要ボックスを生成するすべての要素
継承: no
パーセンテージ: n/a
算出値: 指定通り
正規順序: 文法通り
アニメーション型: 離散

anchor-nameプロパティは、 要素がアンカー要素であることを宣言し、 その主要ボックスがアンカーボックスとなり、 対象となるアンカー名のリストを与えます。 値の意味は以下の通りです:

none

プロパティは効果を持ちません。

<dashed-ident>#

要素が主要ボックスを生成する場合、 その要素はアンカー要素となり、 指定されたアンカー名のリストを持ちます。 各アンカー名はツリースコープ名です。

それ以外の場合は、プロパティは効果を持ちません。

アンカー名は一意である必要はありません。 すべての要素が特定のボックスのターゲットアンカー要素になれるわけではありません。 したがって、適切なスコープで使用されていれば、名前を複数箇所で再利用できます。

注: 複数の要素が同じアンカー名を共有していて、かつそれらが配置対象ボックスから見える場合、 ターゲットアンカー要素はDOM順で最後のものになります。 anchor-scopeプロパティで どの名前が参照ボックスから見えるかをさらに制限できます。

アンカー名は、デフォルトではcontainment（包含）によってスコープされません。 たとえ要素にスタイルやレイアウト包含（または同様の包含）があっても、 子孫のアンカー名はページ内の他の要素から参照可能です。

注: 要素が他の要素のスキップされた内容内にある場合 （例: content-visibility: hiddenなど）、 それは許容可能なアンカー要素とはみなされず、 実質的に名前を持たないものとして扱われます。

2.1.1. 暗黙的アンカー要素

仕様によっては、特定の状況下で、 ある要素が他の要素の暗黙的アンカー要素であると定義できます。

暗黙的アンカー要素は、 autoキーワード（position-anchor）や、 anchor関数でアンカー参照を省略することで参照できます。

暗黙的アンカー要素は、 疑似要素の場合は、その起点要素になります（別途指定がない限り）。

2.1.2. アンカーの位置

この仕様のいくつかの機能は、アンカーボックスの位置とサイズを参照します。 アンカーボックスの位置とサイズはレイアウト後に決定され、 その際にはpositionによる調整 （例: position: relativeやposition: sticky）も含みます。

レイアウト後の効果（transformなど）は、 アンカーボックスの位置には影響しません。

注:アンカーがtransformや同様のプロパティの効果を含めるかどうかを選択できるようにすることは、将来的に許可される可能性があります。

2.2. アンカー名のスコープ設定：anchor-scope プロパティ

名前: anchor-scope
値: none | all | <dashed-ident>#
初期値: none
適用対象: すべての要素
継承: no
パーセンテージ: 該当なし
算出値: 指定通り
正規順序: 文法に従う
アニメーション型: 離散型

このプロパティは、指定したアンカー名と、それらのアンカー名の検索範囲をこの要素のサブツリーに限定します。 詳細は§2 アンカーの決定を参照してください。

値の意味は以下の通りです:

none
アンカー名のスコープに変更はありません。
all
この要素またはその子孫で定義された、スコープがすでに子孫のanchor-scopeによって制限されていないすべてのアンカー名を、この要素の子孫だけで有効になるようにします。 また、子孫はこのサブツリー内のアンカー要素のみをアンカー名に一致させます。

この値は同じツリーのスコープ内アンカー名のみに影響し、 ツリースコープ名と同様に動作します。 （つまり、anchor-scope: allは、anchor-scope: --foo, --bar, ...と同じで、 関連するアンカー名をすべて列挙した場合と同じ挙動になります。）

<dashed-ident>
この要素またはその子孫で定義された、スコープがすでに子孫のanchor-scopeによって制限されていない一致するアンカー名を、この要素の子孫だけで有効になるようにします。 また、子孫はこのサブツリー内のアンカー要素だけをアンカー名に一致させます。 <dashed-ident>はツリースコープ名です。

注: [css-scoping-1]によれば、 2つのツリースコープ名を比較する場合、同じツリー由来である場合のみ一致します。

このプロパティは暗黙的アンカー要素には影響しません。

デザインパターンを再利用する場合、anchor-scopeを使うことで同じコンポーネント間の名前衝突を防ぐことができます。 例えば、リスト内の各リスト項目に配置された要素があり、 それぞれが自分が属するリスト項目を基準に位置決めしたい場合、 
li {
  anchor-name: --list-item;
  anchor-scope: --list-item;
}
li .positioned {
  position: absolute;
  position-anchor: --list-item;
  position-area: inline-start;
}

anchor-scopeなしの場合、 すべての li 要素がすべての配置要素から見えるため、 それらは全て最後の li を基準に位置決めし、重なってしまいます。

2.3. アンカーの検索

この仕様のいくつかの機能は、 ターゲットアンカー要素を、 アンカー指定子 （これは <dashed-ident>（かつ ツリースコープ参照） であり、ページ上の他の場所の anchor-name値と一致すべきもの、 もしくはキーワード auto、 または何も指定しない（指定子なしの場合））を元に検索します。

注: これらの条件で示される一般的なルールは、 要素自身のボックスが、参照しようとする配置ボックスよりも先に完全にレイアウトされている場合に限り、 その要素が配置ボックスのターゲットアンカー要素となれるということです。 CSSのレイアウトルールは、アンカーと配置ボックスの関係やその包含ブロックによって、 この条件に関して有用な保証を提供します。 以下の条件リストは、 この目的のために積み重ねコンテキストのルールを必要部分だけ明確に言い換えたものであり、 アンカーポジショニングに循環参照が生じる可能性がないことを保証します。

クエリ要素 query el とオプションの アンカー指定子 anchor spec が与えられたとき、ターゲットアンカー要素を決定する方法:

  1. anchor spec が渡されていない場合、 デフォルトアンカー要素が存在すればそれを返し、 存在しなければ何も返さない。

  2. anchor spec が auto の場合:

    1. Popover API が query el に対して 暗黙的アンカー要素を定義しており、 それが query el にとって 許容可能なアンカー要素である場合、 その要素を返す。

    2. そうでなければ何も返さない。

    注: 将来的なAPIでも暗黙的アンカー要素が定義される可能性があります。 その場合はこのアルゴリズムで明示的に対応し、連携を確保します。

  3. それ以外の場合、anchor spec は <dashed-ident> です。 以下の条件を満たす最後の要素 el（ツリー順）を返します:

    • el は アンカー要素であり、anchor spec と一致する アンカー名を持つ。

    • el の アンカー名と anchor spec は、 同じ ツリー ルートに関連付けられている。

      注: アンカー名はツリースコープ名、 anchor spec はツリースコープ参照です。

    • el は query el にとって 許容可能なアンカー要素である。

    これらの条件を満たす要素がなければ、何も返さない。

    注: anchor-scopeは 特定のアンカー名の可視性を制限でき、 検索時にどの要素がアンカー要素になれるかに影響します。

注: ある anchor-name が一つの シャドウツリー内で定義されていても、 別の シャドウツリーのスタイルでアンカー関数から参照されることはありません（カプセル化を維持）。 しかし、異なるシャドウツリー間でも、 anchor-nameと アンカー関数が同じツリー内のスタイルで指定されていれば、要素同士でアンカー可能です。 例えば ::part() を使ってシャドウ内の要素をスタイルする場合などです。 (暗黙的アンカー要素も、必ずしもツリー単位で制限されるわけではありませんが、 その詳細は割り当てAPIに依存します)

要素 possible anchor が、絶対位置指定された要素 positioned el にとって 許容可能なアンカー要素となる条件は以下の全てを満たす場合です:

  • possible anchor は 要素または スタイリング可能な ツリー従属疑似要素である。

  • possible anchor は positioned el に対して anchor-scopeの効果により スコープ内である。

  • possible anchor は positioned el よりも厳密に先にレイアウトされていること。 すなわち、以下のいずれかの条件が成立すること:

    • positioned el は より高いトップレイヤーにあり、possible anchorはそれより下位である。

    • 両要素が 同じトップレイヤーに属し、かつ異なる 包含ブロックを持つ場合、 positioned el の 包含ブロックが possible anchor の 包含ブロックよりも 包含ブロックチェーン上で祖先であること。つまり下記いずれか:

      • positioned el の 包含ブロックがビューポートであり、 possible anchor の 包含ブロックがビューポートでない。

      • positioned el の 包含ブロックが 初期包含ブロックであり、 possible anchor の 包含ブロックが要素によって生成されていて、 possible anchor の 包含ブロックチェーン上で 初期包含ブロックまで到達する直前の 最後の 包含ブロックが 絶対位置指定でないか、 またはツリー順で positioned el より前であること。

      • 両要素の 包含ブロックが要素で生成されていて、 positioned el の包含ブロックが フラットツリー上で possible anchor の 包含ブロックの祖先であり、 possible anchor の 包含ブロックチェーンで positioned el の 包含ブロックまで到達する直前の 最後の 包含ブロックが 絶対位置指定でないか、 またはツリー順で positioned el より前であること。

    • 両方の要素が同じトップレイヤー内にあり、同じ包含ブロックを持ち、 かつ両方とも絶対位置指定されていて、 possible anchorがフラットツリー順序でpositioned elより前にある場合。

    • 両方の要素が同じトップレイヤー内にあり、同じ包含ブロックを持つが、 possible anchorが絶対位置指定されていない場合。

  • possible anchorが他の要素のスキップされた内容内にある場合、 positioned elも同じ要素のスキップされた内容内にある。

    注: つまり、positioned elは両方が同じスキップされた「リーフ」にいる場合のみpossible anchorへアンカー可能だが、 リーフをまたいでアンカーすることはできない。 これは、両方を含む要素がスキップされてもpositioned elが突然別のアンカーへ移動することはなく、 ページ内の他の場所にある配置要素からスキップされた要素へのアンカーは防がれることを意味する。

2.4. デフォルトアンカー：position-anchor プロパティ

名前: position-anchor
値: auto | <anchor-name>
初期値: auto
適用対象: 絶対位置指定されたボックス
継承: no
パーセンテージ: 該当なし
算出値: 指定通り
正規順序: 文法に従う
アニメーション型: 離散型

position-anchorプロパティは、デフォルトアンカー要素を指定します。 これは、position-area、position-try、 および（デフォルトで）この要素に適用されるすべてのアンカー関数に使用されます。position-anchorはリセット専用サブプロパティであり、positionの一部です。

auto

暗黙的アンカー要素があればそれを使用し、 なければボックスはデフォルトアンカー要素を持ちません。

<anchor-name>

指定されたターゲットアンカー要素が、指定の<anchor-name>によって選択され、 そのボックスのデフォルトアンカー要素となります。

デフォルトアンカー要素の主ボックスは、 ボックスのデフォルトアンカーボックスです。

例：次のコードでは .fooと.barの両方の要素が、 参照するアンカー要素だけを変えて同じ位置指定プロパティを使えます。 
.anchored {
  position: absolute;
  top: calc(.5em + anchor(outside));
  /* アンカー名が指定されていない場合、
     デフォルトアンカーボックスを自動参照します。 */
}

.foo.anchored {
  position-anchor: --foo;
}
.bar.anchored {
  position-anchor: --bar;
}

2.5. アンカーの関連性

要素elがユーザーに関連するかどうかを判定する場合、 elの子孫が配置されたボックスのターゲットアンカー要素であり（そのボックス自体がスキップされておらず、その包含ブロックがel自身やその子孫でない場合）、 elはユーザーに関連すると見なさなければなりません。

注: これは例えば、 content-visibility: autoサブツリー内のアンカーが、 それに依存する配置ボックスがスキップされていない限り、 サブツリーの内容スキップを防止することを意味します。 （アンカーと配置ボックスの両方が同じ content-visibility: auto要素の下にある場合を除きます。 この場合は互いに表示を維持するために循環参照にはなりません。）

3. アンカー基準の配置

絶対位置指定されたボックスは、ページ上の1つ以上のアンカーボックスを基準に配置できます。

position-areaプロパティは、デフォルトアンカーボックスを基準にした便利なグリッド概念を提供します。 より複雑な配置や複数ボックスを基準にした配置には、 anchor()関数を insetプロパティで使用し、 明示的にアンカーボックスの辺を参照することができます。

3.1.1. ポジションエリアグリッドの解決

position-area grid（ポジションエリアグリッド）は、各軸に4本のグリッドラインからなる 3×3 のグリッドです。 順序は（writing mode（書字モード）とcontaining block（包含ブロック）に従います）：

  • ボックスの修正前の包含ブロックの開始端、またはそれよりも開始側にある場合はデフォルトアンカーボックスの開始端

  • デフォルトアンカーボックスの開始端

  • デフォルトアンカーボックスの終了端

  • ボックスの修正前の包含ブロックの終了端、またはそれよりも終了側にある場合はデフォルトアンカーボックスの終了端

注: デフォルトアンカーボックスが修正前の包含ブロックの外側に部分的または完全に存在する場合、 position-area gridの行や列がゼロサイズになることがあります。

y-start, y-end, y-self-start, y-self-end

このキーワードが指定する軸に応じて、対応する1つの行または列です。

anchor()と同様に、 単純な論理キーワード (start、end など) は、ボックスの書字モードの包含ブロックを参照します。 x-startなどは、同様に指定された物理軸で方向を決定します。

self-*論理キーワード (self-start、x-self-end など) も同じですが、ボックス自身の書字モードを参照します。

span-start, span-end, span-self-start, span-self-end
span-top, span-bottom

span-block-start, span-block-end, span-self-block-start, span-self-block-end
span-inline-start, span-inline-end, span-self-inline-start, span-self-inline-end

（例えば、span-topは、最初の2行―中央の行と上の行―を含みます。）

span-all

このキーワードが指定する軸に応じて、3つすべての行または列です。

いくつかのキーワードは、どの軸を指しているかが曖昧です：center、span-all、 また、ブロック軸やインライン軸を明示しないstartなどのキーワードも同様です。 他方のキーワードが軸について明確であれば、曖昧なキーワードは反対軸を指します。 （例えば、block-start centerの場合、 centerキーワードはインライン軸を指します。） どちらのキーワードも曖昧な場合は、 最初のキーワードがボックスのブロック軸、 2番目が包含ブロックのインライン軸を指します。 （例えば、span-all startはspan-all inline-startと同等です。）

キーワードが1つだけ指定された場合、 そのキーワードが軸について明確であれば2つ目のキーワードはspan-allとみなされます。 明確でない場合は、そのキーワードが繰り返されたかのように動作します。 （例えば、topはtop span-allと同等ですが、 centerはcenter centerと同等です。）

3.1.3. 算出値と<position-area>のシリアライズ

算出値は、各軸で選択されたトラックを示す2つのキーワードとなります。 長い論理キーワード（block-start）と短い論理キーワード（start）は同等に扱います。 シリアライズは文法（上記）の順序で行い、論理キーワードは短縮形で出力されます （例：start start、block-start inline-startではなく）。

3.1.4. エリア固有のデフォルトアラインメント

各<position-area>は、デフォルトの自己アラインメントも意味します。 ボックスの自己アラインメントプロパティがnormalの場合に使われます：

  • 軸で中央トラックのみが選択された場合、その軸のデフォルトアラインメントはcenterです。

  • 3つすべてのトラックが選択された場合、その軸のデフォルトアラインメントはanchor-centerです。

  • それ以外の場合、その軸のデフォルトアラインメントは非指定側のトラック方向になります。 例えば「start」トラックを指定していれば、その軸のデフォルトアラインメントはendなど。

例えば、英語相当の書字モード（horizontal-tb, ltr）で、 span-x-start topは縦軸のstart領域と、横軸のstartおよびcenter領域に解決されるため、 デフォルトのアラインメントはalign-self: end（ボックス下端のマージン端がtop領域下端と揃う）と、 justify-self: end（ボックスの終端側マージン端がspan-start領域終端側と揃う）となります。
ボックスがインセット修正包含ブロックをはみ出しても、 元の包含ブロック内には収まる場合、 通常のアラインメントに違反しても、デフォルトでは元の包含ブロック内に収まるよう「シフト」します。 詳細はCSS Box Alignment 3 § 4.4 Overflow Alignment: the safe and unsafe keywords and scroll safety limitsを参照してください。

この挙動により、配置ボックスが予想より包含ブロックが小さくなった場合でも、表示領域内に収まりやすくなります。

例えば、position-area: bottom span-rightの場合、 配置ボックスはアンカーの左端から包含ブロックの右端まで伸び、左揃えとなります。 しかし配置ボックスがその領域より大きい（例えばアンカーが画面右端近くの場合など）場合、左方向へシフトして表示を維持します。

3.2. アンカー基準の中央揃え：anchor-centerアラインメント値

名前: justify-self, align-self, justify-items, align-items
新しい値: anchor-center

自己アラインメントプロパティは、絶対位置指定ボックスがインセット修正包含ブロック内で自身を揃えることを可能にします。 従来の値と適切なinsetプロパティでほとんどの用途は十分ですが、 アンカー基準の配置でよくある「アンカーボックス上で中央揃え」は複雑な調整が必要でした。

新しいanchor-center値はこのケースを極めて簡単にします。 配置ボックスがデフォルトアンカーボックスを持つ場合、 その軸で（可能な限り）デフォルトアンカーボックス上で中央揃えされます。

さらに、 すべてのauto insetプロパティは0に解決されます。

ボックスが絶対位置指定されていないか、 デフォルトアンカーボックスを持たない場合、 この値はcenterと同じ動作となり、insetプロパティの解決への追加効果はありません。

注: anchor-center利用時、 アンカーがボックスの元の包含ブロックの端に近すぎる場合は、純粋な中央揃えから「シフト」して、 元の包含ブロック内に収まるようになります。 詳細はCSS Box Alignment 3 § 4.4 Overflow Alignment: the safe and unsafe keywords and scroll safety limitsを参照してください。

ターゲットアンカー要素がフラグメント化されている場合は、 フラグメントのボーダーボックスの軸揃えバウンディング矩形が使用されます。

一方、 .bar { inset-block-end: anchor(--foo block-start); }では、 .bar要素のblock-endの辺を --fooアンカーのblock-startの辺に揃える長さとして解決されます。

inset-block-startとinset-block-endは、 それぞれ異なる辺（要素の包含ブロックのblock-startとblock-end）からのインセットを指定するため、 同じanchor()でも通常は異なる長さになります。

anchor()関数は <length>に解決されるため、 他の長さと同様に数式関数でも使用できます。

例えば、以下の例では、 要素のインセット修正包含ブロックを アンカーボックス上で中央揃えし、 包含ブロックをはみ出さない最大幅で配置します： 

.centered-message {
  position: fixed;
  max-width: max-content;
  justify-self: center;

  --center: anchor(--x 50%);
  --half-distance: min(
    abs(0% - var(--center)),
    abs(100% - var(--center))
  );
  left: calc(var(--center) - var(--half-distance));
  right: calc(var(--center) - var(--half-distance));
  bottom: anchor(--x top);
}

これは例えば input 要素のエラーメッセージ表示などに適しており、 中央揃えにすることでどのinputに関連するかが分かりやすくなります。

3.3.1. anchor()の解決

anchor()関数は、以下すべての条件を満たす場合のみ解決可能なアンカー関数となります：

  • 絶対位置指定されたボックスに適用されている。

  • 関数内の<anchor-side>が物理キーワードの場合、そのキーワードはその軸のインセットプロパティで指定されていること。 （例えば、leftはleftやright、または水平方向の論理インセットプロパティでのみ使えます。)

  • 関数が使用されているボックスと、関数内で指定された<anchor-name>値に対して、ターゲットアンカー要素が存在している。

いずれかの条件を満たさない場合、 anchor()関数は指定されたフォールバック値に計算されます。 フォールバック値が指定されていない場合は、参照宣言が算出値の時点で無効となります。

3.4. スクロールの考慮

パフォーマンスの観点から、 実装では通常、スクロール処理を独立したスクロール／「合成」スレッド上で行います。 このスレッドは能力が限られており（移動・変形などの単純処理のみ、レイアウトなど高負荷処理不可）、 そのためスクロールへの応答を人間の知覚上「瞬時」とみなせる速度で処理できることが期待できます。

スクロールが単にアンカーポジション要素を移動させるだけなら、 理論的には問題ありません。 移動はスクロールスレッドで行えるため、配置要素はスクロールコンテンツと滑らかに連動して動きます。 しかし、アンカーポジショニングでは 要素自身の反対側の辺の位置を異なるスクロールコンテキストの要素に依存させられるため、 スクロールが一方の辺だけを動かしてサイズ変更が発生し、レイアウトが必要になる場合があります。 これはスクロールスレッドでは処理できません！

この制約に対応しつつ、できる限り様々な要素へのアンカーを許容するために、アンカーポジショニングでは 記憶されたスクロールオフセットとスクロール補正の組み合わせを使います。

詳細は技術的ですが、要点は以下です：

  • 配置要素が初めて表示されるとき、またはフォールバックが変更されるとき、 すべてのアンカー参照の最新位置に基づいて正しく配置が計算されます。

    これらのアンカー参照が異なるスクロールコンテキストの場合、 その総スクロールオフセットが記憶され、 後でその要素がスクロールされても、レイアウトではその記憶されたオフセットが使われ続けます。 （記憶されるのはスクロールオフセットのみで、実際のレイアウト位置は毎回新しく計算され正確です。） 再計算は、配置要素が表示されなくなり再び表示されるときや、フォールバックが変更されたときのみ行われます。

  • 唯一の例外はデフォルトアンカー要素です。 これが記憶されたスクロールオフセットからスクロールされた場合、配置要素も一緒に移動します。 これは位置のシフトのみのため、配置要素のサイズ変更やレイアウトは不要です。

結果として、アンカーポジショニングは、どんな要素へのアンカーでも「そのまま動作」するはずですが、 スクロールへの応答の仕方には制限がある場合があります。

アンカー再計算ポイントは、 絶対位置指定された要素がボックス生成を開始した時（つまり、display:noneやdisplay:contentsから他のdisplay値に切り替わる時）に発生します。 これはCSSアニメーション開始時と同じタイミングです。

要素に対してフォールバックスタイルの決定が行われる場合にも アンカー再計算ポイントが発生し、 フォールバックスタイルが変化した場合は選択されたフォールバックスタイルに対応するアンカー再計算ポイントの結果を使います。

要素absposにアンカー再計算ポイントが発生した場合、 absposのアンカー参照で参照されている全要素anchorに対し、 その時点の現在のスクロールオフセットの合計（スクロールコンテナ祖先の合計、ただしabsposの包含ブロックは含まない）を 記憶されたスクロールオフセットとして関連付けます。 記憶されたスクロールオフセットには、 position: stickyなど、 スクロール依存の他の位置変更も含みます。 absposがデフォルトアンカー要素を持つ場合は、 実際にアンカー参照しなくても必ず記憶されたスクロールオフセットを計算します。

すべてのアンカー参照は、 すべてのスクロールコンテナが初期スクロール位置にあると仮定して計算され、 その後関連する記憶されたスクロールオフセットが加算されます。

transformはスクロールと同じ問題があるため、アンカーポジショニングは通常transformを考慮しません。ここでtransformの効果も組み込めるか？

上記により、配置要素はアンカー参照のスクロール位置に一度だけ応答できますが、 それらがスクロールされた場合、配置要素はもはやそれらにアンカーされているようには見えなくなります（ただし非スクロールの動きには引き続き応答します）。 この問題は一般的には解決できませんが、 1つのアンカー参照のスクロールには応答できます。 具体的には、デフォルトアンカー要素です：

絶対位置指定されたボックスabsposが、横軸または縦軸でスクロール補正を行うのは、以下両方の条件を満たす場合です：

  • absposはデフォルトアンカーボックスを持つ。

  • absposは自身のデフォルトアンカーボックス、または少なくとも同じスクロールコンテキスト内の何かへのアンカー参照を持つ。 つまり、少なくとも次のいずれか：

    • absposのその軸における自己アラインメントプロパティの使用値がanchor-centerである；

    • absposのposition-areaの値がnone以外である

    • absposのその軸のinsetプロパティのanchor()関数のいずれかが、absposのデフォルトアンカーボックスと同じ最も近いスクロールコンテナ祖先を持つターゲットアンカー要素を参照している。

注: absposにposition options listがある場合、 ある軸でスクロール補正を行うかどうかは、適用されたフォールバックスタイルにも影響される。

absposのデフォルトスクロールシフトは、水平および垂直軸それぞれに対する長さのペアである。 各長さは次のように計算される：

  • absposがその軸でスクロール補正している場合、 長さはデフォルトアンカー要素の記憶されたスクロールオフセットと、 それを再計算した場合の現在の記憶されたスクロールオフセットとの差となる。

  • それ以外の場合、長さは0となる。

レイアウトがabsposに対して完了した後、 さらにデフォルトスクロールシフトによってシフトされる。 これは他のtransformの前にtransformの影響を受けるかのように動作する。

スナップショットの正確なタイミングを定義すること： フレームごとに更新し、スタイル再計算前。

記憶されたスクロールオフセットと同様に、 デフォルトアンカー要素のtransformも考慮できるか？

注: 記憶されたスクロールオフセットはanchor()関数の値に影響するが、デフォルトスクロールシフトは要素自体を直接シフトする（insetプロパティの値決定後、 アラインメント適用後、など）。 これは通常は区別できないが、round(anchor(outside), 50px)のように デフォルトアンカー要素の位置を非線形に変換するケースでは、 挙動の違いが表面化する。

4. アンカー基準のサイズ指定

絶対位置指定ボックスは、anchor-size()関数を サイズ指定プロパティで使い、 1つ以上のアンカーボックスのサイズを参照できます。 anchor-size()関数は<length>に解決されます。 この関数は許可された@position-tryプロパティでのみ有効です（それ以外は無効）。

4.1. anchor-size()関数

名前: width, height, min-width, min-height, max-width, max-height, top, left, right, bottom, margin-top, margin-left, margin-right, margin-bottom
新しい値: <anchor-size()> 
anchor-size() = anchor-size( [ <anchor-name> || <anchor-size> ]? , <length-percentage>? )
<anchor-size> = width | height | block | inline | self-block | self-inline

anchor-size()関数はanchor()と似ており、 引数も同じですが、<anchor-side>キーワードが<anchor-size>に置き換わり、 これは対になる2辺の間の距離を示します。

物理的な<anchor-size>キーワード (widthとheight) はそれぞれターゲットアンカー要素の幅と高さを指します。 anchor()とは異なり、軸の一致制限はありません。 例えば、width: anchor-size(--foo height);は有効です。

論理的な<anchor-size>キーワード (block、inline、self-block、 self-inline) は物理キーワードのいずれかに対応し、 書字モードにより決定されます。 （self-blockとself-inlineはボックス自身の書字モード、 blockとinlineはボックスの包含ブロックの書字モードによります。）

<anchor-size>キーワードが省略された場合、 anchor-size()が使用されているプロパティの軸に一致するキーワードとして動作します。 （例えば、width: anchor-size()はwidth: anchor-size(width)と同等です。）

anchor-size()関数が解決可能なanchor-size関数を表す場合、 算出値のタイミングで （style & layout interleavingを介して） 指定軸に対応する境界辺（左/右または上/下）間の <length>に解決されます。 これはターゲットアンカー要素の該当する境界辺の長さとなります。

4.1.1. anchor-size()の解決

anchor-size()関数は、以下すべての条件が満たされた場合にのみ解決可能なanchor-size関数となります：

  • 絶対位置指定されたボックスに適用されていること。

  • そのボックスおよび関数で指定された<anchor-name>値に対するターゲットアンカー要素が存在すること。

これらの条件のいずれかが満たされない場合、 anchor-size()関数は指定されたフォールバック値に解決されます。 フォールバック値が指定されていない場合、 それを参照する宣言は算出値時に無効となります。

5. オーバーフロー管理

アンカーポジショニングは強力ですが、予測が難しいこともあります。 アンカーボックスはページ上のどこにでもあり得るため、 例えばアンカーの上や右など、特定の方法でボックスを配置しようとすると、 配置されたボックスが包含ブロックをはみ出したり、画面の一部が隠れてしまうことがあります。

これを緩和するため、絶対位置指定されたボックスは、 position-try-fallbacksプロパティを使って、 ボックスの既存スタイルから生成された、または@position-tryルールで指定された、複数の位置・アラインメントプロパティの組み合わせを参照できます。 UAは初期位置でボックスがはみ出した際に、それぞれを順番に適用し、 包含ブロックからはみ出さない最初のものを適用します。

position-try-orderを使えば、 これらのオプションを利用可能な空間の広さでソートでき、 空間の広さが重要な場合に宣言順より優先できます。

5.1. フォールバックオプション指定：position-try-fallbacksプロパティ

名前: position-try-fallbacks
値: none | [ [<dashed-ident> || <try-tactic>] | <'position-area'> ]#
初期値: none
適用対象: 絶対位置指定ボックス
継承: no
パーセンテージ: 該当なし
算出値: 指定通り
正規順序: 文法に従う
アニメーション型: 離散型

このプロパティは、絶対位置指定されたボックスがインセット修正包含ブロックからはみ出した際に試す代替配置スタイルのリストを提供します。 このposition options listは最初は空です。

カンマ区切りの各エントリは個別のオプションです： @position-tryブロック名、またはボックスの計算スタイルを自動変換する<try-tactic>です。

値の意味は以下の通りです：

none

このプロパティは効果を持ちません。 ボックスのposition options listは空です。

<dashed-ident>

指定した名前の@position-tryルールが存在する場合、 そのposition optionをposition options listに追加します。

存在しない場合は何も起こりません。

<try-tactic>

ボックスの計算スタイルからposition optionを自動生成し、 指定キーワードに従ってtry-tacticによる入れ替えを行い、 生成したposition optionをボックスのposition options listに追加します。 さらにposition options listに加えます。 

<try-tactic> = flip-block || flip-inline || flip-start
flip-block

ブロック軸の値（例えばmargin-block-startとmargin-block-end）を入れ替え、 本質的にインライン軸線で左右反転します。

flip-inline

インライン軸の値を入れ替え、 本質的にブロック軸線で上下反転します。

flip-start

startプロパティ同士、 およびendプロパティ同士の値（例えばmargin-block-startとmargin-inline-start）を入れ替え、 本質的にstart-start隅からend-end隅への対角線で反転します。

複数キーワードが指定された場合は、 それらの変換を順に合成して一つのposition optionを生成します。

<dashed-ident> || <try-tactic>

前述2つのオプションの効果を組み合わせます： 指定名の@position-tryルールが存在する場合、 そのposition optionをベーススタイルに適用し、 指定の<try-tactic>で変換し、 結果をボックスのposition options listに追加します。

存在しない場合は何も行いません。

<'position-area'>

指定値のposition-areaプロパティのみで構成されたposition optionを自動生成します。

박스의 요소 el의 스타일을 두 방향 directions 사이에서 try-tactic에 따라 교환하려면, 스타일 집합을 반환합니다:

  1. directionsが同じ軸上の反対側であれば「対向」である。 異なる軸の場合は「直交」である。

  2. el上のaccepted @position-try propertiesの指定値を決定し、その結果をstylesとする。

  3. styles内の変数の代入、env()関数、 その他任意の置換関数を代入する。

    env()関数については、 参照された環境変数が方向や軸に関連する場合 （例：safe-area-inset-topなど）、 参照する環境変数をdirectionsに合わせて切り替える。

    例：top: env(safe-area-inset-top);が指定されていてdirectionsが上と左なら、 env()はenv(safe-area-inset-left)と同様に解決される。 （次のステップでleftプロパティに実際に入れ替わる。）

  4. stylesの値をdirectionsに対応する関連プロパティ間で入れ替える。

    例：topとleftを入れ替える場合、 margin-topとmargin-leftの値を入れ替え、 widthとheightも入れ替える、など。

    注: directionsが同じ軸の反対側の場合、 widthやalign-selfなど一部のプロパティは入れ替わらないが、 次のステップで値が変更される場合がある。

  5. 入れ替え先の方向に合わせて、プロパティ値を次のように調整する：

    • insetプロパティの場合、 anchor()関数内の指定側を変更し、 新旧方向で同じ相対関係を保つようにする。

      <percentage>が使われていて、 directionsが対向の場合は、値を「100%－元の値」に変える。

      例：topとleftを入れ替える場合、 margin-top: anchor(bottom)はmargin-left: anchor(right)となる。

      topとbottomを入れ替える場合、 margin-top: anchor(20%)はmargin-bottom: anchor(80%)になる。

    • サイズ指定プロパティの場合、 anchor-size()関数内の指定軸を変更し、 新旧方向で同じ相対関係を保つようにする。

      例：topとleftを入れ替える場合、 width: anchor-size(width)はheight: anchor-size(height)となる。

    • 自己アラインメントプロパティの場合、 directionsが対向なら、 指定された<self-position>（またはleft/rightキーワード）があれば、 新旧方向で同じ相対関係になるよう変更する。

      例：topとbottomを入れ替える場合、 align-self: startはalign-self: endとなる。

      ただし、align-self: centerは両方向とも関係が同じのため変更されない。

      また、align-self: first baselineも、 <baseline-position>であり、 <self-position>ではないので変更されない。

    • position-areaの場合、 position-area gridの選択行・列が 新旧方向で同じ相対関係になるよう値を変更する。

      例：topとleftを入れ替える場合、 position-area: left span-bottomは position-area: top span-rightになる。

  6. stylesを返す。

5.4. @position-tryルール

@position-tryルールは、 指定された名前でposition optionを定義します。 これはボックスに適用できる1つ以上の位置指定プロパティセットを指定し、 position-try-fallbacks経由で使用します。

@position-tryルールの構文は以下の通りです：

@position-try <dashed-ident> {
  <declaration-list>
}

前置で指定される<dashed-ident>はルール名です。 同じ名前で複数の@position-tryルールが宣言された場合、 文書順で最後のものが有効となります。

@position-tryルールは、次のプロパティのみ許可します：

  • insetプロパティ

  • marginプロパティ

  • サイズ指定プロパティ

  • 自己アラインメントプロパティ

  • position-anchor

  • position-area

<declaration-list>内のプロパティで!importantを使用するのは無効です。 それを使用したプロパティは無効となりますが、@property-tryルール全体は無効になりません。

@position-try内のすべてのプロパティは、 Position Fallback Origin経由でボックスに適用されます。 これは新しいカスケードオリジンであり、 作者オリジンとアニメーションオリジンの間に位置します。

アニメーションオリジンと同様に、 revert値の使用は、 そのプロパティが作者オリジンの一部であるかのように振る舞い、 代わりにユーザーオリジンに戻ります。 （アニメーションオリジンと同様、revert-layerは特別な挙動はなく指定通り動作します。）

注: 許可される@position-tryプロパティは、 ボックス自身のサイズと位置だけに影響し、内容やその他のスタイリングを変更しない最小限のプロパティ群です。 これによりposition fallbackの実装が大幅に簡素化され、 アンカー配置ボックスを空間に応じて移動するという本質的なニーズに対応できます。 これらのルールは作者オリジンの通常宣言を上書きするため、 @position-try宣言が 他のプロパティの通常のカスケードや継承と悪い相互作用を起こしにくくなっています。 今後のコンテナクエリの拡張により、 使用中のposition fallbackに応じて要素をクエリし、 この制限されたリストではできない条件付きスタイリングも可能になると期待されています。

注: 異なるアンカーを使う複数のボックスが同じフォールバック位置を使用したい場合、 それぞれのアンカー要素に対して相対的に配置するだけなので、 <anchor-name>はanchor()で省略し、 それぞれのボックスでposition-anchorを指定します。

注: 最も一般的なフォールバック位置の種類 （通常はアンカーの一方に配置し、必要なら反対側に反転） は、position-try-fallbacksのキーワードで自動的に実現できるため、 @position-tryを使う必要はありません。

5.5. 位置フォールバックの適用

配置ボックス（default scroll shiftでシフト済み）が インセット修正包含ブロックからはみ出し、 position options listが空でない場合、 position fallback stylesの決定を行い、 はみ出しを回避できるオプションを探します。

これら変更後のスタイルはインターリーブ経由で要素に適用されるため、 算出値に影響し（トランジション等も発生し得る）、 レイアウトや使用値に依存していても反映されます。

position optionの適用アルゴリズム： ボックス要素elに対し、 position option new stylesを与えて：

  1. new stylesをposition fallback originのカスケードに挿入し、 カスケードを解決して、 elの使用スタイルを決定する。

    これらのスタイル計算のために、 仮想的なanchor recalculation pointを計算し、 そこから得られる仮想の記憶されたスクロールオフセットを用いてelのスタイルを決定する。

  2. elの使用スタイルを返す。

位置フォールバックスタイルの決定アルゴリズム（要素absposに対して）：

  1. current stylesをabsposの現在の使用スタイル（過去のフォールバックの結果も含む場合あり）とする。

  2. 各optionについて、position options listを反復：

    1. optionが現在absposの直近成功position optionであれば、continue（次のoptionへ）。

    2. adjusted stylesをposition optionの適用でoptionをabsposに適用した結果とする。

    3. el rectをabsposのmargin boxのサイズ・位置、 cb rectをabsposのインセット修正包含ブロックのサイズ・位置 （adjusted stylesでレイアウト時）とする。

    4. cb rectがいずれかの軸で負のサイズとなり、ゼロサイズに修正された場合は、continue（次のoptionへ）。

      注: ゼロサイズel rectが負サイズcb rect内とみなされて成功と判定されるのを防ぐため。

    5. el rectがcb rect内に完全に収まらない場合、continue（次のoptionへ）。

    6. adjusted stylesと、それに仮想的に計算された記憶されたスクロールオフセットのセットを返す。

  3. アサート：前ステップで、はみ出しを回避できるposition optionが見つからず終了した。

  4. current stylesを返す。

注: elの子孫がはみ出してもこの計算には影響しない。el自身のmargin boxのみ判定対象。

注: 現在有効なposition optionは意図的にスキップするため、 その記憶されたスクロールオフセットは更新されない。 他のフォールバックが使われず現状維持の場合は、すべての記憶されたスクロールオフセットもそのまま。

全レイアウトパス中、 ボックスがフォールバックスタイル（または未使用と判定）を決定した後は、 それ以降にレイアウトされる他のボックスがこの決定を変更することはできません。

例：二つの配置ボックスAとBがあり、Aが先にレイアウトされ、BがAの包含ブロックにスクロールバーを追加するほどはみ出す場合でも、 Aは再度フォールバックスタイルを再決定しようとはしません。 （最悪の場合、レイアウトが循環し収束しません。）

つまりレイアウトは「後戻り」しません。

ResizeObserver イベントが決定・配信されるタイミング：

  • ボックスelが絶対位置指定の場合、 直近成功position optionを、現在使用中のaccepted @position-try properties（と値）に設定する。

  • それ以外の場合、elが直近成功position optionを持ち、 以下のいずれかがtrueなら、その直近成功position optionを削除する：

    • elが絶対位置指定でなくなった場合

    • elのposition-try-fallbacksの算出値が変化した場合

    • elの@position-tryルールのいずれかが追加・削除・変更された場合

    その後、位置フォールバックスタイルの決定をelに対して実行し、 直近成功position optionを 現在使用中のaccepted @position-try properties（と値）に設定する。

注: この記録・削除のタイミングは、直近記憶サイズの扱いと意図的に一致しています。

実装は、position options listの長さに実装定義の上限を設けてもよいです。 この上限は最低でも5でなければなりません。

例：以下のCSSは「popover」をまず要素の下に配置し、画面に収まらなければ上に切り替えます。 デフォルトでは左揃えですが、収まらなければ右揃えにも切り替えます。 
#myPopover {
  position: fixed;
  top: anchor(--button bottom);
  left: anchor(--button left);
  position-try-fallbacks: flip-inline, flip-block, flip-block flip-inline;

  /* popover の幅は少なくともボタンと同じ */
  min-width: anchor-size(--button width);

  /* popover の高さは少なくともメニュー項目2つ分 */
  min-height: 6em;
}

6. アクセシビリティへの影響

アンカーポジショニングでは、配置されたボックスとそのアンカーとの間に自動的に意味的な関係が確立されるわけではないことに注意が必要です。 この手法は様々な用途で使えるためです。 著者は、要素同士を意味的に関連付ける際に、位置による視覚的な結びつきだけに頼ってはいけません。 追加の配慮がなければ、要素間に意味のあるDOM関係が全く存在しないことも多く、 視覚以外のユーザーエージェント（スクリーンリーダー等）では利用が困難または不可能になります。

ウェブプラットフォーム上には、既存および今後登場する多くの機能があり、 非視覚的なユーザーエージェントでも利用できるよう、こうした関係を明示的に確立できます。

例えば、HTMLのPopover APIでは、 呼び出しボタンとポップオーバー要素が自動的にリンクされ、 タブ順序も自動調整されます。 また、呼び出しボタンはポップオーバーの暗黙のアンカー要素としても確立されるため、 アンカーポジショニングも簡単に利用できます。

より一般的なケースでは、 ARIAの aria-details属性や aria-describedby属性などを アンカー要素に付与することで、 やや手動ですがこの情報を提供できます。 さらに、配置要素側に role 属性を指定すれば、 非視覚的なユーザーエージェントも 要素間の関係をユーザーに伝えたり、自動で要素間を移動できるようになります。

8. 付録：スタイル＆レイアウトのインターリーブ

スタイル＆レイアウトのインターリーブは、 レイアウト処理中にサブツリーのスタイル更新が発生し、 要素の算出スタイルが遡及的に更新される技法です。

この概念の正しい仕様書ではありません。 本来は Cascade に入れるべきですが、参照用のドラフトが必要です。

注: スタイル＆レイアウトのインターリーブは 既にコンテナクエリや コンテナクエリ長で使われています。 10cqwのような長さは、クエリコンテナのサイズのレイアウト情報を使って算出長に解決されるため、 レイアウト間でコンテナサイズが変化するとトランジションが発生します。

accepted @position-try propertiesも フォールバック解決時にインターリーブされます（position-try参照）。

明らかに詳細が必要ですが、 現状「コンテナクエリと同様に動作する」で十分です。 その挙動も未定義ですが、少なくともある程度相互運用可能です。
