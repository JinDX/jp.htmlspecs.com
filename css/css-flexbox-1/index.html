概要

この仕様は、ユーザーインターフェース設計に最適化されたCSSボックスモデルを説明します。フレックスレイアウトモデルでは、フレックスコンテナの子要素は任意の方向に配置でき、サイズを「フレックス」させることが可能です。余ったスペースを埋めるために拡大したり、親要素からはみ出さないように縮小したりできます。子要素の水平方向および垂直方向の配置は簡単に操作できます。これらのボックスを（水平の中に垂直、または垂直の中に水平など）入れ子にすることで、二次元のレイアウトを構築できます。

CSSは、構造化文書（例えばHTMLやXML）の描画を画面や紙などで記述するための言語です。

1. はじめに

このセクションは規定ではありません。

CSS 2.1では4つのレイアウトモード ― 兄弟や祖先ボックスとの関係に基づいて ボックスのサイズと位置を決定するアルゴリズム ― が定義されていました:

  • ブロックレイアウト（文書のレイアウト用）

  • インラインレイアウト（テキストのレイアウト用）

  • テーブルレイアウト（二次元データを表形式でレイアウトするため）

  • 配置レイアウト（他の要素をあまり考慮せず明示的な位置決め用）

このモジュールは新しいレイアウトモードフレックスレイアウト を導入します。これはより複雑なアプリケーションやウェブページのレイアウトに適しています。

1.1. 概要

このセクションは規定ではありません。

フレックスレイアウトは見た目上はブロックレイアウトと似ていますが、 floatやカラムなど ブロックレイアウトで使える複雑なテキスト・文書中心のプロパティが多くありません。 その代わりに、ウェブアプリや複雑なウェブページが必要とする スペース配分やコンテンツの揃え方に関する シンプルかつ強力なツールが得られます。 フレックスコンテナの内容:

  • 任意のフロー方向（左、右、下、上！）にレイアウト可能

  • 表示順序を反転したり、並び替えたりできる（つまり、視覚的な順序はソースや音声の順序と独立できます）

  • 単一の(メイン)軸に直線的にレイアウトしたり、第二の(交差)軸に複数行へ折り返しも可能

  • 「フレックス」サイズで利用可能なスペースに応じて伸縮できます

  • 第二の(交差)軸でコンテナや他要素と揃えることができます

  • メイン軸に沿って動的に折りたたみ／解除でき、 コンテナの交差サイズは保持されます

これは各アイテムにタイトル、写真、説明、購入ボタンがあるカタログの例です。 デザイナーの意図は、各エントリが同じ大きさで、 写真がテキストの上にあり、 購入ボタンが説明文の長さに関係なく下で揃うことです。 フレックスレイアウトはこの設計の多くの面を簡単にします:

  • カタログはアイテムの行を横方向にフレックスレイアウトで並べ、 行内のアイテムが全て同じ高さになるようにします。 各エントリ自体は縦方向のフレックスコンテナとなり、 内容を縦に並べます。

  • 各エントリ内のソース文書はタイトル→説明→写真の順で論理的に配置されます。 これにより音声レンダリングや非CSSブラウザでも適切な順序となります。 より魅力的な視覚的表示では、orderで 画像を内容の後方から上部に移動し、 align-selfで水平方向に中央揃えします。

  • 購入ボタンの上にautoマージンを設定することで、 説明文の高さに関係なくボタンが各エントリボックスの下部に配置されます。

#deals {
  display: flex;        /* フレックスレイアウトでアイテム 同じ高さにする  */
  flex-flow: row wrap;  /* アイテムを複数行に折り返し可能に */
}
.sale-item {
  display: flex;        /* 各アイテムをフレックスレイアウトで並べる */
  flex-flow: column;    /* アイテム内容を縦方向に配置  */
}
.sale-item > img {
  order: -1;            /* 画像を他の内容より前に（視覚順で） */
  align-self: center;   /* 画像を交差方向（水平）中央揃え         */
}
.sale-item > button {
  margin-top: auto;     /* 自動トップマージンでボタンを下へ */
}

<section id="deals">
  <section class="sale-item">
    <h1>コンピュータ スターターキット</h1>
    <p>これは、あまりお金がなくても買える最高のコンピュータです。
    <ul>
      <li>コンピュータ
      <li>モニター
      <li>キーボード
      <li>マウス
    </ul>
    <img src="images/computer.jpg"
         alt="内容物: 白いコンピュータと付属周辺機器">
    <button>今すぐ購入</button>
  </section>
  <section class="sale-item">
    …
  </section>
  …
</section>

1.2. モジュールの相互作用

このモジュールはdisplay プロパティ[CSS21]の定義を拡張し、 新しいブロックレベル・新しいインラインレベルのdisplay型を追加、 新しいフォーマットコンテキストの種類とそのレイアウト制御プロパティを定義します。 このモジュールで定義されたプロパティは、::first-lineや::first-letter疑似要素には適用されません。

CSSボックスアラインメントモジュールは 揃えプロパティ (justify-content, align-items, align-self, align-content) の定義を拡張し、置き換えます。

2. フレックスレイアウトのボックスモデルと用語

フレックスコンテナは、 計算されたdisplayが flex または inline-flexである要素が生成するボックスです。 フレックスコンテナ内のインフローの子はフレックスアイテムと呼ばれ、フレックスレイアウトモデルを使って配置されます。

ブロックやインラインレイアウトとは異なり、 それらのレイアウト計算はブロックおよびインラインのフロー方向に偏っていますが、 フレックスレイアウトはフレックス方向に偏っています。 フレックスレイアウトについて話しやすくするために、 このセクションではフレックスフロー相対の用語セットを定義します。 flex-flow値と書字モードによって、 これらの用語が物理的な方向（上／右／下／左）、 軸（垂直／水平）、サイズ（幅／高さ）にどのように対応するかが決まります。

メイン軸
メイン次元
メイン軸は、フレックスコンテナのフレックスアイテムが主に並ぶ軸です。 これはメイン次元方向に伸びます。
main-start
main-end
フレックスアイテムは、 コンテナのmain-start側から開始され、 main-end側へ向かって配置されます。
メインサイズ
メインサイズプロパティ
フレックスコンテナまたはフレックスアイテムの幅または高さのうち、 メイン次元に該当するものが、 そのボックスのメインサイズです。 そのメインサイズプロパティは width またはheightプロパティのうち、 メイン次元に該当するものとなります。 同様に、minおよびmax main size propertiesは min-width/max-widthまたはmin-height/max-heightプロパティのうち メイン次元に該当するものとなり、 min/max main sizeを決定します。
クロス軸
クロス次元
メイン軸に対して垂直な軸は クロス軸と呼ばれます。 これはクロス次元方向に伸びます。
cross-start
cross-end
フレックスラインはアイテムで満たされ、 コンテナのcross-start側から開始され、 cross-end側へ向かって配置されます。
クロスサイズ
クロスサイズプロパティ
フレックスコンテナまたはフレックスアイテムの幅または高さのうち、 クロス次元に該当するものが、 そのボックスのクロスサイズです。 そのクロスサイズプロパティは width またはheightプロパティのうち クロス次元に該当するものとなります。 同様に、minおよびmax cross size propertiesは min-width/max-widthまたはmin-height/max-heightプロパティのうち クロス次元に該当するものとなり、 min/max cross sizeを決定します。

この仕様で使われる追加のサイズ用語は、CSS 内在・外在サイズで定義されています。 [CSS-SIZING-3]

3. フレックスコンテナ: flex および inline-flex の display値

名前: display
新しい値: flex | inline-flex
flex
この値は、要素がフレックスコンテナボックスを生成し、 ブロックレベルとしてフローレイアウト内に配置されます。
inline-flex
この値は、要素がフレックスコンテナボックスを生成し、 インラインレベルとしてフローレイアウト内に配置されます。

フレックスコンテナは、 その内容に対して新しいフレックスフォーマットコンテキストを確立します。 これはブロックフォーマットコンテキストを確立するのと同じですが、 ブロックレイアウトの代わりにフレックスレイアウトが使われる点が異なります。 例えば、floatはフレックスコンテナ内に侵入せず、 フレックスコンテナのマージンは内容のマージンと折り重なりません。フレックスコンテナは、 内容に対してブロックコンテナ同様に包含ブロックを形成します。[CSS21] overflowプロパティはフレックスコンテナに適用されます。

フレックスコンテナはブロックコンテナではないため、 ブロックレイアウトを前提に設計された一部のプロパティは フレックスレイアウトでは適用されません。 特に:

  • floatとclearはフレックスアイテムの フロートやクリアランスを作成せず、アウトオブフローにもなりません。

  • vertical-alignはフレックスアイテムには効果がありません。

  • ::first-lineや::first-letter疑似要素はフレックスコンテナには適用されず、 フレックスコンテナは 祖先に最初の整形行や最初の文字を提供しません。

要素の指定されたdisplayが inline-flexである場合、 ある条件下でdisplayプロパティは flexに計算されます: CSS 2.1 セクション 9.7の表に "Specified Value"列にinline-flex、 "Computed Value"列にflexの行が追加されます。

4. フレックスアイテム

ざっくり言えば、フレックスアイテムは フレックスコンテナのインフロー内容を表すボックスです。

フレックスコンテナのインフローの各子は フレックスアイテムとなり、 子のテキストランの連続した部分は 匿名のブロックコンテナ フレックスアイテムでラップされます。 ただし、子のテキストラン全体が 空白文字（つまりwhite-spaceプロパティの影響を受ける文字）のみで構成される場合は、 レンダリングされません（テキストノードがdisplay:noneであるかのように扱われます）。

フレックスアイテムの例:

<div style="display:flex">

    <!-- フレックスアイテム: ブロックの子 -->
    <div id="item1">block</div>

    <!-- フレックスアイテム: float要素（floatは無視される） -->
    <div id="item2" style="float: left;">float</div>

    <!-- フレックスアイテム: インライン内容を囲む匿名ブロックボックス -->
    anonymous item 3

    <!-- フレックスアイテム: インラインの子 -->
    <span>
        item 4
        <!-- フレックスアイテムは ブロックの周囲で分割されません -->
        <q style="display: block" id=not-an-item>item 4</q>
        item 4
    </span>
</div>

要素間の空白は消えることに注意してください。 それ自体がフレックスアイテムにはなりませんが、 要素間テキストは匿名フレックスアイテムでラップされます。

また、匿名アイテムのボックスはスタイル不可であることにも注意してください。 要素がないためスタイルルールを割り当てできません。 ただし内容はフレックスコンテナから（フォント設定など）スタイルを継承します。

フレックスアイテムは 内容に対して独立したフォーマットコンテキストを確立します。 ただし、フレックスアイテム自体はフレックスレベルボックスであり、ブロックレベルボックスではありません: フレックスアイテムはコンテナのフレックスフォーマットコンテキストに属し、 ブロックフォーマットコンテキストには属しません。

注: この仕様を読む作者は以下のボックス生成や静的ポジションの詳細を飛ばしてもよいかもしれません。

display値は フレックスアイテムの場合 ブロック化されます: displayが インフローの子でインラインレベル値の場合は、ブロックレベル相当値に計算されます。 （詳細はCSS2.1§9.7 [CSS21] および CSS Display [CSS3-DISPLAY] を参照。）

注: displayの一部の値は 通常匿名ボックスの生成を引き起こします。 そのようなボックスがフレックスアイテムの場合、 先にブロック化されるため匿名ボックス生成は行われません。 たとえば、フレックスアイテムが2つ連続し、 display: table-cellの場合、 それぞれが独立したdisplay: block フレックスアイテムとなり、 1つの匿名テーブルにはなりません。

display: tableのフレックスアイテムの場合、 テーブルラッパーボックスがフレックスアイテムとなり、 orderやalign-selfプロパティが適用されます。 captionボックスの内容はテーブルラッパーボックスのmin-contentとmax-contentサイズ計算に寄与します。 また、widthや heightと同様に flexのロングハンドはテーブルボックスに以下のように適用されます: フレックスアイテムの最終サイズは テーブルラッパーボックスの端とテーブルボックスの内容端との距離が すべてテーブルボックスのborder+padding領域と見なされ、 テーブルボックス自体がフレックスアイテムであるかのように レイアウトすることで計算されます。

4.1. 絶対位置のフレックスの子

アウトオブフローであるため、 フレックスコンテナの絶対位置の子はフレックスレイアウトに参加しません。

静的ポジションは フレックスコンテナの絶対位置の子の場合、 子が唯一のフレックスアイテムであるかのように 配置されるよう決定されます。 このとき子とフレックスコンテナは使用サイズの固定サイズボックスと仮定します。 この目的のため、autoマージンはゼロ扱いとなります。

つまり、 静的位置矩形は フレックスコンテナの コンテンツボックスとなります。 静的位置矩形は 絶対位置ボックスの静的位置オフセット決定に使われる アラインメントコンテナです。

（ブロックレイアウトでは静的位置矩形は CSS2.1§10.3.7で説明される「仮想ボックス」の位置に対応します。 アラインメントプロパティがないため、 CSS2.1では常にblock-startおよび inline-startで 絶対位置ボックスを静的位置矩形内で揃えます。 この定義は将来的にCSS Positioningモジュールに移動予定です。）

たとえば、align-self: center;を フレックスコンテナの絶対位置の子に指定すると、 子のautoオフセットでフレックスコンテナの 交差軸の中央に配置されます。

ただし絶対位置ボックスは「固定サイズ」と見なされるため、 stretchは flex-startと同じ扱いとなります。

4.2. フレックスアイテムのマージンとパディング

隣接するフレックスアイテムのマージンは 折り重なりません。

フレックスアイテムの パーセンテージ指定のマージンとパディングは ブロックボックスと同様に 包含ブロックのインラインサイズに対して解決されます。 例えば左右上下のパーセンテージは 水平書字モードでは 全て包含ブロックのwidthに対して解決されます。

autoマージンは対応する方向で余ったスペースを吸収するように拡大します。 揃えや隣接するフレックスアイテム同士の間隔調整に使えます。 詳細はautoマージンによる揃えを参照してください。

4.3. フレックスアイテムのZ順序

フレックスアイテムは インラインブロックと全く同じように描画されます[CSS21]。 ただし orderによって修正された文書順が 通常の文書順の代わりに使われ、 z-index が auto 以外の場合は、 positionが staticでも スタッキングコンテキストを作ります （positionが relativeである場合と同じ挙動）。

注: フレックスアイテム外に配置された子孫も フレックスアイテムが確立したスタッキングコンテキストには参加します。

4.4. 折りたたみアイテム

visibility:collapseをフレックスアイテムに指定すると、 そのアイテムは折りたたみフレックスアイテムとなります。 これはvisibility:collapseをテーブル行やテーブル列に指定した場合と似た効果を生みます。 折りたたみフレックスアイテムは完全にレンダリングから除外されますが、 フレックスラインの交差サイズを安定させるための「支柱（strut）」が残ります。 そのため、フレックスコンテナにフレックスラインが1本しかない場合、 アイテムを動的に折りたたみ・解除すると フレックスコンテナのメインサイズは変わる可能性がありますが、 交差サイズには影響せず、他のページレイアウトが「ぐらつく」ことはありません。 ただし折りたたみ後はフレックスラインの折り返しが再計算されるため、 複数行がある場合は交差サイズが変わる場合もあります。

折りたたみフレックスアイテムはレンダリングされませんが、 フォーマット構造には現れます。 そのため、display:noneアイテム[CSS21]とは異なり、 ボックスがフォーマット構造に存在することに依存する効果 （カウンターのインクリメントやアニメーション・トランジションの実行など）は 折りたたみアイテムにも適用されます。

次の例では、 サイドバーが内容に合わせてサイズ調整されます。visibility: collapseを使って、 ナビゲーションサイドバーの一部を動的に非表示にしますが、幅には影響しません。 一番幅広いアイテム（「Architecture」）が折りたたみセクションにあっても、 サイドバーの幅は変わりません。

    order: 1;        /* ナビゲーションの右側（後）に配置 */
    min-width: 12em; /* メインコンテンツ領域サイズの最適化 */
  }
}

  display: flex;
  flex-flow: column;
}
/* ターゲットでないサブメニューを動的に折りたたみ */
nav > ul > li:not(:target):not(:hover) > ul {
  visibility: collapse;
}

支柱（strut）のサイズを計算するには、まずすべてのアイテムを折りたたまない状態でフレックスレイアウトを実行し、 その後、折りたたみアイテムを元のラインの交差サイズを維持する支柱に置き換えて再度実行します。 visibility:collapseがフレックスレイアウトとどのように作用するかの規定は フレックスレイアウトアルゴリズムを参照してください。

4.5. フレックスアイテムの自動最小サイズ

注: auto キーワードは 自動最小サイズを表し、 min-widthとmin-heightプロパティの新しい初期値です。 このキーワードは以前は本仕様で定義されていましたが、現在はCSS Sizingモジュールで定義されています。

最小サイズのより妥当なデフォルト値を フレックスアイテムに提供するため、 スクロールコンテナでないフレックスアイテムにおける メイン軸の自動最小サイズの使用値は 内容ベースの最小サイズとなります。 スクロールコンテナの場合は従来通り自動最小サイズはゼロです。

一般的に、内容ベースの最小サイズは 内容サイズの提案値と 指定サイズの提案値 のうち小さい方です。 ただし、ボックスにアスペクト比があり指定サイズがなければ、 内容ベースの最小サイズは 内容サイズの提案値と 転送サイズの提案値のうち小さい方です。 指定サイズの提案値もアスペクト比もなければ、 内容ベースの最小サイズは 内容サイズの提案値です。

内容サイズの提案値、 指定サイズの提案値、 転送サイズの提案値 はこの計算で関連するmin/max/希望サイズプロパティを考慮し、 内容ベースの最小サイズが 作者が指定した制約を妨げないようになっています。 以下で定義します：

指定サイズの提案値
アイテムの計算されたメインサイズプロパティが確定値の場合、 指定サイズの提案値はそのサイズ （max main size propertyが確定値ならそれでクランプ）。 それ以外の場合は未定義。
転送サイズの提案値
アイテムに固有のアスペクト比があり、 計算されたクロスサイズプロパティが確定値なら、 転送サイズの提案値はそのサイズ （min/maxクロスサイズプロパティが確定値ならそれでクランプ）、 アスペクト比で変換。 それ以外の場合は未定義。
内容サイズの提案値
内容サイズの提案値は min-contentサイズ（メイン軸において）であり、 アスペクト比があれば、min/maxクロスサイズプロパティが確定値ならそれでクランプし、さらにmax main size propertyが確定値ならそれでもクランプ。

ボックスの内在サイズ（例：min-contentサイズ）計算の目的では、 内容ベースの最小サイズは その軸でサイズが未確定（indefinite）になります （たとえば、widthプロパティが確定値を指定していても）。 このため、このサイズに対して計算されるパーセンテージは autoと同様に扱われます。

それでも、場合によってはパーセンテージ再解決のため追加レイアウトパスが必要ですが、 この値は （min-content、max-content、fit-contentなど [CSS-SIZING-3]で定義される値と同様）、 アイテム内のパーセンテージサイズ解決を妨げません。

内容ベースの最小サイズは多くの場合適切で、 内容がコンテナからはみ出たり重なるのを防ぐ助けになりますが、 場合によっては適切でないこともあります。

特に、文書の主要な内容領域でフレックスサイズを使う場合は、 min-width: 12emのようなフォント相対の明示的な最小幅を指定する方が良いです。 内容ベースの最小幅では、大きなテーブルや画像が 全体の内容領域をオーバーフローするほど引き伸ばし、 行の長さが不必要に長くなって読みづらくなることがあります。

また、内容ベースのサイズが大きな内容を持つアイテムに使われる場合、 レイアウトエンジンは最小サイズ決定のために全内容を走査する必要がありますが、 作者が明示的に最小値を指定していればこの必要はありません。 （内容量が少ないアイテムなら走査はわずかなのでパフォーマンス問題にはなりません。）

5. 並び順と方向

フレックスコンテナの内容は、任意の方向・順序でレイアウトできます。 これにより、以前は複雑または不安定な手法（例えばfloatや clearプロパティを使ったハック）を必要とした効果も、簡単に実現できます。 この機能はflex-direction、 flex-wrap、 orderプロパティで提供されます。

注: フレックスレイアウトの並び替え機能は、視覚的なレンダリングのみに影響するよう設計されています。 音声順やソース順に基づくナビゲーションには影響しません。 これにより、作者は視覚的な表示を操作しつつ、ソース順を非CSSユーザーエージェントや 音声・線形ナビゲーションモデル用に維持できます。 この二面性を利用したアクセシビリティ向上の例は 並び替えとアクセシビリティや フレックスレイアウト概要を参照してください。

作者は、orderや *-reverseの値（flex-flow/flex-direction）を 正しいソース順の代わりに使ってはいけません。 これは文書のアクセシビリティを損なう原因となります。

5.1. フレックスフロー方向：flex-direction プロパティ

名前: flex-direction
値: row | row-reverse | column | column-reverse
初期値: row
適用対象: フレックスコンテナ
継承: no
パーセンテージ: 該当なし
算出値: 指定されたキーワード
規範順序: 文法順
アニメーションタイプ: 離散型

flex-directionプロパティは、 フレックスアイテムが フレックスコンテナ内でどのように並ぶかを指定し、 コンテナのメイン軸の方向を設定します。 これによってフレックスアイテムの並び方向が決まります。

row
フレックスコンテナのメイン軸は 現在のインライン軸と同じ向きです。 main-startと main-end方向は、 それぞれ現在のinline-startおよびinline-end方向に等しいです。
row-reverse
rowと同じですが、 main-startと main-end方向が入れ替わります。
column
フレックスコンテナのメイン軸は 現在のブロック軸と同じ向きです。 main-startと main-end方向は、 それぞれ現在のblock-startおよびblock-end方向に等しいです。
column-reverse
columnと同じですが、 main-startと main-end方向が入れ替わります。

注: reverse値はボックスの順序自体を反転しません。 writing-modeやdirection（[CSS3-WRITING-MODES]）と同様に、 フロー方向だけが変化します。 描画順序、音声順序、シーケンシャルナビゲーション順には影響しません。

5.2. フレックスラインの折り返し：flex-wrapプロパティ

名前: flex-wrap
値: nowrap | wrap | wrap-reverse
初期値: nowrap
適用対象: フレックスコンテナ
継承: no
パーセンテージ: 該当なし
算出値: 指定されたキーワード
規範順序: 文法順
アニメーションタイプ: 離散型

flex-wrapプロパティは、 フレックスコンテナが単一行か複数行か、 また新しい行が積み重なる交差軸の方向を制御します。

nowrap
フレックスコンテナは単一行です。
wrap
フレックスコンテナは複数行です。
wrap-reverse
wrapと同じです。

wrap-reverse以外の値の場合、 cross-start方向は 現在のinline-startまたはblock-start方向 （交差軸のいずれか）に等しく、 cross-end方向は cross-startの逆方向です。 flex-wrapがwrap-reverseの場合、 cross-startと cross-end方向が入れ替わります。

5.3. フレックス方向と折り返し：flex-flowショートハンド

名前: flex-flow
値: <‘flex-direction’> || <‘flex-wrap’>
初期値: 各プロパティを参照
適用対象: 各プロパティを参照
継承: 各プロパティを参照
パーセンテージ: 各プロパティを参照
算出値: 各プロパティを参照
アニメーションタイプ: 各プロパティを参照
規範順序: 文法順

flex-flowプロパティは、 flex-directionと flex-wrapプロパティをまとめて設定するショートハンドであり、 両者でフレックスコンテナのメイン軸と交差軸を定義します。

英語（左から右、横書き）文書における有効なフローの例： 
div { flex-flow: row; }
/* 初期値。メイン軸はインライン方向、折り返しなし。
   （アイテムは縮小して収まるか、はみ出す。） */

div { flex-flow: column wrap; }
/* メイン軸はブロック方向（上から下）
   行はインライン方向（右）へ折り返し。 */

div { flex-flow: row-reverse wrap-reverse; }
/* メイン軸はインライン方向の逆（右から左）。
   新しい行は上方向に折り返し。 */
注: flex-flowの方向は書字モードに依存します。 たとえば日本語の縦書きでは、 rowのフレックスコンテナは内容を上から下へ並べます。 この例を見てください：
英語 日本語 
flex-flow: row wrap;        
writing-mode: horizontal-tb;
 
flex-flow: row wrap;        
writing-mode: vertical-rl;

5.4. 表示順序：orderプロパティ

フレックスアイテムは、デフォルトではソース文書の順序で表示・レイアウトされます。 orderプロパティを使ってこの順序を変更できます。

名前: order
値: <integer>
初期値: 0
適用対象: フレックスアイテム
継承: no
パーセンテージ: 該当なし
算出値: 指定された整数値
規範順序: 文法順
アニメーションタイプ: 算出値型による

orderプロパティは、 フレックスアイテムを序数グループに割り当てて、フレックスコンテナ内の表示順序を制御します。 値は1つの<integer>で、どの序数グループに属すかを指定します。

フレックスコンテナは内容をorder修正文書順でレイアウトします。 序数グループの値が低いものから高いものへ並び、 同じ序数グループ内ではソース文書の順序で並びます。 これは描画順序[CSS21]にも影響し、 フレックスアイテムがソース順で並び替えられた場合と同じ挙動となります。 フレックスコンテナの絶対位置の子は、 order: 0とみなして、 フレックスアイテムとの描画順序を決定します。

下図はアクティブなペインのタブが常に一番最初になるシンプルなタブインターフェースの例です：

これは以下のCSSで実装できます（主要部分のみ）：

.tabs {
  display: flex;
}
.tabs > .current { 
  order: -1; /* デフォルトの0よりも低い値 */
}

今後の仕様で特に指定されない限り、このプロパティはフレックスアイテム以外のボックスには効果がありません。

5.4.1. 並び替えとアクセシビリティ

orderプロパティは 非視覚メディア（音声など）での順序に影響しません。 同様に、orderは シーケンシャルナビゲーションモード（例えばリンクの循環、tabindex [HTML]など）のデフォルト巡回順序にも影響しません。

作者はorderを 視覚的な並び替えだけに使い、論理的な並び替えには使わなければなりません。 orderで論理的並び替えを行うスタイルシートは非適合です。

注: 非視覚メディアや非CSSユーザーエージェントは通常、内容を線形に提示するため、 論理的なソース順に依存できます。 一方でorderは視覚順序を調整するのに利用されます。 （視覚的認識は二次元かつ非線形であるため、望ましい視覚順が論理順と一致しない場合があります。）

多くのウェブページは、マークアップ上は 上部にヘッダー、下部にフッター、中央にコンテンツと左右に追加のカラムという形をしています。 一般的には、コンテンツを追加カラムより先にページのソースコードに記述した方が望ましいです。 しかしこれでは、たとえば追加カラムを左、コンテンツを右にしたい一般的なレイアウトが難しくなります。 これは様々な方法で工夫されてきましたが、追加カラムが2つある場合「Holy Grail Layout」と呼ばれることが多かったです。 orderを使えばこれも簡単です。 例として、ページコードと希望レイアウトのスケッチ： 
<!DOCTYPE html>
<header>...</header>
<main>
   <article>...</article>
   <nav>...</nav>
   <aside>...</aside>
</main>
<footer>...</footer>

このレイアウトはフレックスレイアウトで容易に実現できます：

main { display: flex; }
main > article { order: 2; min-width: 12em; flex:1; }
main > nav     { order: 1; width: 200px; }
main > aside   { order: 3; width: 200px; }

さらに、各カラムはデフォルトで同じ高さとなり、 メインコンテンツは画面幅いっぱいまで広がります。 さらにメディアクエリと組み合わせれば、画面が狭い場合に全て縦並びにもできます：

@media all and (max-width: 600px) {
  /* 3カラムにするのは幅が足りない場合は不可 */
  main { flex-flow: column; }
  main > article, main > nav, main > aside {
    /* ドキュメント順に戻す */
    order: 0; width: auto;
  }
}

（複数行フレックスコンテナを使えばさらに賢い折り返しも可能。詳細は読者への課題。）

すべての表示モードで作者の意図した順序を保つため、 オーサリングツール（WYSIWYGエディタやWebベースのオーサリング支援など）は 基本的にドキュメントソースを並び替える必要があり、 orderによる並び替えは 作者が明示的に「ドキュメント順（音声やナビゲーション順）」と「視覚順」が一致しないことを望んだ場合のみ使うべきです。

例えば、あるツールはフレックスアイテムのドラッグ＆ドロップ並べ替えと、画面サイズごとのレイアウト変更のためのメディアクエリ対応の両方を提供する場合があります。

多くの場合、並べ替えはすべての画面サイズ範囲だけでなく、ナビゲーション順や音声順にも影響するべきなので、ツールはDOM層でドラッグ＆ドロップ並べ替えを行います。 しかし場合によっては、著者が画面サイズごとに異なる視覚的な順序を希望することもあります。 このような場合、ツールはorderとメディアクエリを組み合わせて機能を提供しつつ、 最も小さい画面サイズの順序はDOMの順序に紐づける（これは論理的な線形提示順である可能性が高いため）一方、 その他のサイズ範囲ではorderで提示順を決定することができます。

このツールは適合となりますが、ドラッグ＆ドロップ並べ替えをorderだけで処理する（実装は簡便でも）ようなツールは非適合となります。

注: ユーザーエージェント（ブラウザや支援技術、拡張機能など）は空間ナビゲーション機能を提供する場合があります。 このセクションは、そうした空間ナビゲーションモードでorderプロパティを順序決定に考慮することを否定するものではありません。 実際、この機能を実現するには考慮が必要です。 ただし、orderは空間ナビゲーション機能で考慮すべきCSSプロパティの一部に過ぎず、 主なものですらありません。 空間ナビゲーションを十分に実装するには、 空間関係を変更するCSSのすべてのレイアウト機能を考慮する必要があります。

6. フレックスライン

フレックスアイテムはフレックスコンテナ内で フレックスライン （レイアウトアルゴリズムがグループ化や揃えに使う仮想的なコンテナ）に配置・整列されます。 フレックスコンテナは単一行にも複数行にもなり得ます。 これはflex-wrapプロパティによって決まります：

  • 単一行フレックスコンテナ（flex-wrap: nowrap指定時）は、 すべての子要素を1行に並べます。 その結果内容がはみ出しても1行に収めます。

  • 複数行フレックスコンテナ（flex-wrap: wrapまたはflex-wrap: wrap-reverse指定時）は、 フレックスアイテムを複数行に分割して並べます。 これは、テキストが行に収まらない時に改行されるのと似ています。 追加の行が作られる際は、 交差軸方向にflex-wrapプロパティに従って積み重ねられます。 各行には必ず1つ以上のフレックスアイテムが含まれます。 （フレックスコンテナが完全に空の場合のみ例外。）

この例は、4つのボタンが横並びで収まらず、 複数行に折り返される様子を示しています。 
#flex {
  display: flex;
  flex-flow: row wrap;
  width: 300px;
}
.item {
  width: 80px;
}

<div id="flex">
  <div class="item">1</div>
  <div class="item">2</div>
  <div class="item">3</div>
  <div class="item">4</div>
</div>

コンテナの幅が300pxなので、1行には3つのアイテムしか収まりません。 3つで240pxを使い、残り60pxのスペースが余ります。 flex-flowプロパティにより （値に複数行指定キーワードである wrapが含まれているため）、 フレックスコンテナは残りのアイテム用に追加の行を作成します。

内容が行に分割されると、各行は独立してレイアウトされます。 フレキシブル長やjustify-content、 align-selfプロパティは 1行ごとにその行のアイテムのみ考慮します。

複数行のフレックスコンテナ（1行しかない場合も含む）では、 各行の交差サイズは その行内のフレックスアイテムを収めるために必要な最小サイズになります （align-selfによる整列後）。 行同士はalign-contentプロパティで フレックスコンテナ内に揃えられます。 単一行のフレックスコンテナでは、 行の交差サイズは フレックスコンテナ自身の交差サイズとなり、 align-contentは効果を持ちません。 行のメインサイズは常に フレックスコンテナのコンテンツボックスのメインサイズと同じです。

前の例と同じですが、 今度はすべてのフレックスアイテムにflex: autoが指定されています。 1行目には60pxの余剰スペースがあり、 全アイテムの柔軟性が同じなので、3つのアイテムそれぞれに20pxずつ追加幅が割り当てられ、 最終的に各アイテムは100px幅となります。 残りのアイテムは独立した行に配置され、その幅は行全体（300px）に広がります。

7. 柔軟性

フレックスレイアウトの特徴は、フレックスアイテムが「フレックス」する能力にあります。 メイン次元の利用可能なスペースに合わせて 幅や高さを変化させます。 これはflexプロパティによって行われます。 フレックスコンテナはアイテムのflex grow factorに比例して 余ったスペースを分配したり、 flex shrink factorに比例して はみ出しを防ぐために縮小したりします。

フレックスアイテムは、 flex-growとflex-shrinkの値がともに0なら 完全に柔軟性なし、 それ以外は柔軟性ありです。

7.1. flex ショートハンド

名前: flex
値: none | [ <‘flex-grow’> <‘flex-shrink’>? || <‘flex-basis’> ]
初期値: 0 1 auto
適用対象: フレックスアイテム
継承: no
パーセンテージ: 各プロパティを参照
算出値: 各プロパティを参照
アニメーションタイプ: 算出値型による
規範順序: 文法順

flex プロパティは可変長の構成要素を指定します： flex要素（growとshrink）、 およびflex basisです。 ボックスがフレックスアイテムの場合、flexが main size propertyの代わりにボックスのメインサイズを決定するために参照されます。 ボックスがフレックスアイテムでない場合、 flexは効果を持ちません。

<‘flex-grow’>
この<number> 成分はflex-grow ロングハンドを設定し、 flex grow要素を指定します。 これはフレックスアイテムが フレックスコンテナ内の他のフレックスアイテムに対して、 正の余剰スペース分配時にどれだけ拡大するかを決めます。 省略時は1となります。
flex値が0～1の間は少し特殊です： 行内のflex値合計が1未満の場合、余剰スペースの100%未満しか使われません。

アイテムのflex-grow値は 余剰スペースの割合のリクエストとして機能し、 1は「余剰スペースの100%」を意味します。 行内アイテムの合計が100%を超えてリクエストした場合は、 比率は維持したままちょうど100%分配されます。 逆に合計が100%未満（例：3つのアイテムがflex-grow: .25の場合）は、 それぞれリクエスト通り（25%ずつ）割り当てられ、 残りの25%は未使用となります。 詳しくは§9.7 柔軟長解決を参照してください。

この挙動はflex-growがゼロに近づく時の連続性維持に必要です。 これがなければflex-grow: 1のアイテムも flex-grow: 0.1のアイテムも flex-grow: 0.01のアイテムも すべて余剰スペースを全て使うことになり、 最終的にゼロにアンダーフローした瞬間だけ余剰スペースを全く使わなくなります。 この挙動により、値が1未満になるにつれ余剰スペースの割当は徐々に減り、 ゼロで完全に使わなくなります。

この「部分充填」挙動が特に欲しい場合以外は、値は1以上にするのが推奨されます。 例えば1と2を使う方が .33と.67より、 アイテム追加、削除、折り返し時にも望ましい挙動になりやすいです。

<‘flex-shrink’>
この<number> 成分はflex-shrink ロングハンドを設定し、 flex shrink要素を指定します。 これはフレックスアイテムが フレックスコンテナ内の他のフレックスアイテムに対して、 負の余剰スペース分配時にどれだけ縮小するかを決めます。 省略時は1となります。

注: flex shrink要素は、 flex基本サイズと 負のスペース分配時に掛け算されます。 これにより、アイテムが縮小可能な度合いに応じて縮小割合が決まり、 例えば小さいアイテムが大きいアイテムより先にゼロになることなく、 大きいアイテムが十分縮小された後に小さいアイテムが縮小されます。

<‘flex-basis’>
この成分はflex-basis ロングハンドを設定します。 これはflex基準値を指定します： フレックスアイテムの 初期メインサイズであり、 flex要素による余剰スペース分配前のサイズです。

<‘flex-basis’>はwidthやheightプロパティ（ただしautoは特別扱い）と、 contentキーワードと同じ値を受け付けます：

auto
フレックスアイテムで指定した場合、 autoキーワードは main size propertyの値を 使用flex-basisとして取得します。 その値自体がautoなら、使用値はcontentになります。
content
自動サイズをフレックスアイテムの内容に基づいて決定します。 （通常はmax-contentサイズに等しいですが、 アスペクト比、内在サイズ制約、直交フローなどに対応する調整があります。 詳細は§9 フレックスレイアウトアルゴリズムを参照。）

注: この値はFlexible Box Layout初期リリースでは存在しなかったため、 古い実装ではサポートされない場合があります。 同等の効果は、autoとwidthまたはheightのautoを組み合わせることで実現可能です。

<‘width’>
その他の値では、flex-basisはwidthやheightと同様に解決されます。

flexショートハンドで省略した場合、指定値は0となります。

none
noneキーワードは0 0 autoに展開されます。

flexの初期値はflex: 0 1 autoと同等です。

注: flex-growと flex-basisの初期値は、 flex ショートハンドで省略した場合のデフォルト値とは異なります。 これはflexショートハンドが一般的なケースにより適合するようにするためです。

単位なしゼロが2つのflex要素より前にない場合は、flex要素として解釈しなければなりません。 誤解釈や無効な宣言を避けるため、 作者はゼロの<‘flex-basis’>成分を単位付きで指定するか、 2つのflex要素を先に記述してください。

7.1.1. flexの基本値

下記リストは、最も一般的に使われるflexの4つの値の効果をまとめたものです：

flex: initial
flex: 0 1 autoと同等。（これが初期値。） アイテムのサイズはwidth/heightプロパティで決定されます。 （アイテムのmain size propertyがautoに算出された場合は、内容に基づいてサイズ決定。） 正の余剰スペースがあればフレックスアイテムは拡大しませんが、 スペースが不足すれば最小サイズまで縮小します。 揃え機能やautoマージンでメイン軸上での整列が可能です。

デフォルトでは、フレックスアイテムは最小内容サイズ（最長の単語や固定サイズ要素の長さ）より縮小しません。 これを変えるにはmin-widthまたはmin-heightプロパティを設定してください。 （§4.5 フレックスアイテムの自動最小サイズ参照。）

8. 揃え（アラインメント）

フレックスコンテナの内容がフレックス処理を終え、 すべてのフレックスアイテムの寸法が確定した後、 アイテムをフレックスコンテナ内で揃えることができます。

marginプロパティは、ブロックレイアウトよりも強力な方法で アイテムの揃えに使うことができます。フレックスアイテムはまたCSS Box Alignmentのアラインメントプロパティにも対応し、 メイン軸（main axis）と交差軸（cross axis）の両方で簡単にキーワード指定で揃えられます。 これらのプロパティにより、CSS 2.1では困難だった 水平・垂直中央揃えなど多くの一般的な揃えが簡単に実現できます。

注: 揃えプロパティはCSS Box Alignment [CSS-ALIGN-3]で定義されていますが、 Flexible Box Layoutでは、仕様の進行を遅らせないよう 関連する定義をここでも再現しています。 これらのプロパティは、CSS Box Alignment Level 3が完成し 他のレイアウトモードへの作用が規定されるまでは、フレックスレイアウトだけに適用されます。 また、Box Alignmentモジュールで新しく定義された値は Flexible Box Layoutにも適用されます。 つまり、Box Alignmentモジュール完成時には ここでの定義が置き換えられます。

8.1. autoマージンによる揃え

このセクションは規定ではありません。 マージンがフレックスアイテムに与える影響の規定はフレックスレイアウトアルゴリズムセクションにあります。

フレックスアイテムのautoマージンは、ブロックフローのautoマージンと非常によく似た効果を持ちます：

  • フレックスベースやフレキシブル長計算時、autoマージンは0として扱われます。

  • justify-contentやalign-selfによる揃えの直前に、 その方向の正の余剰スペースがautoマージンに分配されます。

  • はみ出したボックスはautoマージンを無視し、end方向にはみ出します。

注: 余剰スペースがautoマージンに分配される場合、 その方向のアラインメントプロパティは効果を持ちません。 なぜならflex処理後に残ったスペースはすべてマージンで消費されてしまうからです。

メイン軸でのautoマージンの使い方の一例は、 フレックスアイテムを「グループ」に分けることです。 下記例は、よくあるUIパターン ― 左側にいくつかのアクション、右側に別のアクションを揃えたバー ― の再現方法です。 
nav > ul {
  display: flex;
}
nav > ul > #login {
  margin-left: auto;
}

<nav>
  <ul>
    <li><a href=/about>About</a>
    <li><a href=/projects>Projects</a>
    <li><a href=/interact>Interact</a>
    <li id="login"><a href=/login>Login</a>
  </ul>
</nav>

名前: justify-content
値: flex-start | flex-end | center | space-between | space-around
初期値: flex-start
適用対象: フレックスコンテナ
継承: no
パーセンテージ: 該当なし
算出値: 指定キーワード
規範順序: 文法順
アニメーションタイプ: 離散型

justify-contentプロパティは、 フレックスコンテナの現在の行のメイン軸に沿ってフレックスアイテムを揃えます。 これはフレキシブル長やautoマージンの解決後に行われます。 通常は、行内のフレックスアイテムが 柔軟性なしの場合や最大サイズに到達した場合に残る余剰スペースの分配に使われます。 また、アイテムが行をオーバーフローした場合も整列に影響します。

flex-start
フレックスアイテムを行の開始側に寄せます。 行の最初のフレックスアイテムの main-startマージン端が 行のmain-start端にぴったり合い、 続く各フレックスアイテムは 直前のアイテムにぴったり合うよう配置されます。
flex-end
フレックスアイテムを行の終了側に寄せます。 行の最後のフレックスアイテムの main-endマージン端が 行のmain-end端にぴったり合い、 それ以外の各フレックスアイテムは 次のアイテムにぴったり合うよう配置されます。
center
フレックスアイテムを行の中央に寄せます。 行内のフレックスアイテムは 互いにぴったり合い、行の中央で整列されます。 行のmain-start端と最初のアイテムの間、 行のmain-end端と最後のアイテムの間に 等しいスペースが取られます。 （余剰スペースが負なら、フレックスアイテムは 両端に均等にはみ出します。）
space-between
フレックスアイテムを行内で均等に分配します。 余剰スペースが負の場合や行内にフレックスアイテムが1つしかない場合は flex-startと同じになります。 それ以外の場合、 行の最初のフレックスアイテムの main-startマージン端は 行のmain-start端にぴったり合い、 行の最後のフレックスアイテムの main-endマージン端は 行のmain-end端にぴったり合い、 残りのフレックスアイテムは 隣り合うアイテム間のスペースがすべて等しくなるよう分配されます。
space-around
フレックスアイテムを行内で均等に分配しますが、 両端は半分のスペースになります。 余剰スペースが負の場合や行内にフレックスアイテムが1つしかない場合は centerと同じになります。 それ以外の場合、行内のフレックスアイテムは 隣り合うフレックスアイテム間のスペースがすべて等しく、 最初／最後のフレックスアイテムとフレックスコンテナ端とのスペースは アイテム間スペースの半分の大きさになります。

3つの色付きアイテムを持つフレックスコンテナに対するjustify-contentキーワード5種の効果の図解。

8.3. 交差軸方向の揃え：align-itemsおよびalign-selfプロパティ

名前: align-items
値: flex-start | flex-end | center | baseline | stretch
初期値: stretch
適用対象: フレックスコンテナ
継承: no
パーセンテージ: 該当なし
算出値: 指定キーワード
規範順序: 文法順
アニメーションタイプ: 離散型
名前: align-self
値: auto | flex-start | flex-end | center | baseline | stretch
初期値: auto
適用対象: フレックスアイテム
継承: no
パーセンテージ: 該当なし
算出値: 指定キーワード
規範順序: 文法順
アニメーションタイプ: 離散型

フレックスアイテムは、フレックスコンテナの現在の行の交差軸方向に揃えることができます。 （justify-contentと似ていますが、直交方向で作用します。） align-itemsはコンテナのすべてのアイテム（匿名フレックスアイテムも含む）のデフォルトの揃えを設定します。 align-selfは個々のフレックスアイテムごとに このデフォルトの揃えを上書きできます。 （匿名フレックスアイテムのalign-self値は常にコンテナのalign-items値と一致します。）

フレックスアイテムの交差軸方向マージンがどちらかautoなら、 align-selfは効果がありません。

値の意味は以下の通りです：

auto
親ボックスの交差軸揃え制御に委ねます。 （align-selfの初期値。）
flex-start
フレックスアイテムのcross-startマージン端が 行のcross-start端にぴったり合います。
flex-end
フレックスアイテムのcross-endマージン端が 行のcross-end端にぴったり合います。
center
フレックスアイテムのマージンボックスが 行内の交差軸の中央に配置されます。 （フレックスラインの交差サイズがアイテムより小さい場合、 両端にはみ出します。）
baseline
フレックスアイテムは ベースライン揃えに参加します。 参加する全フレックスアイテムは ベースラインが揃うよう配置され、 ベースラインからcross-startマージン端までの距離が最大のアイテムは 行のcross-start端にぴったり合います。 必要軸にベースラインがなければ、 ボーダーボックスから合成されます。
stretch
フレックスアイテムの交差サイズプロパティがautoに算出され、 交差軸方向のマージンがどちらもautoでなければ、 フレックスアイテムは伸張状態となります。 使用値はアイテムのマージンボックスの交差サイズが 行のサイズにできるだけ近くなる値で、 min-height/min-width/max-height/max-widthの制約を守ります。

注: フレックスコンテナの高さが制約されている場合、 この値でアイテムの内容がはみ出すことがあります。

フレックスアイテムのcross-startマージン端は 行のcross-start端にぴったり合います。

8.4. フレックスラインのパッキング：align-contentプロパティ

名前: align-content
値: flex-start | flex-end | center | space-between | space-around | stretch
初期値: stretch
適用対象: 複数行のフレックスコンテナ
継承: no
パーセンテージ: 該当なし
算出値: 指定キーワード
規範順序: 文法順
アニメーションタイプ: 離散型

align-contentプロパティは、 フレックスコンテナ内のラインを、交差軸方向の余剰スペースがある場合に コンテナ内で揃えます。 （justify-contentがアイテムをメイン軸で揃えるのに似ています。） このプロパティは単一行のフレックスコンテナには効果がありません。 値の意味は以下の通りです：

flex-start
ラインはフレックスコンテナの開始側に寄せて配置されます。 コンテナ内の最初のラインのcross-start端は コンテナのcross-start端にぴったり合い、 続く各ラインは直前のラインにぴったり合うよう配置されます。
flex-end
ラインはフレックスコンテナの終了側に寄せて配置されます。 コンテナ内の最後のラインのcross-end端は コンテナのcross-end端にぴったり合い、 それ以外の各ラインは次のラインにぴったり合うよう配置されます。
center
ラインはフレックスコンテナの中央に寄せて配置されます。 コンテナ内のラインは互いにぴったり合い、中央に整列されます。 コンテナのcross-start端と最初のラインの間、 コンテナのcross-end端と最後のラインの間に 等しいスペースが取られます。 （余剰スペースが負なら、ラインは両端にはみ出します。）
space-between
ラインはフレックスコンテナ内で均等に分配されます。 余剰スペースが負の場合やコンテナ内にフレックスラインが1本しかない場合は flex-startと同じになります。 それ以外の場合、 コンテナ内の最初のラインのcross-start端は コンテナのcross-start端にぴったり合い、 最後のラインのcross-end端は コンテナのcross-end端にぴったり合い、 残りのラインは隣り合うライン間のスペースがすべて等しくなるよう分配されます。
space-around
ラインはフレックスコンテナ内で均等に分配されますが、両端は半分のスペースになります。 余剰スペースが負の場合は centerと同じになります。 それ以外の場合、コンテナ内のラインは隣り合うライン間のスペースがすべて等しく、 最初／最後のラインとフレックスコンテナ端とのスペースは ライン間スペースの半分の大きさになります。
stretch
ラインが残りのスペースを均等に伸張します。 余剰スペースが負の場合は flex-startと同じになります。 それ以外の場合は余剰スペースが全ラインで均等に分配され、交差サイズが増加します。

注: 複数行のフレックスコンテナだけが、ラインを交差軸方向で揃える余剰スペースを持ちます。 単一行フレックスコンテナでは、唯一のラインが自動的にスペースいっぱいに伸張します。

8.5. フレックスコンテナのベースライン

フレックスコンテナ自身がベースライン揃えに参加（例えばフレックスコンテナ自身が外側のフレックスアイテムとして別のフレックスコンテナ内にある場合など）するには、 内容を最もよく表すベースライン位置を提出する必要があります。 そのために、フレックスコンテナのベースラインは次のように決定されます （orderによる並べ替えとflex-directionを考慮した後）:

first/last メイン軸ベースラインセット
インライン軸がフレックスコンテナのメイン軸と一致する場合、 ベースラインは以下のように決定されます:

  1. フレックスコンテナの開始／終了側のフレックスライン上のフレックスアイテムのうち いずれかがベースライン揃えに参加する場合、 フレックスコンテナのfirst/last メイン軸ベースラインセットは それらフレックスアイテムの 共有アラインメントベースラインから 生成されます。

  2. それ以外で、フレックスコンテナに1つ以上のフレックスアイテムがある場合、 first/last メイン軸ベースラインセットは 開始／終了側のフレックスアイテムの アラインメントベースラインから 生成されます。 （そのアイテムがフレックスコンテナのメイン軸に平行なアラインメントベースラインを持たない場合は、 まずボーダー端から合成されます。)

  3. それ以外の場合、フレックスコンテナはfirst/lastメイン軸ベースラインセットを持ちません。 必要に応じ合成され、 アラインメントコンテキストの規則に従います。

first/last 交差軸ベースラインセット
インライン軸がフレックスコンテナの交差軸と一致する場合、 ベースラインは以下のように決定されます:

  1. フレックスコンテナに1つ以上のフレックスアイテムがある場合、 first/last 交差軸ベースラインセットは 開始／終了側のフレックスアイテムの アラインメントベースラインから 生成されます。 （そのアイテムがフレックスコンテナの交差軸に平行なアラインメントベースラインを持たない場合は、 まずボーダー端から合成されます。)

  2. それ以外の場合、フレックスコンテナはfirst/last交差軸ベースラインセットを持ちません。 必要に応じ合成され、 アラインメントコンテキストの規則に従います。

上記規則でベースラインを計算する際、 ベースラインを提供するボックスのoverflow値がスクロール可能な場合は、 ベースライン決定のためにボックスは初期スクロール位置として扱われます。

テーブルセルのベースライン決定時には、 フレックスコンテナはline boxやテーブル行と同様にベースラインを提供します。[CSS21]

ベースラインの詳細は、CSS Writing Modes 3 §4.1 ベースラインの概要や CSS Box Alignment 3 §9 ベースライン揃えの詳細を参照してください。

9. フレックスレイアウトアルゴリズム

このセクションはフレックスコンテナとその内容の正確なレイアウト挙動を定義する規定アルゴリズムです。 ここでのアルゴリズムは可読性と理論的な単純さを最適化する形で書かれており、 必ずしも最も効率的とは限りません。 実装は実際のアルゴリズムが何であっても構いませんが、 ここで記載したアルゴリズムと同じ結果を生成しなければなりません。

注: このセクションは主に実装者向けです。 ウェブページを書く作者は個々のプロパティの説明で十分であり、 CSSレイアウトの詳細を深く理解したい特殊な場合以外はこのセクションを読む必要はありません。

以下のセクションでフレックスコンテナとその内容のレイアウトアルゴリズムを定義します。

注: フレックスレイアウトはフレックスアイテムをorder修正文書順で処理し、 元の文書順ではありません。

9.1. 初期セットアップ

  1. 匿名フレックスアイテムを生成（§4 フレックスアイテム参照）。

9.2. ライン長の決定

  1. フレックスアイテムのメイン／交差スペースの利用可能領域を決定。 各次元について、 フレックスコンテナの内容ボックスのその次元が確定サイズならそれを使い、 フレックスコンテナがその次元でminまたはmax-content制約下でサイズ決定される場合は その制約が利用可能スペース、 それ以外はフレックスコンテナのマージン・ボーダー・パディングを その次元の利用可能スペースから差し引いた値を使う（無限値になる場合もあり得ます）。

    例えば、利用可能スペースは floatされた autoサイズのフレックスコンテナの場合：

    • フレックスコンテナの包含ブロックの幅から フレックスコンテナの 水平方向のマージン・ボーダー・パディングを引いた値
    • 垂直方向は無限大
  2. 各アイテムのflex基本サイズと仮メインサイズを決定：
    1. アイテムの使用flex basisが確定値なら、 それがflex基本サイズ。
    2. アイテムが...
      • 固有アスペクト比を持つ
      • 使用flex basisがcontent
      • 交差サイズが確定値
      の場合、flex基本サイズは 内部交差サイズと アイテムの固有アスペクト比から計算されます。
    3. 使用flex basisがcontentまたは利用可能スペースに依存する場合、 かつフレックスコンテナがmin-contentやmax-content制約下でサイズ決定されている場合 （例：自動テーブルレイアウト [CSS21]実行時）、 アイテムはその制約下でサイズ決定する。 flex基本サイズは アイテムの結果メインサイズになる。
    4. それ以外で、使用flex basisがcontentまたは利用可能スペースに依存し、 メインサイズが無限大であり、アイテムのインライン軸がメイン軸に平行な場合は、 直交フローボックスの規則[CSS3-WRITING-MODES]でレイアウト。 flex基本サイズは アイテムのmax-contentメインサイズ。

      注: このケースは例えば、 英語（横書き）文書内のカラムフレックスコンテナに、 縦書き日本語（縦書き）フレックスアイテムが入っている場合に起こります。

    5. それ以外の場合は 使用flex basisをメインサイズの代わりに使い 利用可能スペースでサイズ決定。 content値はmax-contentとして扱う。 メインサイズ決定に交差サイズが必要（例：アイテムのメインサイズがブロック軸の場合）で アイテムの交差サイズがautoかつ確定値でない場合は、 この計算では交差サイズにfit-contentを使う。 flex基本サイズは アイテムの結果メインサイズ。

    flex基本サイズ決定時、 アイテムのmin/maxメインサイズは無視され（クランプしない）、 box-sizing適用時にcontent boxサイズを0で切り捨てる計算も無視される。 （例えば、サイズ指定が0・パディングが正・box-sizing: border-boxの場合、 外部flex基本サイズは0となり 内部flex基本サイズは負になる。）

    仮メインサイズは アイテムのflex基本サイズを 使用min/max使用値でクランプし（content boxサイズは0で切り捨て）たもの。

  3. フレックスコンテナのメインサイズを 参加しているフォーマットコンテキストの規則で決定。 この計算時、フレックスアイテムのautoマージンは0として扱う。

9.3. メインサイズの決定

  1. フレックスアイテムをフレックスラインに収集：
    • フレックスコンテナが単一行の場合、 すべてのフレックスアイテムを1つのフレックスラインに収集。
    • それ以外の場合は、 収集されていない最初のアイテムから始めて、 連続するアイテムを1つずつ収集し、 次の収集アイテムがフレックスコンテナの内部メインサイズに 収まらなくなる（または強制改行に遭遇する、§10 フレックスレイアウトの分割参照）まで収集。 最初の未収集アイテムが収まらない場合は、それだけをそのラインに収集。

      このステップでは フレックスアイテムのサイズは外部仮メインサイズ（負値もあり得る）として扱う。

      すべてのフレックスアイテムがフレックスラインに収集されるまで繰り返す。

      「できるだけ収集」のラインでは、 サイズゼロのフレックスアイテムも直前のラインの末尾に収集される（最後の非ゼロアイテムがちょうどラインを埋めても）。

  2. 柔軟長を解決し、すべてのフレックスアイテムの使用メインサイズを求める。§9.7 柔軟長の解決参照。

9.4. 交差サイズの決定

  1. 各アイテムの仮交差サイズを決定 使用メインサイズと利用可能スペースでレイアウトを行い、 autoはfit-contentとして扱う。
  2. 各フレックスラインの交差サイズを計算。

    フレックスコンテナが単一行で、確定の交差サイズを持つ場合、 フレックスラインの交差サイズは フレックスコンテナの内部交差サイズとなる。

    それ以外の場合は、各フレックスラインごとに:

    1. インライン軸がメイン軸と平行で、align-selfがbaseline、 かつ交差軸マージンが両方ともautoでないアイテムを収集。 各アイテムのベースラインから仮外部cross-start端までの距離の最大値と、 ベースラインから仮外部cross-end端までの距離の最大値をそれぞれ求めて、 その2つを合計する。
    2. 上記で収集されなかったアイテムのうち、 仮交差サイズの最大値を求める。
    3. フレックスラインの使用交差サイズは、 上記2ステップで求めた数値と0のうち最大のもの。

      フレックスコンテナが単一行の場合、 ラインの交差サイズをコンテナの計算されたmin/max交差サイズの範囲にクランプする。CSS 2.1のmin/max-width/height定義が一般化されていれば、 この挙動は自動的に導出されることになる。

  3. 'align-content: stretch'の処理。 フレックスコンテナが確定交差サイズを持ち、align-contentがstretchで、 フレックスラインの交差サイズ合計がコンテナの内部交差サイズより小さい場合は、 各フレックスラインの交差サイズを均等に増やして 合計がコンテナの内部交差サイズと一致するようにする。
  4. visibility:collapseアイテムの折りたたみ。 フレックスアイテムにvisibility: collapseが指定されている場合、 そのアイテムが属するラインの交差サイズをアイテムのstrutサイズとして記録し、 レイアウトを最初からやり直す。

    2回目のレイアウトラウンドでは、 アイテムをラインに収集する際に、 折りたたみアイテムはメインサイズがゼロとみなす。 以降のアルゴリズムでは折りたたみアイテムは完全に無視する （display:noneと同様）、 ただしラインの交差サイズを計算した後に、 ラインの交差サイズがそのライン内の折りたたみアイテムのstrutサイズの最大値未満の場合は、 そのstrutサイズに交差サイズを設定する。

    2回目のレイアウトラウンドではこのステップはスキップ。

  5. 各フレックスアイテムの使用交差サイズを決定。 フレックスアイテムがalign-self: stretchで、 計算された交差サイズプロパティがautoで、 交差軸マージンがどちらもautoでない場合は、 使用外部交差サイズはそのフレックスラインの使用交差サイズで、 アイテムの使用min/max交差サイズでクランプする。 それ以外の場合は、 使用交差サイズはアイテムの仮交差サイズとなる。

    フレックスアイテムがalign-self: stretchの場合は、 この使用サイズを確定交差サイズとして内容の再レイアウトを行い、 子要素のパーセンテージサイズを解決できるようにする。

    このステップは、アイテムのメインサイズには影響しません（固有アスペクト比があっても）。

9.5. メイン軸方向の揃え

  1. 残った余剰スペースを分配。 各フレックスラインごとに:
    1. 残りの余剰スペースが正で、このライン上にメイン軸マージンがautoのものが1つ以上ある場合は、 余剰スペースをそれらマージンで均等分配。 それ以外は、すべてのautoマージンをゼロに設定。
    2. justify-contentに従って アイテムをメイン軸で揃える。

9.6. 交差軸方向の揃え

  1. 交差軸autoマージンの解決。 フレックスアイテムが交差軸マージンにautoを持つ場合:
    • その外部交差サイズ（autoマージンはゼロ扱い）と ラインの交差サイズの差が正なら、 その差をautoマージンで均等分配。
    • それ以外で、block-startまたはinline-startマージン（交差軸側）がautoならゼロに設定。 反対側マージンはアイテムの外部交差サイズがラインの交差サイズと一致するように設定。
  2. すべてのフレックスアイテムを交差軸方向に揃える アイテムの交差軸マージンが両方ともautoでなければalign-selfに従う。
  3. フレックスコンテナの使用交差サイズを決定:
    • 交差サイズプロパティが確定値ならそれを使い、 フレックスコンテナの使用min/max交差サイズでクランプ。
    • それ以外は、 フレックスラインの交差サイズ合計を使い、 フレックスコンテナの使用min/max交差サイズでクランプ。
  4. すべてのフレックスラインを揃える align-contentに従う。

9.7. 柔軟長の解決

フレックスライン内のアイテムの柔軟長を解決するには:

  1. 使用flex要素を決定。 ライン上のすべてのアイテムの外部仮メインサイズを合計。 合計がフレックスコンテナの内部メインサイズ未満なら、 以下のアルゴリズムでflex grow要素を使用; そうでなければflex shrink要素を使用。

9.8. 確定サイズと非確定サイズ

CSS Sizing [CSS-SIZING-3] は 確定と非確定長を定義していますが、 Flexboxでは長さが確定とみなされる追加ケースがあります：

  1. 単一行のフレックスコンテナが確定交差サイズを持つ場合、 任意の伸張状態のフレックスアイテムの外部交差サイズはコンテナの内部交差サイズ（アイテムのmin/max交差サイズでクランプ）となり、確定とみなされる。
  2. フレックスコンテナが確定メインサイズを持つ場合、 フレックスアイテムのフレックス処理後のメインサイズは 確定とみなされる（同じライン内の他アイテムが非確定サイズでも）。
  3. フレックスラインの交差サイズが決定された後は、 autoサイズのフレックスコンテナ内のアイテムも レイアウト上は確定とみなされる。ステップ11参照。

注: 完全に柔軟性なしアイテムで、確定のflex basisを持つ場合、定義上確定となる。

9.9. 内在サイズ

内在サイズは、フレックスコンテナの 内容ベースの自動サイズ決定で使われます。 例えば shrink-to-fit 論理幅（fit-content式を使用）や 内容ベースの論理高さ（max-content size式を使用）などです。

このセクションの用語定義は [CSS-SIZING-3] を参照してください。

9.9.1. フレックスコンテナの内在メインサイズ

max-content メインサイズ（最大内容メインサイズ）は、 フレックスコンテナが 各フレックスアイテムの max-content貢献値を アイテム自身の柔軟性の範囲で保ちつつ取り得る最小サイズです：

  1. 各フレックスアイテムについて、 外部flex基本サイズを max-content貢献値サイズから引きます。 その結果が正なら、 flex grow要素（1で切り捨て）で割ります。 負なら、 スケール済みflex shrink要素（flex shrink要素を1で切り捨て）で割ります。 これがアイテムのmax-content flex fractionです。
  2. すべてのフレックスアイテムを 無限長のラインに配置します。
  3. 各ライン内で、 全フレックスアイテムの max-content flex fractionの最大値を求めます。 各アイテムのflex基本サイズに flex grow要素（または選択されたmax-content flex fractionが負なら スケール済みflex shrink要素）を掛け、 その結果に選択されたmax-content flex fractionを乗じて加算し、 さらにmax main size（min main sizeで切り捨て）でクランプします。
  4. フレックスコンテナの max-content sizeは、 以上で計算したサイズのうち、単一ライン内のアイテム合計が最大となる値です。

min-content メインサイズ（最小内容メインサイズ）は、 単一行フレックスコンテナの場合、 max-contentメインサイズと同様に計算しますが、 フレックスアイテムの min-content貢献値を max-content貢献値の代わりに使います。 ただし複数行コンテナの場合、 コンテナ内すべてのmin-content貢献値の最大値になります。

flex合計が1未満の場合のこのアルゴリズムの意味

上記アルゴリズムは主に以下2つのケースで正しい挙動を与え、 両者間でフレックスコンテナサイズが連続的に変化するよう設計されています：

  1. すべてのアイテムが柔軟性なしの場合、 フレックスコンテナのサイズは アイテムのflex基本サイズの合計。 （柔軟性なしのflex基本サイズは 幅・高さと同じ役割を持ち、 それが指定されればBlock Layoutでのmax-content貢献値の基準になります。）

  2. すべてのアイテムが柔軟性あり（flex要素≥1）の場合、 フレックスコンテナのサイズは アイテムのmax-content貢献値合計 （あるいはそれより少し大きい値となり、 すべてのアイテムが 少なくとも max-content貢献値サイズになり、 かつ柔軟性による比率も正しく維持されます）。

例えば、フレックスコンテナに フレックスアイテムが1つあり flex-basis: 100px;だがmax-contentサイズが200pxの場合、 flex-grow: 0のときは フレックスコンテナ（とアイテム）は100px幅、 flex-grow: 1以上なら フレックスコンテナ（とアイテム）は200px幅。

この2ケース間の挙動を連続的にする方法は複数ありえますが、どれも欠点があります。 本仕様では最も悪影響が少ないと思われる方法を採用しています。 ただし、flex合計が1未満の場合は柔軟性を「二重適用」することになります。 上記例でflex-grow: .5 の場合、フレックスコンテナの幅は150pxとなり、 アイテムは通常どおりそのスペースにサイズ決定され、最終的に125px幅となります。

9.9.2. フレックスコンテナの内在交差サイズ

min-content/max-content 交差サイズは、 単一行 フレックスコンテナの場合、 コンテナ内すべてのmin-content貢献値／max-content貢献値（それぞれ）の最大値となります。

複数行 フレックスコンテナの場合、 min-content/max-content 交差サイズは、 コンテナを交差軸のmin-content制約/max-content制約（それぞれ）でサイズ決定したときの フレックスライン交差サイズ合計となります。 ただし、フレックスコンテナが flex-flow: column wrap;の場合は、 まずすべてのmin-content／max-content 交差サイズ貢献値（それぞれ）の最大値を求め、 そのサイズをレイアウト時に各フレックスアイテムの利用可能スペース（交差軸側）として使います。

注: この column wrap フレックスコンテナのアルゴリズムは フレックスコンテナの理想サイズの近似値を与えてくれます。 各フレックスアイテムは min(自身のmax-content, 全アイテム中最大min-content) となり、 各フレックスラインは最大のフレックスアイテムより大きくなりません。 一部のケースでは完全ではありませんが、 完全に正しくやろうとすると非常にコストが高く、 この方法が現実的に十分機能します。

9.9.3. フレックスアイテムの内在サイズ貢献値

メインサイズmin-content貢献値は、 フレックスアイテムの 外部min-contentサイズと外部希望サイズ（width/height適用）で autoでないもののうち大きい方を、 grow不可ならflex基本サイズで最大値として（shrink不可なら最小値として）クランプし、 さらにmin/max main sizeでクランプした値です。

メインサイズmax-content貢献値は、 フレックスアイテムの 外部max-contentサイズと外部希望サイズ（width/height適用）で grow不可ならflex基本サイズで最大値として（shrink不可なら最小値として）クランプし、 さらにmin/max main sizeでクランプした値です。

10. フレックスレイアウトの分割

フレックスコンテナはページ間で、 アイテム間、 アイテムのライン間（複数行モード時）、 アイテム内部でも分割できます。 break-*プロパティは ブロックレベルやインラインレベルボックスと同様に フレックスコンテナに適用されます。 このセクションではフレックスアイテムおよびその内容への適用方法を定義します。 詳しくはCSS Fragmentation Module [CSS3-BREAK]を参照してください。

以下の分割規則では、 分割コンテナを「ページ」と呼びます。 他の分割コンテキストでも同じ規則が適用されます。 （必要に応じて「ページ」を適切な分割コンテナ種に置き換えてください。） このセクションでは「row」「column」は、 フレックスコンテナが 分割コンテキストのブロックフロー方向に対して どちら向きかを指します（コンテナ自身の方向ではなく）。

分割されたフレックスコンテナの正確なレイアウトは 本Flexible Box Layoutレベルでは定義されていません。 ただしフレックスコンテナ内の分割は、 以下の規則（order修正文書順で解釈）に従います：

  • rowフレックスコンテナでは、 フレックスアイテム上のbreak-beforeやbreak-after値は フレックスラインに伝播します。 最初のラインのbreak-beforeや 最後のラインのbreak-after値は フレックスコンテナに伝播します。

    注: breakの伝播 （text-decoration伝播のように） は算出値には影響しません。

  • columnフレックスコンテナでは、 最初のアイテムのbreak-before値や 最後のアイテムのbreak-after値は フレックスコンテナに伝播します。 強制的な分割は他のアイテム自身に適用されます。
  • フレックスアイテム内部で強制分割がある場合、 内容サイズが増加する効果となり、 兄弟アイテム内部では強制分割は発生しません。
  • rowフレックスコンテナでは Class A break機会は兄弟フレックスライン間、 Class C break機会は最初／最後のフレックスラインとコンテナ内容端間に発生。 columnフレックスコンテナでは Class A break機会は兄弟フレックスアイテム間、 Class C break機会は ライン内の最初／最後のフレックスアイテムとコンテナ内容端間に発生。 [CSS3-BREAK]
  • 分割後にフレックスコンテナが継続される場合、 フレックスアイテムの利用可能スペース（分割コンテキストのブロックフロー方向）は 先行ページのフレックスコンテナ断片が消費したスペース分だけ減少します。 コンテナ断片が消費したスペースは そのページの内容ボックスのサイズです。 この調整で利用可能スペースが負になる場合はゼロに設定。
  • フレックスコンテナの最初の断片がページの上端でなく、 残りスペースにフレックスアイテムが1つも収まらない場合は 断片全体が次ページへ移動します。
  • 複数行のcolumnフレックスコンテナが分割される場合、 各断片は自身の「フレックスラインのスタック」を持ち、 これはマルチカラムコンテナの各断片が自身のカラムボックスの行を持つのと同様です。
  • 前項によるアイテム並び替え以外は、 ユーザーエージェントは 非分割フローとの歪みを最小限にするよう努めるべきです。

10.1. フレックスレイアウト分割のサンプルアルゴリズム

この参考情報セクションでは、フレックスコンテナの分割アルゴリズム例を示します。 実装者はこのアルゴリズムを改良し、CSSワーキンググループへフィードバックを提供することが推奨されます。

このアルゴリズムはページネーションが常に前方にのみ進むと仮定しています。 そのため、以下のアルゴリズムではページネーション前の揃えはほぼ無視されます。 高度なレイアウトエンジンでは断片間でも揃えを考慮できる場合があります。

単一行カラムフレックスコンテナ
  1. フレックスレイアウトアルゴリズムを（ページ分割を考慮せずに）交差サイズの決定まで実行。
  2. 最初から連続するフレックスアイテムまたはその断片を、できるだけ多く（最低1つまたはその断片）ページ上にレイアウト。 ページの残りスペースがなくなるか、強制的な改ページに遭遇するまで繰り返す。
  3. 前ステップでスペースがなくなり、余剰スペースが正の場合は、 UAはこのページで分配された余剰スペースを減らす（ゼロまで、ただしマイナスは不可）ことで 次の分割できないフレックスアイテムや断片のスペースを確保してもよい。 それ以外の場合は、収まらないアイテムや断片は次ページに送る。 残りスペースの50%以上が断片に収まるなら引き上げ（pull up）、それ以外は送り出し（push）推奨。
  4. まだレイアウトされていないフレックスアイテムや断片がある場合は、 フレックスレイアウトアルゴリズムを ライン長の決定から交差サイズの決定まで 次ページのサイズ・全内容（既にレイアウト済みも含む）で再実行し、 前ステップに戻る。ただし未レイアウトの最初のアイテム・断片から再開。
  5. 各フレックスコンテナ断片ごとに、 メイン軸の揃えから終わりまで フレックスレイアウトアルゴリズムを継続。

このアルゴリズムの意図は、カラム方向の単一行フレックスコンテナが ブロックフローと非常に似たページ分割になることです。 試しに、justify-content:startかつ柔軟性なしのアイテムのみのフレックスコンテナは、 同じ内容・同じ使用サイズ・同じマージンのブロックと同じページ分割になるべきです。

複数行カラムフレックスコンテナ
  1. フレックスレイアウトアルゴリズムをページ分割を考慮して （フレックスコンテナの最大ライン長をページ残スペースに制限しながら） 交差サイズの決定まで実行。
  2. できるだけ多くのフレックスラインをレイアウト（最低1ライン）、 フレックスコンテナの交差次元でスペースがなくなるか強制改ページまで繰り返す:
    1. 最初から連続するフレックスアイテムを、できるだけ多く（最低1つ）ページ上にレイアウト。 ページの残りスペースがなくなるか強制改ページに遭遇するまで繰り返す。 アイテム内部の強制改ページは無視。
    2. 最初のフレックスコンテナ断片で このラインが1つだけのフレックスアイテムで、 それがページ残スペースより大きく、かつコンテナがすでにページ上端でない場合は、 フレックスコンテナを次ページに移動し、レイアウトを最初からやり直す。
    3. 前ステップで未レイアウトのアイテムがある場合は、 メインサイズの決定から交差サイズの決定まで そのライン未レイアウトアイテムのみで再実行し、 前ステップに戻る（未レイアウトアイテムの最初から再開）。
  3. 前ステップで未レイアウトのアイテムがある場合は、 ラインサイズの決定から交差サイズの決定まで 次ページのサイズ・未レイアウトアイテムのみで再実行し、 前ステップに戻る（未レイアウトアイテムの最初から再開）。
  4. 各フレックスコンテナ断片ごとに、 メイン軸の揃えから終わりまで フレックスレイアウトアルゴリズムを継続。

フレックスアイテムが1ページに完全に収まらない場合、 複数行カラムフレックスコンテナでは分割されません。

単一行ローフレックスコンテナ
  1. ページ分割を考慮せず、フレックスレイアウトアルゴリズム全体を実行。 ただしalign-selfがflex-startまたはbaseline以外の場合は flex-startとして扱う。
  2. 分割できないアイテムがページ残スペースに収まらず、かつコンテナがページ上端でない場合は、 フレックスコンテナを次ページに移動し、レイアウトを最初からやり直す。
  3. 各アイテムごとに、ページ残スペースに収まる内容だけレイアウトし、 残りは次ページに断片化してレイアウトアルゴリズムを ライン長の決定からメイン軸の揃えまで 新しいページサイズ・全内容（前ページで完了したアイテムも含む）で再実行。

    以前の断片に完全に収まったフレックスアイテムは、 後続断片でもメイン軸スペースを消費する。

  4. 各フレックスコンテナ断片ごとに、 交差軸の揃えから終わりまで フレックスレイアウトアルゴリズムを再実行。 最初以外の断片では、 align-selfとalign-contentは すべてのアイテム断片・ラインでflex-startとして扱う。
  5. アイテムのいずれかが、元のalign-self値で すべてのフレックスコンテナ断片の合計交差サイズ内に完全に収まる場合は、 その断片に移動し、適切に揃えてよい。
複数行ローフレックスコンテナ
  1. フレックスレイアウトアルゴリズムをページ分割を考慮せず、 交差サイズの決定まで実行。
  2. 最初からできるだけ多くのフレックスラインを（最低1ライン）、 ページ残スペースがなくなるか強制改ページまでレイアウト。

    ラインがページに収まらず、かつラインがページ上端でない場合は、 ラインを次ページに移動しレイアウトアルゴリズムを最初からやり直し、 そのライン以降のアイテムだけ使う。

    アイテム自身が強制改ページを引き起こす場合は、 フレックスレイアウトアルゴリズムを メインサイズの決定からメイン軸の揃えまで このライン以降のアイテムだけ使い再実行し、 改ページを引き起こしたアイテムはライン分割ステップで自動的に新しいライン開始、 その後このステップを継続。 アイテム内部の強制改ページは無視。

  3. 前ステップで未レイアウトのアイテムがある場合は、 ライン長の決定からメイン軸の揃えまで 次ページサイズ・未レイアウトアイテムのみで再実行。 前ステップに戻る（未レイアウトラインの最初から再開）。
  4. 各フレックスコンテナ断片ごとに、 交差軸の揃えから終わりまで フレックスレイアウトアルゴリズムを継続。

    フレックスコンテナのメインサイズはメインサイズプロパティで決定される。 この計算では、フレックスコンテナのmin-contentメインサイズはアイテムのmin-contentサイズ貢献値の最大値、 max-contentメインサイズはアイテムのmax-contentサイズ貢献値の合計。 アイテムのmin-content/max-contentメインサイズ貢献値は min-content/max-content制約下でサイズ決定した場合の外部仮メインサイズ。 この計算では、フレックスアイテムのautoマージンは0として扱う。

    max-content メインサイズは フレックスコンテナ内アイテムのmax-content貢献値（メイン軸）の合計。 min-content メインサイズは 単一行フレックスコンテナではアイテムのmin-content貢献値（メイン軸）の合計、 複数行コンテナではその最大値。

    min-content 交差サイズとmax-content 交差サイズは、 与えられた利用可能なメイン軸スペースと無限の交差軸スペースでレイアウトした後の交差サイズ。

    アイテムのメインサイズmin-content/max-content contributionは min-content/max-content制約下でサイズ決定した場合の外部仮メインサイズ。

    用語定義は[CSS-SIZING-3]参照。

    フレックスコンテナが1本のみフレックスライン（複数行コンテナでも）しかなく かつ確定交差サイズを持つ場合、 フレックスラインの交差サイズは フレックスコンテナの内部交差サイズです。

  • フレックスラインのサイズは0で切り捨てられるよう修正。 （Issue 2）

    フレックスラインの使用交差サイズは 大きい方 上記2ステップで求めた数値と0のうち最大値 です。

  • フレックスコンテナの絶対配置子の静的位置をGrid Layoutと一貫性を持たせて簡略化。 （Issue 6）

    フレックスコンテナの絶対配置子は再並べ替えステップ以外はフレックスレイアウトに参加しません。

    ただし、left・rightやtop・bottomがautoの場合は、使用値は静的位置から算出されます：

    left/right両方autoの場合は、そのアイテムのmain-start/cross-start端（水平軸のいずれか）が静的位置に揃えられます。 top/bottom両方autoの場合は、main-start/cross-start端（垂直軸のいずれか）が静的位置に揃えられます。

    メイン軸において：

    1. 同じフレックスラインに後続のインフローアイテムがあれば、そのmain-start端が静的位置。
    2. 前方のインフローアイテムがあれば、そのmain-end端が静的位置。
    3. それ以外は、静的位置はコンテナのjustify-content値による0サイズアイテムとして決定。

    交差軸において：

    1. 前方のインフローアイテムがあれば、そのフレックスラインのcross-start端が静的位置。
    2. それ以外は最初のフレックスラインのcross-start端が静的位置。

    静的位置は、flex-start揃えの匿名0×0インフローアイテムをフレックスレイアウトに参加させた場合の位置とほぼ一致する想定。 主な違いは、justify-content: space-around や space-betweenによるpackingスペースが仮アイテムの周囲で抑制される点。

    静的位置は絶対配置子を除いたフレックスレイアウトを一度完全に行い、 各絶対配置子を フレックスコンテナ内で唯一のフレックスアイテムとして配置した場合と同じ位置にします。 子・コンテナ両方が使用サイズで固定サイズボックスであると仮定します。

    例：デフォルトでは絶対配置子の静的位置はmain-start/cross-start端に揃います（コンテナのjustify-content/align-contentのデフォルト）。コンテナにjustify-content:centerを設定すると、メイン軸中央に揃います。

11. プライバシーとセキュリティへの配慮

Flexboxは新たなプライバシー情報漏洩や、 「正しく実装すること」以外のセキュリティ上の懸念を導入しません。

適合性

文書規約

適合性要件は、記述的な断言とRFC 2119の用語の組み合わせで表現されています。重要語「MUST」「MUST NOT」「REQUIRED」「SHALL」「SHALL NOT」「SHOULD」「SHOULD NOT」「RECOMMENDED」「MAY」「OPTIONAL」は、本仕様の規範部分においてRFC 2119で定義された通りに解釈してください。 ただし、可読性のため、これらの語は本仕様書ではすべて大文字では記述されていません。

本仕様書のすべての文章は、明示的に非規範と記されたセクション・例・注を除き、規範的です。[RFC2119]

本仕様書の例は「for example」という語で始まるか、class="example"のように規範的な本文と区別されています。例：

これは参考例の一例です。

参考注は「Note」で始まり、class="note"で本文と区別されます。例：

注：これは参考注です。

勧告文は、特別な注意喚起のために規範的なセクションとして強調され、<strong class="advisement">で他の規範本文と区別されます。例： UAはアクセシブルな代替手段を提供しなければならない。

適合クラス

本仕様への適合性は、以下の3つの適合クラスで定義されます：

スタイルシート
CSSスタイルシート。
レンダラー
UA。スタイルシートの意味を解釈し、それらを使う文書を描画します。
オーサリングツール
UA。スタイルシートを書き出します。

スタイルシートは、本モジュールで定義された構文を使用するすべての記述が、汎用CSS文法および本モジュールで定義された各機能の個別文法に従って有効であれば、本仕様に適合しています。

レンダラーは、スタイルシートを適切な仕様通りに解釈することに加え、本仕様で定義されたすべての機能を正しくパースし、文書を正しく描画することで本仕様に適合します。ただし、デバイスの制限によりUAが文書を正しく描画できない場合でも、UAが非適合となるわけではありません。（たとえば、UAがモノクロモニターの場合に色を描画する必要はありません。）

オーサリングツールは、汎用CSS文法および本モジュールの各機能の個別文法に従って文法的に正しいスタイルシートを書き出し、かつ本モジュールで定義されるすべてのスタイルシートの適合要件を満たしていれば、本仕様に適合します。

CSSの責任ある実装のための要件

以下のセクションでは、現在および将来の相互運用性を促進する形でCSSを責任を持って実装するための、いくつかの適合要件を定義します。

部分的実装

著者がフォールバック値を割り当てるために前方互換なパース規則を利用できるよう、CSSレンダラーは、サポートが不十分なatルール、プロパティ、プロパティ値、キーワード、その他構文要素を、無効として 適切に無視しなければなりません。 特にUAは、サポートされていない値のみを選択的に無視し、1つの複数値プロパティ宣言内のサポートされている値のみを適用してはなりません： いずれかの値が無効（＝サポートされていない値）とみなされる場合、CSSは宣言全体を無視することを要求します。

不安定・独自機能の実装

将来の安定したCSS機能との衝突を避けるため、CSSWGはベストプラクティスに従い、CSSの不安定な機能や独自拡張の実装を推奨します。

CRレベル機能の実装

仕様がCandidate Recommendation（CR）段階に達した場合、実装者は正しく仕様どおりに実装できることが示せるCRレベルの機能について、プレフィックスなしの実装を公開すべきです。 また、その機能のプレフィックス付きバリエーションを公開することは避けるべきです。

CSSの相互運用性を確立・維持するため、CSSワーキンググループは、非実験的なCSSレンダラーに対し、何らかのCSS機能のプレフィックスなし実装を公開する前に、実装レポート（必要に応じてその実装レポートで使ったテストケースも）をW3Cに提出するよう要請します。W3Cに提出されたテストケースは、CSSワーキンググループによるレビューおよび修正の対象となります。

テストケースおよび実装レポートの提出方法については、CSSワーキンググループのWebサイト（http://www.w3.org/Style/CSS/Test/）をご覧ください。 質問はpublic-css-testsuite@w3.orgメーリングリストへ。

CR終了基準

本仕様が提案勧告（Proposed Recommendation）へ進むためには、各機能につき少なくとも2つの独立した相互運用可能な実装が必要です。各機能は異なる製品で実装されていてもよく、すべての機能が単一製品で実装される必要はありません。この基準のために、以下の用語を定義します：

独立
各実装は別の当事者によって開発され、他の適格な実装で使用されたコードを共有、再利用、派生してはなりません。仕様実装と関係ないコード部分はこの要件の対象外です。
相互運用可能
公式CSSテストスイートの該当テストケースに合格すること、またはWebブラウザでない場合は同等のテストに合格すること。該当するすべてのテストは、該当UAを相互運用性主張に使う場合、同等のテストが用意されるべきです。さらに、そのUAを相互運用性主張に使う場合は、同じ方法でその同等テストに合格できる他のUAが最低1つ以上必要です。同等テストはピアレビューのために公開されなければなりません。
実装
ユーザーエージェント（UA）。次の条件を満たすもの：
  1. 仕様を実装していること。
  2. 一般公開されていること。リリース済み製品またはベータ版、プレビュー版、nightly build等でも可。 未リリース製品の場合、少なくとも1ヶ月間その機能を実装して安定性を示す必要あり。
  3. 実験的でないこと（テストスイート合格だけ目的のものでなく、今後通常利用されるもの）。

本仕様は少なくとも6ヶ月間Candidate Recommendationの状態を維持します。
