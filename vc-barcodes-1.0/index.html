概要

本仕様は、運転免許証（PDF417）や渡航文書（MRZ）に見られるような 光学バーコードを、検証可能な資格情報 [VC-DATA-MODEL-2.0] を使用して 保護する仕組みについて説明する。検証可能な資格情報の 表現は、150 バイト未満に収まるほどコンパクトであり、 そのため、標準的な印刷工程を使用して物理カードに 印刷される従来の二次元バーコードと統合できる。

                  1. はじめに

                  運転免許証、パスポート、渡航用クレデンシャルなどの物理的なクレデンシャルには、 文書から情報を迅速に読み取るために使用できる機械可読データが含まれていることがよくあります。 この情報は、PDF417 [ISO15438-2015]、 機械可読領域（MRZ）[ICAO9303-3]、 および 1次元または2次元の「バー」としてフォーマットされた、 その他の光学的にスキャン可能なコードなどの形式でエンコードされます。このため、 「バーコード」という用語が使用されます。この情報は改ざんから保護されていないことが多く、 バーコードの生成およびスキャン用ライブラリが容易に入手できるため、誰でもこれらの バーコードを生成することは比較的容易です。

                  したがって、これらのバーコードの発行者が、 バーコード内に含まれる情報だけでなく、そのバーコードを生成したエンティティも 保護できることは有用です。

                  検証可能なクレデンシャル・データモデルv2.0 仕様は、運転免許証や 渡航文書に含まれるようなクレデンシャル情報を表現するための世界的な標準を提供します。 検証可能なクレデンシャル・データ整合性1.0 仕様は、クレデンシャル情報を保護するための 世界的な標準を提供します。これら2つの仕様を組み合わせることで、 クレデンシャルを改ざんから保護し、 クレデンシャルの作成者を表現し、プライバシーを保護する方法でクレデンシャルの現在の ステータスを提供できます。ただし、これらのデータ形式は、 光学バーコードで表現するには大きすぎる傾向があります。

                  リンクトデータ向け コンパクトバイナリオブジェクト表現v1.0仕様は、保護された 検証可能な クレデンシャルを、情報を光学バーコードとして表現したり、 光学バーコード内に埋め込んだりすることが可能になる程度まで 圧縮する手段を提供します。

                  この仕様は、運転免許証（PDF417）や渡航文書（MRZ）に見られるような 光学バーコードを保護するための仕組みについて説明します。これは、 検証可能なクレデンシャル [VC-DATA-MODEL-2.0]を使用して、 バーコード自体またはバーコードの主体に関する情報を表現するものです。 次に、この検証可能なクレデンシャルはデータ整合性 [VC-DATA-INTEGRITY]を使用して保護され、 その後CBOR-LD [CBOR-LD]を使用して圧縮されます。

                  検証可能なクレデンシャルを使用してバーコードを保護するための 主な仕組みは2つあります。

                  • 検証可能なクレデンシャルを含むバーコードを、現在バーコードが付いていない 物理文書（新しい文書を含む）に追加できます。バーコード内の 検証可能なクレデンシャルには、文書内のデータの一部または全部を 含めることができます。このユースケースでは、通常、バーコードには 検証可能なクレデンシャルのみが含まれます。
                  • 文書にすでにバーコードが含まれている場合、そのバーコードを、 セキュリティを向上させる検証可能なクレデンシャルを含む新しいバーコードに 置き換えることで、セキュリティを改善できます。このシナリオでは、新しい拡張バーコードに、 元のバーコードの情報と、その他の情報にデジタル署名するために新たに追加された 検証可能なクレデンシャルの両方が含まれることが想定されます。

                  どちらのユースケースでも、検証可能なクレデンシャルによって保護されたデータについて、 真正性、整合性、および改ざん耐性が実現されます。さらに、それを必要とする ユースケースでは、これらの検証可能なクレデンシャルをバーコードごとに 失効または一時停止できます。

                  1.1.2 就労許可

                  このセクションでは、この仕様の技術を使用して、カードの裏面に 機械可読領域（MRZ）を使用する就労許可文書の機械可読領域を 保護する方法の例を示します。まず、就労許可文書の例から始めます。

                  就労許可文書の裏面には、光学文字認識によって読み取られるように 設計された情報を含む機械可読領域（MRZ）があります。

                  MRZデータには、この仕様で説明されているアルゴリズムを使用して 保護された情報が含まれています。具体的には、カードの表面にあるQRコードには、 カードの裏面の情報を保護する、次の形式の検証可能な クレデンシャルが含まれています。

                  例 3：カードの表面にQRコードとして表現された検証可能な クレデンシャル 
                  {
  "@context": [
    "https://www.w3.org/ns/credentials/v2",
    "https://w3id.org/citizenship/v2",
    "https://w3id.org/citizenship/utopia/v1"
  ],
  "type": [
    "VerifiableCredential",
    "OpticalBarcodeCredential"
  ],
  // 以下の値は、上記の「utopia/v1」コンテキストでURLとして定義されています
  "issuer": "did:web:immigration.utopia.example",
  "credentialSubject": {
    "type": "MachineReadableZone",
  },
  "proof": {
    "type": "DataIntegrity",
    "cryptosuite": "ecdsa-xi-2023",
    // 以下の値は、上記の「utopia/v1」コンテキストでURLとして定義されています

                  1.2 設計目標

                  この仕様における技術の設計目標は次のとおりです。

                  • 真正性：検証に成功した検証可能なクレデンシャル・バーコードは、 主張された発行機関から発行されたものであることが保証されます。

                  2.1 OpticalBarcodeCredential

                  OpticalBarcodeCredentialは、1. 作成者情報、2. 改ざん耐性、 および3. 任意で失効および一時停止ステータスを提供する方法で、光学 バーコードの内容を保護するために使用されます。言い換えると、このクレデンシャルは、 光学バーコードを誰が発行したか、最初に発行されてから光学バーコードが 改ざんされたかどうか、および光学バーコードの発行者が、 その文書が現在も有効であることを引き続き保証しているかどうかを示すことができます。 これらの機能は、物理文書に対して重要な不正防止保護を提供します。

                  OpticalBarcodeCredentialのcredentialSubjectの型は、 検証可能なクレデンシャル 自体に表現された情報に加えて、物理文書上にある、検証可能なクレデンシャルによって 保護される機械可読データの種類を識別します。この仕様では、 OpticalBarcodeCredentialとともに使用されるすべての credentialSubject型のスーパークラスとして機能する一般型 MachineReadableInformationを定義します。この仕様の主要部分では、 1つの具体的なサブクラス、MachineReadableZone（セクション 2.1.1 MachineReadableZoneを参照）を定義します。これは、 検証可能なクレデンシャルが 物理文書上の機械可読領域を保護することを示します。さらに2つのサブクラスが附属書で定義されています。 AamvaDriversLicenseScannableInformation（規範的な附属書A. AAMVA運転免許証および 身分証明書を参照）およびIsoDrivingLicenceMachineReadableInformation（ 参考情報である附属書B. ISO準拠の運転免許証を参照）です。その他の仕様は、 出生証明書や教育証明書上のPDF417バーコードなど、クレデンシャルが他の種類の 機械可読データを保護することを示す追加のcredentialSubject型を 定義してもかまいません（MAY）。

                  この仕様または別の仕様によって定義されるかどうかにかかわらず、 OpticalBarcodeCredentialとともに使用されるすべての credentialSubject型は、MachineReadableInformationの サブクラスとして定義されなければなりません（MUST）。 値MachineReadableInformationは、より具体的なサブクラス値とともに credentialSubjectのtype配列に含めてもかまいません（MAY）。特定のサブクラスを認識しないプロセッサは、 スーパークラス値を使用して、credentialSubjectが 検証可能なクレデンシャルによって 保護された機械可読データを記述していることを判断できます。

                  • その型のクレデンシャルが保護する、物理文書上の機械可読光学データ （例：PDF417バーコード、機械可読領域）。
                  • コンポーネント抽出規則、すなわちセクション 3.2.4.4 opticalDataBytesを作成の アルゴリズムで使用するため、光学データの保護対象コンポーネントを 文字列のリストとして抽出する方法。
                  • 抽出されたコンポーネントを決定論的なリストに並べ替えるために使用される 順序付け規則（その型が順序付け規則を定義しない場合は、セクション 3.2.4.4 opticalDataBytesを作成で説明されているように、Unicodeコードポイント順が使用されます）。
                  • 光学データのコンポーネントの一部のみがデジタル署名される場合、 ビット文字列の長さ、パディングビットの値、および各ビットが参照する コンポーネントの決定論的リストを含む、protectedComponentIndex プロパティの使用方法。

                  2.1.1 MachineReadableZone

                  型がMachineReadableZoneであるcredentialSubjectは、 検証可能な クレデンシャルが物理文書上の機械可読領域（MRZ） [ICAO9303-3] を保護することを示します。MachineReadableZoneは MachineReadableInformationのサブクラスです。機械可読領域のすべての行が 保護されるため、この型ではprotectedComponentIndexプロパティは不要です。

                  実装者は、CBOR-LDでエンコードされたMachineReadableZoneクレデンシャルを QRコードにエンコードする前に、文字列「VC1-」を先頭に付加した base45-multibaseとして再エンコードする必要があります（REQUIRED）。

                  TerseBitstringStatusListEntryは、ビット文字列ステータスリストv1.0 仕様で定義されているBitstringStatusListEntryのコンパクトな表現です。

                  型がTerseBitstringStatusListEntryであるオブジェクトは、次の2つの 追加プロパティを持たなければなりません（MUST）。

                  • terseStatusListBaseUrl。このクレデンシャルに関連付けられた ステータスリストの場所を識別します。 terseStatusListBaseUrlはURL [URL]でなければなりません （MUST）。
                  • terseStatusListIndex。上記のURLにある個別のステータスを指定します。 terseStatusListIndexは、32ビット符号なし整数として 表現可能でなければなりません（MUST）。

                  TerseBitstringStatusListEntryを処理するには、セクション 3.2.3.1 TerseBitstringStatusListEntryから BitstringStatusListEntryへのアルゴリズムを適用して BitstringStatusListEntryに変換し、その後ビット文字列ステータスリストv1.0に 従って処理します。

                  実装者は、個々のステータスリストの長さとして値listLengthを設定する必要があります。 これにより、32ビットのterseStatusListIndexに対して、ステータスリストの数は listCount = 2^32 / listLengthとなります。 TerseBitstringStatusListEntryからBitstringStatusListEntryに 変換するには、listLengthが必要です。 listLengthの一部の値は、これらのステータスリストのプライバシー保護特性を 損なうことに注意してください。実装は、listLength = 2^26および listCount = 2^6を使用しなければなりません（MUST）。

                  2.3 バーコードへのエンコードおよび バーコードからのデコード

                  この仕様のクレデンシャルは、CBOR-LDを使用して検証可能なクレデンシャルを バイナリ形式で効率的にエンコードしますが、バイナリデータを標準的なバーコード画像形式に 直接変換することは、非効率または互換性がない場合がよくあります。そのため、実装の 文字エンコーディング要件は型ごとに提供され、通常は対象となるバーコード形式によって異なります。

                  各credentialSubject型は、その型のCBOR-LDエンコード済み クレデンシャルをバーコードに配置する前に、どのように文字エンコードするかを 指定することが想定されます。MachineReadableZoneの文字エンコーディング要件は、 セクション2.1.1 MachineReadableZoneで提供されます。 その他の型の要件は、それらを定義する仕様または附属書で提供されます（例： 附属書A. AAMVA運転免許証および身分証明書および附属書B. ISO準拠の 運転免許証）。

                  3. アルゴリズム

                  以下のセクションでは、運転免許証や渡航文書などの物理媒体上にある、 バーコードや機械可読領域などの光学情報を保護するデジタル証明を 追加および検証するためのアルゴリズムについて説明します。

                  3.1 一般アルゴリズム

                  3.1.1 VCをQRコードにエンコード

                  次のアルゴリズムは、検証可能な クレデンシャルを、QRコードで表現できるテキスト文字列に エンコードする方法を指定します。必須入力は、検証可能な クレデンシャル（マップinputDocument）および オプションの集合（マップoptions）です。 出力は、エンコードされた検証可能な クレデンシャル（文字列）またはエラーです。 このアルゴリズムが文字列をエンコードする場合は常に、 UTF-8エンコーディングを使用しなければなりません（MUST）。

                  1. typeTableをoptions.typeTableの値とします。
                  2. registryEntryIdをoptions.registryEntryIdの値とします。
                  3. cborldDocumentを、inputDocument、 typeTable、およびregistryEntryIdを入力として渡した JSON-LDからCBOR-LDへのエンコードアルゴリズムの結果とします。 エンコードエラーが発生した場合は、エラーを発生させなければなりません（MUST）。
                  4. base45Valueを、cborldDocumentをbytesとして、 整数45をtargetBaseとして、および 0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ $%*+-./:を baseAlphabetとして渡した 基数エンコードアルゴリズムの 結果とします。
                  5. qrCodeTextを、次の値を連結した結果とします。 VC1-、 R（base45用のMultibase 接頭辞）、および base45Value。
                  6. qrCodeTextを、エンコードされた検証可能な クレデンシャルとして返します。その後、QRコード画像としてエンコードする テキストとして、任意のQRコードライブラリに渡すことができます。

                  3.1.2 QRコードをVCにデコード

                  次のアルゴリズムは、QRコードにエンコードされた検証可能な クレデンシャルをデコードする方法を指定します。必須入力は、 テキスト文字列（文字列inputDocument）および オプションの集合（マップoptions）です。 出力は、検証可能な クレデンシャル（マップ）またはエラーです。 このアルゴリズムが文字列をエンコードする場合は常に、 UTF-8エンコーディングを使用しなければなりません（MUST）。

                  1. inputDocumentがテキストVC1-Rで始まることを確認します。 そうでない場合は、エラーを発生させなければなりません（MUST）。
                  2. base45Valueを、inputDocumentから最初の5文字 （VC1-R）を削除したものとします。
                  3. cborldDocumentを、base45Valueを sourceEncodingとして、整数45を sourceBaseとして、および 0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ $%*+-./:を baseAlphabetとして渡した 基数デコードアルゴリズムの 結果とします。
                  4. jsonldDocumentを、cborldDocumentを入力として渡した CBOR-LDからJSON-LDへのデコードアルゴリズムの結果とします。 デコードエラーが発生した場合は、エラーを発生させなければなりません（MUST）。
                  5. jsonldDocumentを、デコードされた検証可能な クレデンシャルとして返します。

                  1. opticalDataを、保護対象の光学バーコード内のデータに設定します。
                  2. issuerUrlを、発行者がクレデンシャルの検証に使用することを望むURLに設定します。
                  3. unsignedCredentialを、unsignedCredential.issuerが issuerUrlに設定されたOpticalBarcodeCredentialに設定します。
                  4. クレデンシャルのステータスが必要な場合：
                    1. statusListEntryを、発行者が OpticalBarcodeCredentialに追加することを望む BitstringStatusListEntry（ビット文字列ステータスリスト v1.0仕様で定義）に設定します。

                  5. signedCredentialを、 opticalDataおよびunsignedCredentialに署名するため、 3.2.4.1 証明を追加 （ecdsa-xi-2023）のアルゴリズムを使用した結果に設定します。
                  6. signedCredentialをCBOR-LD [CBOR-LD]を使用して エンコードし、credentialSubject.typeの目的の値に対する 処理規則およびユースケースに従って、opticalDataの指定された領域に追加します。

                  3.2.2 OpticalBarcodeCredentialをデコード

                  1. opticalDataを、物理文書からスキャンされた機械可読データ （例：PDF417またはMRZデータ）に設定します。
                  2. vcbPayloadを、opticalDataの指定された領域から抽出された 検証可能な クレデンシャルペイロードに設定します。
                  3. cborldBytesを、クレデンシャル型およびバーコード形式に関連付けられた 文字エンコーディングからvcbPayloadをデコードし、存在する場合は multibaseヘッダーを削除した結果に設定します（セクション 2.3 バーコードへのエンコードおよびバーコードからのデコードを参照）。

                  ecdsa-xi-2023暗号スイートは、実質的には ecdsa-rdfc-2019アルゴリズム [VC-DI-ECDSA]に、 元の光学バーコードデータなどの「追加情報」（xi）を入力として受け取り、 そのデータをデジタル署名によって保護される情報に含める手順を追加したものです。 このセクションのアルゴリズムでは、このような署名を作成および検証する方法を詳述します。

                  3.2.4.4 opticalDataBytesを作成

                  このセクションのアルゴリズムは、デジタル署名が生成または検証される 光学データ（opticalDataBytes）を生成します。このアルゴリズムは、 あらゆる種類の機械可読光学データに対して一般的です。 credentialSubjectの型は、光学データの保護対象コンポーネントを 抽出する方法を定義し（以下の型固有のコンポーネント抽出セクションを参照）、 このアルゴリズムは、それらのコンポーネントを正規化およびハッシュする前に、 決定論的なリストへ並べ替えます。

                  このアルゴリズムへの必須入力は、機械可読光学データ （opticalData）およびクレデンシャルの主体 （credentialSubject）です。保護対象光学データの正規化形式を ハッシュしたものを表す単一の値が出力として生成されます。

                  1. dataToCanonicalizeを、credentialSubjectの型に 関連付けられたコンポーネント抽出規則に従って、opticalDataの 保護対象コンポーネントを文字列の配列として抽出した結果に設定します。 その型で定義されている場合は、 credentialSubject.protectedComponentIndexを使用します。 MachineReadableZoneのコンポーネント抽出規則は、セクション コンポーネント抽出： MachineReadableZoneで示されます。その他の型の規則は、 それらを定義する仕様または附属書で示されます（例：附属書 A.2 コンポーネントの抽出は、附属書 A.                   2. canonicalizedDataを、credentialSubjectの型に 関連付けられた順序付け規則を使用して、dataToCanonicalizeを 決定論的なリストに並べ替え、その後、配列から単一の文字列を作成するために 結合操作を適用した結果に設定します。credentialSubjectの型が 順序付け規則を定義しない場合は、dataToCanonicalizeを Unicodeコードポイント順で並べ替えます。
                  3. canonicalizedDataをハッシュし、その結果を返します。

                  型がMachineReadableZoneである credentialSubjectの場合、opticalDataは 機械可読領域内のデータであり、保護対象コンポーネントは次のように抽出されます。 この型の順序付け規則は、各行がクレデンシャル上に現れる順序です。

                  1. dataToCanonicalizeを空の配列に設定します。
                  2. クレデンシャル上の機械可読領域にある各行について：

                  3. dataToCanonicalizeを返します。

                  4.1 光学データの重複

                  OpticalBarcodeCredentialsに対する攻撃ベクトルの1つは、 不正な文書で使用するために、デジタル署名を含む光学バーコードを複製することです。 複製されたバーコードは元のバーコードと同様に署名検証に合格しますが、この攻撃は、 文書の検証者が次の3点を確認することで軽減されます。署名されたデータが文書上に 表示されているデータと一致すること、署名されたデータが利用者の身体的特徴と一致すること、 および表示されているデータが利用者の身体的特徴と一致することです。これら3つがすべて 同等である場合、OpticalBarcodeCredentialが複製であり得る唯一の状況は、 不正文書の作成者が、不正文書の利用者の身体的特徴と署名済みの身体的特徴が完全に一致する 本物のOpticalBarcodeCredentialにアクセスできた場合です。任意の人物の外見と 完全に一致し、なおかつ盗難が検出されていないOpticalBarcodeCredentialが 存在する可能性は低いため、この攻撃が成功する可能性は低くなります。

                  4.2 部分的に署名された光学データ

                  場合によっては、既存の光学データ全体に対してデジタル署名を作成できない可能性があります。 たとえば、署名時には発行者が把握していないシリアル番号が、物理クレデンシャルの製造者に よって挿入される場合を考えます。この場合、検証者は、デジタル署名されていないデータが 光学バーコード内で変更されても、OpticalBarcodeCredentialの検証成功能力には 影響しない可能性があると想定します。

                  光学バーコードのデータが文書上に表示されるデータおよび文書保持者の特徴と一致することを 確認する際、実装者には、デジタル署名されたフィールドのみを使用することが推奨されます。 検証者には、 署名されていない文書で他のフィールドが不正検出にどれほど一般的に使用されているかに かかわらず、デジタル署名によって保護されたフィールドのみを使用することが推奨されます。 たとえば、目の色と髪の色が署名によって保護されている一方で、 保持者の 顔写真が保護されていない場合、検証者には、 顔写真よりも目の色と髪の色を重視して不正を検出することが推奨されます。

                  検証者が使用する ソフトウェアの実装者には、検証プロセス中、デジタル署名によって保護されたカードデータのみを 表示することが推奨されます。改ざんされている可能性のある署名されていないデータを表示すると、 不正検出を妨げる可能性があります。

                  4.3 安全な検証

                  検証者には、 OpticalBarcodeCredentialの有効性を確認する際、常に信頼できるプログラムおよび インターフェイスを使用することが推奨されます。信頼できないソフトウェアを使用して文書を 検証すると、不正なクレデンシャルが受け入れられたり、本物のクレデンシャルが盗まれたりする 可能性があります。

                  A.1 AamvaDriversLicenseScannableInformation

                  型がAamvaDriversLicenseScannableInformationである credentialSubjectは、検証可能なクレデンシャルが、 AAMVA準拠の運転免許証または身分証明カード上のPDF417バーコードの内容を保護することを 示します [aamva-dl-id-card-design-standard]。

'DDG']）です。 protectedComponentIndexの位置iにあるビットが1の場合、 並べ替えられた要素IDの位置iにあるAAMVA必須フィールドはデジタル署名によって 保護されます。protectedComponentIndexの最後の2ビットはパディングビットであり、 0でなければなりません（MUST）。詳細については、 セクション3.2.4.4 opticalDataBytesを作成を参照してください。

                  A.2 コンポーネントの抽出

                  型がAamvaDriversLicenseScannableInformationである credentialSubjectの場合、opticalDataはPDF417バーコード内の データであり、保護対象コンポーネントは次のように抽出されます。この型の順序付け規則は、 既定のUnicodeコードポイント順です。

                  2. bitfieldDecodedを、 credentialSubject.protectedComponentIndexをmultibase-base64urlから バイナリにデコードして得られる長さ24のビット文字列の最初の22ビットに設定します。
                  3. fieldsAlphabetizedを、この型のコンポーネントの決定論的リスト、すなわち、 AAMVAの22個の必須PDF417要素ID [aamva-dl-id-card-design-standard]を Unicodeコードポイント順で並べ替えた配列 （すなわち、['DAC', 'DAD' ... 'DDG']）に設定します。

                  B. ISO準拠の運転免許証

                  このセクションは非規範的です。

                  この附属書では、MachineReadableInformationのサブクラスとして定義される credentialSubject型であり、ISO準拠の運転免許証（IDL） [ISO18013-2]上の 機械可読データを保護するために使用できる IsoDrivingLicenceMachineReadableInformationについて説明します。 これは、セクション2.1 OpticalBarcodeCredentialの要件を満たす credentialSubject型をIDLについて定義する方法を示すものであり、 使用前に、この仕様の将来の改訂または別の仕様によって改良されることが想定されます。

                  B.1 IsoDrivingLicenceMachineReadableInformation

                  型がIsoDrivingLicenceMachineReadableInformationである credentialSubjectは、検証可能なクレデンシャルが、 ISO準拠の運転免許証（IDL）上にある、[ISO18013-2]で 定義されるコンパクトエンコードされた機械可読データ、たとえば免許証上のPDF417バーコードの 内容を保護することを示します。機械可読IDLデータは、[ISO18013-2]の 論理データ構造で定義されるデータグループ（DG1からDG11）に 編成されます。これらのデータグループは、この仕様における光学データのコンポーネントです。 IsoDrivingLicenceMachineReadableInformationは、 MachineReadableInformationのサブクラスです。

                  IsoDrivingLicenceMachineReadableInformationオブジェクトには、 どのデータグループがデジタル署名されるかに関する情報を含む、必須の protectedComponentIndexプロパティがあります。 protectedComponentIndexは、JSON-LDクレデンシャル内で合計4文字になるように multibase-base64urlでエンコードされた2バイト／16ビットの値です。この型に関連付けられる コンポーネントの決定論的リストは、[ISO18013-2]で 定義されるデータグループを、データグループ番号の昇順で並べ替えたリスト （すなわち、['DG1', 'DG2' ... 'DG11']）であり、 protectedComponentIndex値の最初の11ビットがこれらのデータグループに 対応します。protectedComponentIndexの位置iにあるビットが 1の場合、コンポーネントの決定論的リストの位置iにある データグループがデジタル署名によって保護されます。 protectedComponentIndexの残りのビットはパディングビットであり、 0に設定されます。詳細については、セクション 3.2.4.4 opticalDataBytesを作成を参照してください。

                  実装者は、CBOR-LDでエンコードされた IsoDrivingLicenceMachineReadableInformationクレデンシャルをPDF417に エンコードする前に、base64urlとして文字エンコードすることが想定されます。 基数エンコードされたVCBデータの先頭にmultibaseヘッダーを付加できます。 エンコードされたVCBは、任意および管轄区域固有のデータ用のデータグループ11に 配置されることが想定されます。VCBが11個のデータグループのいずれかに配置される場合、 protectedComponentIndexの対応する要素は0に設定されます。

                  B.2 コンポーネントの抽出

                  型がIsoDrivingLicenceMachineReadableInformationである credentialSubjectの場合、opticalDataは免許証上の コンパクトエンコードされた機械可読データ [ISO18013-2]であり、 保護対象コンポーネントは次のように抽出されます。この型の順序付け規則は、 データグループ番号の昇順です。

                  1. dataToCanonicalizeを空の配列に設定します。
                  2. bitfieldDecodedを、 credentialSubject.protectedComponentIndexをmultibase-base64urlから バイナリにデコードして得られる長さ16のビット文字列の最初の11ビットに設定します。
                  3. dataGroupsOrderedを、この型のコンポーネントの決定論的リスト、すなわち、 [ISO18013-2]で 定義されるデータグループ識別子を、データグループ番号の昇順で並べ替えた配列 （すなわち、['DG1', 'DG2' ... 'DG11']）に設定します。
                  4. bitfieldDecoded内で値が1である各ビットについて：
                    1. 文字列dataGroupNameを dataGroupsOrdered[i]に設定します。ここで、 iはbitfieldDecoded内のビットのインデックスです。
                    2. 文字列dataGroupDataを、データグループ区切り文字によって区切られた opticalData内でdataGroupNameによって識別される データグループの内容をbase64urlでエンコードしたものに設定します。
                    3. dataGroupDataをdataGroupNameの末尾に連結し、その末尾に 改行文字（\n、U+000A）を連結して、その結果を dataToCanonicalizeに追加します。
                  5. dataToCanonicalizeを返します。

                  C.3 実装上の注意事項

                  C.3.1 CBOR-LD

                  コンテキスト用語からCBOR-LD整数へのマップを構築する際、一部のコンテキストには、 特定の型のオブジェクトの下にネストされた別のコンテキストが含まれることに 注意してください。これらのネストされたコンテキストは「型スコープコンテキスト」と 呼ばれ、関連付けられた型がデータ内で使用される場合にのみアクティブになります。 コンテキスト内の用語には、そのコンテキストがアクティブになったときにのみIDが 割り当てられるため、これは用語IDの割り当てに重要です。これらのテストベクトルで、 2つのクレデンシャルが同一のコンテキストを使用しているにもかかわらず、 運転免許証と就労許可文書について作成されるマップが異なるのは、このためです。

                  また、CBOR-LDでは、関連付けられた値が複数である場合、用語を表現するために奇数が 使用されることに注意してください。たとえば、上記のCBOR-LDの用語からIDおよび IDから用語へのマップでは、「type」は156にマッピングされていますが、VC内で複数の型が 表現される箇所では、代わりに157が使用されます。
